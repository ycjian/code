Exercise 09: API Integration & Contract Testing

Duration: 3 hours Difficulty: Beginner+ Prerequisites: Exercises 06, 08 complete

ç›®æ ‡ï¼ˆObjectiveï¼‰

Build a layered test suite for the model-serve API from exercise 08: unit tests (pure functions), contract tests (in-process FastAPI æœåŠ¡æ¡†æž¶ï¼ˆFastAPIï¼‰ client), integration tests (containerized service), and consumer-driven contract tests (mock client â†’ real server contract enforcement). By the end you'll know which test type catches which class of bug and have a CI workflow that runs all four.

Why this matters

A model-serving service has at least four interfaces: input schema, output schema, observability shape (metrics + logs), and operational endpoints (health/ready/admin). Each needs a different test strategy. Conflating them â€” or only writing the "easy" ones â€” is the most common source of production regressions.

è¦æ±‚ï¼ˆRequirementsï¼‰

1. Unit tests

Pure-function tests for everything that doesn't need an HTTP server: feature validation, schema serialization, batch splitting, prediction post-processing.

2. Contract tests (in-process)

FastAPI æœåŠ¡æ¡†æž¶ï¼ˆFastAPIï¼‰'s TestClient against your app instance. These should cover every request/response shape exhaustively but never touch a real model file â€” use a tiny MagicMock or in-test trained scikit-learn model.

3. Integration tests (real container)

Spin up the actual Docker å®¹å™¨ï¼ˆDockerï¼‰ image (via testcontainers or docker compose), drive real HTTP traffic, verify ç«¯åˆ°ç«¯ï¼ˆend-to-endï¼‰ including metrics scraping. Slower but catches Dockerfile, network, and signal-handling bugs.

4. Consumer-driven contract tests

Use Pact (or a hand-rolled equivalent): a fake client publishes a contract (expected requests + expected response shapes), the server's CI verifies it can satisfy that contract. Prevents the server from changing its API in a way that breaks downstream consumers.

å¸¸è§å‘ç‚¹ï¼ˆCommon pitfallsï¼‰

  • Mocking the model at the wrong layer â€” Mock at joblib.load or Model.predict, not at the request handler. Tests should exercise as much of your code as possible.
  • Flaky integration tests â€” Always wait on a health endpoint, never time.sleep(5).
  • Pact contracts that just mirror the OpenAPI â€” Contracts should encode consumer expectations, not duplicate spec.
  • Coverage targets driving behavior â€” A test that runs every line but asserts nothing is worse than no test. Tests must assert observable behavior.
  • Rate-limit middleware enabled in tests â€” Test parallelism trips rate limits; disable via env var override.
