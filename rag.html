B. Ray Serve / Triton / vLLM -> Layered Serving Architecture

Resume evidence

  • Implemented orchestration services for a layered inference stack.
  • Ray Serve for routing and DAG composition.
  • Triton for embedding/ranking inference.
  • vLLM for generation.
  • Supported generative editing, semantic retrieval, media understanding, LLM assistants, and multi-step AI workflows.

Project mapping

这个项目里的 serving layer 可以明确讲成三层：

  1. Ray Serve - Orchestrates the full RAG / Agent DAG. - Handles fan-out / fan-in. - Coordinates query rewrite, retrieval, rerank, context building, LLM generation, citation validation, and tool execution. - Good for Python-native business logic and workflow composition.

  2. Triton - Serves dense embedding models. - Serves cross-encoder rerankers. - Serves classifiers such as intent classification, safety classification, document type classification. - Uses dynamic batching and multiple model instances for GPU efficiency.

  3. vLLM - Serves autoregressive generation. - Handles query rewrite, answer generation, summarization, and agent planning. - Uses continuous batching and KV cache management to improve throughput.

What to say in interview

I would not put the whole RAG system behind a single LLM endpoint. RAG has multiple workloads with different latency and throughput profiles. Embeddings and reranking are tensor inference workloads, generation is autoregressive, and orchestration is workflow logic. So I split the serving layer into Ray Serve, Triton, and vLLM.

中文表达：

我不会把整个 RAG 系统做成一个单一 LLM endpoint。因为 embedding、reranking、classification、generation、tool execution 的 workload shape 完全不同。Embedding/reranking 更适合 Triton，generation 适合 vLLM，pipeline orchestration 和 business logic 适合 Ray Serve。

Senior-level tradeoffs

  • Central gateway vs direct model access
  • Central gateway gives governance, quota, audit, fallback, and observability.

  • Direct model access is faster to prototype but creates duplicated cost and inconsistent safety.

  • One large model vs model routing

  • One large model is simple.

  • Model routing is cheaper and faster: small model for classification, embedding model for retrieval, reranker for top candidates, larger model only for final generation.

  • Prompt versioning

  • Prompts are production artifacts.
  • Every prompt change can affect behavior, so prompt version, evaluation result, owner, and rollback path should be tracked.

Billion-Scale Multimodal System Design Notes

From the Adobe study reference, the most important scale lesson is: billion-scale multimodal retrieval is constrained by physics and economics, not just code.

Hard constraints

  • 1B+ users cannot synchronously upload huge raw assets to one region.
  • 100B+ dense vectors cannot be searched with flat KNN.
  • A global vector index must not leak tenant data.
  • Raw video/3D/PDF ingress cost can dominate compute cost.
  • Freshness and index stability fight each other.

Scale architecture

Global clients
  -> edge orchestrator SDK
  -> modality parser / preprocessor
  -> regional PoP / geo-routed API gateway
  -> presigned object storage for heavy assets
  -> Kafka/Flink ingestion backbone
  -> modality-specific embedding workers
  -> vector + lexical + metadata indexes
  -> retrieval/ranking API

Vector index strategy

100B vector problem:
  - flat KNN is impossible
  - use approximate nearest neighbor retrieval
  - partition vectors with IVF-style clustering
  - compress vectors with PQ / quantization where acceptable
  - shard by tenant, modality, region, and freshness tier
  - use metadata pre-filtering before similarity search

Tenant and ACL strategy

Every vector/chunk/asset has:
  - tenant_id
  - owner_id
  - ACL
  - license class
  - region
  - asset version
  - modality

Retrieval must filter before candidate scoring:
  - never retrieve unauthorized vectors into LLM context
  - never rely on the LLM to ignore unauthorized evidence
  - fail closed if ACL metadata is missing

Late-fusion alignment

Modality-specific encoders:
  - text encoder
  - image encoder
  - audio/transcript encoder
  - video keyframe encoder
  - 3D rendered-view encoder

Projection:
  - map outputs into shared representation space
  - retrieve candidates across modalities
  - rerank with modality-aware features and business rules

Interview line

At Adobe scale, multimodal RAG is not just “put embeddings in a vector DB.” You need edge preprocessing, PoP routing, object-storage ingress, approximate vector indexing, tenant-aware filtering, modality-aware ranking, and traceable citations.

中文表达：

Adobe 规模下，多模态 RAG 不是“把 embedding 放进 vector DB”。必须考虑 edge preprocessing、PoP routing、object storage ingress、ANN index、tenant-aware filtering、modality-aware ranking 和可追溯 citation。

4. End-to-End Workflow 1: Document Ingestion Pipeline

Goal

把企业里的 PDF、扫描件、Word、Excel、HTML、ticket、wiki、database records 变成可检索、可引用、可权限过滤的 knowledge chunks。

Raw Enterprise Sources

  -> Vector Index + BM25 Index

Detailed pipeline

1. Connector
   - SharePoint, Confluence, Google Drive, S3, Jira, ServiceNow, PDF folder, DB dump

2. Parse

   - Scanned PDF: OCR
   - Tables: Camelot / Tabula / custom layout parser
   - HTML: BeautifulSoup
   - Word: python-docx
   - Excel: pandas/openpyxl

3. Clean
   - Remove headers/footers
   - Normalize whitespace
   - Fix OCR artifacts
   - Language detection
   - Duplicate detection by hash / near-duplicate similarity

4. Structure-aware chunking
   - By section heading
   - By clause number
   - By operation step
   - Keep table as one logical chunk
   - Preserve source/page/section metadata

5. Metadata enrichment
   - tenant_id
   - doc_id
   - doc_version
   - source_uri
   - page
   - section
   - doc_type
   - domain
   - equipment_type
   - effective_date
   - expired/deleted flag
   - ACL / access level

6. Index
   - Dense embedding -> Vector DB
   - Text -> BM25 index
   - Metadata -> Postgres / OpenSearch / Qdrant payload

7. Publish
   - Build candidate index version
   - Run validation
   - Canary / shadow test
   - Promote to production

RAG 文档 pipeline 的核心就是 parse -> clean -> chunk -> metadata -> embeddings -> vector/search index；query time 则是 rewrite -> retrieve -> rerank -> context -> generate -> cite -> evaluate/log。

9. Code Logic / Implementation Skeleton

9.1 Chunker

from dataclasses import dataclass
from typing import Any

@dataclass
class ParsedBlock:
    text: str
    page: int
    section: str | None
    block_type: str  # paragraph, heading, table, code, figure
    metadata: dict[str, Any]

@dataclass
class Chunk:
    chunk_id: str
    text: str
    source: str
    page_start: int
    page_end: int
    section_path: str
    metadata: dict[str, Any]

def structure_aware_chunk(blocks: list[ParsedBlock], max_tokens: int = 1000) -> list[Chunk]:
    chunks = []
    current = []
    current_tokens = 0
    current_section = None

    for block in blocks:
        # tables should stay together
        if block.block_type == "table":
            if current:
                chunks.append(make_chunk(current))
                current = []
                current_tokens = 0
            chunks.append(make_chunk([block]))
            continue

        # heading starts new semantic unit
        if block.block_type == "heading":
            if current:
                chunks.append(make_chunk(current))
            current = [block]
            current_tokens = estimate_tokens(block.text)
            current_section = block.section
            continue

        block_tokens = estimate_tokens(block.text)

        # split only at safe boundaries
        if current_tokens + block_tokens > max_tokens and current:
            chunks.append(make_chunk(current))
            current = [block]
            current_tokens = block_tokens
        else:
            current.append(block)
            current_tokens += block_tokens

    if current:
        chunks.append(make_chunk(current))

    return chunks

9.4 Hybrid retrieval + RRF

def reciprocal_rank_fusion(result_lists: list[list[str]], k: int = 60) -> list[tuple[str, float]]:
    scores = {}

    for results in result_lists:
        for rank, chunk_id in enumerate(results, start=1):
            scores[chunk_id] = scores.get(chunk_id, 0.0) + 1.0 / (k + rank)

    return sorted(scores.items(), key=lambda x: x[1], reverse=True)

9.5 Permission-aware retrieval

def retrieve_candidates(query: str, user: UserContext, filters: dict, top_k: int = 50):
    # Never rely on prompt-only ACL.
    # Apply permission filter before context reaches the LLM.
    acl_filter = {
        "tenant_id": user.tenant_id,
        "access_level": {"$in": user.allowed_access_levels},
        "doc_status": "active",
    }

    final_filter = {**filters, **acl_filter}

    dense_hits = vector_db.search(
        query_vector=embed_query(query),
        filters=final_filter,
        top_k=top_k,
    )

    bm25_hits = search_index.search(
        query=query,
        filters=final_filter,
        top_k=top_k,
    )

    fused_ids = reciprocal_rank_fusion([
        [h.chunk_id for h in dense_hits],
        [h.chunk_id for h in bm25_hits],
    ])

    return load_chunks([chunk_id for chunk_id, _ in fused_ids[:top_k]])

Access control must happen before retrieved text enters the LLM; do not rely only on prompt-level instructions for permissions.

9.6 Reranking

def rerank(query: str, candidates: list[Chunk], top_n: int = 5):
    pairs = [(query, c.text) for c in candidates]
    semantic_scores = cross_encoder.predict(pairs)

    ranked = []
    for chunk, score in zip(candidates, semantic_scores):
        business_score = business_priority(chunk.metadata)
        freshness_score = freshness_priority(chunk.metadata)
        final_score = 0.75 * score + 0.15 * business_score + 0.10 * freshness_score
        ranked.append((chunk, final_score))

    ranked.sort(key=lambda x: x[1], reverse=True)
    return [c for c, _ in ranked[:top_n]]

9.7 Context builder

def build_context(chunks: list[Chunk], max_chars: int = 8000) -> str:
    parts = []
    total = 0

    for c in chunks:
        header = (
            f"[chunk_id={c.chunk_id}; "
            f"source={c.source}; "
            f"page={c.page_start}-{c.page_end}; "
            f"section={c.section_path}]"
        )
        text = header + "\n" + c.text

        if total + len(text) > max_chars:
            break

        parts.append(text)
        total += len(text)

    return "\n\n---\n\n".join(parts)

RAG context should remove duplicates, group by source/page, sort by relevance, include metadata, and fit the token budget instead of dumping top-20 chunks blindly.

9.9 Agent loop with tool validation

MAX_STEPS = 6

def run_agent(goal: str, user: UserContext):
    state = {
        "goal": goal,
        "messages": [{"role": "user", "content": goal}],
        "tool_results": [],
        "needs_approval": False,
    }

    for step in range(MAX_STEPS):
        response = call_llm(
            messages=state["messages"],
            tools=allowed_tool_schemas(user),
        )

        if response.type == "final":
            return response.text

        if response.type == "tool_call":
            tool_name = response.tool_name
            raw_args = response.arguments

            tool = tool_registry.get(tool_name)
            if tool is None:
                raise ValueError("Unknown tool")

            if not policy.can_call(user, tool_name, raw_args):
                raise PermissionError("User cannot call this tool")

            args = tool.input_schema.validate(raw_args)

            if tool.requires_approval:
                approval_id = create_approval_request(user, tool_name, args)
                return {
                    "status": "pending_approval",
                    "approval_id": approval_id,
                    "proposed_action": {"tool": tool_name, "args": args},
                }

            result = execute_tool_with_timeout_and_retry(
                tool_name=tool_name,
                args=args,
                idempotency_key=f"{state['goal']}:{step}:{tool_name}",
            )

            log_tool_call(user, tool_name, args, result)

            state["messages"].append(response.as_message())
            state["messages"].append({
                "role": "tool",
                "tool_call_id": response.tool_call_id,
                "content": json.dumps(result),
            })

    return "I could not complete the task within the allowed step limit."

Tool calling should validate tool name, JSON schema, timeout, retry, auth, rate limit, and logging; the model suggests tool calls, but the application executes them and enforces permission.

16. Interview Deep-Dive Questions

Q1: Why not fine-tune first?

Answer

I would not start with fine-tuning because the dominant failure modes in enterprise RAG are usually bad parsing, bad chunking, stale versions, missing metadata, weak retrieval, and lack of evaluation. Fine-tuning cannot fix wrong or missing evidence. I would first improve document quality, chunking, hybrid retrieval, reranking, source grounding, and golden-set evaluation. Fine-tuning becomes useful later if we need consistent format, domain style, or specialized classification behavior.

中文：

先不要微调。企业 RAG 大部分问题不是模型“不懂”，而是证据没找对、文档脏、chunk 切坏、版本冲突、权限和 metadata 不对。先把 retrieval pipeline 做好。

Q2: Why hybrid retrieval?

Answer

Dense retrieval handles semantic paraphrases, but it can miss exact IDs, equipment names, version numbers, and rare domain terms. BM25 is strong for exact match but weak for synonyms. Hybrid retrieval combines both, then reranking improves precision. This is especially important for enterprise documents where exact terms and semantic intent both matter.

Q3: Why does corpus growth hurt quality?

Answer

As the corpus grows, semantically similar but operationally wrong documents enter the candidate set. For example, the same term may appear in operation manuals, maintenance manuals, safety standards, and historical versions. Increasing TopK often increases noise. The better fix is domain routing, metadata filtering, version filtering, and business-rule filtering before reranking.

Q4: How do you handle conflicting documents?

Answer

I would not let the LLM resolve conflicts blindly. I would encode document priority rules: latest version wins, local/site-specific policy beats generic policy, active documents beat expired documents, and special-case regulation beats general rule. The retrieval layer should avoid sending contradictory evidence when priority can be resolved deterministically.

Q5: How do you make agents safe?

Answer

I would use controlled workflows instead of unconstrained autonomy. All tools are allowlisted, arguments are schema-validated, permission is checked outside the model, write actions require human approval, tool calls use idempotency keys, and every step is logged for audit. The agent can propose actions, but the platform decides whether those actions are allowed.

Interview Q&A: Overview, Architecture, Ingestion, Retrieval & Generation

1. Project Overview / 项目总览

Q1: Can you walk me through this project?

A:
I built an enterprise AI Knowledge & Agent Platform. At the bottom, it is a production-grade RAG system that ingests messy enterprise documents, cleans and chunks them, builds vector and lexical indexes, retrieves relevant evidence, reranks it, and generates grounded answers with citations. On top of that, it adds an agent workflow layer so the assistant can safely call tools, draft tickets, query internal systems, compare policies, and execute approved actions.

中文补充：
这个项目不是简单 chatbot，而是企业级 AI execution layer：把企业知识、权限、业务规则、工具调用、评估、审计、监控放在一个统一平台里。

Q2: What problem were you solving?

A:
The core problem was that enterprise knowledge is messy and fragmented. Documents are scattered across PDFs, wikis, tickets, databases, spreadsheets, and legacy systems. Users ask vague questions, documents have duplicates and conflicting versions, and actions require strict permission control. A simple vector search demo cannot solve this.

So the goal was to build a governed AI platform that can answer questions reliably, cite evidence, respect access control, and perform safe workflow actions.

Q3: What is the key business value?

A:
The business value is reducing time-to-answer and time-to-action. Instead of engineers, operators, or support agents manually searching across dozens of systems, they can ask one assistant and get a grounded answer with source citations. For workflow tasks, the system can draft tickets, summarize incidents, compare policies, and prepare actions for human approval.

中文：
核心价值是把“找知识”和“执行流程”合并起来，但又不牺牲企业需要的权限、审计和可控性。

2. High-Level Architecture / 高层架构

Q4: What are the major components?

A:
The system has five major layers:

  1. Ingestion layer — connectors, OCR, table extraction, parsing, chunking, metadata enrichment.
  2. Index layer — vector index, BM25 index, metadata store, document/version store.
  3. RAG runtime — query rewrite, domain routing, hybrid retrieval, rerank, grounded generation.
  4. Agent runtime — tool registry, state machine, permission checks, approval gates, audit logs.
  5. Evaluation and ops layer — golden set, LLM-as-judge, metrics, traces, canary rollout, rollback.

中文：
架构上我会强调 control plane 和 data plane 分离。Control plane 决定 intent、policy、routing、model/index/prompt version；data plane 执行 parsing、retrieval、rerank、generation 和 tool calls。

Q5: Why did you separate RAG and Agent runtime?

A:
Because RAG and agents have different risk profiles. RAG mainly answers based on retrieved evidence. Agent workflows may call tools and create side effects, such as creating tickets, sending messages, or updating records.

So I keep RAG as the evidence and reasoning foundation, while the agent runtime adds state, tool permissions, human approval, retries, idempotency, and audit logs.

中文：
RAG 是“查证据并回答”，Agent 是“基于证据去行动”。行动必须更严格控制。

Q6: Why not just use LangChain or LlamaIndex?

A:
Frameworks are useful for prototyping, but enterprise production needs stronger boundaries: ACL filtering, versioned indexes, golden-set evaluation, prompt/version rollout, audit logs, cost tracking, tool permissions, and failure recovery. I would use frameworks selectively, but not let the framework define the production architecture.

中文：
开源框架能帮你半天跑通 demo，但不能替你解决权限、版本冲突、审计、评估、回滚和运维。

3. Ingestion Pipeline / 文档入库

Q7: How does the document ingestion pipeline work?

A:
The ingestion pipeline starts from enterprise connectors like SharePoint, Confluence, S3, Jira, ServiceNow, PDF folders, or database exports. Each document goes through parsing, OCR if needed, table extraction, cleaning, structure-aware chunking, metadata enrichment, embedding, BM25 indexing, and index validation before publishing.

中文流程：

Raw sources
 -> connectors
 -> parse / OCR / table extraction
 -> clean / dedup
 -> structure-aware chunking
 -> metadata + ACL enrichment
 -> embedding
 -> vector index + BM25 index
 -> validation
 -> publish index version

Q8: What is the biggest ingestion challenge?

A:
The biggest challenge is document quality. Enterprise PDFs often include scanned pages, broken tables, repeated headers, outdated versions, and inconsistent terminology. If we embed noisy text, retrieval quality drops immediately. So data preprocessing has the highest ROI.

中文：
RAG 不是先调模型，而是先把知识库弄干净。OCR、表格、版本、去重、metadata 都非常关键。

Q9: How do you handle scanned PDFs?

A:
I run OCR on scanned PDFs and store OCR confidence as metadata. For high-risk pages such as equipment parameters, operation steps, or compliance clauses, I support manual validation or human-in-the-loop review. Low-confidence OCR chunks can be excluded from production retrieval or marked as lower priority.

中文：
不能盲信 OCR。OCR 识别率 97% 听起来高，但一个 chunk 里可能还有很多错字，这会污染 embedding。

Q10: How do you handle tables?

A:
Tables are treated as first-class chunks, not flattened randomly into text. I extract tables into structured JSON or markdown, preserve row/column relationships, and store table metadata such as page, caption, and section. For parameter lookup, table-aware chunks are much more reliable than normal paragraph chunks.

中文：
表格不能随便 OCR 成一坨文本。参数查询、设备配置、阈值类问题，表格结构非常重要。

Q11: How do you chunk documents?

A:
I prefer structure-aware chunking over fixed-size chunking. For policy or regulation documents, one clause or operation step is usually a complete semantic unit. So I chunk by heading, clause number, procedure step, table boundary, and paragraph. Fixed token size is only a fallback.

中文：
chunk size 不是最重要的，chunk boundary 才是最重要的。完整、自洽、可引用的 chunk 才适合 RAG。

Q12: What metadata do you store per chunk?

A:
Each chunk stores tenant ID, document ID, version, source URI, page number, section path, document type, domain, effective date, expiration date, ACL policy, content hash, embedding model version, and index version.

中文：
metadata 不是装饰品。它决定权限过滤、版本过滤、source citation、debug、rerank 和 rollout。

4. Retrieval Design / 检索设计

Q13: Why use hybrid retrieval instead of only vector search?

A:
Vector search is good for semantic similarity, but it can miss exact identifiers, equipment names, numbers, error codes, and rare terms. BM25 is strong for exact lexical matching but weak for paraphrases. Hybrid retrieval combines both: dense retrieval gives semantic recall, BM25 gives exact-match precision.

中文：
企业搜索里 exact term 很重要，比如设备编号、法规条款、告警代码、版本号。纯向量检索不够。

Q14: How does hybrid retrieval work?

A:
For each rewritten query, I run vector retrieval and BM25 retrieval in parallel. Then I merge candidates using reciprocal rank fusion or weighted score fusion, deduplicate by chunk/document, apply metadata and ACL filters, and send the top candidates to reranking.

query
 -> dense retrieval topK
 -> BM25 retrieval topK
 -> merge / RRF
 -> dedup
 -> metadata + ACL filter
 -> rerank

Q15: What is query rewriting and why do you need it?

A:
Users rarely ask questions in the same language as enterprise documents. For example, a user may say “the transformer keeps tripping,” while the document says “relay protection device frequent operation.” Query rewriting normalizes vague user language into standard domain terminology and expands the query into several retrieval queries.

中文：
query rewrite 不是简单 paraphrase，而是 domain normalization，把用户口语翻译成企业文档里的标准术语。

Q16: How do you prevent query rewrite from hallucinating?

A:
The rewrite prompt is constrained. It must preserve exact entities, avoid inventing facts, use only conversation context when filling missing details, and output multiple search queries rather than a final answer. I also log original query and rewritten queries for debugging.

中文：
rewrite 可以补全上下文，但不能编造上下文。比如用户没说设备名，就不能乱加设备名。

Q17: Why do you need domain routing?

A:
As the knowledge base grows, semantically similar but operationally wrong documents enter the candidate set. For example, “protection trip” may appear in operation manuals, maintenance manuals, safety policies, and historical incident reports. Domain routing sends the query to the right sub-index or applies metadata filters before retrieval.

中文：
知识库越大，不是 TopK 越大越好。更好的办法是先判断领域，再去对应知识库里搜。

Q18: How do you handle ACL in retrieval?

A:
Access control must happen before context reaches the LLM. I apply tenant, role, document-level, and chunk-level ACL filters inside retrieval. The LLM should never see text the user is not allowed to access. Prompt-level instruction is not enough.

中文：
权限过滤不能靠 prompt。必须在 retrieval 层过滤掉用户无权访问的 chunk。

Q19: How do you rerank retrieved chunks?

A:
After hybrid retrieval, I use a cross-encoder reranker on the top 20–50 candidates. The reranker scores query-document relevance more accurately than bi-encoder vector similarity because it sees the query and document together. I also add business rules such as version priority, document type priority, and source reliability.

中文：
rerank 负责 precision，但业务规则负责 correctness。通用 reranker 不一定懂“主变”和“线路”在业务上不能混用。

Q20: Why not send top-50 chunks directly to the LLM?

A:
More context is not always better. Too many chunks increase latency, token cost, and the chance that the model gets distracted by irrelevant or conflicting information. I prefer retrieving broadly, reranking carefully, and sending only the top evidence with source metadata.

中文：
TopK 越大不一定越准。RAG 要控制 context quality，不是堆 context 数量。

5. Generation / Grounded Answer

Q21: How do you make the answer grounded?

A:
The generation prompt includes structured evidence with source IDs, document names, pages, and section numbers. The model is instructed to answer only from retrieved evidence, cite sources, and abstain if evidence is insufficient. After generation, I validate that citations reference retrieved chunks.

中文：
核心是：带来源、要求引用、允许不知道、citation validation。

Q22: How do you reduce hallucination?

A:
I reduce hallucination through multiple layers: better retrieval, reranking, source-grounded prompts, citation validation, abstention behavior, and evaluation. Prompting alone cannot fix bad retrieval, so the main focus is evidence quality.

中文：
幻觉不是只靠 prompt 解决。证据找错了，LLM 再聪明也会答错。

Q23: What happens if retrieved documents conflict?

A:
I avoid sending unresolved conflicts to the model when possible. I use deterministic rules: active version beats expired version, latest version beats older version, local/site-specific policy beats generic policy, and special-case procedure beats general procedure. If conflict remains, the system explicitly reports the conflict with citations instead of inventing a resolution.

中文：
规程冲突不是模型问题，是 governance 问题。要有版本优先级和适用范围规则。

Q24: How do you decide whether to answer or abstain?

A:
The system abstains when retrieval confidence is low, reranker scores are below threshold, citations are weak, or evidence does not cover the question. In high-risk domains, I prefer safe abstention over a plausible but unsupported answer.

中文：
企业场景里“不知道”比“编一个看起来对的答案”安全得多。

6. Agent Workflow / Agent 设计

Q25: What is the agent layer doing?

A:
The agent layer turns a user goal into a controlled workflow. It can search documents, read evidence, query internal systems, compare policies, draft tickets, request approval, and execute approved actions. It is not an unconstrained autonomous loop; it is a governed workflow runtime.

中文：
Agent 不是让 LLM 随便行动，而是让 LLM 在可控工具、状态机、权限和审批框架里行动。

Q26: How do you define a tool?

A:
Each tool has a name, input schema, output schema, owner, timeout, retry policy, permission policy, risk level, and whether human approval is required. The LLM can propose a tool call, but the application validates and executes it.

Example:

{
  "tool_name": "create_jira_ticket",
  "input_schema": {
    "title": "string",
    "description": "string",
    "priority": "enum"
  },
  "risk_level": "write_action",
  "requires_approval": true
}

Q27: How do you make tool calling safe?

A:
I use strict schemas, permission checks outside the model, allowlisted tools, timeouts, idempotency keys, human approval for side effects, and audit logs. The model does not directly execute tools; it only emits a structured tool request.

中文：
模型只能“建议调用工具”，真正能不能调用，由平台 policy 决定。

Q28: How do you prevent infinite agent loops?

A:
The agent has a maximum number of steps, tool-call budget, token budget, and timeout. It also has explicit stopping criteria. If it cannot complete the task within limits, it returns a partial result or asks for human clarification.

中文：
production agent 必须有边界：max steps、max cost、timeout、tool allowlist。

Q29: When do you require human approval?

A:
Human approval is required for destructive or external side-effect actions, such as creating or updating tickets, sending emails, deleting records, changing production settings, issuing refunds, or modifying customer data. Read-only actions can usually run automatically if permissions pass.

中文：
read-only 可以自动化，write action 要审批，尤其是会影响外部系统或客户数据的操作。

Q30: How do you recover from agent failure?

A:
The workflow state is checkpointed after each step. Tool calls use idempotency keys, so retries do not duplicate side effects. Failed workflows can be retried, resumed, or sent to human review. All intermediate observations are logged for replay and debugging.

Interview Q&A: Agents, Serving, Evaluation & Governance

7. Model Serving / Serving 架构

Q31: What serving stack would you use?

A:
I would use a layered serving stack:

  • Ray Serve for request orchestration, DAG composition, routing, fan-out, and business logic.
  • Triton for high-throughput embedding, reranking, and classification models.
  • vLLM for generation, summarization, query rewrite, and agent planning.
  • CPU workers for parsing, metadata processing, policy checks, and lightweight business rules.

中文：
不同 workload bottleneck 不同，所以不要用一个 serving system 解决所有问题。

Q32: Why Ray Serve?

A:
Ray Serve is good for Python-native orchestration and model composition. It lets me build a serving DAG: query rewrite, retrieval, rerank, context build, LLM generation, post-validation. It is better suited for workflow orchestration than raw GPU inference.

Q33: Why Triton?

A:
Triton is strong for high-throughput tensor inference, especially embedding models, rerankers, and classifiers. It supports dynamic batching, multiple model instances, and GPU-efficient serving. For non-generative models, it is usually more efficient than putting everything in Python.

Q34: Why vLLM?

A:
vLLM is designed for autoregressive LLM serving. It handles KV cache efficiently and supports continuous batching, which improves throughput for generation workloads. I use it for query rewrite, grounded answer generation, summarization, and agent planning.

Q35: How do you optimize latency?

A:
I optimize latency by separating the pipeline into stages, setting timeouts per stage, running retrieval branches in parallel, using rerank only on top candidates, caching frequent embeddings and answers, streaming generation, and routing simple requests to smaller models.

中文：
不要让每个请求都走最大模型、最大 context、最大 TopK。要按 intent 和 complexity 做 routing。

Q36: How do you optimize cost?

A:
I reduce cost by using smaller models for rewrite/classification, batching embedding and reranking, limiting rerank depth, using retrieval instead of long-context prompting, caching hot queries, applying model routing, and avoiding LLM calls when deterministic logic is enough.

中文：
如果 answer 错了，要知道是 retrieval 错、rerank 错、prompt 错，还是模型生成错。

Q38: What is the golden dataset?

A:
The golden dataset is a set of real user questions labeled by domain experts. Each item includes the question, expected answer points, relevant source documents, must-not-use documents, difficulty, and domain. Every model, prompt, retriever, and index change is tested against this set before rollout.

Q39: Why not rely only on generic RAG benchmarks?

A:
Generic benchmarks do not represent enterprise failure modes. Real enterprise queries are vague, context-dependent, permission-sensitive, and affected by document conflicts. A domain-specific golden set is more useful for production quality.

中文：
真实用户问题比 benchmark 难得多，比如“昨天那个告警怎么处理”。

Q40: How do you use LLM-as-Judge?

A:
LLM-as-Judge is used as a scalable evaluation layer, not the only source of truth. It scores faithfulness, completeness, citation correctness, unsupported claims, and policy compliance. For high-risk domains, I calibrate judge results against human labels.

中文：
LLM-as-Judge 可以扩大评估覆盖，但关键集合还是要有人类专家标注。

Q41: How do you evaluate agents?

A:
Agent evaluation includes task success rate, correct tool selection, valid tool arguments, number of steps, tool error rate, approval correctness, unsafe action prevention, and user correction rate. I also replay historical tasks to test new prompts and tool policies.

9. Security / Governance / Compliance

Q42: How do you ensure tenant isolation?

A:
Every document, chunk, index entry, request, tool, workflow, and audit event carries tenant ID. Retrieval filters include tenant ID by default. We also separate encryption keys, access policies, and sometimes physical indexes for large or regulated tenants.

Q43: How do you protect against prompt injection?

A:
I treat retrieved documents and tool outputs as untrusted input. The system prompt explicitly says retrieved text is evidence, not instruction. Tool calls are controlled by policy outside the model. Sensitive actions require approval. I also scan retrieved content for suspicious instructions when needed.

中文：
文档里写“ignore previous instructions”不能被模型当成系统指令。

Q44: How do you handle PII and sensitive data?

A:
PII detection and redaction happen during ingestion and logging. The retrieval layer respects ACLs. Logs store redacted payloads or hashes where possible. Tenant retention policies control how long prompts, answers, and evidence snapshots are stored.

Q45: How do you audit an answer?

A:
Each answer logs request ID, user, tenant, original query, rewritten queries, retrieved chunks, reranked chunks, source documents, prompt version, model version, index version, answer, citations, latency, cost, and feedback. This allows replay and root-cause analysis.

中文：
企业客户一定会问：这个答案为什么这么答？你必须能 replay。

10. Operations / 运维

Q46: What are the most important production metrics?

A:
For RAG: retrieval recall, citation accuracy, hallucination rate, stale document recall, and user feedback.
For serving: p95/p99 latency, error rate, timeout rate, queue depth, GPU utilization, tokens/sec, and cost per request.
For ingestion: parse failure rate, OCR confidence, indexing lag, duplicate rate, and index freshness.

Q47: What alerts would you set?

A: 

- p95 latency above SLO
- LLM timeout rate spike
- vector DB error rate spike
- index freshness lag too high
- OCR/parse failure rate spike
- ACL filter violation
- hallucination rate regression
- cost/request above budget
- GPU memory pressure
- tool failure rate spike

Q48: How do you roll out a new embedding model or index?

A:
I build a new index version in parallel, run offline golden-set evaluation, compare candidate overlap, run shadow traffic, then canary a small percentage of users. If retrieval quality, latency, cost, and citation accuracy are healthy, I promote it. Otherwise I rollback to the previous index version.

中文：
embedding model 变了，index 也要一起 version。不能只换模型不重建 index。

Q49: How do you roll out prompt changes?

A:
Prompts are versioned artifacts. Each prompt change runs through golden-set regression, LLM-as-Judge evaluation, safety checks, shadow traffic, and canary rollout. If hallucination or citation accuracy regresses, the rollout is stopped or rolled back.

Q50: How do you debug a bad answer?

A:
I trace the pipeline step by step:

1. Was the original query understood correctly?
2. Did query rewrite preserve intent?
3. Did domain routing choose the right corpus?
4. Did retrieval return relevant chunks?
5. Did metadata / ACL filters remove good evidence?
6. Did reranker rank the right evidence high?
7. Did context packing include enough information?
8. Did the LLM ignore evidence or hallucinate?
9. Were citations valid?
10. Was the document stale or conflicting?

中文：
不要一上来怪 LLM。先拆开看 retrieval、rerank、context、prompt、generation 哪一层坏了。

Interview Q&A: Operations, Scaling, Tradeoffs, Failures & Close

11. Scaling / 大规模设计

Q51: How would you scale to thousands of enterprise tenants?

A:
I would use tenant-aware sharding, separate large tenants into dedicated collections or indexes, apply metadata filters for smaller tenants, isolate quotas, and use per-tenant index versions. Hot tenants can get dedicated capacity, while smaller tenants share multi-tenant infrastructure.

Q52: How do you handle very large knowledge bases?

A:
I use hierarchical retrieval and domain routing. First classify the domain or intent, then search the relevant subset. I also use metadata filters, partitioned vector collections, BM25 prefiltering for exact entities, and reranking only on a limited candidate set.

中文：
大库不能一锅炖。要先缩小搜索空间，再精排。

Q53: How do you keep indexes fresh?

A:
I use event-driven ingestion. Document updates generate events through Kafka. The indexing pipeline reprocesses only changed documents, updates metadata, rebuilds affected chunks, and publishes a new index version or incremental update. Deleted or expired documents are tombstoned and excluded from retrieval.

Q54: How do you handle deleted documents?

A:
Deleted documents are marked with tombstones and removed from active retrieval. The vector index and BM25 index are updated asynchronously. The system also prevents old evidence snapshots from being reused in new answers unless retention policy allows it.

12. Tradeoffs / Design Decisions

Q55: Fixed-size chunking vs structure-aware chunking?

A:
Fixed-size chunking is simple and generic, but it can break semantic units. Structure-aware chunking is more complex, but it preserves clauses, procedures, tables, and headings. For enterprise documents, structure-aware chunking gives better answer quality and citations.

Q56: Single index vs multiple domain indexes?

A:
A single index is easier to operate, but it can mix unrelated domains and increase noisy retrieval. Multiple indexes or logical partitions improve precision and access control, but add operational complexity. My default is metadata-based partitioning first, then dedicated indexes for large or high-value domains.

Q57: Fine-tuning vs RAG improvement?

A:
Fine-tuning improves style, format, or domain behavior, but it does not fix missing or wrong evidence. For enterprise knowledge systems, I would first improve ingestion, chunking, retrieval, reranking, and evaluation. Fine-tuning comes later if we need specialized behavior.

Q58: Long-context LLM vs retrieval?

A:
Long context is useful, but it is expensive and still needs ranking. Retrieval gives better cost control, freshness, and source attribution. I would use retrieval to select high-quality evidence and reserve long context for cases where many related documents are truly needed.

Q59: Build vs buy?

A:
I would buy commodity components where possible, such as vector database, search engine, OCR, and model serving runtime. I would build the enterprise-specific control plane: metadata model, ACL enforcement, index/version management, evaluation, agent permissions, audit, and workflow orchestration.

中文：
真正有差异化的是企业治理层和 workflow 层，不是自己从零写一个 vector DB。

Q62: What is the hardest part of this project?

A:
The hardest part is not the vector database or the LLM call. The hardest part is making the system trustworthy under real enterprise conditions: messy documents, vague questions, changing policies, permission boundaries, conflicting sources, tool side effects, and the need for auditability.

中文：
难点不是 demo，而是 production trust。

Because it connects four things that enterprise customers care about: knowledge, action, governance, and measurement. A normal chatbot only answers. This platform retrieves trusted evidence, generates cited answers, safely executes workflows, respects permissions, logs every decision, and continuously evaluates quality.

这不是一个 chatbot，而是企业 AI operating layer。它把私有知识、业务规则、权限系统和工具流程连接起来，并且做到可治理、可审计、可评估、可回滚。
