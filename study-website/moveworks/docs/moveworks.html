4. Probable Interview Questions & How to Answer Them

Q1 (Architecture): "How did you balance precision and latency when applying dense retrieval (vectors) across billions of documents?"

Your Strategy: Do not just say "we added more GPUs."
Answer: "We utilized hierarchical indexing (like IVF-PQ) to shard the vector space and drop our search parameters drastically. But the key was our Parallel Hybrid Architecture. We ran sparse keyword queries and dense embedding searches asynchronously across isolated compute nodes, and merged them at the API gateway layer using Reciprocal Rank Fusion (RRF). To hit strict SLAs, we cached hot vectors in Redis and ensured our fallback mechanisms would downgrade to sparse-only search if the GPU Embedding cluster tripped a latency circuit breaker."

🇨🇳 中文提示（Q1答法）：
核心论点：不是靠堆 GPU，而是靠架构设计。IVF-PQ 分层索引将1000亿向量的 ANN 搜索时间从数小时压缩到毫秒级（PQ 量化节省95% 内存）。Parallel Hybrid Architecture：稀疏检索和密集检索并发执行、RRF 融合，在 API 网关层合并结果。Redis 热向量缓存 + 熔断器降级（Dense 超时自动回退到 Sparse）保障 SLA。

Q2 (GenAI Ops): "Enterprise data is notoriously messy. How do you ensure your RAG pipelines don't just 'hallucinate' bad internal facts?"

Your Strategy: Lean entirely on your Acrobat AI Assistant Evals experience.

If the retrieval score is below a certain confidence threshold, the agent rejects rather than guesses. Second, post-generation evals: I architected an LLM-as-a-judge pipeline sitting in front of our API gateway. It executes deterministic safety scoring and artifact checks on the generated response. By running 5 million of these evals a day on EKS with sub-300ms latency, we enforced extremely tight error budgets before user-facing delivery."

🇨🇳 中文提示（Q2答法）：
两层防护：第一层在检索端——先做意图分类，检索置信分低于阈值直接拒绝作答（宁可说"我不知道"也不猜）。第二层在生成端——LLM-as-a-Judge 管道在响应到达用户前执行确定性安全评分和引用核查（artifact check）。每天 500 万次、延迟 <300ms 的规模保证了这不是实验性方案，而是生产级基础设施。

C. Serving & Inference Infrastructure (EKS / Triton / vLLM)

Adobe Experience: The AI Platform ran on AWS EKS, with specialized node pools per workload:
- CPU node pool: Ran lightweight preprocessing (tokenization, query expansion, BM25 sparse retrieval via FAISS, metadata filtering).
- T4 GPU node pool: Ran online embedding inference (bi-encoder), neural reranking (cross-encoder), and the LLM-as-a-judge eval pipelines—5M+ inferences/day at <300ms p95 latency.
- A10G/A100 GPU node pool (spot instances): Runs async batch embedding jobs (delta-indexing overnight), Firefly image generation queues, and fine-tuned LoRA adapter merging.

Key optimizations you implemented:
1. Dynamic batching with Triton: Instead of sending one request at a time to the embedding model, Triton's dynamic batching groups up to 32 concurrent requests into a single GPU kernel call, massively improving GPU utilization from ~35% to ~82%.
2. KV-Cache management for LLM evals: When running LLM-as-a-judge evals, the system prompt and rubric are constant across millions of calls. By pre-computing and caching the KV state of the fixed prefix, each eval call skips ~40% of the prefill phase.
3. KEDA-based autoscaling on Kafka lag: Rather than scaling on CPU/memory (which lags behind real load by minutes), the EKS deployment scaled GPU node groups directly on the Kafka consumer lag metric. If the eval queue backlogged > 5,000 messages, KEDA immediately provisioned additional T4 nodes.

How to frame for Praveen:
"The secret to sub-300ms at 5M evals/day wasn't just raw hardware—it was eliminating every wasted GPU cycle. Dynamic batching turned our embedding servers from 35% to 82% utilized. KV-cache prefix sharing cut the latency of our LLM-as-a-judge calls nearly in half. And we scaled reactively on queue depth, not server metrics, which meant we provisioned capacity before users felt latency, not after."

🇨🇳 中文提示（推理基础设施优化）：
三个关键优化最值得深入展开：1) Triton 动态批处理（dynamic batching）——将最多32个并发请求合并成单次 GPU 内核调用，GPU 利用率从35%提升到82%；2) KV-Cache 前缀共享——LLM评估的 system prompt 和评估标准在数百万次调用中是固定的，预计算并缓存这段固定前缀的 KV 状态，使每次评估调用跳过约40%的 prefill 阶段，延迟减半；3) 基于 Kafka 消费者 lag 的 KEDA 自动扩容——不等 CPU/内存告警（这会滞后几分钟），直接监控队列积压深度，超过5000条消息立即触发 T4 节点扩容，在用户感知到延迟之前就完成扩容。

7. DEEP DIVE: Orchestration Layer (Directly Maps to MMM Agent Architecture)

What the orchestration layer does at Adobe

The orchestration layer is the "traffic controller" sitting between user requests and the backend model/retrieval services. For Acrobat's AI Assistant, a single user question like "Summarize this contract and flag unusual clauses" triggers:
1. Intent classification → route to "summarize" vs "QA" vs "comparison" flow.
2. RAG retrieval → fetch top-K paragraph chunks from the PDF's vector index.
3. Context assembly → merge retrieved chunks + conversation history + system prompt.
4. LLM call → stream response from the primary model (Claude/Gemini/GPT-4).
5. Eval gate → LLM-as-a-judge scores the output for safety, factuality, artifact quality.
6. Tool-call loop → if the model decides to "look up the definition of a legal term", it triggers a secondary tool call (web search or internal KB lookup) and re-enters the pipeline.
7. Response delivery → stream tokens to the user with citation metadata.

🇨🇳 中文提示（编排层全流程）：
编排层是用户请求和后端模型/检索服务之间的"流量控制器"。以 Acrobat AI 助手为例，"总结这份合同并标记异常条款"这个问题触发了一个完整的7步流水线：意图分类（路由到哪个子流程）→ RAG 检索（从 PDF 向量索引取 top-K 段落）→ 上下文组装（检索内容+对话历史+系统 Prompt）→ LLM 生成（流式响应）→ 评估门（安全性/事实性/引用核查）→ 工具调用循环（模型决定查外部知识库时重新入队）→ 带引用元数据的响应流式输出。这与 MMM 的架构几乎完全一致。

A. The DAG-Based Orchestration Engine

You built a DAG-structured workflow engine rather than a simple linear chain. Reasons:
- Parallelism: Query understanding, intent classification, and initial sparse retrieval can run in parallel before dense retrieval needs the intent signal.
- Conditional branching: If intent = "document comparison," the DAG fans out to retrieve context from two documents simultaneously and joins the results before LLM context assembly.
- Retry semantics: Each DAG node has its own retry policy and fallback behavior (e.g., if dense retrieval times out, re-route to sparse-only path rather than hard failing).

🇨🇳 中文提示（DAG编排引擎）：
选择 DAG（有向无环图）结构而非简单线性链的原因：并行性——意图分类和初始稀疏检索可以并发执行，Dense 检索在拿到意图信号后再启动；条件分支——"文档对比"意图时，DAG 同时展开从两份文档检索上下文，再合并后进行 LLM 组装；重试语义——每个 DAG 节点有独立的重试策略和 fallback 行为（Dense 检索超时 → 自动降级到 Sparse-only，而不是整体报错）。这使系统具备局部容错能力，而不是"一个节点失败全链崩溃"。

B. Rate Limiting, Circuit Breakers, and Backpressure

This is where MMM will probe you hardest—enterprise SLAs require the system to fail gracefully, not catastrophically.

Your Adobe Implementation:
- Token Bucket Rate Limiter (Redis): Every API caller (product team, enterprise tenant) gets a token bucket tracked in Redis. Tokens refresh at a configured rate (e.g., 100 requests/min per tenant). The orchestration layer checks the bucket atomically with Lua scripts on every incoming request—no distributed lock needed.
- Circuit Breaker Pattern (Resilience4j / custom): Each downstream dependency (embedding service, LLM API, reranker) had a circuit breaker with three states:
- CLOSED (healthy): All requests pass through normally.
- OPEN (failing): After 10 consecutive failures or 50% error rate in a 10-second window, the circuit opens. Requests immediately return a fallback (cached result or degraded response) without hitting the failing service.
- HALF-OPEN (recovering): After 30 seconds, one probe request is sent. If it succeeds, the circuit closes. If it fails, it resets the open timer.
- Backpressure via Kafka: Async tasks (batch indexing, eval scoring, LoRA fine-tune triggers) are queued in Kafka topics. If consumers fall behind (> configurable lag threshold), the API gateway throttles new task submissions, preventing OOM crashes on the processing tier.

How to frame for Praveen:
"Enterprise systems can't hard-fail during a supplier outage. I implemented circuit breakers around every external dependency—including our own LLM endpoints. When Anthropic had a brief API degradation, our circuit opened automatically, served cached responses, and the user never saw an error. Meanwhile, the system was paging our on-call and preparing to reroute to a fallback model."

🇨🇳 中文提示（限流/熔断/背压）：
这是 MMM 必问的系统可靠性主题。三个机制形成纵深防御：1) Token Bucket 限流（Redis + Lua 原子脚本）——在租户层而非用户层限流，Lua 脚本确保原子性检查，无需分布式锁；2) 熔断器（三态：CLOSED健康/OPEN熔断/HALF-OPEN探测恢复）——每个下游依赖（Embedding 服务、LLM API、重排序器）独立保护，10次连续失败或10秒内50%错误率触发熔断，30秒后进入半开状态发送探针请求；3) Kafka 背压——异步任务队列的消费者 lag 超过阈值时，API 网关停止接受新任务提交，防止处理层 OOM。当 Anthropic API 短暂降级时，熔断自动触发、缓存响应兜底、用户无感知、on-call 同时收到报警——这就是企业级容错架构。

C. Caching Strategy (Semantic + Exact)

LLM inference is expensive. Every unnecessary LLM call at 5M/day scale costs real money.

Your two-tier caching architecture at Adobe:
1. Exact-match cache (Redis, TTL=1hr): The full MD5 hash of (query + doc_id + model_version) is the cache key. If a user asks "what is the termination clause?" on the same document twice, the second call returns in <1ms from Redis.
2. Semantic similarity cache (Vector-based, TTL=24hr): For near-duplicate queries that aren't byte-identical ("What does the termination section say?" vs "Summarize the termination clause"), a fast bi-encoder embedding is computed and compared against a small in-memory ANN index of recently answered queries. If cosine similarity > 0.92, the cached response is returned. Cache hit rate: ~34% reduction in live LLM calls.

How to frame for Praveen:
"We ran a two-tier cache. The exact-match layer is trivial but surprisingly effective for enterprise chatbots where employees ask the same VPN setup question repeatedly. The semantic layer was more interesting—we embedded every cached query and ran a fast ANN lookup before hitting the LLM. At 34% cache hit rate on 5M daily evals, that's 1.7 million LLM calls saved per day. At $0.01 per call average, that's $17,000/day in cost avoidance."

🇨🇳 中文提示（两层缓存策略）：
LLM 调用很贵，在每天500万次的规模下，节省每一次多余的调用都相当于真金白银。两层缓存：第一层精确匹配缓存（Redis，TTL=1小时）——缓存键是 MD5(租户ID + 查询 + 模型版本)，完全相同的问题第二次 <1ms 返回；第二层语义相似缓存（向量ANN，TTL=24小时）——对字节不完全相同但语义相近的查询（"终止条款是什么"vs"总结终止部分"），计算查询 Embedding 并与最近已回答问题的内存 ANN 索引比对余弦相似度，超过0.92直接返回缓存答案。34%的缓存命中率意味着每天节省170万次 LLM 调用，按均价$0.01/次计算，每天省17000美元——这是为 Director 准备的语言。

8. Key Interview Q&A: Platform, Infra, and Orchestration Specifics

Q: "How do you handle non-determinism in your orchestration pipeline for enterprise-grade reliability?"

A: "Non-determinism is the enemy of enterprise trust. We attacked it at three levels. First, at model selection: for safety-critical paths (flagging content, generating legal summaries), we always used temperature=0 and fixed seed to minimize variance. Second, at the eval layer: we didn't trust the LLM to be consistent, so we ran every response through a separate, deterministic scoring function (regex-anchored rubrics + lightweight classifier) alongside the LLM judge. Third, at the deployment level: we shadowed every new model on live traffic for 48 hours before promoting it, comparing against the old model's output distribution with Jensen-Shannon divergence. If JSD exceeded 0.05, the promotion was blocked."

🇨🇳 中文提示（非确定性处理）：
三层打击：模型层（temperature=0，固定 seed）→ 评估层（在 LLM judge 之外叠加确定性评分函数：正则锚定 rubric + 轻量分类器，不完全信任 LLM judge 自身的一致性）→ 部署层（每次新模型上线前做48小时影子流量测试，用 Jensen-Shannon 散度比较新旧模型输出分布，JSD > 0.05 则阻断发布）。JSD 是衡量两个概率分布差异的统计量，用它来检测"看起来通过离线测试但在线行为已发生漂移"的静默回归——这是最危险的 ML 回归类型。

Q: "Walk me through how you'd debug a latency regression in the retrieval pipeline."

A: "I'd start with our per-stage Prometheus traces—we instrumented every hop (intent classification, sparse retrieval, dense retrieval, RRF merge, rerank, LLM call) with p50/p95/p99 histograms. If the aggregate latency spiked, the traces would show which stage's p95 had changed. Most often it was reranker throughput degrading on GPU nodes—either thermal throttling on spot instances, or a batch size regression from a code change. We also maintained a synthetic canary query that ran every 60 seconds across all regions and alarmed if p95 exceeded our SLO by 20%. Catching regressions before users do is the only acceptable strategy."

🇨🇳 中文提示（延迟回归调试）：
方法论：从 Prometheus 每阶段 trace 入手——对每个流水线节点（意图分类、稀疏检索、密集检索、RRF 融合、重排序、LLM 调用）都有 p50/p95/p99 直方图。总延迟上升时，trace 会精确指向哪个阶段的 p95 变了。最常见的原因是 GPU 节点的重排序器吞吐量下降——可能是 Spot 实例热降频（thermal throttling），也可能是代码变更导致批处理大小回退。此外还维护了一个综合 Canary 查询，每60秒在所有 Region 执行，p95 超出 SLO 20% 立即告警。核心理念：在用户感知到问题之前发现并处理，是唯一可接受的策略。

Q: "How do you design the orchestration layer to support multimodal inputs at MMM scale—not just text queries but images, files, and audio?"

A: "The orchestration DAG needs a modality normalization step at the very front. Every input—regardless of whether it's a Slack message, a PDF attachment, or a voice memo—passes through a Modality Router that extracts a unified semantic representation. A PDF goes through our document parser to extract text chunks. An image gets a CLIP embedding. An audio clip goes through Whisper for transcription before entering the same text pipeline. By the time a request hits the retrieval stage, the orchestration layer is always working with the same canonical data types: text chunks and embeddings. This is exactly what I built for Acrobat—a multimodal intake that normalized PDFs, images embedded in PDFs, and typed queries into a single retrieval context before any LLM call."

🇨🇳 中文提示（多模态输入处理）：
架构原则：在 DAG 最前端加一个 Modality Router（模态路由器），将所有类型的输入归一化为统一的标准表示形式。PDF → 文档解析器提取文本块；图片 → CLIP 编码生成视觉 Embedding；音频 → Whisper ASR 转写为文本。到达检索阶段时，编排层面对的永远只有两种标准数据类型：文本块（text chunks）和 Embedding 向量。这意味着检索引擎和 LLM 调用代码完全不需要针对模态做特殊处理——模态复杂度被完全封装在 Router 层内。这正是你在 Acrobat 上构建的多模态进入管道的核心设计思路。
