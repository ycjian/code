Section 03

1. Classic ML evaluation

Classification

from sklearn.metrics import (
    accuracy_score, precision_score, recall_score, f1_score,
    classification_report, confusion_matrix, roc_auc_score
)

print("accuracy", accuracy_score(y_true, y_pred))
print("f1 macro", f1_score(y_true, y_pred, average="macro"))
print(classification_report(y_true, y_pred))
print(confusion_matrix(y_true, y_pred))

# binary classification probability metric
auc = roc_auc_score(y_true, y_prob[:, 1])

When to use

  • accuracy: balanced labels, simple overview
  • precision: false positives are expensive
  • recall: false negatives are expensive
  • F1: balance precision/recall
  • macro F1: every class equally important
  • weighted F1: account for class frequency
  • AUC: ranking/separation quality for binary classifier

Regression

from sklearn.metrics import mean_absolute_error, mean_squared_error, r2_score

mae = mean_absolute_error(y_true, y_pred)
mse = mean_squared_error(y_true, y_pred)
rmse = mse ** 0.5
r2 = r2_score(y_true, y_pred)

Ranking/search

  • recall@k: relevant items found in top k
  • precision@k: top k relevance density
  • MRR: first relevant result rank
  • NDCG@k: ranking quality with graded relevance
Section 04

2. Slice-based evaluation

Average metrics hide failures.

Examples of slices

  • language = English vs Chinese
  • device = mobile vs desktop
  • query length short vs long
  • new users vs returning users
  • high-value customers
  • rare categories
  • long-tail documents
  • cold-start items

Simple slice eval

import pandas as pd
from sklearn.metrics import f1_score

df = pd.DataFrame({
    "y_true": y_true,
    "y_pred": y_pred,
    "country": country,
    "device": device,
})

for name, group in df.groupby("device"):
    print(name, f1_score(group.y_true, group.y_pred, average="macro"))

Interview phrase

"I would never trust only the aggregate metric. I would inspect failure slices and build regression tests from those slices."

Section 05

3. RAG evaluation framework

RAG pipeline

query -> query rewrite -> retrieval -> reranking -> context building -> LLM answer -> citations

Evaluate each component separately.

Retrieval metrics

  • recall@k: did we retrieve the supporting docs?
  • MRR: how high is the first relevant doc?
  • NDCG@k: are more relevant docs ranked higher?
  • context precision: are retrieved chunks mostly useful?
  • context recall: did we miss necessary chunks?

Generation metrics

  • faithfulness: answer supported by retrieved context?
  • answer relevance: answer addresses the question?
  • correctness: matches known answer?
  • citation correctness: cited source supports claim?
  • refusal correctness: refuses when context is insufficient?

Operational metrics

  • latency by stage
  • token cost
  • retrieval timeout rate
  • empty retrieval rate
  • answer fallback rate

Example row

  "expected_answer": "Full refund before X date, minus registration fee.",
  "required_doc_ids": ["policy_2026.pdf"],
  "slice": "policy_question",
  "difficulty": "medium"
}

Build dataset from

  • real user queries
  • support tickets
  • known failure cases
  • synthetic variations reviewed by humans
  • edge cases: ambiguous, no-answer, outdated-doc, conflicting-doc

5. RAGAS-style evaluation example

RAGAS evaluates RAG dimensions such as faithfulness, answer relevancy, context precision, and context recall.

# pip install ragas datasets
from datasets import Dataset
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_precision, context_recall

data = {
    "question": ["What is our PTO policy?"],
    "answer": ["Employees get 15 days of PTO per year."],
    "contexts": [["The company provides 15 PTO days per year for full-time employees."]],
    "ground_truth": ["Full-time employees receive 15 PTO days per year."],
}

dataset = Dataset.from_dict(data)

result = evaluate(
    dataset,
    metrics=[faithfulness, answer_relevancy, context_precision, context_recall],
)
print(result)

Use with caution

  • LLM-as-judge can be noisy.
  • Use consistent judge model and prompt.
  • Calibrate judge with human labels.
  • Track judge version.

6. MLflow evaluation and registry

Track experiments

import mlflow
from sklearn.metrics import f1_score

with mlflow.start_run():
    model.fit(X_train, y_train)
    y_pred = model.predict(X_test)
    f1 = f1_score(y_test, y_pred, average="macro")

    mlflow.log_param("model_type", "logistic_regression")
    mlflow.log_metric("f1_macro", f1)
    mlflow.sklearn.log_model(model, "model")

Register model

model_uri = "runs:/<run_id>/model"
mlflow.register_model(model_uri, "search-ranking-model")

Evaluate model

import mlflow

result = mlflow.models.evaluate(
    model="runs:/<run_id>/model",
    data=eval_df,

Section 09

7. Online A/B testing

Offline win does not guarantee online win.

A/B test design

  • Randomize by user_id or request_id.
  • Keep assignment stable.
  • Define primary metric before launch.
  • Define guardrail metrics.
  • Run long enough for statistical power.

Example

Treatment: new reranker
Control: old reranker
Primary: conversion rate / task success / NDCG from human labels
Guardrails: latency p95, error rate, cost/query, complaint rate

Common metrics

  • CTR
  • conversion rate
  • revenue/session
  • retention
  • search success rate
  • zero-result rate
  • reformulation rate
  • latency

Interleaving

  • Useful for search/ranking.
  • Mix results from two rankers and infer preference from clicks.
  • Faster sensitivity than full A/B in some ranking systems.
Section 10

8. Regression gates for LLM/RAG systems

Before deploy, run gates

1. Unit tests for prompt templates and parsers
2. Retrieval recall@k must not drop > threshold
3. Faithfulness must not drop > threshold
4. Safety eval must pass all critical tests
5. Latency/cost must stay under budget
6. Golden failure cases must not regress

Example gate code

assert metrics["retrieval_recall_at_5"] >= 0.92
assert metrics["faithfulness"] >= 0.85
assert metrics["p95_latency_ms"] <= 2500
assert metrics["cost_per_query_usd"] <= 0.01

9. Human evaluation

Human rubric example

Correctness: 1-5
Completeness: 1-5
Faithfulness to sources: 1-5
Citations support claims: yes/no
Tone/helpfulness: 1-5
Safety/privacy issue: yes/no

Best practices

  • Use blinded comparison when possible.
  • Sample across slices.
  • Include no-answer cases.
  • Measure inter-rater agreement.
  • Convert human-rated failures into golden tests.
Section 12

10. Error analysis loop

Loop

1. Collect failures.

2. Categorize failure type.

3. Add slice or golden test.

4. Fix data/retrieval/prompt/model.

5. Re-run eval.

6. Monitor online.

Failure taxonomy for RAG

  • Retrieval miss
  • Wrong chunk ranking
  • Context too long/noisy
  • Prompt misunderstood question
  • LLM hallucinated beyond context
  • Citation unsupported
  • Outdated document used
  • Permission leakage
Section 13

11. Interview quick answers

Q: Model accuracy improved from 90% to 92%. Ship it?
A: Not immediately. Check confidence intervals, slice metrics, business metrics, latency/cost, and online A/B results.

Q: RAG answer is wrong. How debug?
A: First check retrieval: did we retrieve required context? Then check context assembly: was it included in prompt? Then generation: did the LLM ignore/misread context? Then evaluate citations.

Q: How to evaluate no-answer cases?
A: Include questions where answer is not in docs. Metric should reward correct refusal and penalize hallucinated answers.

Q: How to prevent metric gaming?
A: Keep hidden test sets, slice-based eval, human review, online guardrails, and periodically refresh eval sets from real traffic.

RESUME-ALIGNED INTERVIEW DEEP DIVE

A. Evaluation stack for your resume
------------------------------------------------------------
Use this as the answer when interviewers ask: How do you know the model or GenAI system is better?

Layered answer

  • Offline ML metrics for classifier/ranker/retriever.
  • RAG component metrics: retrieval recall, context precision/recall, faithfulness, answer relevance, citation correctness.
  • LLM-as-judge only after calibration against human labels.
  • Golden regression dataset for recurring failures.
  • Slice metrics for product, modality, language, tenant, content type, long-tail queries.
  • Online A/B or interleaving for real user impact.
  • Cost/latency/error budget as rollout gates.

B. RAG debug split
------------------------------------------------------------

A bad RAG answer comes from one of four places

1. Query understanding failed.

2. Retrieval missed evidence.

3. Reranker/context builder selected poor evidence.

4. Generator ignored or misused correct evidence.

Never debug RAG only by looking at final answer text. Trace stage-level evidence.

C. Ranking/LLM evaluation tools
------------------------------------------------------------
MLflow: experiment tracking, model registry, classic evaluation, and GenAI evaluation APIs.
Weights & Biases / W&B: experiment tracking, artifact/version tracking, sweeps, dashboards, reports, and collaborative experiment review.
Ragas/DeepEval-style libraries: RAG faithfulness, context precision/recall, answer relevance, judge-based metrics.
Evidently-style monitoring: drift, data quality, production monitoring.

D. Interview Q&A
------------------------------------------------------------
Q: How do you prevent LLM-as-judge from becoming circular?
A: Calibrate with human labels, use multiple judges for high-risk areas, keep adversarial examples, track judge prompt/model version, and inspect disagreement slices.

Q: How do you evaluate ranking online?
A: Use A/B tests or interleaving, monitor CTR/task success/reformulation/zero-result rate, and guard against novelty effects and position bias.

Q: What goes in a golden dataset?
A: High-value queries, historical regressions, hard negatives, permission-sensitive cases, long-context cases, multilingual cases, and product-critical workflows.
