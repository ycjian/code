Section 03

1. Tool calling basics

Tool = function the model can ask the app to call.

Example tool schema idea

def search_docs(query: str, top_k: int = 5) -> list[dict]:
    """Search internal docs."""
    return vector_search(query, top_k=top_k)

Tool calling flow

1. Send user request + available tools to model.

2. Model returns tool call name + arguments.

3. App validates arguments and permissions.

4. App executes tool.

5. App sends tool output back to model.

6. Model responds or calls more tools.

Critical rule

  • The model suggests tool calls; your application executes them.
  • Validate all tool args.
  • Enforce permissions outside the model.

3. LangGraph-style state machine

LangGraph is useful for stateful/durable agents and graph-based control flow.

Conceptual example

# Pseudocode close to LangGraph style.
from typing import TypedDict, List
from langgraph.graph import StateGraph, END

class State(TypedDict):
    question: str
    docs: List[str]
    answer: str
    need_human: bool


def retrieve(state: State):
    docs = search_docs(state["question"])
    return {"docs": docs}


def generate(state: State):
    answer = call_llm(question=state["question"], docs=state["docs"])
    return {"answer": answer}


def check_risk(state: State):
    risky = contains_sensitive_action(state["answer"])
    return {"need_human": risky}


def route_after_risk(state: State):
    return "human_review" if state["need_human"] else "done"


def human_review(state: State):
    # pause / wait for human approval in real system
    approved = request_human_approval(state["answer"])
    if not approved:
        return {"answer": "I cannot proceed without approval."}
    return state

builder = StateGraph(State)
builder.add_node("retrieve", retrieve)
builder.add_node("generate", generate)
builder.add_node("check_risk", check_risk)
builder.add_node("human_review", human_review)

builder.set_entry_point("retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", "check_risk")
builder.add_conditional_edges("check_risk", route_after_risk, {
    "human_review": "human_review",
    "done": END,
})
builder.add_edge("human_review", END)

graph = builder.compile()
result = graph.invoke({"question": "Can I refund this customer?"})

4. Temporal workflow basics

Temporal is for durable workflows

  • workflow state is persisted
  • activities can retry
  • workflow can resume after worker crash
  • good for long-running business processes

Concepts

  • Workflow: deterministic orchestration logic.
  • Activity: external side effect, e.g. call API, send email.
  • Retry policy: automatic retries for transient failure.
  • Signal: external event sent to workflow.
  • Timer: durable sleep.

Python sketch

from temporalio import workflow, activity
from datetime import timedelta

@activity.defn
async def call_llm_activity(prompt: str) -> str:
    return await call_llm(prompt)

@activity.defn
async def send_email_activity(to: str, body: str) -> None:
    await send_email(to, body)

@workflow.defn
class RefundAgentWorkflow:
    def __init__(self):
        self.approved = False

    @workflow.signal
    async def approve(self):
        self.approved = True

    @workflow.run
    async def run(self, user_request: str) -> str:
        draft = await workflow.execute_activity(
            call_llm_activity,
            user_request,
            start_to_close_timeout=timedelta(seconds=30),
        )

        # wait for human approval signal
        await workflow.wait_condition(lambda: self.approved)

        await workflow.execute_activity(
            send_email_activity,
            "customer@example.com",
            draft,
            start_to_close_timeout=timedelta(seconds=30),
        )
        return "sent"

5. Idempotency

Tool calls may retry. Make them safe.

Bad

def charge_card(user_id, amount):
    payment_api.charge(user_id, amount)

Good

def charge_card(user_id, amount, idempotency_key):
    payment_api.charge(
        user_id=user_id,
        amount=amount,
        idempotency_key=idempotency_key,
    )

Idempotency key examples

workflow_id + activity_name + attempt-independent operation id
request_id + tool_name + business_entity_id

11. Interview quick answers

Q: Why not just use a ReAct loop?
A: A ReAct loop is fine for prototypes. Production needs persisted state, retries, idempotent tools, human approval, and audit logs. A graph/workflow engine makes execution resumable and safer.

Q: How prevent agents from doing dangerous actions?
A: Enforce tool-level authorization, validate args, require approval for risky tools, sandbox execution, limit cost/iterations, and log actions for audit.

Q: How recover if agent crashes halfway?
A: Use durable execution/checkpoints. Completed steps are persisted, activities are idempotent, and the workflow resumes from the last safe state.

Q: How debug bad agent behavior?
A: Inspect trace by node/tool call, prompt/model versions, retrieved context, tool outputs, and decision points. Add failure to regression eval.

Section 15

RESUME-ALIGNED INTERVIEW DEEP DIVE

A. Production agent stance
------------------------------------------------------------
For production agents, do not rely on a simple while loop. Use a graph/state machine or durable workflow so execution can be resumed, audited, retried, and approved by humans.

B. LangGraph vs Temporal
------------------------------------------------------------
LangGraph: useful for stateful agent control flow, tool loops, branching, human-in-the-loop, and durable agent state.
Temporal: durable workflow engine for long-running business processes, retries, timeouts, activities, compensation, and event history.

Use together conceptually

  • LangGraph controls agent reasoning/tool graph.
  • Temporal controls durable execution around external side effects and long-running jobs.

C. Reliability rules
------------------------------------------------------------

  • Workflow state must be checkpointed.
  • Tool calls must be idempotent or protected with dedup keys.
  • Risky actions need approval gates.
  • Activities need timeouts and retry policy.
  • Compensation/saga steps are needed for partial failure.
  • Every tool call needs audit metadata.

D. Interview Q&A
------------------------------------------------------------
Q: Why is deterministic workflow code important?
A: Durable workflow engines replay history to rebuild state. Non-deterministic workflow logic can break replay; side effects belong in activities.

Q: How do you resume an agent after worker failure?
A: Persist state/checkpoints after each step, use durable workflow history, retry idempotent activities, and resume from the last committed state.

Q: How do you handle human approval?
A: Represent approval as an explicit workflow state, pause execution, show decision context, record approver/action/time, then continue or compensate.
