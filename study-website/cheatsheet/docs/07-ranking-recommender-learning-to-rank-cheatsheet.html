request/query/user context
-> candidate generation
-> filtering
-> feature hydration
-> ranking/reranking

2. BM25 vs dense retrieval

BM25

  • lexical keyword matching
  • strong for exact terms, names, IDs, rare words
  • interpretable
  • poor at semantic paraphrase

Dense vector retrieval

  • semantic similarity
  • strong for natural language and paraphrase
  • needs embeddings + ANN index
  • can miss exact rare terms

Hybrid search

  • combine BM25 + dense vector results
  • robust in production
  • often use Reciprocal Rank Fusion (RRF) or weighted score blend

RRF formula idea

def rrf(rankings, k=60):
    scores = {}
    for ranking in rankings:
        for rank, doc_id in enumerate(ranking, start=1):
            scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank)
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)

3. Two-tower retrieval

Two-tower model

user/query tower -> embedding q
item/doc tower -> embedding d
score = dot(q, d)

Why

  • Precompute item embeddings offline.
  • Compute query/user embedding online.
  • Use ANN to retrieve nearest items.

Training data

  • positive pairs: clicked/purchased/relevant
  • negatives: random negatives, in-batch negatives, hard negatives

PyTorch sketch

import torch
import torch.nn as nn
import torch.nn.functional as F

class TwoTower(nn.Module):
    def __init__(self, user_dim, item_dim, hidden=128):
        super().__init__()
        self.user_tower = nn.Sequential(nn.Linear(user_dim, hidden), nn.ReLU(), nn.Linear(hidden, hidden))
        self.item_tower = nn.Sequential(nn.Linear(item_dim, hidden), nn.ReLU(), nn.Linear(hidden, hidden))

    def forward(self, user_x, item_x):
        u = F.normalize(self.user_tower(user_x), dim=-1)
        v = F.normalize(self.item_tower(item_x), dim=-1)
        return u @ v.T  # in-batch scores

model = TwoTower(user_dim=50, item_dim=100)
scores = model(user_features, item_features)
labels = torch.arange(scores.size(0)).to(scores.device)  # diagonal positives
loss = F.cross_entropy(scores, labels)

4. Learning to Rank with LightGBM

Learning-to-rank data format

  • Each row = query-item pair
  • label = relevance grade/click/conversion
  • group = number of items per query

Example

import lightgbm as lgb
import numpy as np

X = np.array([
    [0.9, 3.2, 1.0],  # query1-doc1 features
    [0.2, 1.1, 0.0],  # query1-doc2
    [0.7, 2.0, 1.0],  # query1-doc3
    [0.1, 0.5, 0.0],  # query2-doc1
    [0.8, 1.9, 1.0],  # query2-doc2
])
y = np.array([3, 1, 2, 0, 2])
group = np.array([3, 2])  # query1 has 3 docs, query2 has 2 docs

ranker = lgb.LGBMRanker(
    objective="lambdarank",
    metric="ndcg",
    n_estimators=100,
    learning_rate=0.05,
)
ranker.fit(X, y, group=group)

scores = ranker.predict(X)

6. Ranking metrics

Precision@k

among top k, how many are relevant?

Recall@k

of all relevant items, how many appear in top k?

MRR

1 / rank of first relevant item

DCG@k

sum((2^rel_i - 1) / log2(i + 1))

NDCG@k

DCG@k / ideal_DCG@k

Simple NDCG

import math

def dcg(rels):
    return sum((2**rel - 1) / math.log2(i + 2) for i, rel in enumerate(rels))

def ndcg(rels):
    ideal = sorted(rels, reverse=True)
    return dcg(rels) / dcg(ideal) if dcg(ideal) > 0 else 0.0

print(ndcg([3, 2, 0, 1]))

7. Bias and counterfactual logging

Clicks are biased

  • position bias: top results get more clicks
  • presentation bias
  • selection bias
  • popularity bias

Log enough for training

{
  "request_id": "r1",
  "user_id_hash": "u123",
  "query": "running shoes",
  "shown_items": ["i1", "i2", "i3"],
  "positions": [1, 2, 3],
  "scores": [0.9, 0.8, 0.7],
  "features_version": "v5",
  "clicked_item": "i2",
  "timestamp": "2026-05-14T10:00:00Z"
}

Why log impressions, not just clicks

  • Need negatives.
  • Need position info.
  • Need reconstruct training examples.

8. Exploration vs exploitation

Problem

  • If you only show top predicted items, new items never get exposure.

Strategies

  • epsilon-greedy
  • Thompson sampling
  • UCB
  • freshness boost
  • quota for new items

Simple epsilon-greedy

import random

def choose_item(ranked_items, epsilon=0.05):
    if random.random() < epsilon:
        return random.choice(ranked_items[:50])
    return ranked_items[0]

10. Common deep-dive questions

Q: How to build search for billions of assets?
A: Use inverted index for lexical retrieval, ANN vector index for semantic retrieval, retrieve top candidates from both, merge/dedupe, hydrate features, rerank with LTR/cross-encoder, and cache popular queries. Keep index fresh via Kafka/Flink and batch rebuilds via Spark.

Q: How to handle cold-start items?
A: Use content-based features, embeddings from item text/image, freshness boosts, exploration traffic, and seller/category priors.

Q: How to personalize search?
A: Add user context and user-item/category affinity features at ranking stage. Keep retrieval broad enough to avoid over-personalization.

Q: Why offline NDCG improves but online CTR drops?
A: Offline labels may be biased/stale, slice regression may be hidden, model may reduce diversity/freshness, or online latency changed user behavior.

RESUME-ALIGNED INTERVIEW DEEP DIVE

A. Billion-scale search/recommendation answer
------------------------------------------------------------

Flow

1. Query understanding and intent classification.

2. Candidate generation from BM25, vector ANN, collaborative/personalized sources.

3. Business and permission filters.

4. Feature hydration from online feature store/cache.

5. First-stage ranker for hundreds/thousands of candidates.

6. Cross-encoder/LLM/richer reranker for top candidates.

7. Diversity/freshness/dedup policy.

8. Online metrics, interleaving/A-B tests, and offline regression suites.

B. Learning-to-rank models
------------------------------------------------------------
Classic stack: LambdaMART/XGBoost/LightGBM with dense feature vectors. Modern stack: two-tower retrieval, cross-encoder reranking, transformer rankers, and LLM-assisted reranking for small candidate sets.

C. Metrics to know
------------------------------------------------------------
NDCG: graded relevance with position discount.
MRR: rank of first relevant result.
Recall@K: whether retrieval includes relevant items.
CTR/conversion/task success: online impact.
Calibration: whether predicted probabilities match observed rates.

D. Interview Q&A
------------------------------------------------------------
Q: How do you avoid optimizing only CTR?
A: Track long-term satisfaction, diversity, freshness, reformulation rate, saves/conversions, and guardrail metrics. CTR can reward clickbait.

Q: How do you handle cold start?
A: Use content features, metadata, embeddings, exploration, popularity priors, and rapid online feature updates.

Q: Why interleaving?
A: It compares rankers within the same user/query context and can be more sensitive than coarse A/B for ranking changes.
