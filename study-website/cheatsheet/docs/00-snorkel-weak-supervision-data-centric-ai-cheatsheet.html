Section 02

0. Mental model

Snorkel-style data-centric AI starts from a simple observation: in many production ML systems, labeled data is the bottleneck, not the model architecture. Instead of hand-labeling every example, engineers write labeling functions that encode noisy heuristics, weak signals, rules, distant supervision, existing models, or LLM prompts. A label model combines these noisy sources into probabilistic training labels.

Interview one-liner

"I would use weak supervision when expert labels are expensive but we have many noisy labeling signals. The key is to model coverage, conflict, correlation, and noise, then validate with a small high-quality gold set and slice-based error analysis."

Section 03

1. Core concepts

Labeling function

  • A programmatic rule that emits a label, abstain, or sometimes a score.
  • Examples: regex rule, metadata rule, keyword rule, existing classifier, LLM prompt, knowledge-base match.

Coverage

  • Fraction of examples that a labeling function labels instead of abstaining.

Accuracy

  • How often the labeling function agrees with true labels on a labeled validation set.

Conflict

  • Labeling functions disagree on the same example.

Correlation

  • Two labeling functions are not independent, often because they use the same signal.

Label model

  • Learns how to combine noisy labeling function outputs into probabilistic labels.
  • It estimates source reliability and handles conflicts better than majority vote.

End model

  • The final model trained on probabilistic or denoised labels.
  • This model generalizes beyond the exact labeling function rules.
Section 04

2. Simple labeling-function example

ABSTAIN = -1
NEGATIVE = 0
POSITIVE = 1

def lf_refund_policy(text: str) -> int:
    t = text.lower()
    if "refund" in t or "return policy" in t:
        return POSITIVE
    return ABSTAIN

def lf_error_code(text: str) -> int:
    if "error code" in text.lower() or "stack trace" in text.lower():
        return NEGATIVE
    return ABSTAIN

def apply_lfs(rows, lfs):
    matrix = []
    for row in rows:
        matrix.append([lf(row["text"]) for lf in lfs])
    return matrix

In a real system, the labeling matrix goes into a weak-supervision label model rather than directly using majority vote.

Section 05

3. Active learning + weak supervision workflow

1. Start with a small high-quality gold set.

2. Write initial labeling functions.

3. Measure coverage, conflict, and slice performance.

4. Train label model and end model.

5. Use active learning to select uncertain or high-impact examples.

6. Ask human experts to label those examples.

7. Improve labeling functions and repeat.

Why it works

  • Weak supervision gives scale.
  • Active learning spends expert time where labels are most useful.
  • Slice analysis prevents aggregate metrics from hiding failures.
Section 06

4. LLMs as labeling functions

LLMs can be used as weak labelers, but they should not be treated as perfect ground truth.

Good uses

  • bootstrap labels for ambiguous enterprise text.
  • classify intent when rules are hard to write.
  • extract candidate entities for human review.
  • generate rationales to help debug labeling policy.

Controls

  • sample and audit outputs.
  • keep prompt/model version.
  • compare against human-labeled gold data.
  • track cost and latency.
  • use abstain when confidence is low.

Section 08

6. Failure modes

  • Labeling functions encode bias or outdated business rules.
  • Multiple labeling functions are correlated, so majority vote overestimates confidence.
  • Low coverage leaves important slices unlabeled.
  • LLM labeler follows prompt artifacts instead of true task definition.
  • Gold set is too small or not representative.
  • End model learns shortcuts from weak labels.
  • Evaluation leaks weak labeling rules into test data.
Section 09

7. Interview Q&A

Q: Why not just hand-label everything?
A: Hand labels are expensive and slow. Weak supervision scales domain heuristics, then expert labels are reserved for validation, active-learning examples, and hard slices.

Q: Why train an end model if labeling functions already label data?
A: Labeling functions may have limited coverage and brittle rules. The end model learns broader patterns from denoised labels and can generalize beyond the rules.

Q: What is the difference between majority vote and a label model?
A: Majority vote treats all labeling functions as equally reliable and independent. A label model estimates reliability, conflict, and sometimes correlation, producing probabilistic labels.

Q: How do you know weak labels are good?
A: Use a human-labeled gold set, LF summary statistics, conflict analysis, slice metrics, and downstream end-model performance. Never trust aggregate metrics alone.

Q: How does this connect to LLM evaluation?
A: LLM judges can be weak labelers. They are useful, but must be calibrated against human judgments and tracked by prompt/model version.

RESUME-ALIGNED INTERVIEW DEEP DIVE

A. How to connect this to your Adobe story
------------------------------------------------------------
Use this when an interviewer asks how you built evaluation or training data for search, RAG, or GenAI safety without waiting for perfect human labels.

Concrete project framing

  • Search relevance: weak labels from clicks, saves, dwell time, query reformulation, similarity clicks, and manual judgments.
  • RAG quality: weak labels from citation overlap, answer-evidence entailment, retrieval recall, and LLM-as-judge scores.
  • Safety: weak labels from policy rules, human audit queues, blocklists, and red-team prompts.
  • Multimodal: weak labels from captions, OCR text, metadata, user edits, and content similarity.

B. Staff-level design answer
------------------------------------------------------------
I would build a data-quality loop rather than only a model-training loop. First define a small human-labeled gold set for calibration. Then write weak labeling functions from product signals and domain rules. I would measure coverage/conflict, train a label model, train the end model, and run slice-based evaluation. Finally, active learning sends uncertain/high-impact cases to human reviewers.

C. Metrics to mention
------------------------------------------------------------

  • LF coverage and overlap.
  • LF conflict rate.
  • estimated LF accuracy on gold data.
  • label distribution drift.
  • slice-level F1/NDCG/faithfulness.
  • human agreement with LLM-as-judge.
  • cost per newly useful label.

D. Hard interview questions
------------------------------------------------------------
Q: How do you avoid reinforcing noisy click bias?
A: Treat clicks as one weak source, not truth. Combine with dwell, reformulation, explicit labels, debiasing, randomization/interleaving data, and slice evaluation.

Q: How do you use an LLM judge safely?
A: Use it as a weak signal. Version the prompt/model, calibrate against humans, evaluate disagreement slices, and avoid making high-risk rollout decisions from judge scores alone.

Q: What would you do if weak labels conflict heavily?
A: Inspect LF overlap/conflicts, remove redundant correlated LFs, add higher-precision rules, collect gold labels in conflict regions, and train a label model rather than majority vote.
