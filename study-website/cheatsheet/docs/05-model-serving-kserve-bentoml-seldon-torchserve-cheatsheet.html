0. Mental model / 中文总览

Model serving converts a trained model into a production API.

Layers

1. Model artifact: weights + tokenizer + config + preprocessing code

2. Runtime: PyTorch/TensorFlow/ONNX/vLLM/Triton/custom Python

3. Server: FastAPI/BentoML/TorchServe/Triton/vLLM

4. Platform: KServe/Seldon/Ray Serve/Kubernetes/SageMaker/Vertex AI

5. Ops: autoscaling, canary, monitoring, rollback, security

Chinese

  • serving 不只是开一个 Flask API.
  • 要考虑 batching、latency、versioning、canary、GPU scheduling、observability、rollback.

Interview one-liner

"I would package the model with a reproducible runtime, expose a typed inference API, deploy through a serving platform with autoscaling and canary rollout, and monitor latency, error rate, model version, feature skew, and business metrics."

1. Serving architecture patterns

Simple pattern

client -> API service -> model server -> response

Production pattern

client
-> API gateway / auth
-> request validation
-> feature retrieval / preprocessing
-> model server
-> postprocessing / policy filter
-> logging + metrics
-> response

LLM serving pattern

client -> router -> vLLM/TGI/Triton-LLM -> tokenizer/model/KV cache -> streaming response

RAG serving pattern

client -> retriever -> reranker -> context builder -> LLM server -> citation verifier

2. KServe

Use when

  • Kubernetes-native model serving.
  • Need autoscaling/canary/standard inference service abstraction.
  • Need multi-framework serving on K8s.

KServe InferenceService YAML sketch

apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: sklearn-iris
spec:
  predictor:
    model:
      modelFormat:
        name: sklearn
      storageUri: s3://my-bucket/models/sklearn-iris
      resources:
        requests:
          cpu: "1"
          memory: "2Gi"
        limits:
          cpu: "2"
          memory: "4Gi"

Custom container

apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: custom-sentiment
spec:
  predictor:
    containers:
    - image: myrepo/sentiment-server:2026-05-14
      ports:
      - containerPort: 8080
      resources:
        limits:
          nvidia.com/gpu: 1

Canary sketch

spec:
  predictor:
    canaryTrafficPercent: 10
    model:
      modelFormat:
        name: sklearn
      storageUri: s3://bucket/model-v2

KServe components

  • predictor: model inference
  • transformer: preprocessing/postprocessing
  • explainer: explanations

Interview point

"KServe is a platform abstraction; the actual model runtime may still be Triton, vLLM, sklearn server, or custom container."

3. BentoML

Use when

  • Python-friendly model service packaging.
  • Need to serve arbitrary Python/model code.
  • Need local dev -> container -> production path.

BentoML service example

# service.py
import bentoml
from bentoml.io import JSON

# Example: load a model saved in BentoML model store
# model_ref = bentoml.sklearn.get("sentiment_model:latest")
# runner = model_ref.to_runner()

svc = bentoml.Service("sentiment_service")

@svc.api(input=JSON(), output=JSON())
def predict(payload):
    text = payload["text"]
    # pred = runner.predict.run([text])[0]
    pred = "positive" if "love" in text.lower() else "neutral"
    return {"sentiment": pred}

Run locally

bentoml serve service:svc --reload

Containerize

bentoml build
bentoml containerize sentiment_service:latest

Modern BentoML-style example may use class-based services depending on version. Always check project version.

Production concerns

  • batching
  • concurrency
  • model loading time
  • dependency pinning
  • GPU support
  • tracing/metrics

4. Seldon

Use when

  • Kubernetes model serving with inference graphs.
  • Need predictors, transformers, routers, combiners.
  • Need explainability, metrics, request logging integrations.

Conceptual graph

request -> transformer -> model A
-> model B
-> combiner -> response

Seldon-style inference graph idea

apiVersion: machinelearning.seldon.io/v1
kind: SeldonDeployment
metadata:
  name: sentiment-ensemble
spec:
  predictors:
  - name: default
    graph:
      name: router
      type: ROUTER
      children:
      - name: model-a
        type: MODEL
      - name: model-b
        type: MODEL

Interview point

"Seldon is useful when inference is not a single model call but a graph of preprocessing, routing, models, and combiners."

5. TorchServe

TorchServe historically serves PyTorch models, but official PyTorch docs now warn that it is in limited maintenance. For new systems, consider alternatives like KServe + custom runtime, BentoML, Triton, Ray Serve, or vLLM for LLMs.

TorchServe concepts

  • .mar model archive
  • handler.py for preprocess/inference/postprocess
  • management API
  • inference API

Handler sketch

class ModelHandler:
    def initialize(self, context):
        self.model = load_model(context.system_properties["model_dir"])
        self.model.eval()

    def preprocess(self, data):
        return tokenize(data)

    def inference(self, inputs):
        with torch.no_grad():
            return self.model(inputs)

    def postprocess(self, outputs):
        return outputs.argmax(dim=-1).tolist()

6. Model API design

Good request schema

{
  "request_id": "abc",
  "model_version": "optional-pin",
  "instances": [
    {"text": "I love this product"}
  ],
  "options": {
    "return_probabilities": true
  }
}

Good response schema

{
  "request_id": "abc",
  "model_version": "sentiment-v12",
  "predictions": [
    {"label": "positive", "score": 0.93}
  ],
  "latency_ms": 12
}

Design rules

  • Support batch requests.
  • Include request_id and model_version.
  • Return structured errors.
  • Validate inputs.
  • Keep API backward-compatible.

7. Batching

Why batch

  • Better GPU utilization.
  • Higher throughput.

Tradeoff

  • Bigger batch increases throughput but may increase latency.

Dynamic batching pattern

# Pseudocode
queue = []
while True:
    batch = collect_until(max_batch_size=32, max_wait_ms=10)
    outputs = model(batch)
    return_outputs_to_callers(outputs)

Metrics

  • average batch size
  • queue wait time
  • GPU utilization
  • p95 latency
  • throughput

8. Canary and rollback

Rollout plan

1. Deploy v2 with 0% traffic.

2. Run shadow traffic or internal test.

3. Send 1% traffic.

4. Check guardrails.

5. Increase to 10%, 50%, 100%.

6. Roll back if metrics fail.

Guardrails

  • error rate
  • latency p95/p99
  • cost/request
  • prediction distribution drift
  • business metric
  • safety metric

Prediction distribution check

# Compare class distribution old vs new
old_dist = old_preds.value_counts(normalize=True)
new_dist = new_preds.value_counts(normalize=True)
print(new_dist - old_dist)

11. Interview quick answers

Q: How do you deploy a model safely?
A: Register/version artifact, package runtime, deploy behind stable API, run shadow/canary traffic, monitor guardrails, and keep rollback path.

Q: How do you reduce inference latency?
A: Profile stages, optimize preprocessing, batch requests, use faster runtime/quantization, cache results, reduce model size, use GPU efficiently, and avoid network hops.

Q: How to handle model cold start?
A: Preload model, use readiness probes, keep warm replicas, optimize container image/model loading, and avoid scale-to-zero for latency-sensitive services.

Q: KServe vs BentoML?
A: KServe is K8s-native serving platform abstraction; BentoML is Python developer-friendly packaging and serving framework. They can be complementary.

RESUME-ALIGNED INTERVIEW DEEP DIVE

A. Serving-platform comparison
------------------------------------------------------------
Triton: high-throughput tensor inference for embeddings, rankers, vision/audio/classifiers.
vLLM: high-throughput autoregressive LLM serving with KV-cache optimized batching.
Ray Serve: Python-native distributed serving and DAG/workflow composition.
KServe: Kubernetes-native model serving abstraction using InferenceService CRDs.
BentoML: packaging and developer experience for model services.
Seldon: Kubernetes model deployment, inference graphs, governance/enterprise serving.
TorchServe: PyTorch model serving, useful but less central for modern LLM serving.

B. KServe interview model
------------------------------------------------------------
KServe is useful when the organization wants Kubernetes-native model deployment as a platform primitive: InferenceService, autoscaling, canary rollout, predictor/transformer/explainer pattern, model versioning, and standard traffic management.

C. When to choose what
------------------------------------------------------------

  • Need maximum GPU throughput for embedding/ranking: Triton.
  • Need LLM generation throughput: vLLM.
  • Need Python DAG orchestration around models/tools: Ray Serve.
  • Need Kubernetes-native platform abstraction for many teams: KServe/Seldon.
  • Need simple packaging and deployment DX: BentoML.

D. Interview Q&A
------------------------------------------------------------
Q: What is the predictor/transformer/explainer pattern?
A: Predictor runs the model, transformer handles pre/post-processing, and explainer provides model explanations. It separates concerns and lets platform teams standardize deployment.

Q: Why not just use FastAPI?
A: FastAPI is an HTTP framework. Model serving platforms add versioning, autoscaling, batching, canary rollout, model repository integration, GPU placement, and inference-specific observability.

Q: What is hard about scale-to-zero?
A: Cold start. Large models may take minutes to pull/load/warm, so user-facing models often need warm capacity.
