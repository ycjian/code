0. Mental model / 中文总览

A production AI/search system usually has two data paths

1) Batch / offline path

Source DB / logs / S3/GCS -> Spark -> Iceberg tables -> training data / backfill / analytics

2) Streaming / near-real-time path

App events / CDC -> Kafka -> Flink -> online store / search index / vector index / features

ASCII workflow

OLTP DB / app logs / CDC
|
v
Kafka topics -----------------------------+
| |
v v
Flink streaming jobs Raw events in S3/GCS
| |
v v
online features / index updates Iceberg tables
| |
v v
low-latency serving Spark batch jobs
|
v
training sets / backfills

中文

  • Kafka = durable event log / buffer / replay source.
  • Flink = stateful stream processing; handles event time, late data, windows, exactly-once state.
  • Spark = batch ETL and large joins/backfills.
  • Iceberg = table format on object storage with snapshots, schema evolution, partition evolution, time travel.

Interview one-liner

"I would use Kafka for durable ingestion and replay, Flink for low-latency stateful processing, Spark for batch backfill and heavy joins, and Iceberg/S3 as the source-of-truth table layer. The key concerns are idempotency, replayability, schema evolution, data freshness, and monitoring lag."

Section 03

1. Kafka essentials

Core concepts

  • Topic: logical event stream, e.g. order_events, user_clicks, document_updates.
  • Partition: ordered append-only log shard. Parallelism and ordering unit.
  • Offset: position of a message in a partition.
  • Producer: writes events to topic partitions.
  • Consumer group: multiple consumers share partitions. Each partition is consumed by one consumer in a group.
  • Retention: Kafka can keep data for a time/size window, enabling replay.

Key rules

  • Ordering is guaranteed only within a partition, not across the whole topic.
  • Pick a key that matches ordering needs.

Example: key=user_id if events per user must be ordered.
Example: key=doc_id if document update ordering matters.

  • Partitions decide max consumer parallelism in one consumer group.

Producer example

from kafka import KafkaProducer
import json

producer = KafkaProducer(
    bootstrap_servers=["kafka-1:9092", "kafka-2:9092"],
    key_serializer=lambda k: k.encode("utf-8"),
    value_serializer=lambda v: json.dumps(v).encode("utf-8"),
    acks="all",                 # safer durability
    retries=5,
)

event = {
    "event_id": "evt-123",
    "doc_id": "doc-9",
    "event_type": "DOCUMENT_UPDATED",
    "event_ts": "2026-05-14T12:00:00Z",
    "payload": {"title": "new title"},
}

producer.send("document_updates", key=event["doc_id"], value=event)
producer.flush()

Consumer example

from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(
    "document_updates",
    bootstrap_servers=["kafka-1:9092"],
    group_id="indexer-v1",
    enable_auto_commit=False,
    value_deserializer=lambda b: json.loads(b.decode("utf-8")),
)

for msg in consumer:
    event = msg.value
    try:
        # 1. process event idempotently
        update_search_index(event)
        # 2. commit offset only after successful processing
        consumer.commit()
    except Exception:
        # do not commit; retry/replay later
        raise

Idempotency pattern

def update_search_index(event):
    # Use event_id or monotonically increasing source_version.
    # If the same event is replayed, result should be the same.
    existing = index.get(event["doc_id"])
    if existing and existing["source_version"] >= event["source_version"]:
        return
    index.upsert(event["doc_id"], event["payload"], source_version=event["source_version"])

Kafka interview pitfalls

  • Do not say "exactly once" casually. In real systems, you often design for at-least-once + idempotent sinks.
  • Rebalancing can cause duplicate processing unless commits are handled carefully.
  • Poison messages need DLQ / quarantine, not infinite retry.
  • Schema evolution should use Schema Registry or strict versioned JSON/Protobuf/Avro.

Common metrics

  • consumer lag by partition
  • produce rate / consume rate
  • broker disk usage
  • under-replicated partitions
  • rebalance count
  • DLQ rate
  • message age
Section 04

2. Flink essentials

Use Flink when

  • You need stateful streaming.
  • You need event-time windows.
  • You need low-latency updates to online features/search indexes.
  • You need checkpoints and replay after failure.

Key concepts

  • Event time: timestamp when event happened.
  • Processing time: timestamp when system processes the event.
  • Watermark: progress signal for event time; tells Flink how late events can arrive.
  • Window: group events by time, e.g. 5-minute tumbling window.
  • State: per-key memory persisted via checkpoints.
  • Checkpoint: consistent snapshot of operator state + source position.

PyFlink sketch

from pyflink.datastream import StreamExecutionEnvironment
from pyflink.common import WatermarkStrategy, Time
from pyflink.datastream.window import TumblingEventTimeWindows

# Pseudocode-level. Actual connector setup depends on your environment.
env = StreamExecutionEnvironment.get_execution_environment()
env.enable_checkpointing(60_000)  # every 60s

stream = env.from_source(kafka_source, WatermarkStrategy.for_bounded_out_of_orderness(Time.seconds(30)), "kafka")

# event = {"user_id": "u1", "event_ts": ..., "click": 1}
result = (
    stream
    .key_by(lambda e: e["user_id"])
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .reduce(lambda a, b: {"user_id": a["user_id"], "clicks": a["clicks"] + b["clicks"]})
)

result.sink_to(online_feature_sink)
env.execute("realtime-user-click-features")

Flink state interview example

"For per-user rolling counters, I would key by user_id and store per-key state. Checkpointing allows recovery without losing state. For late events, I would use event time and watermarks; late arrivals beyond allowed lateness go to a side output for correction/backfill."

Late data strategy

  • Within watermark delay: include in window.
  • Slightly late: side output + correction event.
  • Very late: write to Iceberg, handle with Spark backfill.

Exactly-once-ish in practice

  • Source supports replay.
  • Flink checkpoints state + offsets.
  • Sink must be transactional/idempotent.
  • If sink is not transactional, use idempotent upsert keys and source_version.

3. Spark essentials

Use Spark when

  • You need batch ETL.
  • You need huge joins.
  • You need backfills.
  • You need training set generation.
  • You need data quality checks across large tables.

PySpark DataFrame basics

from pyspark.sql import SparkSession
from pyspark.sql.functions import col, count, when, to_date

spark = SparkSession.builder.appName("training-data").getOrCreate()

events = spark.read.parquet("s3://bucket/events/date=2026-05-14")
users = spark.read.parquet("s3://bucket/users")

features = (
    events
    .filter(col("event_type") == "click")
    .groupBy("user_id")
    .agg(count("*").alias("click_count_7d"))
)

training = (
    users
    .join(features, on="user_id", how="left")
    .fillna({"click_count_7d": 0})
)

training.write.mode("overwrite").parquet("s3://bucket/training/date=2026-05-14")

Common Spark performance points

1. Avoid unnecessary shuffles.

2. Repartition by join/group key when needed.

3. Broadcast small dimension tables.

4. Handle skew.

5. Cache only if reused.

6. Watch file size; too many small files hurt performance.

Broadcast join

from pyspark.sql.functions import broadcast

large_events.join(broadcast(small_country_table), on="country_code", how="left")

Skew handling

from pyspark.sql.functions import rand, concat_ws

# Add salt to hot key to spread load.
skewed = events.withColumn("salt", (rand() * 10).cast("int"))
# Join with expanded dimension table if needed.

Interview answer for Spark job slow

  • Check Spark UI: stage time, shuffle read/write, spill, skewed tasks.
  • Check join strategy: broadcast vs sort-merge join.
  • Check partition count and small files.
  • Check data skew: one key has huge volume.
  • Check UDFs: Python UDFs can be slow; prefer built-in SQL functions.

4. Iceberg essentials

Why Iceberg

  • Tables on S3/GCS/ADLS with ACID-like table semantics.
  • Snapshot isolation.
  • Time travel.
  • Schema evolution.
  • Partition evolution.
  • Hidden partitioning.
  • Works with Spark/Flink/Trino/etc.

Iceberg SQL examples

CREATE TABLE prod.db.document_features (
  doc_id string,
  embedding_version string,
  quality_score double,
  updated_at timestamp
)
USING iceberg
PARTITIONED BY (days(updated_at));

Append/upsert pattern

MERGE INTO prod.db.document_features t
USING staging_updates s
ON t.doc_id = s.doc_id
WHEN MATCHED AND s.source_version > t.source_version THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;

Time travel

SELECT * FROM prod.db.document_features VERSION AS OF 123456789;
SELECT * FROM prod.db.document_features TIMESTAMP AS OF '2026-05-14 10:00:00';

Schema evolution

ALTER TABLE prod.db.document_features ADD COLUMN language string;
ALTER TABLE prod.db.document_features RENAME COLUMN quality_score TO relevance_quality_score;

Partition evolution

ALTER TABLE prod.db.events ADD PARTITION FIELD bucket(16, user_id);

Operational issues

  • Small file compaction.
  • Snapshot expiration.
  • Manifest compaction.
  • Concurrent writers.
  • Catalog choice: Glue, Hive, REST catalog, Nessie.

5. Freshness + indexing design pattern

Problem: billions of docs, updates daily, search/vector index must be fresh.

Design

1. Source of truth writes document change event to Kafka.

2. Flink consumes events and writes low-latency index update.

3. Raw events also land in Iceberg.

4. Spark nightly job recomputes canonical index snapshot for correctness/backfill.

5. Index update uses idempotent doc_id + source_version.

6. Monitoring tracks event age, consumer lag, index freshness, failure/DLQ count.

ASCII

doc service -> Kafka document_updates -> Flink indexer -> Elasticsearch/Milvus
| |
v v
Iceberg raw_events DLQ bad_events
|
v
Spark backfill job -> rebuild index partitions

7. Interview quick answers

Q: Kafka vs Flink?
A: Kafka stores and transports events; Flink computes over event streams with state, windows, watermarks, and checkpoints.

Q: Spark vs Flink?
A: Spark is usually best for batch ETL/backfills/heavy joins. Flink is stronger for continuous stateful streaming with event-time semantics.

Q: Why Iceberg instead of raw Parquet folders?
A: Raw Parquet folders lack table-level snapshot management, schema/partition evolution, reliable time travel, and strong table metadata semantics.

Q: How to handle duplicate events?
A: Use event_id/source_version and make sinks idempotent. Commit offsets after successful processing. For non-transactional sinks, at-least-once + idempotency is practical.

Q: How to recover from a bad deployment?
A: Stop consumers, fix code, replay Kafka from a known offset or restore Flink checkpoint, quarantine bad messages, and use Iceberg/Spark backfill to correct derived data.

Q: How to avoid training-serving skew?
A: Use the same feature definitions for batch and online paths, version transformations, use point-in-time correct joins, and monitor online/offline feature distributions.

RESUME-ALIGNED INTERVIEW DEEP DIVE

A. End-to-end freshness architecture
------------------------------------------------------------
Deep-dive question: How do you keep a search index, vector index, and feature store fresh when billions of documents or assets change daily?

Strong answer flow

1. Use Kafka as the durable ingestion and replay log for document updates, user events, and CDC.

2. Use Flink for near-real-time transforms: dedup, enrichment, event-time windows, online features, and index-update events.

3. Use Iceberg/S3 or GCS as the source of truth for raw and curated data with snapshots, schema evolution, and time travel.

4. Use Spark for backfills, large joins, embedding regeneration, offline training sets, and consistency repair.

5. Avoid dual-write correctness bugs by writing events once to the log/source-of-truth and deriving downstream materialized views.

6. Make sinks idempotent using document version, event ID, or source timestamp.

7. Monitor Kafka lag, Flink checkpoint health, sink error rate, index freshness, and feature freshness.

B. Spark/Flink/Kafka/Iceberg tradeoff table
------------------------------------------------------------
Kafka: durable ordered partition log, replay, ingestion buffer. Not a database for arbitrary queries.
Flink: low-latency stateful stream processing with event time, watermarks, checkpoints. More operationally complex than batch.
Spark: large batch ETL, joins, backfills, offline features. Higher latency than streaming.
Iceberg: table format with snapshots, schema/partition evolution, and time travel. Needs compaction and metadata maintenance.

C. Production failure modes
------------------------------------------------------------

  • Kafka consumer lag grows after downstream index slowdown.
  • Flink checkpoint duration spikes because state grows or backpressure appears.
  • Spark backfill creates duplicate index writes without idempotency.
  • Iceberg small files slow planning/querying.
  • Schema changes break consumers without compatibility policy.
  • Event-time disorder causes incorrect feature windows if watermarks are too aggressive.

D. Interview Q&A
------------------------------------------------------------
Q: Why use Iceberg instead of just writing Parquet folders?
A: Iceberg gives table metadata, snapshots, schema evolution, partition evolution, hidden partitioning, and time travel. Plain folders make correctness and evolution much harder at scale.

Q: Kafka exactly once?
A: Kafka can support transactional exactly-once semantics inside Kafka pipelines, especially Kafka Streams. Once external sinks are involved, I still design idempotent writes and replay-safe consumers.

Q: Spark skew handling?
A: Detect skewed keys and long tasks, use broadcast join for small dimensions, salting for hot keys, repartitioning by join keys, AQE/skew join where available, and pre-aggregate before joins.

E. Orchestration and transformation layer
------------------------------------------------------------

Airflow

  • Strong for scheduled DAG orchestration, dependency management, retries, backfills, and operational visibility.
  • Good for batch pipelines such as Spark backfills, Iceberg compaction, training dataset generation, and daily feature materialization.
  • Weakness: not a stream processor; task granularity and scheduler load matter.

Dagster

  • Strong asset-centric orchestration, lineage, typed assets, partition-aware backfills, and local testability.
  • Good when the team wants data assets as first-class objects instead of only task DAGs.

dbt

  • SQL transformation, model lineage, testing, documentation, and warehouse/lakehouse transformation discipline.
  • Good for curated tables, feature aggregates, data contracts, and analytics-facing transformations.

Interview phrase

"I would use Kafka/Flink/Spark/Iceberg for data movement and computation, then Airflow or Dagster for orchestration, and dbt-style transformations/tests where SQL-modeled curated datasets need lineage and quality checks."
