Real-time endpoint use

# boto3 sketch
import boto3, json

runtime = boto3.client("sagemaker-runtime")
response = runtime.invoke_endpoint(
    EndpointName="sentiment-endpoint",
    ContentType="application/json",

  • data quality gates
  • access control by dataset/table/prefix

8. IAM and security basics

Principle

  • least privilege
  • separate training/inference roles
  • no long-lived credentials in containers
  • use workload identity / IRSA / service accounts
  • encrypt data at rest and in transit
  • log access to sensitive data

Section 11

9. Cost controls

AI cost drivers

  • GPU hours
  • idle endpoints
  • LLM token usage
  • data scans
  • vector DB memory
  • logging volume
  • cross-zone/network egress

Controls

  • autoscale to zero only for non-latency-sensitive services
  • spot/preemptible for training/backfill
  • right-size GPU
  • quantization
  • cache embeddings and LLM responses
  • partition data to reduce scans
  • retention policies
Section 12

10. Common cloud interview architecture

Prompt

"Design an ML platform that trains nightly and serves online predictions."

Answer

1. Raw events land in S3/GCS.
2. Spark/Dataproc/EMR builds Iceberg/warehouse feature tables.
3. Training job runs on SageMaker/Vertex AI or K8s GPU job.
4. Model and metrics logged to MLflow/Model Registry.
5. CI/CD deploys model to EKS/GKE/KServe or managed endpoint.
6. Online service reads features from Redis/DynamoDB/Bigtable.
7. Metrics/logs/traces go to CloudWatch/Cloud Monitoring + Prometheus/Grafana.
8. Canary rollout and rollback protect production.

RESUME-ALIGNED INTERVIEW DEEP DIVE

A. Interview-level cloud mapping
------------------------------------------------------------
You do not need every service detail. Map architecture concepts to managed services.

Object store: S3 / GCS.
Kubernetes: EKS / GKE.
Streaming: MSK or Kinesis / Pub/Sub.
Batch Spark: EMR / Dataproc.
Warehouse: Redshift or Athena / BigQuery.
ML platform: SageMaker / Vertex AI.
Cache: ElastiCache / Memorystore.
NoSQL: DynamoDB / Bigtable or Firestore.
Monitoring: CloudWatch / Cloud Monitoring.
IAM: AWS IAM / Google IAM.

B. Hybrid AI platform answer
------------------------------------------------------------
For a multi-cloud or hybrid AI platform, keep the abstractions stable: object storage contract, model registry, deployment spec, metrics/tracing contract, identity/tenant contract, and data-governance rules. Avoid leaking cloud-specific APIs into product teams.

C. Cost and reliability topics
------------------------------------------------------------

  • GPU node reservations vs on-demand.
  • spot/preemptible for retryable batch jobs.
  • autoscaling and warm pools.
  • cross-region replication for artifacts.
  • egress cost for large media.
  • IAM least privilege and workload identity.

D. Interview Q&A
------------------------------------------------------------
Q: When use serverless vs Kubernetes?
A: Serverless is good for bursty stateless functions and simple APIs. Kubernetes is better for long-running services, GPU workloads, custom networking, and consistent platform primitives.

Q: How avoid cloud lock-in?
A: Abstract deployment specs, use open formats like Iceberg/Parquet, containerize services, standardize observability, and isolate cloud-specific adapters.

Q: What is most expensive in AI cloud systems?
A: GPUs, long-running idle capacity, LLM tokens, data egress, high-cardinality logging/tracing, and repeated media/model artifact movement.
