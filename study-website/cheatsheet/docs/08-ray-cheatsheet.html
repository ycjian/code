2. Remote tasks

import ray
ray.init()

@ray.remote
def square(x):
    return x * x

refs = [square.remote(i) for i in range(10)]
results = ray.get(refs)
print(results)

Key grammar

  • @ray.remote turns function into distributed task.
  • .remote(...) submits async task.
  • ray.get(ref) blocks and fetches result.

Section 06

4. Actors

Actors are stateful workers.

@ray.remote
class Counter:
    def __init__(self):
        self.value = 0

    def inc(self):
        self.value += 1
        return self.value

counter = Counter.remote()
print(ray.get(counter.inc.remote()))
print(ray.get(counter.inc.remote()))

Use actors for

  • stateful services
  • model loaded once per worker
  • counters/aggregators
  • connection pools

Section 09

7. Parallel map pattern

@ray.remote
def process_one(item):
    return item * 2

refs = [process_one.remote(x) for x in items]
results = ray.get(refs)

Avoid submitting millions of tasks at once

def bounded_parallel(items, max_in_flight=100):
    pending = []
    results = []

    for item in items:
        pending.append(process_one.remote(item))
        if len(pending) >= max_in_flight:
            done, pending = ray.wait(pending, num_returns=1)
            results.extend(ray.get(done))

    results.extend(ray.get(pending))
    return results

12. Ray Serve

Ray Serve = scalable model serving.

from ray import serve
from starlette.requests import Request

@serve.deployment(num_replicas=2)
class SentimentModel:
    def __init__(self):
        self.model = load_model()

    async def __call__(self, request: Request):
        data = await request.json()
        text = data["text"]
        return {"sentiment": self.model.predict(text)}

serve.run(SentimentModel.bind())

Call

curl -X POST http://localhost:8000/ -d '{"text":"I love it"}'

Section 17

15. Performance tips

  • Avoid tiny tasks; batch work.
  • Use actors for expensive model initialization.
  • Use ray.put for large shared objects.
  • Use ray.wait for backpressure.
  • Specify resource requirements.
  • Avoid returning huge objects repeatedly.
  • Monitor object store memory.

Section 19

17. Interview answer pattern

Question: When would you use Ray?

Answer

I would use Ray when I need to scale Python workloads across cores or machines without rewriting everything into Spark or a lower-level distributed system. For stateless parallel work I would use Ray tasks, for stateful model workers I would use actors, for batch inference I would use Ray Data, and for online serving I would use Ray Serve. I would monitor object store memory, task granularity, GPU scheduling, and backpressure.

Section 20

ADVANCED INTERVIEW DEEP DIVE - EXPANDED COVERAGE

A. Ray system mental model
------------------------------------------------------------
Ray is a distributed runtime for Python tasks and actors. In AI infra interviews, explain when Ray is better than multiprocessing, Spark, Kubernetes Jobs, or a pure model server.

B. Architecture basics
------------------------------------------------------------

  • driver: submits tasks/actors.
  • worker: executes tasks.
  • raylet: node-level scheduler/resource manager.
  • global control store: cluster metadata.
  • object store: shared immutable objects.
  • autoscaler: adds/removes nodes based on resource demand.

Ray schedules logical resources like CPU/GPU/custom resources; your code must expose useful parallelism.

C. Task vs actor
------------------------------------------------------------
Use task for stateless fan-out work. Use actor when model/connection/cache/state should be reused.

@ray.remote(num_gpus=1)
class Embedder:
    def __init__(self):
        self.model = load_model().cuda()
    def encode(self, batch):
        return self.model(batch)

D. Object store and memory
------------------------------------------------------------
Object refs point to immutable distributed objects. Memory issues come from too many pending refs, huge returns, unconsumed results, object spilling, and closures accidentally capturing large objects. Use ray.wait for backpressure.

E. Ray Serve
------------------------------------------------------------
Serve concepts: application, deployment, replica, ingress deployment. A common AI pattern is HTTP ingress -> CPU router/DAG -> GPU deployment for embeddings/ranking/generation -> async handles for composition.

F. Ray vs Spark vs Kubernetes
------------------------------------------------------------
Ray: Python-native dynamic task graphs, actors, ML serving/training. Spark: SQL/batch ETL and shuffle-heavy analytics. Kubernetes: container orchestration, not a Python programming model. Use Kubernetes underneath Ray for lifecycle and isolation.

G. Failure modes
------------------------------------------------------------
Driver dies, actor bottleneck, object-store pressure, too many tiny tasks, GPU actors underutilized by small batches, autoscaler reacts slowly, serialization dominates compute.

H. Interview Q&A
------------------------------------------------------------
Q: How prevent millions of tasks overwhelming Ray?
A: Bound in-flight refs with ray.wait, batch small work, use actors to amortize setup, and monitor queue/object-store metrics.

Q: Why Ray Serve vs FastAPI?
A: FastAPI is HTTP. Ray Serve adds distributed deployments, replicas, resource-aware scheduling, autoscaling, handles, and workflow composition.

Q: How debug Ray performance?
A: Check dashboard timeline, task duration, queueing, object store memory, serialization, GPU utilization, and task granularity.

Q: How schedule GPU work?
A: Declare num_gpus; Ray schedules on nodes with logical GPU resources and sets CUDA visibility. Model code still must use GPU correctly.
