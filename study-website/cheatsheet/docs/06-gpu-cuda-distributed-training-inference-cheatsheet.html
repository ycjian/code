Deep learning performance depends on

  • compute: matrix multiplication throughput
  • memory: GPU HBM capacity and bandwidth
  • communication: GPU-GPU / node-node synchronization
  • input pipeline: data loading/preprocessing

1. GPU basics

Terms

  • GPU memory / HBM: stores model weights, activations, gradients, optimizer states, KV cache.
  • CUDA kernel: function executed on GPU.
  • Tensor cores: accelerate matrix operations in fp16/bf16/tf32.
  • PCIe/NVLink: GPU interconnect.
  • NCCL: communication library for all-reduce/all-gather/reduce-scatter.

Memory during training

weights
+ gradients
+ optimizer states, often 2x weights for Adam moments
+ activations
+ temporary buffers

Memory during inference

weights
+ KV cache
+ activations for current step
+ batching buffers

2. Precision

fp32

  • stable, high memory

fp16

  • lower memory, faster
  • may need loss scaling

bf16

  • similar memory to fp16
  • better numeric range, common on newer GPUs/TPUs

int8/int4 quantization

  • reduces memory and can speed inference
  • may reduce accuracy
  • useful for LLM serving

PyTorch mixed precision

import torch

scaler = torch.cuda.amp.GradScaler()

for x, y in loader:
    optimizer.zero_grad()
    with torch.cuda.amp.autocast(dtype=torch.float16):
        pred = model(x.cuda())
        loss = loss_fn(pred, y.cuda())

    scaler.scale(loss).backward()
    scaler.step(optimizer)
    scaler.update()

With bf16

with torch.autocast(device_type="cuda", dtype=torch.bfloat16):
    out = model(x)

3. Data Parallel / DDP

DDP = DistributedDataParallel.
Each GPU has full model replica.
Each GPU processes different mini-batch shard.
Gradients are all-reduced across GPUs.

Command

torchrun --nproc_per_node=8 train.py

Minimal DDP sketch

import os
import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data.distributed import DistributedSampler


def setup():
    dist.init_process_group("nccl")
    torch.cuda.set_device(int(os.environ["LOCAL_RANK"]))

setup()
local_rank = int(os.environ["LOCAL_RANK"])

model = MyModel().cuda(local_rank)
model = DDP(model, device_ids=[local_rank])

sampler = DistributedSampler(dataset)
loader = DataLoader(dataset, sampler=sampler, batch_size=32)

for epoch in range(num_epochs):
    sampler.set_epoch(epoch)
    for x, y in loader:
        x, y = x.cuda(local_rank), y.cuda(local_rank)
        loss = loss_fn(model(x), y)
        optimizer.zero_grad()
        loss.backward()
        optimizer.step()

DDP limitation

  • Full model + optimizer states replicated on every GPU.
  • Not enough for very large models.
Section 06

4. FSDP

FSDP = Fully Sharded Data Parallel.
It shards parameters, gradients, and optimizer states across workers.

Why

  • Reduces GPU memory footprint.
  • Enables training models too large for one GPU.

FSDP sketch

import torch
import torch.distributed as dist
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP


dist.init_process_group("nccl")
local_rank = int(os.environ["LOCAL_RANK"])
torch.cuda.set_device(local_rank)

model = MyLargeModel().cuda()
model = FSDP(model)

for x, y in loader:
    x, y = x.cuda(), y.cuda()
    loss = loss_fn(model(x), y)
    optimizer.zero_grad()
    loss.backward()
    optimizer.step()

DDP vs FSDP

DDP: replicate model everywhere, sync gradients
FSDP: shard model states, all-gather only when needed

Interview phrase

"DDP is simpler and fast when the model fits on each GPU. FSDP trades communication complexity for memory savings by sharding model parameters, gradients, and optimizer states."

5. DeepSpeed ZeRO

ZeRO partitions model states across data-parallel workers.

Stages

  • Stage 1: shard optimizer states
  • Stage 2: shard optimizer states + gradients
  • Stage 3: shard optimizer states + gradients + parameters

DeepSpeed config sketch

{
  "train_batch_size": 1024,
  "bf16": {"enabled": true},
  "zero_optimization": {
    "stage": 3,
    "offload_optimizer": {
      "device": "cpu"
    },
    "offload_param": {
      "device": "cpu"
    }
  }
}

Launch

deepspeed train.py --deepspeed_config ds_config.json

Tradeoff

  • More memory savings.
  • More communication and possible CPU/NVMe bottlenecks if offloading.
Section 08

6. Tensor / pipeline / data parallelism

Data parallelism

  • Split data across replicas.
  • Good when model fits on GPU.

Tensor parallelism

  • Split individual matrix operations/model layers across GPUs.
  • Needed for huge models in inference/training.

Pipeline parallelism

  • Split model layers across GPUs.
  • Micro-batches flow through pipeline.
  • Bubble overhead if not tuned.

Hybrid

  • LLM training often combines data + tensor + pipeline + ZeRO/FSDP.

7. Activation checkpointing

Problem

  • Activations take lots of memory during training.

Solution

  • Save fewer activations, recompute during backward.
  • Reduces memory, increases compute.

PyTorch

from torch.utils.checkpoint import checkpoint

class Block(torch.nn.Module):
    def forward(self, x):
        return expensive_layers(x)

out = checkpoint(block, x)

8. LLM inference: prefill vs decode

Prefill

  • Process input prompt tokens in parallel.
  • Compute KV cache for prompt.
  • More compute-heavy.

Decode

  • Generate one token at a time.
  • Uses KV cache.
  • Often memory-bandwidth/KV-cache constrained.

KV cache

  • Stores key/value tensors for attention for previous tokens.
  • Grows with sequence length and batch size.
  • Major memory consumer in LLM serving.

Serving implications

  • Long prompts consume KV cache.
  • More concurrent requests increase KV cache memory.
  • Continuous batching improves GPU utilization during decode.
Section 11

9. vLLM-related concepts

PagedAttention

  • Manages KV cache in blocks/pages to reduce memory fragmentation.
  • Enables higher batching/concurrency.

Continuous batching

  • New requests can join while other requests are decoding.
  • Avoids waiting for entire batch to finish.

Prefix caching

  • Reuse KV cache for shared prompt prefixes.
  • Useful for system prompts and repeated contexts.

Metrics

  • time to first token (TTFT)
  • inter-token latency
  • output tokens/sec
  • request throughput
  • KV cache utilization
  • queue wait time

10. Performance debug checklist

Training slow

  • GPU utilization low? input pipeline bottleneck.
  • CPU dataloader slow? increase num_workers, prefetch, pin_memory.
  • Communication high? check all-reduce, network, batch size.
  • OOM? use bf16/fp16, gradient accumulation, activation checkpointing, FSDP/ZeRO.
  • Loss unstable? check precision/lr/grad clipping.

Inference slow

  • TTFT high? prompt too long, prefill bottleneck, queueing.
  • Decode slow? batch scheduling/KV cache/model size.
  • GPU util low? batching too small or CPU bottleneck.
  • OOM? reduce max tokens/concurrency, quantize, use tensor parallelism.

12. Interview quick answers

Q: DDP vs FSDP?
A: DDP replicates full model states on each GPU and synchronizes gradients. FSDP shards parameters, gradients, and optimizer states to reduce memory footprint for large models.

Q: What is ZeRO?
A: ZeRO reduces redundancy in data parallel training by partitioning optimizer states, gradients, and parameters across workers, with increasing savings from stage 1 to stage 3.

Q: Why is LLM serving hard?
A: Generation is autoregressive. Decode is token-by-token, KV cache grows with sequence length and concurrency, and inefficient memory management limits batch size and throughput.

Q: How to improve LLM serving throughput?
A: Use continuous batching, efficient KV cache management, quantization, prefix caching, tensor parallelism, and tune max concurrency/max tokens.

RESUME-ALIGNED INTERVIEW DEEP DIVE

A. How to answer GPU bottleneck questions
------------------------------------------------------------
Start by separating training and inference.

Training bottlenecks

  • compute throughput for matrix multiplies.
  • activation/optimizer memory.
  • data-loader and preprocessing throughput.
  • all-reduce/all-gather communication.
  • checkpointing overhead.

LLM inference bottlenecks

  • prefill compute for long prompts.
  • decode memory bandwidth.
  • KV-cache memory capacity and fragmentation.
  • scheduler efficiency and continuous batching.
  • model weight loading and cold start.

B. Parallelism decision tree
------------------------------------------------------------
Data parallel / DDP: model fits on one GPU; scale throughput by data shards.
FSDP / ZeRO: model states do not fit comfortably; shard params/grads/optimizer.
Tensor parallel: one layer's matrix ops too large for one GPU or model needs multiple GPUs.
Pipeline parallel: layer stack split across GPUs; watch pipeline bubbles.
Expert parallel: MoE models route tokens to different experts.

C. MIG and bin-packing
------------------------------------------------------------
MIG is useful for isolating smaller inference workloads on one physical GPU. It is not ideal when models need full GPU memory or dynamic sharing. Bin-packing improves utilization but can hurt fault isolation; spreading improves availability but may waste GPU capacity.

D. Interview Q&A
------------------------------------------------------------
Q: Why can adding GPUs fail to improve training speed?
A: Communication overhead, small batch size, data-loader bottleneck, poor overlap, network bandwidth, or optimizer/checkpoint bottlenecks can dominate.

Q: Why is LLM decode memory-bound?
A: Each generated token reads model weights and KV cache with relatively small compute per token. Memory bandwidth and KV-cache access become limiting.

Q: fp16 vs bf16?
A: Both reduce memory/bandwidth. bf16 has wider exponent range and is often more numerically stable on supported hardware; fp16 may need loss scaling.
