0. Mental model / 中文总览

Vector DB stores embeddings and supports nearest-neighbor search.

Query flow

text/image/query -> embedding model -> query vector
-> ANN index search
-> metadata filtering
-> reranking
-> top k results

Chinese

  • ANN = approximate nearest neighbor, 用近似换速度.
  • 核心 tradeoff: recall, latency, memory, indexing cost.
  • metadata filtering 很重要,不能只看向量相似度.

Interview one-liner

"Vector search is approximate, so I would evaluate recall@k against an exact-search baseline, tune index parameters for recall-latency-memory tradeoffs, and design embedding version migration and metadata filtering carefully."

1. Similarity metrics

Cosine similarity

  • Common for normalized text embeddings.
  • If vectors are normalized, cosine similarity equals dot product ranking.

Dot product

  • Common for retrieval models trained with dot-product objective.

Euclidean / L2

  • Distance metric for many vector indexes.

Normalize example

import numpy as np

def normalize(x):
    return x / np.linalg.norm(x, axis=1, keepdims=True)

scores = vectors @ query
topk = np.argsort(-scores)[:10]

ANN recall@k

def recall_at_k(exact_ids, ann_ids, k):
    return len(set(exact_ids[:k]) & set(ann_ids[:k])) / k

3. HNSW

HNSW = Hierarchical Navigable Small World graph.

Idea

  • Build graph where each vector connects to nearby vectors.
  • Search navigates graph from coarse to fine layers.

Important parameters

  • M: number of graph neighbors per node. Higher M = better recall, more memory.
  • efConstruction: build-time search depth. Higher = better index, slower build.
  • efSearch: query-time search depth. Higher = better recall, higher latency.

When good

  • High recall and low latency.
  • Dynamic-ish workloads.
  • Enough memory.

When painful

  • Large memory footprint.
  • Filtering can hurt if graph traversal ignores metadata selectivity.

Tuning

increase efSearch -> improve recall, increase latency
increase M -> improve recall, increase memory/build time
Section 06

4. IVF

IVF = Inverted File Index.

Idea

  • Cluster vectors into coarse centroids.
  • At query time, search only nearest clusters.

Parameters

  • nlist: number of clusters
  • nprobe: number of clusters searched per query

Tradeoff

higher nprobe -> better recall, higher latency
higher nlist -> smaller clusters, more index overhead

When good

  • Very large datasets.
  • Need memory/latency control.
  • Batch/offline index building acceptable.
Section 07

5. PQ / scalar quantization

Quantization reduces memory.

Scalar quantization

  • Convert float32 dimensions to int8-like representation.
  • Lower memory, some accuracy loss.

Product quantization (PQ)

  • Split vector into sub-vectors.
  • Quantize each sub-vector separately.
  • Strong compression, more recall loss.

Use when

  • Dataset huge.
  • Memory cost matters.
  • Approximate recall acceptable.

6. FAISS examples

Flat exact index

import faiss
import numpy as np

vectors = np.random.random((10000, 768)).astype("float32")
faiss.normalize_L2(vectors)

index = faiss.IndexFlatIP(768)  # inner product
index.add(vectors)

query = np.random.random((1, 768)).astype("float32")
faiss.normalize_L2(query)
D, I = index.search(query, k=10)

HNSW

index = faiss.IndexHNSWFlat(768, 32)  # M=32
index.hnsw.efConstruction = 200
index.add(vectors)
index.hnsw.efSearch = 64
D, I = index.search(query, 10)

IVF

quantizer = faiss.IndexFlatIP(768)
index = faiss.IndexIVFFlat(quantizer, 768, 4096, faiss.METRIC_INNER_PRODUCT)
index.train(vectors)
index.add(vectors)
index.nprobe = 16
D, I = index.search(query, 10)

IVF + PQ

quantizer = faiss.IndexFlatIP(768)
index = faiss.IndexIVFPQ(quantizer, 768, 4096, 64, 8)  # m=64, 8 bits
index.train(vectors)
index.add(vectors)
index.nprobe = 32
D, I = index.search(query, 10)

7. Metadata filtering

Problem

"Find similar docs, but only docs user has permission to see, language=en, date > X."

Filtering strategies

1. Pre-filter then vector search

  • good if filter highly selective and supported by index
  • may be expensive if many small filtered indexes

2. Vector search then post-filter

  • simple
  • can reduce recall if many top results filtered out

3. Hybrid filter-aware ANN

  • vector DB-specific optimized strategy

Rule

  • If permission filtering is strict, never rely on post-filter after returning to user.
  • Apply security filters before response.

10. Hybrid search

Hybrid combines sparse lexical + dense vector.

Options

  • score blending: alpha * vector_score + (1-alpha) * bm25_score
  • RRF: combine by ranks, robust to score scale differences
  • retrieve separately, merge, rerank with cross-encoder

RRF implementation

def reciprocal_rank_fusion(rank_lists, k=60):
    scores = {}
    for results in rank_lists:
        for rank, doc_id in enumerate(results, 1):
            scores[doc_id] = scores.get(doc_id, 0.0) + 1.0 / (k + rank)
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)

11. Embedding version migration

Problem

  • New embedding model changes vector space.
  • Cannot mix old and new embeddings in one index.

Migration plan

1. Create new embedding_version column.

2. Backfill new embeddings to new index.

3. Dual-write new updates to old and new index.

4. Shadow test new index.

5. A/B route small traffic.

6. Switch read path.

7. Retire old index after safety window.

Schema

doc_id
embedding
embedding_model_version
content_version
updated_at
tenant_id
metadata

13. Interview quick answers

Q: HNSW vs IVF?
A: HNSW is graph-based, often high recall/low latency but memory-heavy. IVF clusters vectors and searches selected clusters, good for large datasets with tunable nprobe but requires training/building.

Q: Why vector search results are bad?
A: Could be embedding model mismatch, query/document preprocessing mismatch, stale index, poor ANN recall, missing metadata filters, or lack of reranking.

Q: How to measure ANN quality?
A: Compare ANN top-k against exact brute-force top-k on a sampled dataset and compute recall@k/latency/memory tradeoff.

Q: How handle permissions in vector search?
A: Enforce tenant/security filters in retrieval or before response. Do not trust post-processing alone if there is any chance of leakage.

Section 17

RESUME-ALIGNED INTERVIEW DEEP DIVE

A. Vector DB internals answer
------------------------------------------------------------
Vector search is approximate. I would tune recall, latency, and memory together, then evaluate recall@K against an exact-search baseline on a sampled dataset.

B. ANN choices
------------------------------------------------------------
HNSW: graph traversal, strong recall/latency, high memory.
IVF: partition vectors into clusters and probe selected clusters.
PQ: compress vectors to reduce memory, with quality tradeoff.
Scalar quantization: reduce numeric precision for memory/speed.
Disk-based ANN: handles huge corpora with SSD tradeoffs.

C. Production lifecycle
------------------------------------------------------------

  • embedding model versioning.
  • dual-write or side-by-side index build for migrations.
  • backfill and reindex plans.
  • segment compaction.
  • metadata filtering strategy.
  • tombstones/deletes and freshness.
  • hybrid search fusion with lexical results.

D. Interview Q&A
------------------------------------------------------------
Q: Why can metadata filtering hurt ANN search?
A: If filtering happens after ANN, many nearest candidates may be removed, hurting recall. Better designs push filters into index search or over-retrieve before filtering/reranking.

Q: How migrate embedding versions?
A: Build a new index side-by-side, backfill vectors, dual-read/compare, canary traffic, monitor recall/latency, then switch and retire old index.

Q: HNSW parameters?
A: Higher graph degree/search effort improves recall but increases memory/build/query cost. Tune against an exact baseline.
