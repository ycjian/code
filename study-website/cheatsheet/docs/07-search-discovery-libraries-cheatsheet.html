0. Search mental model

Search is usually a multi-stage ranking system

Query -> understand/rewrite/tokenize -> candidate generation -> ranking/reranking -> filters/business rules -> results

Retrieval stages

Stage 1: retrieve many candidates fast
Stage 2: rank/rerank with better model
Stage 3: diversify/personalize/business rules

中文: 搜索不是一个模型,而是一条 pipeline。

2. Core metrics

Precision@k: relevant items in top k
Recall@k: how many relevant items retrieved
MRR: reciprocal rank of first relevant item
MAP: mean average precision
NDCG@k: ranking quality with graded relevance
CTR: click-through rate
CVR: conversion rate
Dwell time: engagement
Zero-result rate: search failure signal
Latency p50/p95/p99

NDCG intuition: relevant results are better when ranked higher.

3. sklearn text search baseline

from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.metrics.pairwise import cosine_similarity

docs = [
    "reset password instructions",
    "refund policy for returned items",
    "shipping time and delivery tracking",
]

vectorizer = TfidfVectorizer(ngram_range=(1, 2))
X = vectorizer.fit_transform(docs)

def search(query, k=3):
    q = vectorizer.transform([query])
    scores = cosine_similarity(q, X)[0]
    idx = scores.argsort()[::-1][:k]
    return [(docs[i], float(scores[i])) for i in idx]

print(search("how do I return my order"))

Good for small demos, interview baseline, simple internal tools.

    idx = np.argsort(scores)[::-1][:k]

5. SentenceTransformers semantic search

from sentence_transformers import SentenceTransformer
import numpy as np

model = SentenceTransformer("sentence-transformers/all-MiniLM-L6-v2")
docs = ["reset password", "refund order", "track shipment"]
doc_vecs = model.encode(docs, normalize_embeddings=True)

def semantic_search(query, k=3):
    q = model.encode([query], normalize_embeddings=True)
    scores = (q @ doc_vecs.T)[0]
    idx = np.argsort(scores)[::-1][:k]
    return [(docs[i], float(scores[i])) for i in idx]

6. FAISS vector search

import faiss
import numpy as np

vectors = np.array(doc_vecs).astype("float32")
index = faiss.IndexFlatIP(vectors.shape[1])
index.add(vectors)

def faiss_search(query, k=5):
    q = model.encode([query], normalize_embeddings=True).astype("float32")
    scores, ids = index.search(q, k)
    return [(docs[i], float(s)) for i, s in zip(ids[0], scores[0])]

FAISS is good for local fast ANN, prototypes, and custom infra.

11. Reranking

Cross-encoder reranker

from sentence_transformers import CrossEncoder

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def rerank(query, docs):
    pairs = [(query, d) for d in docs]
    scores = reranker.predict(pairs)
    return sorted(zip(docs, scores), key=lambda x: x[1], reverse=True)

Why

Bi-encoder/vector search is fast. Cross-encoder is slower but more accurate. Use vector/BM25 for candidate generation, cross-encoder for reranking.

Reciprocal Rank Fusion

def rrf(rankings, k=60):
    scores = {}
    for ranking in rankings:
        for rank, doc_id in enumerate(ranking, start=1):
            scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank)
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)

RRF is robust because it combines ranks, not raw scores.

Section 18

16. Interview answer pattern

Question: Design search for documents/products.

Answer

I would use a multi-stage ranking architecture. First, query understanding extracts filters and entities. Then candidate generation combines BM25 for exact keyword matches and dense vector retrieval for semantic matches. I would merge results with RRF or score fusion, rerank top candidates with a cross-encoder or learning-to-rank model, apply business rules and diversity, and return results with explanations. Offline, I would measure Recall@k, NDCG, and MRR; online, CTR, conversion, zero-result rate, and latency.

Section 19

ADVANCED INTERVIEW DEEP DIVE - EXPANDED COVERAGE

A. Search system mental model
------------------------------------------------------------
Search is a multi-stage funnel: candidate generation, filtering, ranking, reranking, diversification, and feedback learning. Strong systems balance relevance, latency, freshness, permissions, and business constraints.

B. Core libraries and systems
------------------------------------------------------------

  • Elasticsearch/OpenSearch/Lucene: inverted index, BM25, filters, aggregations.
  • Milvus/Faiss/ScaNN: vector ANN search.
  • Spark/Flink/Beam: indexing and feature generation.
  • Kafka: ingestion/update event stream.
  • Redis: cache, counters, feature cache.
  • PyTerrier: IR research/evaluation.
  • Sentence Transformers/HF: embedding models.

C. Indexing pipeline
------------------------------------------------------------

1. Source event arrives.

2. Validate and normalize metadata.

3. Extract text/OCR/captions/embeddings.

4. Write lexical index and vector index.

5. Publish index version and health metrics.

6. Run backfill and incremental updates separately.

Use immutable index versions for safe rollout and rollback. Query services can A/B old/new indexes.

D. ANN deep dive
------------------------------------------------------------
Approximate nearest neighbor trades recall for speed/memory.

  • HNSW: graph-based, high recall/low latency, memory heavy.
  • IVF: cluster vectors, search selected clusters.
  • PQ: compress vectors, lower memory, quality tradeoff.
  • DiskANN: large indexes with SSD.

Tune recall@K, latency, memory, index build time, update speed, and filter behavior.

E. Ranking features
------------------------------------------------------------

Feature families

  • lexical: BM25, exact match, field match.
  • semantic: vector similarity, cross-encoder score.
  • behavioral: clicks, saves, dwell.
  • freshness/popularity.
  • personalization.
  • permissions/availability.
  • diversity/dedup.

F. Evaluation
------------------------------------------------------------
Offline: Recall@K, MRR, nDCG, coverage, freshness, latency. Online: CTR, conversion, zero-result rate, reformulation rate, task success. Watch for position bias and popularity feedback loops.

G. Failure modes
------------------------------------------------------------
Vector index stale while lexical index is fresh; embedding model changed but old vectors remain; ANN recall too low for reranker; filters applied after ANN cause empty results; popularity bias buries niche content.

H. Interview Q&A
------------------------------------------------------------
Q: Why multi-stage retrieval?
A: Full ranking over all documents is too expensive. Use cheap high-recall retrieval first, then expensive reranking on a small candidate set.

Q: Why Reciprocal Rank Fusion?
A: It combines rankings with incomparable scores, such as BM25 and vector search, by using ranks rather than raw scores.

Q: How handle permissions?
A: Apply tenant/ACL filters during candidate generation where possible, cache carefully, and never generate/rerank from unauthorized candidates.

Q: How debug relevance drop?
A: Compare query mix, index freshness, retrieval recall, ranker features, model/index versions, timeouts, and A/B slices. Build query-level diff tools.
