For agents, I would not allow unconstrained autonomy; I would use tools, state, guardrails, and human approval for side effects.

ADVANCED INTERVIEW DEEP DIVE - EXPANDED COVERAGE

A. Cross-stack mental model
------------------------------------------------------------
AI infra interviews test whether you connect product traffic to hardware constraints. Trace the request path: API gateway -> router -> queues/cache -> orchestration -> CPU/GPU serving -> storage/index -> observability -> rollout/evaluation.

B. Kafka patterns
------------------------------------------------------------
Kafka is for durable event streams: ingestion, index updates, feature events, async jobs, and audit logs.

Key points

  • topic is split into partitions.
  • ordering is only within a partition.
  • consumer group scales consumption; one partition is consumed by one consumer in a group at a time.
  • commit offset after durable processing for at-least-once.
  • idempotent upsert/merge handles retries.
  • exactly-once is practical mainly inside Kafka/Kafka Streams transactions; external sinks still need idempotency.

Q: How handle duplicate messages?
A: Use event IDs, upsert/merge semantics, dedup windows, and commit offsets only after processing is durable.

C. Redis patterns
------------------------------------------------------------
Use Redis for low-latency cache, rate limits, counters, feature/session cache, lightweight streams, and request coalescing.

Rules

  • TTL every key that can go stale.
  • avoid unbounded high-cardinality keys.
  • choose eviction policy intentionally.
  • prevent cache stampede with single-flight, jittered TTL, stale-while-revalidate, and background refresh.

D. Triton patterns
------------------------------------------------------------
Triton is strong for tensor inference: embeddings, classifiers, rankers, vision/audio models. It supports dynamic batching, model repository versions, multiple backends, and ensembles. Use Ray/FastAPI around it for complex business DAGs.

Q: Why Triton for rankers/embeddings?
A: It maximizes GPU throughput with dynamic batching and optimized runtimes while standardizing serving.

E. Spark/Flink patterns
------------------------------------------------------------
Spark: batch ETL, offline features, backfills, large joins. Flink: true streaming, event-time windows, stateful processing, continuous indexing/features.

Q: Spark vs Flink?
A: Spark is excellent for batch/micro-batch analytics; Flink is stronger for low-latency streaming and long-running stateful jobs.

F. Database and storage patterns
------------------------------------------------------------

  • object storage: raw media, model artifacts, large docs.
  • relational DB: transactional metadata.
  • vector DB: ANN retrieval.
  • search index: lexical retrieval/filtering.
  • feature store: online/offline feature consistency.

Do not force one storage system to serve every access pattern.

G. Envoy/API gateway patterns
------------------------------------------------------------
Use gateway/proxy for auth, tenant identity, request IDs, timeouts, retries, circuit breakers, mTLS, and traffic splitting. Caution: retries can amplify overload; use deadlines and retry budgets.

H. GPU resource patterns
------------------------------------------------------------
Separate CPU orchestration from GPU inference. Batch GPU-friendly tensor work. Keep warm capacity. Use MIG for small isolated workloads. Bin-pack compatible jobs for utilization and spread critical replicas for availability. Monitor GPU memory, utilization, tokens/sec, OOM, and XID errors.

I. System design answer template
------------------------------------------------------------

1. Clarify SLA, scale, sensitivity, workload type.

2. Split sync vs async paths.

3. Choose retrieval/model/serving stack.

4. Define CPU vs GPU routing.

5. Add queue/backpressure/admission control.

6. Add cache/versioning/rollback.

7. Define observability and evaluation.

8. Discuss failure modes and tradeoffs.

J. Stress-test Q&A
------------------------------------------------------------
Q: Traffic spikes 10x and GPUs saturate. What do you do?
A: Admit by priority, degrade to smaller/cached answer, queue async work, scale if capacity exists, shed low-priority traffic, and inspect queue time vs GPU utilization.

Q: Model quality regresses but service metrics are green. What catches it?
A: Shadow/canary eval, LLM-as-judge/human-labeled slices, output drift, retrieval recall checks, and product metrics.

Q: Product team wants a new AI feature in two weeks. What helps?
A: Intent registry, standard schemas, model registry, reusable DAG blocks, shared retrieval connectors, guardrails, observability templates, rollout gates.

Q: CPU vs GPU?
A: CPU for routing, metadata joins, orchestration, small ML. GPU for dense tensor workloads, embeddings, rankers, VLM/LLM inference, and training.
