2. Prometheus FastAPI example

# pip install prometheus-client fastapi uvicorn
from fastapi import FastAPI, Request
from prometheus_client import Counter, Histogram, generate_latest, CONTENT_TYPE_LATEST
from starlette.responses import Response
import time

app = FastAPI()

REQUEST_COUNT = Counter(
    "api_requests_total",
    "Total API requests",
    ["path", "method", "status"],
)

REQUEST_LATENCY = Histogram(
    "api_request_latency_seconds",
    "Request latency",
    ["path", "method"],
)

LLM_TOKENS = Counter(
    "llm_tokens_total",
    "LLM tokens used",
    ["model", "direction"],  # direction=input/output
)

@app.middleware("http")
async def metrics_middleware(request: Request, call_next):
    start = time.time()
    response = await call_next(request)
    latency = time.time() - start

    REQUEST_COUNT.labels(request.url.path, request.method, response.status_code).inc()
    REQUEST_LATENCY.labels(request.url.path, request.method).observe(latency)
    return response

@app.get("/metrics")
def metrics():
    return Response(generate_latest(), media_type=CONTENT_TYPE_LATEST)

3. OpenTelemetry tracing example

# pip install opentelemetry-api opentelemetry-sdk opentelemetry-exporter-otlp
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter

provider = TracerProvider()
processor = BatchSpanProcessor(OTLPSpanExporter(endpoint="http://otel-collector:4317"))
provider.add_span_processor(processor)
trace.set_tracer_provider(provider)

tracer = trace.get_tracer(__name__)

def answer_question(query: str, user_id: str):
    with tracer.start_as_current_span("rag_request") as span:
        span.set_attribute("user_id_hash", hash(user_id))
        span.set_attribute("query_length", len(query))
        span.set_attribute("rag.pipeline_version", "2026-05-14")

        with tracer.start_as_current_span("retrieve") as s:
            docs = retrieve(query)
            s.set_attribute("retrieved.count", len(docs))
            s.set_attribute("retrieved.doc_ids", ",".join(d.id for d in docs[:5]))

        with tracer.start_as_current_span("rerank") as s:
            ranked = rerank(query, docs)
            s.set_attribute("reranker.version", "bge-reranker-v2")

        with tracer.start_as_current_span("generate") as s:
            answer, usage = call_llm(query, ranked)
            s.set_attribute("llm.model", "gpt-x")
            s.set_attribute("llm.input_tokens", usage.input_tokens)
            s.set_attribute("llm.output_tokens", usage.output_tokens)

        return answer

Important privacy note

  • Do not log raw prompts/documents unless policy allows.
  • Prefer hashed user IDs and doc IDs.
  • Redact PII.

Section 09

7. Debug playbooks

Case A: latency spike

1. Check p95 by endpoint.

2. Split trace by stage: retrieval/rerank/generate.

3. Check dependency latency and timeout.

4. Check queue depth and GPU utilization.

5. Check recent deployment/model/prompt change.

6. Roll back or scale bottleneck.

Case B: answer quality drop

1. Check prompt/model/retriever version change.

2. Compare retrieval recall@k.

3. Inspect failed traces: retrieved docs? context too noisy?

4. Check index freshness and embedding version mismatch.

5. Add failures to eval set.

Case C: high cost

1. Token count by route/user/model.

2. Long context prompts.

3. Cache hit rate.

4. Too many tool calls/retries.

5. Rerank/context pruning.

6. Use smaller model for easy queries.

9. Interview quick answers

Q: How do you monitor RAG in production?
A: I monitor both system health and answer quality. System metrics include latency per stage, error rate, token cost, and dependency failures. Quality metrics include retrieval recall on sampled labeled queries, citation failures, user feedback, and slice-based regression tests.

Q: How debug hallucination reports?
A: Trace the request, inspect retrieved docs and context. If required docs were missing, fix retrieval/indexing. If docs were present but model ignored them, fix prompt/context formatting or model. If citation unsupported, add citation verification and eval cases.

Q: What do you log?
A: Structured logs with request_id, trace_id, model/prompt/index versions, stage latencies, token counts, fallback reason, and safe identifiers. Avoid raw PII unless explicitly approved.

RESUME-ALIGNED INTERVIEW DEEP DIVE

A. AI observability answer shape
------------------------------------------------------------
For Staff-level AI infra interviews, describe observability by request stage, not only by service.

Trace fields

  • request_id, tenant_id_hash, user/session hash.
  • intent and workflow version.
  • retrieval index version, embedding model version, reranker version.
  • model/version/prompt/template version.
  • token counts, queue time, GPU time, cache hits.
  • retrieved doc IDs or hashed IDs, not raw sensitive text.
  • fallback and safety decision events.

B. Dashboard layout
------------------------------------------------------------

Golden signals

  • traffic, errors, latency, saturation.

AI panels

  • retrieval p95 and empty retrieval rate.
  • reranker latency and top-k distribution.
  • LLM TTFT, TPOT, output tokens, cost.
  • GPU utilization, memory, KV cache usage, batch size.
  • cache hit rate and fallback rate.
  • quality signals: citation failures, hallucination reports, thumbs down, judge regressions.

C. SLO thinking
------------------------------------------------------------
Define separate SLOs for synchronous user-facing paths and async workflows. Example: p95 answer latency for short RAG, p99 job completion for video/document processing, and error budget for fallback rate.

D. Interview Q&A
------------------------------------------------------------
Q: Why OpenTelemetry?
A: It is vendor-neutral instrumentation for traces, metrics, and logs. It lets us propagate context across gateway, retrieval, reranking, model serving, and external LLM calls.

Q: How do you debug a latency spike?
A: Break latency into queue time, retrieval time, rerank time, model prefill/decode time, network/downstream time, and cache misses. Then correlate with QPS, token length, and GPU saturation.

Q: How do you log prompts safely?
A: Prefer prompt/template version and metadata. If raw prompts are needed, redact PII, sample, encrypt, control access, set retention, and avoid logging secrets/tool credentials.
