Section 02

0. What is an AI agent?

An agent is an LLM-powered system that can

1. reason over a goal

2. choose actions/tools

3. observe results

4. update state/memory

5. continue until done

中文

普通 chatbot 主要回答问题。
Agent 可以调用工具、查数据、执行任务、保存状态。

Core loop

User goal -> plan -> choose tool -> execute tool -> observe result -> decide next step -> final answer

Section 04

2. Tool calling mental model

The model does NOT execute code directly. It produces a structured tool call. Your application executes the tool and returns result.

Flow

1. App sends user input + tool definitions to model

2. Model returns tool call: function name + JSON args

3. App validates args

4. App executes function/API

5. App sends tool result back to model

6. Model gives final answer or asks for another tool

Section 05

3. Minimal manual tool-calling pattern

import json

def get_weather(city: str) -> dict:
    return {"city": city, "temp_f": 72, "condition": "sunny"}

tools = {
    "get_weather": get_weather,
}

# Pseudo-response from LLM:
tool_call = {
    "name": "get_weather",
    "arguments": {"city": "Mountain View"},
}

name = tool_call["name"]
args = tool_call["arguments"]

if name not in tools:
    raise ValueError("Unknown tool")

result = tools[name](**args)
print(result)

Production concerns

  • validate tool name
  • validate JSON schema
  • timeout
  • retry
  • auth
  • rate limit
  • log every call
Section 06

4. Tool schema

tools = [{
    "type": "function",
    "name": "search_docs",
    "description": "Search internal documents by query.",
    "parameters": {
        "type": "object",
        "properties": {
            "query": {"type": "string"},
            "top_k": {"type": "integer", "default": 5}
        },
        "required": ["query"]
    }
}]

Good tool design

  • small, deterministic, well-named
  • clear input/output schema
  • avoid too many overlapping tools
  • return concise structured data

Bad tool

do_everything(query: str)

Good tools

search_docs(query, top_k)
read_doc(doc_id)
create_ticket(title, body, priority)
get_calendar_availability(date)

MAX_STEPS = 5

def run_agent(user_input):
    messages = [{"role": "user", "content": user_input}]

    for step in range(MAX_STEPS):
        response = call_llm(messages, tools=tool_schemas)

        if response.type == "final":
            return response.text

        if response.type == "tool_call":
            tool_name = response.tool_name
            args = response.arguments

            validate(tool_name, args)
            result = execute_tool(tool_name, args)

            messages.append(response.as_message())
            messages.append({

                "tool_call_id": response.tool_call_id,
                "content": json.dumps(result),

Section 08

6. Memory types

Short-term memory: current conversation messages, task scratchpad/state
Long-term memory: user preferences, facts about projects, previous decisions
Episodic memory: prior conversations/events
Semantic memory: knowledge base / docs
Procedural memory: saved workflows / skills

When to store memory

  • stable user preference
  • reusable project context
  • explicit "remember this"

When NOT

  • temporary facts
  • sensitive attributes unless explicitly requested
  • noisy one-off details
Section 09

7. RAG vs memory

RAG:
- search external docs
- source-grounded
- usually large corpus

Memory:
- small personalized facts/state
- improves continuity
- often user/session scoped

Agent = LLM + tools + state.
RAG = one important knowledge tool.
Memory = state across conversations/tasks.

Use when

Section 12

10. Multi-agent design

Use multi-agent only when specialization helps

  • Researcher -> retrieves facts
  • Coder -> writes code
  • Reviewer -> checks correctness
  • Planner -> decomposes tasks
  • Executor -> runs tools

Do not overuse. More agents = more latency/cost/failure points.
For many production systems, a single agent with good tools is better.

Section 18

16. Interview answer pattern

Question: Design an AI agent for customer support.

Answer

I would build a tool-augmented agent with a controlled workflow. The agent first classifies the intent, retrieves relevant policy docs using RAG, and decides whether it can answer or needs human escalation. For actions such as refund or account changes, it calls backend tools with schema validation and permission checks. I would require human approval for high-impact actions, log every tool call, and evaluate with task success rate, citation accuracy, latency, and escalation rate.

Section 19

ADVANCED INTERVIEW DEEP DIVE - EXPANDED COVERAGE

A. Agent mental model
------------------------------------------------------------
An AI agent is an LLM-driven control loop that observes state, selects/plans actions, calls tools, reads results, and decides whether to continue or answer. Production difficulty is control, safety, latency, and debuggability.

B. Core components
------------------------------------------------------------

  • intent/router: choose workflow/tool set.
  • planner: decompose if needed.
  • tool registry: typed schemas, auth, rate limits, owners.
  • memory/state: conversation, evidence, intermediate results.
  • executor: tool calls, retries, timeouts, budgets.
  • guardrails: pre/post policy checks.
  • evaluator: success, groundedness, safety.

C. Tool calling design
------------------------------------------------------------
Good tools are narrow, typed, auditable, and permission-aware.

{
  "name": "search_employee_policy",
  "description": "Search approved HR policy snippets by query and tenant.",
  "parameters": {"query": "string", "tenant_id": "string", "top_k": "integer"}
}

Avoid vague tools like run_any_sql. Mutating tools need idempotency, approval gates, and audit logs.

D. Architectures
------------------------------------------------------------

  • ReAct loop: flexible, can be slow.
  • plan-and-execute: better for long tasks.
  • router + specialized agents: good for enterprise domains.
  • deterministic workflow with LLM slots: best for reliability.
  • human-in-the-loop: required for risky actions.

Enterprise answer: prefer deterministic workflows around LLM calls. Use autonomous loops only when bounded and low-risk.

E. Reliability and safety
------------------------------------------------------------
Use tool timeouts, retry only idempotent calls, circuit breakers, max steps/tokens/cost, trace IDs, auth outside the LLM, prompt-injection defenses, and deterministic validation before mutations.

F. Evaluation
------------------------------------------------------------

Measure final answer and trajectory

  • task success rate.
  • tool selection accuracy.
  • unnecessary tool-call rate.
  • hallucinated tool arguments.
  • groundedness.
  • policy violation rate.
  • p95 latency and cost per resolved task.

G. Interview Q&A
------------------------------------------------------------
Q: When should an agent not be autonomous?
A: High-risk, irreversible, expensive, or security-sensitive actions need deterministic validation and human approval.

Q: How prevent prompt injection?
A: Treat retrieved text as untrusted, separate system instructions from data, whitelist tools, validate args, enforce auth outside the LLM, and audit every tool call.

Q: How debug agent failure?
A: Inspect trace: intent, prompt, retrieved evidence, model output, tool args, tool response, guardrail decisions, and final answer. Classify as retrieval, reasoning, tool, policy, or UX failure.

Q: Why are agents expensive?
A: Multi-step loops multiply LLM calls, context size, tool latency, and retries. Use routing, caching, smaller models, and strict budgets.
