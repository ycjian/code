Section 02

0. Kubernetes mental model

Kubernetes is a platform for running containerized workloads.

Core ideas

Pod: smallest deployable unit; one or more containers
Deployment: manages replicated stateless Pods
Service: stable network endpoint for Pods
Ingress/Gateway: external HTTP routing
ConfigMap/Secret: configuration
Volume/PVC: storage
Job/CronJob: batch work
Namespace: isolation
Node: machine
Cluster: group of nodes

中文: Pod 会被创建/删除,IP 不稳定。Service 给 Pod 一个稳定入口。

2. Deployment

apiVersion: apps/v1
kind: Deployment
metadata:
  name: sentiment-api
spec:
  replicas: 3
  selector:
    matchLabels:
      app: sentiment-api
  template:
    metadata:
      labels:
        app: sentiment-api
    spec:
      containers:
        - name: api
          image: myrepo/sentiment-api:1.0.0
          ports:
            - containerPort: 8080

Apply and rollout

kubectl apply -f deployment.yaml
kubectl rollout status deployment/sentiment-api
kubectl rollout undo deployment/sentiment-api
Section 05

3. Service

apiVersion: v1
kind: Service
metadata:
  name: sentiment-service
spec:
  type: ClusterIP
  selector:
    app: sentiment-api
  ports:
    - port: 80
      targetPort: 8080

Meaning: Service port 80 -> container port 8080. Selector chooses Pods by labels.

Types

  • ClusterIP: internal only
  • NodePort: exposes on node port
  • LoadBalancer: cloud load balancer
  • ExternalName: DNS alias

6. Resource requests and limits

resources:
  requests:
    cpu: "500m"
    memory: "1Gi"
  limits:
    cpu: "1"
    memory: "2Gi"

Meaning

  • requests: scheduler uses this to place Pod
  • limits: max allowed usage

GPU

resources:
  limits:
    nvidia.com/gpu: 1

For GPU workloads, NVIDIA device plugin is usually needed.

7. Probes

livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10

readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5

liveness: should container be restarted?
readiness: should Pod receive traffic?
For model serving, readiness should be false until model is loaded.

Section 17

15. Interview answer pattern

Question: Deploy a model on Kubernetes.

Answer

I would containerize the model server, deploy it with a Deployment, expose it through a Service and Ingress, configure resource requests/limits including GPU if needed, and add readiness/liveness probes. For production I would add autoscaling based on latency or queue length, rolling updates, monitoring with Prometheus, centralized logs, and a rollback strategy. For LLMs I would pay special attention to model loading time, GPU memory, readiness probe delay, and long-running request timeouts.

Section 18

ADVANCED INTERVIEW DEEP DIVE - EXPANDED COVERAGE

A. Kubernetes mental model for AI infra
------------------------------------------------------------
Kubernetes is the control plane for container scheduling, networking, rollout, and resource isolation. For AI, hard parts are GPU placement, cold start, autoscaling signals, model artifacts, and safe rollouts.

B. Scheduler and resources
------------------------------------------------------------
Requests affect scheduling; limits affect runtime enforcement. CPU over limit throttles; memory over limit can OOMKill. GPU resources usually come from the NVIDIA device plugin.

Placement tools: nodeSelector, nodeAffinity, podAffinity/antiAffinity, taints/tolerations, priority/preemption, topology spread constraints.

C. GPU workloads
------------------------------------------------------------
Need correct CUDA image, GPU node pools, device plugin, readiness after model warmup, model cache/local SSD strategy, and GPU observability. MIG partitions one GPU into isolated instances; useful for smaller models but less flexible for large models.

D. Autoscaling
------------------------------------------------------------
HPA can use CPU/memory/custom metrics. KEDA adds event-driven scaling from queues/Kafka/Prometheus/etc. For AI serving, better signals include queue depth, p95 latency, token backlog, GPU utilization, KV-cache pressure, and model warm capacity. Avoid scale-to-zero for huge models unless cold start is acceptable.

E. Rollout safety
------------------------------------------------------------
Rollout unit includes image, model artifact, tokenizer, prompt template, schema, and index version. Use canary, blue/green, or shadow. Readiness should remain false until model is loaded and warmed.

F. Networking
------------------------------------------------------------
Service gives stable DNS/IP. Ingress/Gateway handles external HTTP. Envoy/Istio can add retries, timeouts, mTLS, traffic splitting. Be careful: retries can amplify incidents; use deadlines and retry budgets.

G. Artifact storage
------------------------------------------------------------
Options: bake model into image, initContainer download, sidecar/model cache, object storage + local SSD. Trade off rollout speed, startup latency, image size, and cache hit rate.

H. Failure modes
------------------------------------------------------------
CrashLoopBackOff, ImagePullBackOff, Pending due insufficient resources, OOMKilled, readiness false, node pressure, GPU plugin issues, failed scheduling constraints.

I. Interview Q&A
------------------------------------------------------------
Q: Request vs limit?
A: Request is reserved for placement. Limit is max runtime usage. CPU throttles; memory can kill.

Q: How deploy model safely?
A: Immutable versions, readiness after warmup, canary, monitor latency/error/quality, and rollback to prior image/model/tokenizer/config.

Q: How autoscale LLM serving?
A: Use queue/token/latency metrics, keep warm replicas, account for GPU capacity and cold starts, and avoid pure CPU HPA.

Q: Bin-packing vs spreading?
A: Bin-packing improves utilization/cost; spreading improves availability. AI platforms often combine both by workload class.
