Section 04

2. Offline inference

from vllm import LLM, SamplingParams

prompts = [
    "Explain vector databases in one paragraph.",
    "Write a Python function for binary search.",
]

sampling_params = SamplingParams(
    temperature=0.7,
    top_p=0.95,
    max_tokens=128,
)

llm = LLM(model="meta-llama/Llama-3.1-8B-Instruct")
outputs = llm.generate(prompts, sampling_params)

for output in outputs:
    print("PROMPT:", output.prompt)
    print("OUTPUT:", output.outputs[0].text)

Key

  • LLM loads model.
  • SamplingParams controls decoding.
  • generate accepts batch of prompts.
Section 05

3. Sampling parameters

temperature: randomness. lower = deterministic.
top_p: nucleus sampling.
max_tokens: max generated tokens.
stop: stop sequences.
presence_penalty / frequency_penalty: discourage repetition.

Interview

  • deterministic coding/extraction: temperature=0
  • creative generation: temperature around 0.7

4. OpenAI-compatible server

Start server

vllm serve meta-llama/Llama-3.1-8B-Instruct \
  --dtype auto \
  --host 0.0.0.0 \
  --port 8000 \
  --api-key token-abc123

Call with OpenAI Python client

from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="token-abc123",
)

resp = client.chat.completions.create(
    model="meta-llama/Llama-3.1-8B-Instruct",
    messages=[
        {"role": "user", "content": "Explain vLLM in one sentence."}
    ],
    temperature=0.0,

print(resp.choices[0].message.content)

Section 10

8. Memory and KV cache

LLM serving memory includes

  • model weights
  • KV cache for active requests
  • temporary activations
  • CUDA overhead

KV cache grows with

  • batch size
  • sequence length
  • number of concurrent requests
  • hidden size/layers

Common tuning

--gpu-memory-utilization 0.9
--max-model-len 8192
--max-num-seqs 128

Longer context and more concurrent requests need more KV cache.

Section 11

9. Quantization

Use quantization to reduce memory

AWQ
GPTQ
bitsandbytes
FP8
INT8/INT4 depending on model/hardware support

Tradeoff: lower memory, sometimes faster, possible quality loss, hardware compatibility matters.

Section 12

10. Prefix caching

Use when many requests share the same prefix

  • system prompt
  • long instructions
  • few-shot examples
  • RAG template

Benefit: reuse computation for repeated prefixes, reducing latency/cost.

Section 16

14. Common serving issues

OOM

  • reduce max_model_len
  • reduce max_num_seqs
  • lower gpu_memory_utilization
  • use quantization
  • more GPUs/tensor parallelism

High latency

  • reduce output tokens
  • increase replicas
  • separate long/short requests
  • tune batching
  • use prefix caching

Low throughput

  • increase concurrency
  • batching
  • use larger GPU
  • optimize dtype/quantization

Bad model quality

  • check chat template
  • check tokenizer
  • check prompt format
  • check model instruction tuning

Section 18

16. Interview answer pattern

Question: How would you serve an open-source LLM?

Answer

I would use vLLM when the goal is high-throughput online LLM serving. I would expose the OpenAI-compatible server behind an API gateway, run multiple GPU-backed replicas on Kubernetes, configure tensor parallelism if the model does not fit on one GPU, and monitor TTFT, tokens/sec, queue time, GPU memory, and p95 latency. I would tune max context length, concurrency, and KV cache usage, and use quantization or prefix caching if needed.

Section 19

ADVANCED INTERVIEW DEEP DIVE - EXPANDED COVERAGE

A. vLLM mental model
------------------------------------------------------------
vLLM is an LLM serving engine optimized for high-throughput autoregressive generation through continuous batching, PagedAttention/KV-cache management, OpenAI-compatible APIs, and distributed inference.

B. Why LLM serving is special
------------------------------------------------------------
Autoregressive generation has prefill and decode. Prefill processes prompt tokens in parallel and is compute-heavy. Decode emits one token at a time and is often memory-bandwidth/KV-cache bound. Request lengths vary, so static batching wastes capacity.

C. KV cache deep dive
------------------------------------------------------------
The model stores key/value tensors per layer for previous tokens. KV memory grows with batch size, sequence length, layer count, hidden size/heads, and dtype. Long context and high concurrency can OOM even when weights fit.

D. PagedAttention
------------------------------------------------------------
PagedAttention splits KV cache into blocks and maps logical token blocks to physical GPU-memory blocks, similar to virtual memory. It reduces fragmentation, supports larger effective batches, and improves utilization under variable request lengths.

Tradeoff: block scheduling/memory indirection adds complexity. For tiny single-request latency, simpler stacks can be competitive.

E. Serving APIs and knobs
------------------------------------------------------------

python -m vllm.entrypoints.openai.api_server \
  --model meta-llama/Llama-3.1-8B-Instruct \
  --tensor-parallel-size 2 \
  --max-model-len 8192
from vllm import LLM, SamplingParams
llm = LLM(model='meta-llama/Llama-3.1-8B-Instruct')
params = SamplingParams(temperature=0.2, max_tokens=256)
outputs = llm.generate(['Explain KV cache'], params)

Important knobs: max_model_len, gpu_memory_utilization, tensor_parallel_size, quantization, prefix caching, speculative decoding, concurrency/batching limits.

F. Production needs around vLLM
------------------------------------------------------------
vLLM is not the whole platform. You still need auth, routing, rate limits, prompt templates, safety, model registry/versioning, autoscaling, GPU placement, and observability for TTFT, TPOT, tokens/sec, queue time, KV usage, OOM, and cost.

G. Failure modes
------------------------------------------------------------
KV-cache OOM, high time-to-first-token from prefill queue, decode bottleneck from memory bandwidth, bad chat template, too much concurrency hurting p95, prefix cache ineffective due unnormalized prompts.

H. Interview Q&A
------------------------------------------------------------
Q: What problem does PagedAttention solve?
A: KV-cache fragmentation/waste. It allocates cache in blocks so more sequences fit concurrently.

Q: What is continuous batching?
A: Scheduler adds/removes requests at token boundaries so GPU stays busy despite variable prompt/output lengths.

Q: What metrics matter?
A: TTFT, TPOT, tokens/sec, queue time, p95/p99 latency, GPU utilization, KV usage, OOM rate, cost per 1K tokens.

Q: How handle KV-cache OOM?
A: Reduce max context/concurrency, admission-control long prompts, route long-context to larger GPUs, monitor KV usage, and keep separate SLO tiers.
