Section 02

0. Mental model / 中文总览

A feature store is not just a database. It is a system that keeps feature definitions, historical training data, and low-latency serving features consistent.

Core split

offline store online store
------------ ------------
S3/GCS/Iceberg/BigQuery/Snowflake Redis/DynamoDB/Cassandra/Bigtable
historical features latest feature values
training data generation low-latency inference
point-in-time joins materialized features

Chinese

  • offline store = 训练用,历史特征,必须 point-in-time correct.
  • online store = serving 用,低延迟读最新特征.
  • materialization = 把 offline/computed features 推到 online store.
  • entity = feature 的主键,比如 user_id, item_id, driver_id.

Interview one-liner

"The feature store solves two hard problems: point-in-time correct historical features for training, and low-latency fresh features for online inference, using shared feature definitions to reduce training-serving skew."

Section 03

1. Key concepts

Entity

  • The object features describe.
  • Examples: user_id, item_id, query_id, merchant_id, doc_id.

Feature

  • A typed value used by a model.
  • Examples: user_click_7d, item_ctr_1h, doc_quality_score.

Feature view

  • A group of features from a source table, with entity key and timestamp.

Offline store

  • Historical feature data.
  • Used for training datasets and backtesting.

Online store

  • Low-latency key-value store.
  • Used by online model serving.

Point-in-time correctness

  • When building a training row at timestamp T, only use feature values known at or before T.
  • Never use future information.

Training-serving skew

  • Training computed one way, serving computed another way.
  • Model performs well offline but poorly online.

Feature definitions

# feature_repo/features.py
from datetime import timedelta
from feast import Entity, FeatureView, Field, FileSource
from feast.types import Float32, Int64

user = Entity(name="user", join_keys=["user_id"])

user_stats_source = FileSource(
    path="data/user_stats.parquet",
    timestamp_field="event_timestamp",
)

user_stats_fv = FeatureView(
    name="user_stats",
    entities=[user],
    ttl=timedelta(days=7),
    schema=[
        Field(name="click_count_7d", dtype=Int64),
        Field(name="avg_session_length_7d", dtype=Float32),
    ],
    online=True,
    source=user_stats_source,
)

feast apply

Generate historical training data

from feast import FeatureStore
import pandas as pd

store = FeatureStore(repo_path=".")

entity_df = pd.DataFrame({
    "user_id": ["u1", "u2", "u3"],

Section 05

3. Point-in-time correctness example

Bad training row

label timestamp = 10:00 AM
feature used = click_count_7d computed at 11:00 AM

This leaks future behavior.

Correct rule

For each entity row (user_id, event_timestamp), find latest feature row where

feature.entity_id == entity.entity_id
feature.event_timestamp <= entity.event_timestamp
feature.event_timestamp >= entity.event_timestamp - ttl

SQL sketch

WITH ranked AS (
  SELECT
    e.user_id,
    e.event_timestamp AS label_ts,
    e.label,
    f.click_count_7d,
    f.event_timestamp AS feature_ts,
    ROW_NUMBER() OVER (
      PARTITION BY e.user_id, e.event_timestamp
      ORDER BY f.event_timestamp DESC
    ) AS rn
  FROM labels e
  LEFT JOIN user_features f
    ON e.user_id = f.user_id
   AND f.event_timestamp <= e.event_timestamp
   AND f.event_timestamp > e.event_timestamp - INTERVAL '7' DAY
)
SELECT * FROM ranked WHERE rn = 1;
Section 06

4. Feature freshness

Freshness = how old the online feature value is.

Examples

  • user_click_count_5m must be < 2 minutes old.
  • user_profile_country can be hours/days old.
  • fraud velocity features may need seconds-level freshness.

Monitoring

feature_age_seconds = now() - feature_value.event_timestamp

Metrics

  • p50/p95/p99 feature age
  • materialization lag
  • missing feature rate
  • online store read latency
  • online/offline distribution drift
  • default value rate
Section 07

5. Online store design choices

Redis

  • Very low latency.
  • Good for hot features.
  • Memory cost can be high.

DynamoDB / Bigtable / Cassandra

  • Higher scale and durability.
  • Good for massive key-value workloads.
  • Usually single-digit to tens of ms latency.

Key design

feature_key = entity_type + ":" + entity_id + ":" + feature_view_name
value = serialized feature vector + event_timestamp + feature_version

Example online lookup API

class FeatureClient:
    def get_features(self, user_id: str, item_id: str) -> dict:
        user_features = redis.hgetall(f"user:{user_id}:user_stats")
        item_features = redis.hgetall(f"item:{item_id}:item_stats")
        return {**user_features, **item_features}

Production improvements

  • batch get instead of one key at a time
  • cache hot keys
  • timeout and fallback defaults
  • feature freshness validation
  • record feature version with prediction logs
Section 08

6. Streaming features

Example: compute user_click_count_5m with Kafka + Flink.

Flow

click events -> Kafka -> Flink keyed state -> Redis online store

Flink logic

  • key_by user_id
  • maintain event-time window or rolling state
  • update Redis with count and timestamp
  • checkpoint state

Pseudocode

def process_event(event, state):
    user_id = event["user_id"]
    ts = event["event_ts"]
    state[user_id].add(ts)
    state[user_id].remove_older_than(ts - 5 * 60)
    redis.hset(f"user:{user_id}:realtime", mapping={
        "click_count_5m": state[user_id].count(),
        "feature_ts": ts,
    })
Section 09

7. Feature versioning

Every feature should have

  • name
  • owner
  • description
  • entity key
  • type
  • source
  • transformation logic
  • freshness SLA
  • TTL
  • version
  • validation checks

Why versioning matters

  • model v1 trained with old definition
  • model v2 trained with new definition
  • serving must fetch compatible feature versions

Example

user_click_count_7d_v1 = count all clicks
user_click_count_7d_v2 = count only qualified clicks excluding bots

8. Feature drift and skew checks

Offline/online skew check

1. Log online features used at prediction time.

2. Recompute same features offline for the same entity/timestamp.

3. Compare values.

Example

import pandas as pd

joined = online_log.merge(offline_recomputed, on=["request_id", "feature_name"])
joined["abs_diff"] = (joined["online_value"] - joined["offline_value"]).abs()

skew_rate = (joined["abs_diff"] > 1e-6).mean()
assert skew_rate < 0.001

10. Feature store interview questions

Q: Why not just read features from the warehouse?
A: Warehouse is not low-latency enough for online serving and does not automatically enforce point-in-time joins or shared feature definitions.

Q: How do you avoid future leakage?
A: Use event timestamps and point-in-time joins. For each label timestamp, only join feature values with feature_ts <= label_ts.

Q: How to handle missing features online?
A: Use defaults, fallback features, freshness checks, and log missing/default rates. Important models may use feature availability as a separate signal.

Q: How do you backfill a feature?
A: Write feature definition, compute historical values with Spark into offline store, validate, materialize recent values to online store, then update model training/serving config.

Q: What if online feature is stale?
A: Return default/fallback, mark feature freshness in request context, maybe route to a safe model, and alert if freshness SLA is violated.

RESUME-ALIGNED INTERVIEW DEEP DIVE

A. How this maps to ranking and personalization
------------------------------------------------------------
For search/discovery, the feature store provides consistent features for retrieval/ranking models: user affinity, asset popularity, freshness, creator quality, query intent, doc engagement, tenant-level preferences, and safety signals.

The hard part is not only computing features. It is ensuring point-in-time correctness for training and low-latency freshness for serving.

B. Feature-store design for AI search
------------------------------------------------------------

Offline path

  • Spark/Flink writes feature histories to Iceberg/S3/warehouse.
  • Training jobs request point-in-time joins for labels.
  • Feature definitions are versioned with model versions.

Online path

  • Flink or batch materialization writes latest feature values to Redis/DynamoDB/Cassandra/Bigtable.
  • Serving service hydrates features by entity keys with strict latency budget.
  • Missing/stale feature defaults are tracked and monitored.

C. Freshness and skew checks
------------------------------------------------------------

Metrics

  • online feature p95 lookup latency.
  • feature freshness age by feature view.
  • missing feature rate.
  • training-serving skew sample checks.
  • materialization lag.
  • entity hot-key rate.

D. Interview Q&A
------------------------------------------------------------
Q: What is point-in-time correctness?
A: For a training row at time T, features must reflect only information available at or before T. Otherwise future data leaks into training.

Q: How do you handle real-time features?
A: Use streaming jobs to update online features, watermark/event-time logic for windows, TTLs to prevent stale values, and online/offline parity tests.

Q: What causes training-serving skew?
A: Different code paths, different defaults, stale online values, inconsistent time windows, schema changes, or offline features using data unavailable online.
