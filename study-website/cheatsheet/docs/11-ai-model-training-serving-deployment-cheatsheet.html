3. PyTorch training skeleton

def train_one_epoch(model, loader, optimizer, criterion, device):
    model.train()
    total_loss = 0

    for xb, yb in loader:
        xb, yb = xb.to(device), yb.to(device)
        logits = model(xb)
        loss = criterion(logits, yb)

        optimizer.zero_grad()
        loss.backward()
        optimizer.step()

        total_loss += loss.item() * xb.size(0)

    return total_loss / len(loader.dataset)

Evaluation

import torch

@torch.no_grad()
def evaluate(model, loader, device):
    model.eval()
    correct = total = 0

    for xb, yb in loader:
        xb, yb = xb.to(device), yb.to(device)
        logits = model(xb)
        pred = logits.argmax(dim=1)
        correct += (pred == yb).sum().item()
        total += yb.size(0)

    return correct / total

from accelerate import Accelerator

model, optimizer, train_loader = accelerator.prepare(model, optimizer, train_loader)

for batch in train_loader:
    optimizer.zero_grad()
    outputs = model(**batch)
    loss = outputs.loss
    accelerator.backward(loss)
    optimizer.step()

12. Batch inference

def batch_predict(model, dataloader, device):
    model.eval()
    results = []

    with torch.no_grad():
        for xb in dataloader:
            xb = xb.to(device)
            logits = model(xb)
            pred = logits.argmax(dim=1)
            results.extend(pred.cpu().tolist())

    return results

Section 22

20. Interview answer pattern

Question: Build and deploy an ML model.

Answer

I would start by defining the target metric and creating train/validation/test splits with data leakage checks. I would build a simple baseline first, track experiments with MLflow, and register the best model with metrics and artifacts. For serving, I would package the model behind a FastAPI or dedicated serving system, containerize it, deploy to Kubernetes with readiness/liveness probes, and use canary rollout. In production I would monitor latency, error rate, prediction distribution, data drift, and business metrics, and set up retraining based on drift or scheduled refresh.

Section 24

ADVANCED INTERVIEW DEEP DIVE - EXPANDED COVERAGE

A. End-to-end ML platform mental model
------------------------------------------------------------
A production AI system is data ingestion, labeling, training, evaluation, registry, deployment, monitoring, feedback, and rollback. The model is only one artifact in a larger control loop.

B. Training pipeline
------------------------------------------------------------

1. Define product objective and offline metric.

2. Collect/version data and labels.

3. Split by time/user/document to avoid leakage.

4. Train baseline.

5. Tune features/model.

6. Evaluate slices and hard cases.

7. Register model artifact and metadata.

8. Shadow/canary deploy.

9. Monitor drift and product metrics.

C. Data and feature correctness
------------------------------------------------------------
Track dataset version, label logic, preprocessing hash, feature definitions, model config, random seed, environment, and owner. Feature skew happens when online features differ from training features; this can destroy quality even with a correct model artifact.

D. Model choices
------------------------------------------------------------

  • classic ML: fast, interpretable, strong for tabular.
  • deep model: strong for unstructured data, more compute/data.
  • foundation model: flexible, expensive, requires safety/eval.
  • distilled model: lower latency/cost, good for routing/classification.
  • fine-tuned model: stable repeated behavior, needs data and governance.

E. Serving patterns
------------------------------------------------------------

  • sync online API: low-latency user path.
  • async job: long video/document/batch workflows.
  • streaming: token UX for LLMs.
  • batch inference: scheduled large workloads.
  • edge/on-device: privacy, latency, cost.

Serving stack mapping

FastAPI for simple CPU service; Triton for tensor inference; vLLM for generation; Ray Serve for Python DAG composition; Kubernetes for rollout/orchestration.

F. Registry and rollout
------------------------------------------------------------
Registry should track model URI, version, training data, metrics, tokenizer/schema compatibility, owner, approval, and deployment status. Rollout gates should include quality, latency, cost, safety, canary metrics, and rollback readiness.

G. Monitoring
------------------------------------------------------------
System: QPS, latency, error rate, saturation, queue depth, CPU/GPU/memory, OOM, tokens/sec. Model: input drift, prediction drift, calibration, groundedness, hallucination, human feedback, business KPI.

H. Cost controls
------------------------------------------------------------
Mixed precision, gradient checkpointing, LoRA/QLoRA, distillation, quantization, dynamic batching, model cascade/router, cache embeddings/prompts, right-size CPU/GPU, and use spot/preemptible for fault-tolerant batch jobs.

I. Failure modes
------------------------------------------------------------
Offline metric improves but online drops; data leakage; model/tokenizer mismatch; new model slower at p95; index/model version mismatch; canary slice not representative; stale feature definitions.

J. Interview Q&A
------------------------------------------------------------
Q: Fine-tune or prompt?
A: Prompt first. Fine-tune when the task is repeated, data exists, output style is stable, latency/cost justify smaller model, or prompting is unreliable.

Q: How deploy with rollback?
A: Immutable versions, registry metadata, canary/blue-green routing, compatibility checks, dashboards, and automated rollback on latency/error/quality regression.

Q: What is shadow deployment?
A: Copy real traffic to new model without affecting users. Compare outputs, latency, safety, and costs before canary.

Q: How reduce inference cost?
A: Route by intent/complexity, cache, batch, quantize, use smaller models, reduce context, and choose CPU/GPU by workload profile.
