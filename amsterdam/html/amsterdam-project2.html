Motivation and Product Context

The motivation was to make creative discovery feel natural. Creative users often do not search like database users. They search by intent, style, mood, visual reference, incomplete memory, or the current project context. A keyword-only system can retrieve assets with matching tags, but it struggles with style and visual similarity. A vector-only system can retrieve semantically close assets, but it can miss hard constraints such as format, license, orientation, product availability, or enterprise access control.

动机是让 creative discovery 更自然。创意用户并不像数据库用户那样搜索，他们会按 intent、风格、氛围、视觉参考、不完整记忆或当前项目上下文来搜。只靠关键词的系统可以找到标签匹配的资产，但在风格和视觉相似度上会很弱；只靠向量的系统可以找到语义接近的资产，但又可能漏掉格式、license、orientation、产品可用性或 enterprise access control 这类硬约束。

The platform had to serve both explicit search and exploratory discovery:

这个平台需要同时支持显式搜索和探索式 discovery：

  • Search: the user has a query and expects precise results.

搜索：用户有一个查询并期望得到精确的结果。

  • Similarity: the user has an example asset and wants more like it.

相似性：用户有一个示例资产并想要更多类似的资产。

  • Discovery: the user has partial or evolving intent and wants inspiration.

End-to-End System Design Walkthrough

The system has two main paths: offline/nearline ingestion and online serving.

这个系统主要有两条路径：offline/nearline ingestion 和 online serving。

Offline and nearline ingestion start when an asset is created or updated. The ingestion event enters Kafka, is normalized through stream processing, and is expanded into modality-specific tasks. Images are resized and embedded. Videos are sampled into keyframes and transcripts. Documents are parsed into text and layout features. 3D assets may be rendered into representative views. The pipeline writes metadata, feature records, lexical documents, and vector embeddings. New index versions are validated and published so serving reads a consistent view.

离线与nearline ingestion 从资产创建或更新开始：事件进入 Kafka，经由流式处理做规范化，然后被拆分为按模态的任务。图片会被缩放并生成 embedding；视频会被采样成 keyframes 和 transcript；文档会被解析出文本与版面特征；3D 资产可能会渲染成代表性视图。流水线写入元数据、特征记录、词法文档与向量 embeddings，并在验证后发布新的索引版本，使在线 serving 读取到一致视图。

Online serving starts when the user searches with text, a reference image, or a mixed prompt. The query API normalizes the request, extracts filters, computes or retrieves query embeddings, fans out to vector and lexical retrieval, merges candidates, applies policy filters, joins ranking features, and returns ranked results. Discovery surfaces may also add graph, cluster, co-engagement, diversity, and novelty candidate sources.

在线 serving 从用户提交文本、参考图像或混合 prompt 开始：Query API 规范化请求、提取过滤条件、计算或读取 query embeddings，并并行触发向量与词法检索；随后合并候选、应用策略过滤、拼接排序特征并返回排序结果。Discovery 场景还可能引入 graph、cluster、co-engagement、多样性与新颖性等候选来源。

The training and evaluation loop connects the two paths. Search and engagement logs are used to evaluate recall, ranking quality, zero-result rate, discovery diversity, and product metrics such as CTR or save/download rate. Better models, features, and indexes are rolled out through offline validation and online A/B testing.

训练和评估闭环把这两条路径连接起来。Search 和用户行为日志被用来评估 recall、排序质量、zero-result rate、discovery diversity 以及 CTR 或 save/download rate 等产品指标。更好的模型、特征和索引会通过离线验证和线上 A/B testing 逐步上线。

For search, end-to-end latency is usually a sum of several smaller latencies rather than one large model call:

对搜索来说，端到端延迟通常是多个小阶段延迟的累加，而不是一次大型模型调用的单点延迟：

  • Query parse and normalization.

查询解析和规范化。

  • Query encoding.

查询编码。

  • Vector retrieval.

向量检索。

  • Lexical retrieval.

词汇检索。

  • Metadata filtering.

元数据过滤。

  • Candidate merge and deduplication.

候选合并和重复数据删除。

  • Feature joins.

特征加入。

  • Ranking.

  • Result formatting and cache write.

结果格式化和缓存写入。

At scale, the bottleneck can move:

在规模扩大后，瓶颈会在不同阶段之间迁移：

  • If vector retrieval is slow, tune ANN parameters, sharding, filtering, and index memory layout.

如果向量检索很慢，请调整 ANN 参数、分片、过滤和索引内存布局。

  • If lexical retrieval is slow, tune mappings, filter selectivity, shard count, and query construction.

如果词汇检索很慢，请调整映射、过滤器选择性、分片计数和查询构造。

  • If feature joins are slow, cache hot features and reduce per-candidate fan-out.

如果功能连接很慢，请缓存热门功能并减少每个候选者的扇出。

  • If ranking is slow, reduce rerank depth, use a two-stage ranker, or move the heavier ranker to top candidates only.

如果排序很慢，请减少重新排序深度，使用两阶段排序器，或仅将较重的排序器移至顶级候选者。

  • If freshness is slow, separate nearline updates from full rebuilds.

如果freshness很慢，请将nearline更新与完整重建分开。

The interview point is that search performance is pipeline performance. A fast vector database alone does not guarantee a fast product search experience.

面试时最值得强调的一点是：search performance 本质上就是 pipeline performance。单独一个快的 vector database，并不等于一个快的产品搜索体验。

Hardware and Scale Constraints

Large-scale multimodal retrieval has different hardware constraints than LLM serving:

大规模多模态检索和 LLM serving 面临的硬件约束并不相同：

  • Vector search is memory and cache sensitive. Large vector indexes require careful compression, sharding, and memory layout.

向量搜索对内存和缓存敏感。大型向量索引需要仔细的压缩、分片和内存布局。

  • GPU is useful for embedding generation and heavy reranking, but online candidate retrieval is often CPU/memory/network bound.

GPU 对于嵌入生成和重排序非常有用，但在线候选检索通常受 CPU/内存/网络限制。

  • Batch embedding jobs can consume large GPU capacity but are usually less latency-sensitive than online serving.

批量嵌入作业可能会消耗大量 GPU 容量，但通常比在线服务对延迟不太敏感。

  • Elasticsearch performance depends on shard sizing, filter selectivity, heap pressure, disk layout, and query patterns.

Elasticsearch 性能取决于分片大小、过滤器选择性、堆压力、磁盘布局和查询模式。

  • Kafka and Flink throughput depend on partitioning, checkpointing, backpressure, and exactly-once or at-least-once semantics.

Kafka 和 Flink 吞吐量取决于分区、checkpoint、背压以及恰好一次或至少一次语义。

  • Metadata joins can become a hidden bottleneck if ranking requires many per-candidate features.

如果排序需要许多每个候选特征，则元数据连接可能成为隐藏的瓶颈。

  • Cross-region or multi-cloud deployment introduces consistency, replication, and latency tradeoffs.

跨区域或多云部署引入了一致性、复制和延迟权衡。

This project is a good way to show backend scalability because the hardest problems are not only ML quality. They are index freshness, sharding, tail latency, cache design, feature joins, and safe rollout at very large asset scale.

这个项目很适合展示后端可扩展性，因为最难的问题不只是 ML quality，还包括 index freshness、sharding、tail latency、cache design、feature joins 以及在超大规模资产下的安全发布。

Design and Implementation Challenges

Challenge 1: Semantic Relevance vs Exact Constraints

Creative users expect semantic understanding, but the product still needs strict constraints. A search for "transparent PNG logo" should not return a visually similar JPG with the wrong license.

挑战 1（中文理解）：

创意用户希望系统“懂语义”，但产品仍然必须满足严格约束。例如搜索“transparent PNG logo”时，不能返回视觉相似但格式/授权错误的 JPG。

Design response: 设计响应：

  • Use vector retrieval for semantic recall.

使用向量检索进行语义回忆。

  • Use lexical retrieval and metadata filters for exact constraints.

使用词法检索和元数据过滤器来实现精确约束。

  • Apply hard filters for license, moderation, tenant access, and unsupported asset types.

对许可证、审核、tenant访问和不支持的资产类型应用硬过滤器。

  • Let ranking combine soft relevance features after hard constraints are satisfied.

在满足硬约束后，让排序结合软相关特征。

Challenge 2: Scaling Vector Search Without Losing Quality

At 50B+ assets, brute-force nearest neighbor search is impossible. Approximate indexes and sharding are necessary, but they introduce recall tradeoffs.

挑战 2（中文理解）：

在 50B+ 资产规模下，暴力最近邻搜索不可行，必须使用近似索引与分片；但这会带来召回损失与参数权衡。

Design response: 设计响应：

  • Use approximate nearest neighbor indexes for candidate generation.

使用近似最近邻索引来生成候选。

  • Tune recall/latency parameters by query class.

按查询类别调整召回/延迟参数。

  • Keep reranking separate from vector retrieval.

将重新排序与向量检索分开。

  • Validate with offline Recall@K and online product metrics.

使用离线 Recall@K 和在线产品指标进行验证。

  • Use index versioning and rollback for embedding/index changes.

使用索引版本控制和回滚来进行嵌入/索引更改。

Challenge 3: Freshness vs Index Stability

Users expect new assets to appear quickly, but constant index mutation can hurt consistency and performance.

挑战 3（中文理解）：

用户希望新资产快速可见，但索引频繁变更会损害一致性与性能，需要在 freshness 与稳定性之间取舍。

Design response: 设计响应：

  • Use nearline updates for fresh assets.

对新资产使用nearline更新。

  • Use periodic full rebuilds for compaction and consistency.

使用定期完全重建来实现压缩和一致性。

  • Publish index versions atomically.

以原子方式发布索引版本。

  • Track freshness lag separately from query latency.

单独跟踪freshness滞后和查询延迟。

Challenge 4: Multimodal Alignment and Model Drift

Text, image, video, document, and 3D-derived representations must remain compatible. If model versions drift, cross-modal retrieval degrades.

挑战 4（中文理解）：

文本、图像、视频、文档与 3D 衍生表示必须保持空间兼容；一旦模型版本漂移，跨模态检索质量会显著下降。

Design response: 设计响应：

  • Store embedding version and model ID with every vector.

存储每个向量的嵌入版本和模型 ID。

  • Tie query encoder versions to compatible index versions.

将查询编码器版本与兼容的索引版本联系起来。

  • Use offline regression sets for cross-modal retrieval.

使用离线回归集进行跨模态检索。

  • Run staged backfills for major embedding upgrades.

运行分阶段回填以进行主要嵌入升级。

Challenge 5: Ranking Complexity vs Tail Latency

Ranking quality improves with more features and heavier models, but every feature join and reranker call adds latency.

挑战 5（中文理解）：

更多特征与更重的模型通常能提升排序质量，但每一次 feature join 与 reranker 调用都会增加尾延迟，必须做延迟预算与分层策略。

Design response: 设计响应：

  • Use multi-stage ranking.

使用多阶段排序。

  • Keep candidate generation broad but bounded.

保持候选人的产生范围广泛但有限。

  • Apply heavy reranking only to top candidates.

仅对顶尖候选人进行大幅重新排序。

  • Cache hot features.

缓存热门功能。

  • Degrade gracefully by reducing candidate depth or using lighter rankers under load.

通过减少候选深度或在负载下使用较轻的排序器来优雅地降级。

Stress Test 5: Personalization Without Breaking Privacy Or Diversity

Interviewer challenge:

"How would you personalize discovery for creative users without making results creepy, repetitive, or privacy-risky?" “如何为创意用户提供个性化的发现，而不会使结果令人毛骨悚然、重复或存在隐私风险？”

Strong answer:

Keep personalization bounded and explainable. Use coarse preference signals such as recent asset types, style clusters, product context, and project intent rather than sensitive raw content. Apply diversity constraints so personalization does not collapse exploration. Respect tenant/user privacy policy, provide opt-out where needed, and separate personalization features from eligibility filters.

强回答：

让个性化保持有边界且可解释。使用较粗粒度的偏好信号，例如近期 asset types、style clusters、product context 和 project intent，而不是敏感原始内容。应用多样性约束，避免个性化让探索收缩。遵守 tenant/user privacy policy，在需要时提供 opt-out，并把 personalization features 和 eligibility filters 分开。

What this demonstrates:

  • Product sensitivity and privacy judgment.

产品敏感性和隐私判断。

  • Understanding of recommendation/search tradeoffs.

了解推荐/搜索权衡。

  • Awareness of diversity and filter-bubble risks.

对多样性和过滤泡沫风险的认识。

Stress Test 6: Near-Duplicate Flood In Search Results

Interviewer challenge:

"A search result page is filled with near-identical assets. How would you fix it?" “搜索结果页面充满了几乎相同的资源。您将如何解决它？”

Strong answer:

Detect near-duplicates using perceptual hashes, embedding similarity, asset lineage, creator/source metadata, and visual clusters. Add diversity-aware reranking or result grouping so the top page covers multiple concepts, styles, colors, and compositions. Tune the diversity strength by query type: exact search may tolerate similar results, while discovery needs broader exploration.

强回答：

使用 perceptual hashes、embedding similarity、asset lineage、creator/source metadata 和 visual clusters 检测 near-duplicates。加入 diversity-aware reranking 或结果分组，让首页覆盖更多概念、风格、颜色和构图。按 query type 调整多样性强度：精确搜索可以容忍相似结果，而 discovery 需要更广泛探索。

What this demonstrates:

  • Ranking beyond relevance score.

排序超出相关性score。

  • Understanding of discovery user experience.

了解发现用户体验。

  • Ability to balance similarity and exploration.

平衡相似性和探索的能力。
