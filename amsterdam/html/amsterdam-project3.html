Amsterdam 项目 3：面向产品的模型级联与 Intent Router

This project is a product-engineering-heavy AI platform story. It uses the same Amsterdam background as Project 1 and Project 2, but the emphasis is different: instead of leading with Kubernetes, GPU scheduling, or vector databases, the project starts from product AI experiences and asks how to route each user request to the cheapest, fastest, safest model path that can still meet product quality.

这个项目是偏 product engineering 的 AI platform 叙事。它使用和 Project 1、Project 2 相同的 Amsterdam 背景，但侧重点不同：不是从 Kubernetes、GPU 调度或向量数据库讲起，而是从产品 AI 体验出发，讨论如何把每个用户请求路由到最便宜、最快、最安全，同时仍能满足产品质量要求的模型路径。

The project idea is directly aligned with a real industry trend: model routing, LLM cascading, semantic routing, intelligent prompt routing, and small-model triage. AWS Bedrock has Intelligent Prompt Routing. RouteLLM studies routing between stronger and weaker LLMs using preference data. FrugalGPT studies LLM cascades for cost/quality tradeoffs. Red Hat's LLM Semantic Router uses an Envoy external processor, embeddings, semantic cache, and routing to backend LLMs. NVIDIA NeMo Guardrails and Meta Llama Guard show another related pattern: small or specialized guard models can sit before or after larger LLMs to enforce safety, policy, and task control.

这个项目想法和真实行业趋势高度一致：model routing、LLM cascading、semantic routing、intelligent prompt routing、小模型前置 triage。AWS Bedrock 有 Intelligent Prompt Routing。RouteLLM 研究如何基于 preference data 在强模型和弱模型之间路由。FrugalGPT 研究用 LLM cascade 平衡成本和质量。Red Hat 的 LLM Semantic Router 使用 Envoy external processor、embedding、semantic cache 和后端 LLM routing。NVIDIA NeMo Guardrails 与 Meta Llama Guard 也体现了类似模式：小型或专用 guard model 可以位于大 LLM 前后，用于安全、策略和任务控制。

Two-Minute Pitch

两分钟 Pitch

Amsterdam AI workflows are diverse. A Photoshop-style generative edit, a Premiere-style video understanding request, a Lightroom-style semantic search, an Acrobat-style document summary, and an Express-style design assistant may all look like AI requests, but they do not need the same model path. Some only need intent classification or retrieval. Some need a lightweight model. Some require a large LLM or GenAI backend. Some should never leave the device or should be blocked by policy.

Amsterdam AI workflow 非常多样。Photoshop 风格的生成式编辑、Premiere 风格的视频理解请求、Lightroom 风格的语义搜索、Acrobat 风格的文档摘要、Express 风格的设计助手，从外部看都像 AI 请求，但它们并不需要相同的模型路径。有些只需要 intent classification 或 retrieval，有些需要轻量模型，有些必须走大 LLM 或 GenAI backend，有些由于隐私或策略原因不应该离开设备或应该被拦截。

The model cascade platform introduced a product-facing routing layer. It performed small-model triage at the front: classify task type, estimate complexity, detect privacy/safety risk, check semantic cache, estimate token/media cost, and decide whether to answer locally, call retrieval, use a small model, call a specialized model, or escalate to a larger model. The important engineering challenge was not only routing; it was calibration, evaluation, fallback, observability, and safe rollout. A bad router can save cost but hurt product quality, so the system needed shadow evaluation, confidence thresholds, LLM-as-Judge review, human routing labels, and product telemetry.

model cascade 平台引入了一个面向产品的 routing 层。它在前端做 small-model triage：判断 task type、估计 complexity、检测 privacy/safety risk、检查 semantic cache、估算 token/media cost，并决定本地回答、调用 retrieval、使用小模型、调用 specialized model，还是升级到更大模型。核心工程挑战不只是 routing，而是 calibration、evaluation、fallback、observability 和 safe rollout。一个差的 router 可能省成本但伤害产品质量，所以系统需要 shadow evaluation、confidence threshold、LLM-as-Judge review、人工 routing label 和 product telemetry。

1. Confidence-Aware Escalation

1. 置信度感知的升级策略

The router should not simply pick the top class. It should reason about confidence margin, calibration, product risk, and cost. If the small router is 99% confident that a query is simple retrieval, use retrieval. If it is 55% confident and the request is a paid enterprise workflow, escalate or ask clarification.

router 不应只选概率最高的类别，而应考虑 confidence margin、calibration、product risk 和 cost。如果小 router 99% 确定 query 是简单 retrieval，就走 retrieval。如果只有 55% 且请求来自付费企业 workflow，就升级或询问澄清。

Good metrics:

好的指标：

  • Expected calibration error.
  • 预期校准误差。
  • False cheap-routing rate: cases routed to small model but large model was needed.
  • 错误低成本路由率：被路由到小模型但其实需要大模型的 case。
  • False escalation rate: cases escalated unnecessarily.
  • 错误升级率：不必要升级的 case。
  • Cost per successful task.
  • 每个成功任务的成本。
  • Quality delta vs large-model baseline.
  • 相对大模型 baseline 的 quality delta。

Interview Q&A

面试 Q&A

Q: Why not send everything to the largest model?

Q：为什么不把所有请求都发给最大模型？

Answer:

回答：

Because product AI workflows have different bottlenecks. Some need retrieval, some need classification, some need edge preprocessing, some need safety blocking, and only some need large-model generation. Sending everything to the largest model increases latency, cost, and GPU pressure while sometimes reducing privacy or determinism. The router chooses the cheapest safe path that can meet quality.

因为产品 AI workflow 的瓶颈不同。有些需要 retrieval，有些需要 classification，有些需要 edge preprocessing，有些需要 safety blocking，只有一部分需要大模型生成。所有请求都发给最大模型会增加 latency、cost 和 GPU pressure，有时还会降低 privacy 或 determinism。router 选择能满足质量要求的最便宜安全路径。

Q: How do you know the small model is good enough?

Q：你如何知道小模型足够好？

Answer:

回答：

I would not trust raw confidence alone. I would calibrate the router on labeled product slices, compare against large-model or human baselines, track false cheap-routing and false escalation, and start with low-risk intents. In production, I would use shadow evaluation and canary rollout before enabling it broadly.

我不会只相信原始 confidence。我会在带 label 的产品 slice 上校准 router，与大模型或人工 baseline 对比，跟踪 false cheap-routing 和 false escalation，并从低风险 intent 开始。生产中会先做 shadow evaluation 和 canary rollout，再大范围启用。

Q: What happens if the router is wrong?

Q：如果 router 错了怎么办？

Answer:

回答：

Wrong routing has different severity. If a simple query escalates unnecessarily, cost increases but quality is safe. If a hard query is routed to a weak model, quality may drop. So I bias thresholds by risk: high-risk or low-confidence requests escalate; low-risk high-confidence requests can use smaller paths. Every route decision is logged with model versions, confidence, product surface, and outcome so we can debug and retrain.

错误 routing 的严重程度不同。如果简单 query 被不必要升级，成本增加但质量安全；如果困难 query 被路由到弱模型，质量可能下降。所以我按风险设置 threshold：高风险或低 confidence 请求升级，低风险高 confidence 请求可走小路径。每个 route decision 都记录 model version、confidence、product surface 和 outcome，用于 debug 和 retrain。

Q: Is this just a rules engine?

Q：这只是规则引擎吗？

Answer:

回答：

It starts with rules for safety and product constraints, but the core router is a learned or calibrated decision system. It can use embeddings, classifiers, preference data, production feedback, and large-model shadow labels. Rules define hard constraints; models estimate intent, complexity, and confidence.

它从 safety 和产品约束规则开始，但核心 router 是 learned 或 calibrated decision system。它可以使用 embedding、classifier、preference data、production feedback 和大模型 shadow label。规则定义硬约束，模型估计 intent、complexity 和 confidence。

Q: How is this different from Mixture of Experts?

Q：这和 Mixture of Experts 有什么不同？

Answer:

回答：

Mixture of Experts routes inside a model architecture. This project routes at the application/platform level across models, retrieval systems, caches, edge preprocessing, and policy paths. It is more visible to product constraints: privacy, latency, model cost, tenant policy, and workflow type.

Mixture of Experts 是在模型架构内部 routing。这个项目是在应用/平台层 across models、retrieval system、cache、edge preprocessing 和 policy path 做 routing。它更能感知产品约束：privacy、latency、model cost、tenant policy 和 workflow type。

Q: How would you train the router?

Q：你会如何训练 router？

Answer:

回答：

I would combine product labels, historical outcomes, model comparison data, and shadow large-model judgments. Features include prompt embedding, product surface, modality, privacy class, token/media size, retrieval confidence, prior route, and outcome. The label can be the cheapest route that met quality, or whether escalation was needed. I would evaluate by cost-quality frontier, calibration, and slice-level false-routing rates.

我会结合产品 label、历史 outcome、模型对比数据和大模型 shadow judgment。特征包括 prompt embedding、product surface、modality、privacy class、token/media size、retrieval confidence、prior route 和 outcome。label 可以是满足质量的最便宜 route，或是否需要 escalation。我会用 cost-quality frontier、calibration 和 slice-level false-routing rate 评估。

Q: How would this interact with RAG?

Q：它如何与 RAG 交互？

Answer:

回答：

The router decides whether the request needs RAG at all. For factual document or asset questions, retrieval should happen before generation. If retrieved evidence is strong and answer format is simple, a smaller model may be enough. If retrieval is weak or synthesis is complex, escalate to a stronger model. The router should also enforce citation and grounding checks before returning output.

router 先决定请求是否需要 RAG。对于 factual document 或 asset question，retrieval 应在 generation 前发生。如果 retrieved evidence 强且答案格式简单，小模型可能足够。如果 retrieval 弱或 synthesis 复杂，就升级到强模型。router 还应该在返回输出前强制 citation 和 grounding check。

Q: How do you make this product-facing instead of infra-only?

Q：如何让它听起来是 product-facing，而不是 infra-only？

Answer:

回答：

I start from the product workflow. For example, a Premiere user asking to find a moment in a video should not pay the cost of a large creative generation model. A Photoshop user doing complex generative editing may need that backend. An Acrobat enterprise user may need private RAG with citations. The routing platform maps product intent to the right model path while preserving UX, privacy, and quality.

我从产品 workflow 开始讲。例如，Premiere 用户想在视频里找某个时刻，不应该付出大创意生成模型的成本；Photoshop 用户做复杂生成式编辑可能需要那个 backend；Acrobat 企业用户可能需要带 citation 的 private RAG。routing 平台把 product intent 映射到正确模型路径，同时保持 UX、privacy 和 quality。
