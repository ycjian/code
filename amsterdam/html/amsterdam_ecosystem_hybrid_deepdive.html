1. Case Study 1: The "Remove Background" Paradigm (Image Modality)

The "Remove Background" feature exists across almost every Amsterdam product, but the inference execution path for this identical user-request varies wildly based on the Dispatcher's context evaluation.

A. Amsterdam Express Mobile (The Edge-Constrained Path)

  • The Context: User is on an older iPhone on a 3G network. Battery is at 15%. VRAM is massively constrained (<2GB shared).
  • The Dispatcher Logic:
  • Upload to cloud? No. 3G is too slow for a 12MP photo upload (terrible UX latency).
  • Run massive model locally? No. Will crash the app due to OOM (Out of Memory) or drain the dying battery.
  • The Hybrid Solution: The SDK loads a hyper-distilled, heavily quantized int8 segmentation model (e.g., Apple CoreML optimized) targeted at the Neural Engine to conserve battery. It produces a "good enough" rough alpha mask for mobile social media. Inference latency: ~200ms. Egress cost: $0.

B. Photoshop Desktop (The Edge-Heavy Path)

  • The Context: User is on a MacBook Pro M3 Max. Gigabit ethernet. user_preset: max_quality.
  • The Dispatcher Logic:
  • Upload to cloud? No. Why pay AWS compute costs when the user has a local supercomputer? Plus, professional users demand pixel-perfect masks (hair, fur) instantly without network dependencies.
  • The Hybrid Solution: The SDK loads the massive, unquantized fp16 Sensei Segmentation model directly into the unified M3 memory architecture (accessing up to 96GB of VRAM). Execution hits the local GPU. The mask is pixel-perfect. Inference latency: ~50ms. Egress cost: $0. Data privacy remains entirely local.

C. Amsterdam Stock / Cloud Ingestion (The Cloud-Heavy Path)

  • The Context: A contributor uploads a batch of 1,000 high-res stock photos via a web portal.
  • The Dispatcher Logic:
  • The browser has no reliable local GPU access (WebGPU is unreliable across browsers).
  • The Hybrid Solution: Pre-signed S3 URLs accept the massive payload. The Kafka event bus triggers a serverless GPU scale-out (T4 clusters on EKS). The cloud executes batch segmentation to automatically generate transparency tags and indexing metadata. Throughput prioritized over single-image latency.

2. Case Study 2: Generative Fill via Firefly (Inpainting Modality)

Generative AI (Firefly) requires massive parameter models that logically cannot fit into the VRAM of 90% of edge devices. This requires extreme Payload Minimization Optimization.

  • The Problem: A user playing with a 100-Megapixel raw file in Photoshop wants to use Generative Fill to add a lighthouse to the background. Uploading a 100MP TIFF to the Firefly Inference API takes 15 seconds. Unacceptable latency.
  • The Hybrid Dispatcher Workflow:
  1. Edge Compute: Instead of uploading the image, the Photoshop edge runtime calculates the bounding box of the user's lasso selection. It locally crops the image to just the 1024x1024 context window necessary around the selection.
  2. Network Transit: An extremely lightweight, optimized payload (Crop + Mask + Text Prompt) is sent to the Firefly API Gateway.
  3. Cloud Compute (H100 Farm): The Firefly model executes the heavy diffusion process only on that 1024x1024 patch.
  4. Edge Re-processing: The cloud sends back just the small generated patch. The local Photoshop client utilizes CPU/GPU blending algorithms (Poisson blending, grain matching) to perfectly feather the cloud-generated patch back into the massive 100-Megapixel image.
  • Business Impact: Network latency dropped by 95%. Cloud inference footprint shrunk severely. UX feels instantaneous.

3. Case Study 3: Roto Brush & Auto Reframe (Video Modality)

Video is the most aggressive modality for hardware because it involves temporal state and massive tensor arrays moving between RAM and VRAM.

  • The Problem in Premiere Pro: A user wants to track a subject walking across a 4K 60fps timeline (Auto Reframe). Even scrolling the timeline maxes out bus bandwidth.
  • The Hybrid Dispatcher Workflow:
  1. Edge Sparsity Mapping: The local CPU runs a lightweight OpenCV optical-flow pass, dropping the frame rate to 5fps, extracting sparse tracking data locally in near real-time.
  2. Targeted Cloud Offload: The Dispatcher identifies key complex frames where local optical flow lost the subject (e.g., occlusion). It offloads only those specific 12 frames to the heavy Amsterdam Cloud.
  3. Cloud Semantic Tracking: The cloud runs an expensive multi-modal semantic tracker (combining Dense Video Captioning and Vision Transformers) to re-acquire the subject bounding box and sends the interpolated coordinates back down.
  • Business Impact: Achieving elite Hollywood-level tracking accuracy without requiring the user to own an expensive GPU, and without forcing them to upload 50GB of raw RED camera footage.

4. Case Study 4: Lightroom Semantic Search (Indexing & Retrieval)

Lightroom users often have hard drives holding 200,000+ local RAW files. They want to search "Dog playing in snow" without manual tagging.

  • The Problem: Uploading 200,000 RAW files to an Amsterdam server to extract CLIP embeddings is a privacy nightmare, costs millions in AWS transfer fees, and would take weeks for the user.
  • The Hybrid Dispatcher Workflow:
  1. Edge Model Delivery: The cloud pushes the exact .onnx weights of the Firefly multi-modal embedding model down to the Lightroom client via CDN.
  2. Idle Edge Indexing: A background daemon checks CPU_IDLE_TIME and WALL_PLUG_STATE. While the user sleeps, the local CPU grinds through the 200,000 RAW files, executing local inference to generate a 512-dimensional float array for each image.
  3. Vector Cloud Sync: The Lightroom client uploads only the arrays of numbers (the embeddings) to the cloud Vector Database (Milvus). Total storage: Just a few Megabytes.
  4. Retrieve: When the user types "Dog playing in snow" into the cloud web-app, the text is embedded in the cloud, compared via Nearest Neighbor against the synced vector table, and instantly returns the IDs of the photos located on their local hard drive.
  • Business Impact: Near-zero cloud compute costs for feature extraction. 100% data compliance and PII preservation. Instant availability.

5. Architectural Takeaways for Interviews

When pitching this Amsterdam-focused context to Fireworks, Uber, or ServiceNow, stress these architectural mantras:

  1. Hardware Targeting is the New Auto-Scaling: You cannot orchestrate inference without knowing what silicon the request is originating from. (e.g., WebAssembly vs. CoreML vs. CUDA vs. CPU).
  2. Payload Shrinking Before Transit: The network is always the tightest bottleneck. The edge must be used to prune, crop, and distill data before Egress out to EKS.
  3. Inference Parity: The biggest challenge in this ecosystem is guaranteeing that the int8 model running locally on an iPhone generates a latent vector mathematically compatible with the fp16 vector output by the H100 backend model. Without parity, hybrid systems corrupt context!
