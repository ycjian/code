End-to-End System Design Walkthrough

An end-to-end request starts outside the platform. A product feature service receives a user action, such as asking for a generative edit, requesting semantic retrieval, analyzing a media asset, or invoking an LLM assistant. The product service does not directly call a GPU model. Instead, it sends an intent request with the payload reference, modality, privacy requirements, latency class, and device/session context.

端到端请求从平台之外开始：产品特性服务接收用户动作，例如发起生成式编辑、请求语义检索、分析媒体资产或调用 LLM 助手。产品服务不会直接调用 GPU 模型，而是发送一个 intent 请求，并携带负载引用、模态、隐私要求、延迟等级以及设备/会话上下文。

The first platform stage is admission and policy. The intent gateway validates the intent version, checks schema compatibility, resolves the owner and policy, applies quota, and attaches trace IDs. This stage is deliberately CPU-bound and lightweight because it sits on the hot path for every request.

第一阶段是准入与策略：intent gateway 校验 intent 版本、检查 schema 兼容性、解析 owner 与策略、执行配额控制，并附加 trace IDs。该阶段刻意保持 CPU 绑定且轻量，因为它处在每个请求的关键路径上。

The second stage is planning. The dispatcher uses device context, network quality, payload size, privacy class, model requirements, and live capacity signals to choose an execution mode. A small text rewrite can go directly to an LLM pool. A high-resolution media request may use edge preprocessing first. A retrieval request may call embedding inference, vector search, and reranking. A batch indexing request may be routed away from interactive GPU pools.

第二阶段是规划：dispatcher 结合端侧下文、网络质量、负载大小、隐私等级、模型要求与实时容量信号来选择执行模式。轻量的文本改写可以直接走 LLM pool；高分辨率媒体请求可能先做端侧预处理；检索请求可能会调用 embedding 推理、向量检索和 reranking；批量索引请求通常会被从交互式 GPU 池分流出去。

The third stage is workflow execution. Simple interactive requests can run synchronously. Long-running image/video/document workflows are expanded into a DAG with checkpoints. Each step produces durable artifacts or metadata so failures do not force a restart from raw input.

第三阶段是工作流执行：简单的交互式请求可以同步完成；长时间运行的图像/视频/文档工作流会被展开为带 checkpoint 的 DAG。每个 step 都会产出可持久化的 artifacts 或元数据，从而在失败时无需从原始输入重新开始。

The fourth stage is layered inference. CPU orchestration handles fan-out, feature joins, policy checks, and result composition. Triton handles high-throughput embedding, ranker, and CV inference. vLLM handles autoregressive generation. Batch workers handle lower-priority backfills and evaluation jobs. This separation is central to the design because each layer has a different bottleneck.

第四阶段是分层推理：CPU 编排层负责 fan-out、特征拼接、策略检查与结果组装；Triton 负责高吞吐的 embedding、ranker 和 CV 推理；vLLM 负责自回归生成；批处理 worker 负责低优先级的回填与评估任务。分层是设计核心，因为每一层的瓶颈都不同。

The fifth stage is resource control and rollout safety. Metrics from queues, model servers, GPU nodes, and token streams feed autoscaling and placement decisions. New model, prompt, index, or routing changes go through evaluation, canary rollout, monitoring, and rollback gates.

第五阶段是资源控制与发布安全：来自队列、模型服务、GPU 节点与 token 流的指标会驱动 autoscaling 与 placement 决策。新的模型、prompt、索引或路由策略变更都会经过评估、canary发布、监控与回滚门禁。

Performance Model

The most important performance model is that total latency is not only model latency. For hybrid AI workflows, end-to-end latency includes client preprocessing, upload time, queueing, orchestration, model inference, retrieval, postprocessing, and result transfer.

最关键的性能认知是：总延迟不等于模型推理延迟。对于混合式 AI workflows，端到端延迟包含客户端预处理、上传耗时、排队、编排、模型推理、检索、后处理以及结果回传等多个部分。

Useful mental formula: 有用的心理公式：

  • Interactive latency = admission + planning + queueing + inference + postprocessing.

交互延迟=准入+规划+排队+推理+后处理。

  • Media workflow latency = edge preprocess + upload compact artifact + cloud DAG execution + callback/poll.

媒体 workflow延迟 = 边缘预处理 + 上传压缩artifact + 云端 DAG 执行 + 回调/轮询。

  • LLM latency = admission + queue wait + prefill + time to first token + decode time per token.

LLM 延迟 = 准入 + 队列等待 + 预填充 + 第一个token的时间 + 每个token的解码时间。

  • Cost = GPU time + CPU orchestration + storage + network transfer + evaluation/backfill overhead.

成本 = GPU 时间 + CPU 编排 + 存储 + 网络传输 + 评估/回填开销。

This is why the system cannot optimize only GPU utilization or only model throughput. A GPU can be highly utilized while users still see poor p95 latency because queueing or token backlog is too high. Conversely, low GPU utilization is not always bad if the pool protects interactive latency for high-value workloads.

因此系统不能只优化 GPU 利用率或只优化模型吞吐。即使 GPU 利用率很高，用户仍可能因为排队或 token backlog 过大而看到糟糕的 p95 延迟；反过来，如果 GPU 池是在保护高价值交互请求的延迟，较低的 GPU 利用率也未必是坏事。

Hybrid AI serving is constrained by hardware as much as software:

网络带宽和上传稳定性可以主导图像/视频/文档负载的总延迟。

  • A100/H100-class GPUs are expensive and should be reserved for workloads that actually benefit from them.

A100/H100 级 GPU 价格昂贵，应保留给真正从中受益的workload。

它通常不是压缩最大单模型 GPU 吞吐量的最佳引擎。

  • Python orchestration can become a bottleneck if too much tensor work is kept there.

如果 Python 编排中保留过多的张量工作，则可能会成为瓶颈。

  • Cold starts and dependency packaging need attention.

冷启动和依赖打包需要注意。

  • Operationally, Ray clusters add another control plane on top of Kubernetes.

从操作上来说，Ray 集群在 Kubernetes 之上添加了另一个控制平面。

NVIDIA Triton

Basics: 基础知识：

  • Triton is an inference server optimized for serving models from multiple frameworks.

Triton 是一款推理服务器，针对多个框架的模型服务进行了优化。

  • It supports dynamic batching, concurrent model execution, model repositories, HTTP/gRPC APIs, and Prometheus metrics.

它支持动态批处理、并发模型执行、模型存储库、HTTP/gRPC API 和 Prometheus 指标。

  • It is widely used for embeddings, rankers, CV models, classifiers, and other non-LLM inference.

它广泛用于嵌入、排序器、CV 模型、分类器和其他非 LLM 推理。

How it fits here: 它如何适合这里：

  • Triton serves high-throughput embedding, ranking, CV, moderation, and quality models.

Triton 提供高通量嵌入、排序、CV、审核和质量模型。

  • It is useful when many small or medium tensor requests can be batched to improve GPU utilization.

当可以批量处理许多小型或中型张量请求以提高 GPU 利用率时，它非常有用。

  • It provides per-model configuration knobs such as batch size, queue delay, and instance count.

它提供每个模型的配置旋钮，例如批量大小、队列延迟和实例计数。

Tradeoffs: 权衡：

  • Dynamic batching improves throughput but can hurt tail latency.

动态批处理可以提高吞吐量，但会损害尾部延迟。

  • More model instances can improve concurrency but increase memory pressure.

更多的模型实例可以提高并发性，但会增加内存压力。

  • Triton is less natural for autoregressive token generation than vLLM.

Triton 在自回归token生成方面不如 vLLM 自然。

  • Model repository and deployment versioning must be managed carefully.

必须仔细管理模型存储库和部署版本控制。

Interview angle: 面试角度：

面试表达要点：

"对 embedding 和 ranker 模型来说，Triton 让你可以在延迟与吞吐之间做可控调参：如果 p95 延迟健康但 GPU 利用率偏低，可以通过 batching 提升吞吐；如果 p95 主要被 queue time 拖慢，更好的做法是扩容或降低 queue delay。"

vLLM

Basics: 基础知识：

  • vLLM is optimized for autoregressive LLM serving.

vLLM 针对自回归 LLM 服务进行了优化。

  • It focuses on memory-efficient KV-cache management, batching of sequences, streaming output, and high token throughput.

它专注于内存高效的 KV 缓存管理、序列批处理、流输出和高token吞吐量。

  • Important metrics include time to first token, time per output token, request backlog, tokens/sec, and KV-cache utilization.

重要指标包括第一个token的时间、每个输出token的时间、请求积压、token/秒和 KV 缓存利用率。

  • Its PagedAttention-style memory management treats KV cache like paged memory, reducing waste from variable-length sequences and making continuous batching more practical.

它的 PagedAttention-style memory management 把 KV cache 视为类似分页内存的资源，减少 variable-length sequence 带来的浪费，并让 continuous batching 更可行。

  • For production serving, prefill and decode have different bottlenecks: prefill is compute-heavy over the prompt, while decode is memory-bandwidth and KV-cache heavy as tokens are generated one by one.

在生产 serving 中，prefill 与 decode 的瓶颈不同：prefill 对 prompt 做并行计算，偏 compute-heavy；decode 逐 token 生成，更受 memory bandwidth 与 KV-cache 压力影响。

How it fits here: 它如何适合这里：

  • vLLM serves LLM assistants, query rewrite, explanation, summarization, multi-step AI workflow steps, and LLM-as-judge evaluation.

vLLM 为LLM助手、查询重写、解释、总结、多步 AI workflow步骤以及LLM-as-Judge 评估提供服务。

  • It is separated from embedding/ranking paths because generation has different capacity behavior.

它与嵌入/排序路径分开，因为一代具有不同的容量行为。

  • It provides the runtime foundation for LLM-as-a-service, but the platform still needs admission control, prompt/model routing, safety gates, quota, and rollback around it.

它提供 LLM-as-a-service 的运行时基础，但平台仍然需要围绕它实现 admission control、prompt/model routing、safety gate、quota 与 rollback。

  • Token telemetry flows back to autoscaling and routing so the system can react to actual generated-token pressure rather than raw request count.

Token telemetry 会回流到 autoscaling 与 routing，让系统根据真实 generated-token pressure 做决策，而不是只看 request count。

Tradeoffs: 权衡：

  • Long prompts and long outputs can dominate memory and latency.

长prompt和长输出可能会影响内存和延迟。

  • Request count is a poor scaling signal; token backlog is more meaningful.

请求计数是一个较差的扩展信号；代币积压更有意义。

  • Stronger models may improve quality but reduce throughput and increase cost.

更强大的模型可能会提高质量，但会降低吞吐量并增加成本。

  • Multi-tenant LLM serving requires careful admission control, max-token policies, and priority classes.

多tenant LLM 服务需要仔细的准入控制、最大token策略和优先级。

  • Prefix caching and prompt reuse can reduce repeated prefill work, but they increase cache-management complexity and need guardrails around tenant isolation and prompt-version compatibility.

Prefix caching 与 prompt reuse 可以减少重复 prefill 工作，但会增加 cache management 复杂度，并且需要围绕 tenant isolation 与 prompt-version compatibility 设置 guardrail。

  • Disaggregating prefill and decode can improve specialization at large scale, but it adds scheduling, routing, and cache-transfer complexity; for many teams, colocated serving is simpler until traffic scale justifies the split.

将 prefill 与 decode 解耦可以在大规模下提升专业化程度，但会引入 scheduling、routing 与 cache-transfer 复杂度；对很多团队来说，在流量规模足够之前，colocated serving 更简单。

Interview angle: 面试角度：

"For LLM serving I care less about raw request QPS and more about token pressure, KV-cache utilization, TTFT, TPOT, and queue behavior. Two requests can cost very different amounts depending on prompt and output length."

面试表达要点：

"做 LLM serving 时，我不会只看 request QPS，而更关注 token 压力、KV-cache 利用率、TTFT、TPOT 以及队列行为。因为两个请求的成本可能完全不同，主要取决于 prompt 长度和输出长度。"

Why Not Force Everything Through One Serving Stack?

Because the workloads have different bottlenecks:

Raw media is often the bottleneck:

Design and Implementation Challenges

Challenge 1: Avoiding A Platform That Is Too Generic

The risk of an intent platform is that it becomes too abstract. If every workflow is forced into the same schema, product teams will bypass the platform. The design needs a stable core and flexible edges.

intent 平台最大的风险是抽象过度：如果所有工作流都被强行塞进同一种 schema，产品团队会选择绕开平台。设计必须在“稳定的核心能力”和“灵活的扩展边界”之间取得平衡。

What worked: 什么有效：

  • Keep common platform concepts stable: intent, policy, execution plan, workflow state, model/index version, rollout gate, and observability.

保持通用平台概念稳定：intent、策略、执行计划、工作流状态、模型/索引版本、发布门和可观测性。

允许特定于产品的有效负载模式和后处理。

  • Treat partner teams as owners of user experience and product semantics.

将合作伙伴团队视为用户体验和产品语义的所有者。

让平台所有权重点关注执行、可靠性、路由和资源控制。

Challenge 2: Planning With Incomplete Runtime Information

Device and network context can be stale. A device can move from Wi-Fi to cellular, the user can close the app, a GPU pool can saturate after the plan is created, or a model can be rolled back.

设备与网络上下文可能会过期：设备从 Wi-Fi 切换到蜂窝网络、用户关闭应用、execution plan 创建后 GPU 池突然饱和、或模型被回滚，这些都可能发生。

Design response: 设计响应：

  • Treat execution plans as adaptive rather than permanent.

将执行计划视为适应性的而不是永久性的。

使用checkpoint，以便workflow 可以在飞行中更改路径。

  • Use live capacity signals and circuit breakers.

使用带电容量信号和熔断器。

  • Keep fallback plans in the intent policy.

将fallback计划保留在意向政策中。

  • Make each step idempotent and observable.

使每一步都是幂等且可观察的。

Challenge 3: Protecting Interactive Latency While Improving Utilization

GPU 利用率与 p95 延迟经常互相冲突：batch 作业能提升利用率，但如果不隔离，会伤害交互式 serving 的尾延迟。

Design response: 设计响应：

  • Separate latency classes.

单独的延迟类别。

  • Keep warm capacity for interactive pools.

为互动池保温。

  • Use batch/eval workloads as flexible fill capacity.

使用批处理/评估workload作为灵活的填充容量。

  • Scale on queue and latency signals, not only GPU utilization.

扩展队列和延迟信号，而不仅仅是 GPU 利用率。

  • Apply preemption or throttling when interactive pressure rises.

当交互压力上升时应用抢占或节流。

Challenge 4: Managing Model, Prompt, and Index Compatibility

AI workflows often depend on multiple versioned artifacts. A model change can require an index rebuild. A prompt change can affect evaluation behavior. A reranker change can shift product metrics.

AI workflows 往往依赖多个“可版本化的artifact”（model、prompt、index 等）。模型升级可能需要重建索引；prompt 变更会影响评估行为；reranker 变更会改变产品指标。

Design response: 设计响应：

  • Use canary rollout and rollback pointers.

使用canary发布和回滚指针。

需要离线和在线评估门。

  • Store enough evaluation context to debug regressions.

存储足够的评估上下文来调试回归。

长时间运行的多模态工作流会以不同于普通 request/response 服务的方式失败。用户通常不关心故障来自网络、对象存储、GPU 容量还是客户端预处理，他们只关心能否继续完成任务。

Design response: 设计响应：

  • Checkpoint steps.

checkpoint步骤。

  • Persist intermediate artifacts.

保留中间artifact。

  • Expose workflow state.

公开workflow状态。

  • Provide degraded results when possible.

尽可能提供降级的结果。

  • Keep failure semantics simple for product teams.

对于产品团队来说，保持故障语义简单。

Q: How did you decide edge vs cloud?

We evaluated payload size, device capability, network quality, latency SLO, privacy class, model profile, and live capacity. Large media with available local preprocessing often used hybrid execution. Tensor-heavy workloads used GPU pools. CPU orchestration handled routing and feature joins. Q: Why mention Ray Serve, Triton, and vLLM?

They solve different bottlenecks. Ray Serve is good for DAG composition and Python orchestration. Triton is good for high-throughput embedding and ranking inference with dynamic batching. vLLM is good for autoregressive generation where KV-cache and token throughput matter.

它们分别解决不同的瓶颈：Ray Serve 擅长 DAG 组合与 Python 编排；Triton 擅长通过动态 batching 做高吞吐的 embedding 与 ranking 推理；vLLM 擅长自回归生成，重点解决 KV-cache 与 token 吞吐相关的瓶颈。

Q: How did you improve GPU utilization without hurting latency?

We separated latency classes, used MIG-aware placement, kept warm capacity for interactive workloads, applied bin-packing to reduce fragmentation, and scaled on queues, p95 latency, and token telemetry rather than raw utilization alone. Batch and evaluation workloads could fill spare capacity but were preempted or throttled when interactive load increased.

我们把workload按延迟等级拆分，采用 MIG-aware placement，给交互式负载保留 warm capacity，并通过 bin-packing 降低碎片化；扩缩容信号更多依赖队列、p95 延迟和 token telemetry，而不是只看利用率。batch/评估类负载可以填充空闲容量，但在交互压力上升时会被抢占或限流。

Q: What did live token telemetry mean?

For LLM serving, requests are not equal. A short query and a long generation consume very different capacity. Token telemetry tracked waiting tokens, generated tokens, TTFT, TPOT, and KV-cache pressure so autoscaling and routing could react to actual generation load instead of request count alone.

在 LLM serving 中，“请求数量”并不等价。短请求与长生成消耗的资源差异很大。token telemetry 会跟踪 waiting tokens、generated tokens、TTFT、TPOT 以及 KV-cache 压力，从而让 autoscaling 和路由能够对真实的生成负载做出反应，而不是只看请求数。

Q: How did checkpointed execution work?

Each workflow step wrote a durable state transition and artifact reference. If a network or compute failure occurred, the workflow runner resumed from the last completed step instead of restarting from raw input. Step keys were idempotent to prevent duplicate writes.

每个 workflow step 都会写入可持久化的状态迁移记录与 artifact 引用。一旦出现网络或算力故障，workflow runner 会从最后完成的 step 继续，而不是从原始输入重跑。step key 采用幂等设计来避免重复写入。

Q: What was the hardest part?

The hard part was balancing several constraints simultaneously: product latency, privacy, GPU cost, model quality, workflow reliability, and partner-team usability. A routing decision that optimizes one dimension can hurt another, so the platform needed explicit policy, metrics, and safe fallback behavior.

最难的是同时平衡多个约束：产品延迟、隐私、GPU 成本、模型质量、工作流可靠性以及合作团队的易用性。一个优化某个维度的路由决策，可能会伤害另一个维度，因此平台必须具备明确的策略、指标体系以及安全的 fallback 行为。

I would also improve multi-region failover and make rollout guardrails more standardized across intent types.

下一步我会投入在更强的预测式容量规划、更细粒度的按 intent 成本归因、更好的路由策略变更仿真，以及端云输出兼容性的自动化验证上。同时提升多区域 failover，并让不同 intent 类型的发布 guardrails 更标准化。

Creative media workflows are not simple request/response calls. A single workflow may include upload, preprocessing, OCR/frame extraction, embedding, retrieval, reranking, generation, moderation, and result assembly. Represent long workflows as DAGs with durable step state.
