System Design deep Dive: Unified Multimodal Hybrid Engine (1 Billion User Scale)

unified_billion_scale_multimodal_system_design.md ? 7 top sections ? Billion-scale design

This document expands the Hybrid Orchestration Engine to its maximum architectural limit. We are designing a unified system capable of serving 1 Billion+ MAU (Monthly Active Users), ingesting and retrieving across 100 Billion+ diverse assets, and handling formats ranging from massive 4K video payloads to complex nested PDFs and 3D substances.

This represents the apex of Amsterdam and enterprise-scale multimodal infrastructure.

---

1. The Scale & Constraints

Designing for 1B users is not just about scaling up instances; it introduces fundamental physical and economic limits:

  • Massive Heterogeneity: Input can be a 15-page heavily formatted PDF (Document Cloud), a 30-second 4K ProRES video (Premiere), a RAW image (Lightroom), or a 3D .obj file.
  • Network & Ingress Physics: You cannot have 1 billion users uploading heavy raw files to us-east-1 synchronously. The API Gateway will melt, and AWS ingress costs will bankrupt the project.
  • The Unification Problem: How do you mathematically compare a PDF text chunk to a Video frame inside a single Vector Database?
  • Vector Storage at 100B Scale: Storing 100B dense vectors (e.g., 768 dimensions in float32) takes ~300 Terabytes of raw RAM. Standard Flat Nearest Neighbor search is impossible.

3. The Unified Ingestion Pipeline (Format-Agnostic)

The core principle at 1B user scale is Edge-First Data Pruning. Raw payload must be minimized before it hits the Cloud API.

Modality 1: PDF & Complex Documents (Acrobat AI)

  • The Edge Step: PDFs are complex byte-blobs. The Amsterdam Reader/Acrobat edge SDK locally parses the hierarchical layout tree (bounding boxes for paragraphs, images, tables). It extracts raw text streams locally.
  • The Transfer: Instead of uploading a 50MB marketing PDF, the edge SDK JSONifies the raw extracted text, table structures, and isolated raster images. Upload payload drops to 2MB.
  • The Cloud Step: A specialized Document AI worker slices the text into semantic chunks, runs an OCR pass on the raster images, and passes the chunks into the embedding model.

Modality 2: Massive Video (Premiere / Express)

  • The Edge Step: Uploading a 5GB 4K video for "semantic tagging" is forbidden. The edge SDK locally runs a cheap scene-detection heuristic and decodes the audio track.
  • The Transfer: The client uploads only 1 key-frame per scene change (e.g., 30 JPEGs total) and a highly compressed .aac audio file. Upload payload drops from 5GB to 50MB.
  • The Cloud Step: Video/Audio cluster passes the .aac to a Whisper model for transcription, and the JPEGs to a Vision Transformer.

Modality 3: 3D Substances / Meshes

  • The Cloud Step: 3D vectors require heavy spatial rendering. The asset goes directly to S3 via Presigned URL. The GPU cluster drops the mesh into a headless rendering engine, generates 6 orthogonal extremely high-res 2D preview images, and embeds those 2D images.

4. Unification: The Late-Fusion Alignment Model

How do we query "Show me docs about climate change" and get back a PDF, a Video of a glacier, and a 3D model of a wind turbine?

The "Single Embedding Space" Solution:

  1. Aligned Encoders: We use architectures like CLIP (Contrastive Language-Image Pre-Training) extended to other modalities. Text, Image, Audio, and 3D previews all pass through modality-specific encoders (towers).
  2. The Projection Head: All encoders outputs are mapped into the exact same 768-D representation space.
  3. Result: The vector math for [PDF Text Chunk: "wind turbine"] natively points to the exact same region in the hyperspace as the [3D Mesh: Turbine Object] and the [Video: Frame showing turbine].

5. Storage & Retrieval at 100-Billion Scale

Storing 100 Billion multimodal vectors requires aggressive distributed systems design.

A. Hierarchical Vector Indexing (IVF-PQ)

A flat KNN (K-Nearest Neighbors) search across 100B vectors would literally take hours per query.

  • Inverted File Index (IVF): We use K-Means clustering to divide the 100 Billion vectors into 100,000 "Voronoi clusters" (neighborhoods). When a search queries comes in, we only compare against the absolute closest 2 or 3 clusters (dropping the search space from 100B to a few million).
  • Product Quantization (PQ): To fit this in memory, we compress the 768-D float32 vectors into short 8-bit integer codes. This drops the RAM footprint of the Vector DB by ~95%, allowing massive horizontal scaling.

B. Multi-Tenant Sharding Strategy (Enterprise Compliance)

At 1 billion users, a global index poses a security risk. If a user queries for "Q3 Financials", they cannot accidentally retrieve an embedding from another enterprise client's unreleased PDF.

  • The DB Architecture: Every vector is stored with a Tenant_ID.
  • Pre-Filtering: The Vector Database (e.g., Milvus) is configured for hardware-level pre-filtering. Before the vector similarity cosine math is calculated, a hard bitmask filter completely walls off any vectors where Owner_ID != Requester_ID or ACL != Public.

6. Global Traffic Routing & Resiliency (The PoP Architecture)

When deploying to 1B users across the globe (Tokyo, Frankfurt, NY):

  1. Geo-DNS & Edge CDNs: A user uploading a heavy asset hitting api.amsterdam.com is routed via DNS (e.g., Route53/Cloudflare) to the nearest Point of Presence (PoP).
  2. S3 Transfer Acceleration: Heavy 3D/Video files utilize AWS S3 Transfer Acceleration, riding the internal fiber backbone rather than the public internet.
  3. Async Fallback (Circuit Breaking): If the us-east-1 ML cluster experiences an outage, Kafka topics simply buffer the index requests. The user sees a "Processing..." UI state instead of a hard HTTP 500 error, and the queues drain automatically once EKS nodes respawn or failover to us-west-2.
