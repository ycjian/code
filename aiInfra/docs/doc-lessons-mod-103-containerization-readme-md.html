Module 03: 容器化（Containerization） with Docker 容器（Docker）

Duration: 35 hours Difficulty: Intermediate Prerequisites: Module 01 and 02 complete

Module Overview

This module dives deep into Docker 容器（Docker） containerization specifically for machine learning applications. You'll learn to build optimized Docker 容器（Docker） images, manage ML-specific dependencies, handle GPU 加速（GPU） access in containers, and orchestrate multi-container ML systems with Docker 容器（Docker） Compose 多服务编排（Docker Compose）.

By the end of this module, you'll be proficient in containerizing any ML application, from simple inference services to complex multi-component ML pipelines.

学习产出（Learning Outcomes）

By the end of this module, you will be able to:

  1. Understand Docker 容器（Docker） Architecture - Explain how Docker 容器（Docker） works and its benefits for ML workloads
  2. Write 生产环境（Production） Dockerfiles - Create optimized Dockerfiles for ML applications
  3. Optimize Image Size - Use multi-stage builds and layer caching to reduce image sizes by 50-80%
  4. Manage ML Dependencies - Handle PyTorch 深度学习框架（PyTorch）, TensorFlow 深度学习框架（TensorFlow）, CUDA 并行计算（CUDA）, and system dependencies correctly
  5. Enable GPU 加速（GPU） Access - Configure Docker 容器（Docker） containers to use NVIDIA GPUs
  6. Use Docker 容器（Docker） Compose 多服务编排（Docker Compose） - Orchestrate multi-container ML systems (model + API + database + cache)
  7. Work with Container Registries - Push and pull images from Docker 容器（Docker） Hub, ECR, GCR, ACR
  8. Debug Container Issues - Troubleshoot common Docker 容器（Docker） problems in ML contexts

Why 容器化（Containerization） Matters for ML

The Problem Without Containers

"It works on my machine!" - The classic developer problem, amplified for ML:

  • ML models depend on specific library versions (PyTorch 2.0 vs 2.1)
  • CUDA 并行计算（CUDA） versions must match between driver, toolkit, and libraries
  • System libraries (libcudnn, libnccl) have complex dependencies
  • Python environments are fragile and version-sensitive
  • 部署（Deployment） environments differ from development environments
The Solution: Docker 容器（Docker） Containers

Docker 容器（Docker） solves these problems by:

  1. Packaging Everything - Code, dependencies, system libraries, all in one image
  2. Reproducibility - Same container runs identically everywhere
  3. Isolation - No conflicts with other applications
  4. Portability - Run on laptop, cloud VMs, Kubernetes 容器编排（Kubernetes）, anywhere
  5. Version Control - Tag and version container images like code
Real-World Impact

Example: 大语言模型（LLM） 部署（Deployment）

  • Without Docker 容器（Docker）: 2-3 days setting up CUDA 并行计算（CUDA）, PyTorch 深度学习框架（PyTorch）, dependencies on new machines
  • With Docker 容器（Docker）: 5 minutes to pull image and start serving

Example: Multi-Model Serving

  • Without Docker 容器（Docker）: Different models require different Python versions → separate VMs
  • With Docker 容器（Docker）: Run 10 models with different dependencies on the same machine

Common Pitfalls and Solutions

Pitfall 1: Huge Image Sizes

Problem: ML images often exceed 5GB Solution: Multi-stage builds, slim base images, layer optimization Target: < 500MB for CPU, < 2GB for GPU 加速（GPU） images

Pitfall 2: Build Cache Invalidation

Problem: Small code changes rebuild all layers Solution: Proper layer ordering, separate dependency installation Impact: 20 minute builds → 30 second builds

Pitfall 3: GPU 加速（GPU） Not Accessible

Problem: CUDA 并行计算（CUDA） errors inside containers Solution: NVIDIA Container Toolkit, correct base images, GPU 加速（GPU） flags Verification: nvidia-smi works inside container

Pitfall 4: Model Files Too Large for Image

Problem: Cannot fit 10GB 大语言模型（LLM） model in image Solution: Volume mounts, model registries, download on startup Alternative: Use model serving platforms with external storage

Pitfall 5: Container as Black Box

Problem: Cannot debug issues inside containers Solution: docker exec, docker logs, attach debugger Practice: Learn Docker 容器（Docker） debugging techniques
