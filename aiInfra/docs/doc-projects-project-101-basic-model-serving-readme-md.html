Project 01: Basic 模型服务（Model Serving） System

Duration: 30 hours Difficulty: Medium Prerequisites: Completed Modules 01-04

Project Overview

Build a complete model serving system that deploys a pre-trained image classification model (ResNet) as a REST API. The system will be containerized with Docker 容器（Docker）, deployed on Kubernetes 容器编排（Kubernetes）, monitored with Prometheus 指标监控（Prometheus） and Grafana 可视化（Grafana）, and deployed automatically via CI/CD 持续集成与交付（CI/CD） pipeline.

This project demonstrates fundamental skills in:

  • Model serving and inference
  • 容器化（Containerization） with Docker 容器（Docker）
  • Kubernetes 容器编排（Kubernetes） orchestration
  • 监控（Monitoring） and observability
  • CI/CD 持续集成与交付（CI/CD） for ML systems

Learning Objectives

By completing this project, you will be able to:

  1. Deploy ML Model as REST API - Serve a PyTorch 深度学习框架（PyTorch） model via FastAPI 服务框架（FastAPI）
  2. Containerize ML Application - Create optimized Docker 容器（Docker） images for ML services
  3. Deploy on Kubernetes 容器编排（Kubernetes） - Manage deployments, services, and resources
  4. Implement 监控（Monitoring） - Track custom metrics with Prometheus 指标监控（Prometheus） and visualize with Grafana 可视化（Grafana）
  5. Build CI/CD 持续集成与交付（CI/CD） Pipeline - Automate testing and deployment with GitHub Actions
  6. Document Infrastructure - Create comprehensive documentation for your system

Functional Requirements

Must Have (Required)
FR-1: REST API for Image Classification
  • POST /predict endpoint accepting image upload or URL
  • Returns top-5 predictions with confidence scores
  • API response time <100ms for small images (<1MB)
  • Proper error handling for invalid inputs
  • API documentation available at /docs (FastAPI auto-generates)
FR-2: 容器化（Containerization）
  • Dockerfile with multi-stage builds
  • Image size <2GB
  • Container runs as non-root user
  • Environment variables for configuration
  • Health check endpoint at /health
FR-3: Kubernetes 容器编排（Kubernetes） 部署（Deployment）
  • 部署（Deployment） manifest with resource requests/limits
  • Service exposing the API (LoadBalancer or NodePort)
  • ConfigMap for application configuration
  • Liveness and readiness probes configured
  • Zero-downtime updates supported
FR-4: 监控（Monitoring）
  • Prometheus 指标监控（Prometheus） scraping application metrics
  • Custom metrics: request_count, request_duration, prediction_count
  • Grafana 可视化（Grafana） dashboard showing 5-7 key metrics
  • Alert configured for service downtime
  • Metrics retained for 7+ days
FR-5: CI/CD 持续集成与交付（CI/CD） Pipeline
  • Pipeline runs on every commit to main branch
  • Automated linting and testing
  • Docker 容器（Docker） image built and pushed to registry
  • Automated deployment to Kubernetes 容器编排（Kubernetes） (optional for learning environment)
  • Pipeline completes in <10 minutes
Should Have (Recommended)
FR-6: Performance
  • API latency p95 <100ms
  • Handle 10+ concurrent requests
  • Model loaded in memory (no cold start delays)
  • CPU utilization <70% under normal load

Technical Stack

Required Technologies
  • Python 3.9+ - Application language
  • PyTorch 深度学习框架（PyTorch） - ML framework
  • FastAPI 服务框架（FastAPI） - Web framework for API
  • Uvicorn - ASGI server
  • Docker 容器（Docker） - 容器化（Containerization）
  • Kubernetes 容器编排（Kubernetes） - Orchestration (minikube or cloud)
  • Prometheus 指标监控（Prometheus） - Metrics collection
  • Grafana 可视化（Grafana） - Visualization
  • GitHub Actions - CI/CD 持续集成与交付（CI/CD）
Optional Technologies
  • TorchServe - Alternative to custom FastAPI 服务框架（FastAPI） serving
  • Nginx - Reverse proxy
  • Redis - Caching layer
  • Locust - Load testing

Common Pitfalls

  1. Docker 容器（Docker） image too large - Use multi-stage builds, slim base images
  2. Model not loaded in memory - Leads to slow inference
  3. Missing health checks - Causes deployment issues
  4. Hardcoded configuration - Use ConfigMap and environment variables
  5. Not testing container locally - Test with Docker 容器（Docker） before K8s
  6. Insufficient resource limits - Can cause OOM kills

Optimization Tips

  1. Use slim Python base image (python:3.11-slim)
  2. Cache pip dependencies in Docker 容器（Docker） layer
  3. Pre-load model at application startup
  4. Use async FastAPI 服务框架（FastAPI） endpoints for better concurrency
  5. Implement request batching if needed
  6. Add caching for repeated predictions

Extensions for Advanced Learners

Once you complete the core requirements, try:

  1. Request Batching - Batch multiple inference requests for throughput
  2. Redis Caching - Cache common predictions
  3. A/B 测试（A/B Testing） - Deploy two model versions simultaneously
  4. API Authentication - Add API key validation
  5. Helm 包管理（Helm） Chart - Package for easier deployment
  6. Horizontal Pod Autoscaling - Auto-scale based on load
  7. Distributed Tracing - Add Jaeger 链路追踪（Jaeger） for request tracing
  8. Image Optimization - Reduce Docker 容器（Docker） image to <1GB
