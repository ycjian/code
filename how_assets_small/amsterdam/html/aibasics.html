1. Transformer 与模型架构原理

【1.1】请介绍 Transformer 的结构组成、各部分作用及底层原理。

Transformer（Vaswani et al., 2017）主要由以下部分组成：

  • 输入嵌入（Input Embedding）
  • 将离散 token ID 映射为连续向量；与位置编码相加后作为模型输入。
  • 位置编码（Positional Encoding）
  • 模型本身对 token 顺序不敏感，需注入位置信息（绝对/相对/RoPE 等）。
  • Encoder（编码器栈，若干层重复）
  • Multi-Head Self-Attention：序列内 token 相互 attend，建模依赖关系。
  • 残差连接 + LayerNorm（Post-LN 或 Pre-LN，视架构而定）。
  • 前馈网络 FFN（通常两层 MLP，中间维度常为 4×hidden）。
  • 编码器常用于理解任务；机器翻译里编码器产出上下文表示。
  • Decoder（解码器栈）
  • Masked Self-Attention：保证当前位置只能看到已生成前缀（因果掩码）。
  • Cross-Attention（Encoder-Decoder Attention）：Query 来自解码器，
     Key/Value 来自编码器输出，对齐源端与目标端。
  • FFN + 残差 + LayerNorm。
  • 输出经线性层映射到词表维度，再 softmax 得到下一 token 分布。
  • 输出层
  • 常与输入共享 embedding 权重（权重绑定）以减少参数。

底层原理：用注意力替代 RNN 的递归，并行计算整句表示；Self-Attention 本质是

对值向量的加权平均，权重由 Query 与 Key 的相似度（经 softmax）决定。
【1.2】Transformer 的 forward 计算包含哪些部件？非线性由什么提供？

典型 forward 链路（单 Transformer 层，简化）：

  • 线性投影得到 Q、K、V（以及多头时的多头拆分）。
  • Attention：softmax(QK^T/√d)V（可加相对位置偏置等）。
  • 注意力输出再线性投影、残差、Norm。
  • FFN：两层（或 SwiGLU 等变体），中间含非线性激活。
  • 再次残差 + Norm。

非线性来源：

  • 注意力里的 softmax（非线性归一化）。
  • FFN 中的激活函数：ReLU/GELU/SwiGLU 等（最主要的大量非线性容量）。
  • （若用）激活后的 dropout。
【1.3】Transformer 为什么能替代 RNN？核心优势是什么？

RNN/LSTM 按时间步递归，难以并行整句计算，长程依赖易被梯度路径拉长而弱化， 且 inference 难以充分 batch。

Transformer 核心优势：

  • 并行性：Self-Attention 对长度为 n 的序列可在矩阵乘法层面并行。
  • 路径长度任意两位置直接通过注意力连接，理论上更易捕获长依赖（仍需足够层数

与位置编码）。

  • 可扩展：堆叠层数、宽度、多头可系统性放大容量；配合现代优化与正则稳定训练。
  • 推理侧配合 KV Cache 使自回归解码逐步高效。
【1.4】详细介绍 Self-Attention，包括本质、数学、步骤、复杂度。

本质：对每个位置 i，用「查询」在所有位置 j 上分配权重，对「值」加权求和， 得到融合了全局上下文的新表示。

数学（Scaled Dot-Product Attention）： 给定 X ∈ R^{n×d_model}，经线性层：

  Q = X W_Q, K = X W_K, V = X W_V
权重：α_ij = softmax_j( (q_i · k_j) / √d_k )
输出：head_i = Σ_j α_ij v_j

多头：h 个头并行，concat 后再线性投影。

计算步骤（单头）：

  • 计算 Q、K、V。
  • 评分 S = Q K^T / √d_k。
  • 可选掩码（因果/填充）。
  • softmax 得注意力权重。
  • 乘 V 得输出。
  • 多头拼接与输出投影。
时间复杂度：朴素实现主要瓶颈在 QK^T 与 attn·V。

序列长度 n、hidden d，单头维度 d_k ≈ d/h：

  • QK^T：O(n² d_k) 或视作 O(n² d)
  • softmax 与乘 V：同量级
总体常记为 O(n² d)（每层）；空间上需存 n×n 注意力 logits（若不稀疏）。
【1.5】为什么要 Multi-Head？切分为多头的作用？

单头投影到单一子空间，表达能力有限。多头使模型在不同子空间中学习多种关系 （句法、语义共指、局部短语等）。类比 CNN 多通道：不同头可关注不同位置模式； concat + 输出投影再融合。

【1.6】Softmax 前为什么要除以 √d_k？

当 d_k 较大时，点积 q·k 方差约为 d_k（假设分量独立零均值单位方差），数值变大，

softmax 进入饱和区（接近 one-hot），梯度变小、训练不稳定。缩放使方差恢复到
与维度无关的稳定量级， softmax 更平滑。
【1.7】位置编码（PE），还了解其他 PE？

原始绝对正弦/余弦编码：每个维度不同频率的正弦函数，使相对位置可通过线性变换 表达；无需学习参数（也可学习位置 embedding）。

其他常见方案：

  • 可学习绝对位置 Embedding。
  • 相对位置编码（Transformer-XL、T5）：在注意力 logits 上加相对位置偏置。
  • Rotary Position Embedding（RoPE）：旋转 Q/K，相对位置以旋转角差体现，

广泛用于 GPT-NeoX、Llama、Qwen 等。

  • ALiBi：注意力分数加与距离相关的线性偏置，外推性好。
  • 其它：XPOS、FoX 等改进外推。
【1.8】QKV 的计算；若去掉 K 变成 QQV（仅编码器内）会有什么问题？
标准：Q = XW_Q，K = XW_K，V = XW_V；注意力权重来自 Q 与 K 的匹配。

若去掉 K、用 Q 代替（相当于用 QQ^T 或与 Q 相关的对称形式）：

  • 相似度度量失去独立「键」空间：键本来可与查询学习不同语义角色；

用 Q 自匹配易导致过度对称、表达力下降，难以学习「谁该关注谁」的灵活对齐。

  • 与 V 的信息通道分离被打破：经典解释是 Q「问问题」、K「索引」、V「内容」；

去掉 K 相当于查询与索引绑定，检索灵活性变差。

  • 实践中注意力需要非平凡的 Key 空间来拟合复杂对齐模式；仅用 Q 会显著削弱

模型容量与优化 landscape。

【1.9】Encoder-Decoder vs Decoder-only（GPT）；为何生成式常舍弃 Encoder？

演变原因：

  • Encoder-Decoder：适合 seq2seq（翻译、摘要），编码器读全文，解码器自回归生成；

两端分工明确。

  • Decoder-only：「下一词预测」统一范式，用因果掩码的单栈即可建模长上下文，

架构更简单、工程扩展路径清晰（缩放定律）；大规模语料预训练目标就是 LM， 无需配对数据。

  • 舍弃 Encoder 的常见理由：单一自回归 LM 足够通过 prompt/in-context 完成多种任务；
减少 cross-attention 与编码器同步复杂度；推理管线统一（KV Cache 一侧）。
【1.10】FFN 为何演进出 MoE？

FFN 参数量占比大（常 2/3）。固定 FFN 对所有输入用同一套参数；MoE 把 FFN 拆成 多个「专家」子网络，每 token 仅激活少数专家：

  • 在总参数量大时提高「容量」而不成比例增加每步计算（稀疏激活）。
  • 大模型扩展时常见做法（GShard、Switch、Mixtral 等）。

代价：路由、负载均衡、通信、显存存放多专家时的工程复杂度。

【1.11】MoE 负载均衡怎么做？偏置 b 如何训练？如何保证变换？

典型 Switch / Top-k 路由：门控网络 g(x) 产 logits，选 top-k 专家。 负载不均会导致部分专家过载、部分闲置（训练不稳定、吞吐差）。

负载均衡常用手段：

  • **辅助损失（auxiliary load balancing loss）**：鼓励各专家分到的 token 比例

接近均匀（如重要性/负载的变分形式）。

  • **专家容量（expert capacity）**：限制每专家每步处理 token 数，溢出则溢出策略

（次选专家或跳过）— 用于避免单专家爆掉。

  • **路由 z-loss / 门控正则**：稳定 router logits，避免坍塌到单专家。
  • **Noisy top-k**：路由加噪声促进探索（早期训练）。

「偏置项 b」在不同论文中指代不同：

  • 若指 router 上的可学习 bias：与门控线性层一同反向传播，目标仍是主任务损失

+ 负载均衡损失；无 magic 变换，靠联合优化。

  • 若指 **Expert bias**（如 DeepSeek 等工作里为平衡而加在 expert score 上）：

常根据 **近期负载统计** 动态更新：利用率低的专家增大 bias 使其更容易被选中， 反之减小；使长期分布更均匀。可设计为滑动平均、或与辅助损失共同作用。

「保证变换」的工程含义一般是：bias 更新规则固定、与统计量绑定，且在训练日志中 监控各 expert 利用率曲线是否拉平。

【1.12】如何降低 Transformer 复杂度？常见稀疏注意力变体？
Full attention O(n²)。降低思路：
  • 稀疏模式：局部窗口（Sliding Window）、扩张窗口、Strided、随机块、Longformer

组合（局部+全局 token）、BigBird、ETC 等。

  • 线性 attention 近似：核方法、Performer、Linear Transformer（用特征映射近似 softmax）。
  • 分层/压缩：把序列先聚合为更短表示再注意（Poolingformer 等）。
  • 长上下文工程：FlashAttention（IO 优化非渐近复杂度但常数快）、分块计算。
【1.13】训练过程显存与计算复杂度（概要）
计算（单前向一层，量级）：O(n² d + n d²) — 注意力项与 FFN 项。

显存主要组成：

  • 参数、优化器状态（Adam：m、v）、梯度。
  • 激活：与 batch、序列长、层数成正比；Attention 中间量 n² 显著。
  • 混合精度：FP16/BF16 权重与激活；梯度 often FP32 主权重 copy。

优化：gradient checkpointing 以时间换显存、ZeRO 分片、序列并行、Context 切分等。

【1.14】Self-Attention 在多模态对齐上的瓶颈？注意力「完全偏掉」的实际场景？

瓶颈：

  • 软注意力可浅层对齐，但细粒度 grounding（像素/phrase 级）需大量数据与结构

（检测框、SAM、OCR）辅助；仅靠 attention 权重可解释噪声大。

  • 不同模态分布差异大，需投影层、对比学习、CoT、工具才能稳定。
  • 长序列多模态下 n² 成本与信息稀释。

「偏掉」场景：图像中多个相似物体、文档中近似重复段落、对抗/扰动输入、OCR 错误、 检索到错误 chunk 等，模型可能 attend 错误区域；RAG 错文档时更明显。缓解： 强 encoder、cross-attention 约束、rerank、grounding 监督、引用约束生成。

【1.15】梯度消失与梯度爆炸
  • RNN 中递归连乘导致；Transformer 仍有深层网络与 softmax 饱和问题。

手段：Pre-LN 更稳、残差、合适初始化（Xavier/固定 scale）、梯度裁剪（clip norm）、

学习率 warmup、BF16/FP16 动态 loss scale、更深的网络用 **子层 scaling**（如 Post-LN

初期不稳定时改 Pre-LN）。

【1.16】LayerNorm vs BatchNorm

BatchNorm：沿 batch 维统计 mean/var，依赖 batch 大小，train/test 行为不同； 对序列变长与自回归推理不友好（小 batch 估计抖）。

LayerNorm：沿特征维对单样本归一化，不依赖 batch 统计，适合变长序列与推理； Transformer 标配（常 Pre-LN）。

【1.17】Agent 多轮对话中 Attention 的局限性（与第 4 章呼应，简答）

上下文窗口有限，早期轮次被截断或压缩后信息丢失；长历史「中间」信息易被稀释 （lost in the middle 现象）；注意力不保证事实性，错信息会一并参与加权；多轮 指代消解仍易错；与外部记忆/检索结合才能补全。

2. 大模型训练流程与微调技术

【2.1】从 txt 预处理到 SFT 全流程
  • 文本清洗：去噪、规范化、可选 PII 处理、语言检测、质量过滤。
  • 分词 Tokenize：BPE/SentencePiece/Unigram 等，文本 → token id 序列。
  • 构造样本：instruction + input + 目标 response；设 max length，padding/truncation。
  • 批处理 DataLoader：attention mask、labels。
  • Forward：model(input_ids) → logits [B, L, V]。
  • Loss：**仅对 assistant/目标段**计算 token-level 交叉熵（见 2.6、2.7）。
  • 反向传播、梯度裁剪、优化器步进；学习率 schedule、正则、早停策略等。
【2.2】Pretrain、SFT、RLHF 的区别

| 阶段 | 目标 | 数据特点 | 解决的问题 | |-----------|----------------|--------------------|----------------------| | Pretrain | 下一 token CE | 海量无标注文本 | 语言与世界知识基础 | | SFT | 监督 CE | 指令-回答对 | 指令遵循、格式、任务 | | RLHF/DPO | 偏好/奖励 | 偏好对、评分 | 安全性、有用性、风格 |

【2.3】Pretrain 与 SFT 优化目标区别

二者常用都是交叉熵，但：

  • Pretrain：对整个文档连续预测下一词，最大化 p(x)；无「指令边界」。
  • SFT：最大化 p(y|x)（条件生成），且常 mask 掉 prompt 上 loss，只对回答部分优化。
【2.4】SFT 核心流程与数据策略

流程：数据收集 → 清洗与去重 → 标准化模板 → 长度与配比控制 → 训练。 多样性：多任务源、多语言、难度混合、拒绝与安全样本、工具调用格式。 质量：人工抽检、模型辅助打分、困惑度过滤、毒性过滤、与基准对齐。

【2.5】【2.6】SFT Loss；10 或 100 个 target token
通常 token-level CrossEntropyLoss（忽略 pad，忽略 prompt 位置）。

若有效 target token 数为 T（10 或 100），常见两种口径：

  • **mean**：loss = (1/T) Σ CE(t) — 对长度归一，长短样本权重可比。
  • **sum** 再 batch 内 mean：实现时需一致。
批级别：`loss = CE.reshape(-1, V); labels` 上 `-100` 忽略。
【2.7】SFT 与 Pretrain 在 Loss 上的区别；如何屏蔽 Prompt loss？

Pretrain 对所有非 pad token 算下一词预测。 SFT 仅在「应答」部分计算：把 prompt 对应位置的 labels 设为 `ignore_index` （如 -100），使 CE 不累计这些位置。

【2.8】手写 SFT loss（shift right）

语言模型预测 token t 需 logits[t] 对比 label t（即输入右移一位）。 设 logits 形状 [B, L, V]，labels 为 [B, L]：

  • 方式 A：logits[:, :-1] 对齐 labels[:, 1:]
  • 方式 B：inputs 右移作为 labels

PyTorch 示例：

```python

import torch
import torch.nn.functional as F
def sft_loss(logits, labels, ignore_index=-100):
    # logits: [B, L, V], labels: [B, L]，已把 prompt 标为 ignore_index
    shift_logits = logits[:, :-1, :].contiguous()
    shift_labels = labels[:, 1:].contiguous()
    loss = F.cross_entropy(
        shift_logits.view(-1, shift_logits.size(-1)),
        shift_labels.view(-1),
        ignore_index=ignore_index,
    )
    return loss

```

【2.9】SFT 后 Post-Training 有哪些？目的区别
  • **RLHF/PPO**：用奖励模型 + KL 约束细调策略。
  • **DPO/ORPO/IPO**：纯偏好数据，无需显式 RM 采样。
  • **RAFT/Rejection sampling**：偏好或拒绝样本上继续 SFT。
  • **Safe RLHF、Constitutional AI**：安全与原则。
  • **长文、工具、代码** 专用阶段：继续 SFT 或 RL。

目的：SFT 仅是模仿；偏好阶段对齐人类价值、减少有害、提高 win rate。

【2.10】为什么 SFT 后还要 RL？为何不全用偏好数据 SFT？

偏好数据常是 **chosen/rejected**，非逐 token 监督；简单 SFT 在 chosen 上训练 仍有分布窄、过拟合演示、无法显式惩罚差行为。RL/ DPO 类可直接优化 **偏好对** 或 **奖励**，并加 KL 保持接近原模型防崩。RL 还能利用非可微反馈（规则、检索 F1）。

【2.11】LoRA 原理；秩 r 影响；如何选 r
LoRA：冻结原权重 W，低秩分解 ΔW = B A（B∈R^{d×r}, A∈R^{r×k}），只训 A,B。
前向：W x + ΔW x = W x + B(A x)。

r 小：参数少、易训、可能欠拟合；r 大：表达力强、易过拟合与显存升。 选型：从小到大扫（8,16,32,64）、看验证集、参数量预算；任务难/数据多可略增大 r。

【2.12】LoRA 只能插 Linear？为何不在 LayerNorm 后？稳定性
主要用于 Linear（QKV/O/FFN）；也有 Conv、部分工作扩到其它层。

LayerNorm 后是 **归一化后** 小尺度稳定子空间，放 LoRA 易与 scale 耦合、 改变矩统计，破坏预训练尺度；通常对 LN 参数直接 full finetune 或不改。

插在 Linear 前后有论文讨论；一般 **Attention 和 FFN 的线性层**收益最大。

【2.13】推理是否挂 Adapter？合并与梯度爆炸
推理可 **合并**：W' = W + B@A，一次 matmul，无额外延迟。

合并是 **加法**，若训练时 LoRA 未爆炸，合并通常不引入新爆炸；若训练发散需 调 rank、scale（如 α/r）、学习率。

【2.14】QLoRA；量化方式；NF4+FP16；NF4 拟合逻辑

QLoRA：4bit 量化基座权重 + 分页优化器 + LoRA 在 FP16/BF16 训练适配器。 常见量化：INT8 PTQ/QAT；GPTQ/AWQ 权重量化；NF4（4bit 正态分位）。

**为何 NF4+FP16**：NF4 对权重大致正态假设的信息量最优；计算与 LoRA 用 FP16/BF16 保梯度稳定。

**NF4**：将权重量化为固定 16 个 level，level 由标准正态分位数确定，使期望下 最大化与全精度分布匹配；配合双重量化进一步压显存。

【2.15】LoRA 冻结层选择

小数据：训 **q_proj, v_proj** 或 **q,k,v,o**；数据足可扩到 FFN。 经验：排行榜常 **全部 attention + 一半 FFN**；依据：下游任务复杂度、数据量、 灾难性遗忘容忍度。

【2.16】Tokenizer；Embedding 实现

Tokenizer：词级/子词（BPE、WordPiece、SentencePiece Unigram）；流程 normalize → pretokenize → 合并规则 → id。

Embedding：nn.Embedding(num, dim) 查表；或 one-hot × 矩阵（等价）。 共享权重：输出层与输入 embedding tie。

【2.17】控制生成多样性

temperature（缩放 logits）、top-k、top-p、重复惩罚 repetition_penalty、 presence/frequency penalty、beam search（多样性差）、约束解码。

【2.18】top-k vs top-p；其它生成策略

top-k：保留概率最高的 k 个再归一化采样。 top-p（nucleus）：累积概率达 p 的最小集合。

其它：beam search、典型解码 typical sampling、对比解码 contrastive decoding、 speculative decoding（加速）。

【2.19】优化算法；为何「Adam 不一定最优而 SGD 最优」？

**一种常见面试表述的解读**：在大规模训练中，SGD+Momentum 有时在 **泛化** 上 更优（锐极小值等讨论，视领域与经验）；Adam 收敛快但对超参敏感。并非绝对定理。

Adam：自适应学习率、稀疏梯度友好；内存大（二阶矩）。 AdamW：解耦权重衰减。 Lion、Sophia 等为新选择。

【2.20】长尾与多峰
长尾：重采样、class-balanced loss、focal loss、合成数据。

多峰：混合模型、显式latent、多任务头、对比学习拉开模态。

【2.21】冷启动；LLM 的作用

规则/模板、知名单物品、迁移学习、主动学习、仿真数据。 LLM：few-shot prompt、合成用户、摘要画像、对话策略冷启动内容生成（需审核）。

3. 强化学习与人类对齐 (RLHF)

【3.1】PPO、DPO、GRPO — 定义、区别、优缺点、场景

**PPO（Proximal Policy Optimization）**

  • 在线策略优化，裁剪目标限制新策略相对旧策略步长；常配合 RM 与 ref 模型 KL。
  • 优点：稳定、通用 RL；缺点：训练重、需采样与多模型。

**DPO（Direct Preference Optimization）**

  • 用偏好数据直接优化策略，等价于隐式最优策略与 BT 模型；无需显式 RM 采样 loop。
  • 优点：简单、偏好数据友好；缺点：分布依赖、长响应与复杂偏好表达受限。

**GRPO（Group Relative Policy Optimization，如 DeepSeek-R1 路线）**

  • 组内样本相对奖励归一化，强化可扩展 RL；常与 rule-based/outcome reward 配合。
  • 优点：减少绝对 RM 偏差、利于推理任务；缺点：设计组采样与方差。
【3.2】PPO Clip；为何 Clip 内外还要 Mean？
比率 r = π_new/π_old，目标 min(r A, clip(r,1-ε,1+ε) A)。

**Mean**：小批量上估计期望；clip 约束单次更新幅度。

【3.3】除 Clip 外限制分布差异的方法

KL penalty（显式加到奖励）、trust region（TRPO）、δ 约束、early stopping、 reference model、decoupled clipping、奖励归一化。

【3.4】PPO vs GRPO 结构与场景

PPO：一般优势函数 + clip，适合 RM 打分场景。 GRPO：组归一化奖励，适合数学/代码等规则奖励、同一 prompt 多样本。

【3.5】DAPO、GSPO 改进（概要）

此类名称随论文演进：**动态采样 / 序列级目标 / 策略梯度 reformulation** 等， 用于稳定长序列 RL、减轻梯度噪声或改进 off-policy 偏差。面试答：关注「序列级 奖励 credit assignment、裁剪与温度、批次构建」三类改进即可。

【3.6】奖励坍塌（reward hacking / collapse）

模型钻 RM 漏洞（冗长讨好、格式作弊），或 RM 本身偏差导致高分低质。 缓解：多目标 RM、人工审计、KL、多样化 prompt、对抗训练 RM、规则检验。

【3.7】多目标奖励冲突

加权求和（调权）、Pareto / 约束优化、分层 RL、分段训练、可用 **Chebyshev** 或 **scalarization**；产品上用显式优先级（安全>事实>风格）。

【3.8】离线 vs 在线；RLHF 属于哪类

离线：固定数据集；在线：与环境交互采新数据。 **RLHF 常混合**：基于静态偏好做 DPO（偏离线）；PPO 用当前策略采样再更新（在线成分）。

【3.9】Reference Model 作用

冻结的 SFT 模型，计算 KL(π||π_ref) 防止策略漂太远、遗忘、模式塌缩。

【3.10】KL 散度与估计
离散：KL(P||Q) = Σ P(x) log(P(x)/Q(x))。
连续/采样：Monte Carlo、K3 估计、近似用 **log ratio 的期望**；
训练里常对 batch 平均 `log π - log π_ref`。

```python
import torch
def top_k_filter(logits, k):
    if k <= 0:
        return logits
    values, _ = torch.topk(logits, k, dim=-1)
    min_values = values[..., -1, None]
    return torch.where(logits < min_values, torch.full_like(logits, float('-inf')), logits)
def top_p_filter(logits, p):
    sorted_logits, sorted_idx = torch.sort(logits, descending=True, dim=-1)
    probs = torch.softmax(sorted_logits, dim=-1)
    cumprobs = torch.cumsum(probs, dim=-1)
    mask = cumprobs - probs > p
    sorted_logits[mask] = float('-inf')
    return torch.zeros_like(logits).scatter_(-1, sorted_idx, sorted_logits)

```

```python
def softmax(x, dim=-1):

def cross_entropy(logits, target, ignore_index=-100):
    log_p = torch.log_softmax(logits, dim=-1)
    mask = target != ignore_index
    ce = -log_p.gather(-1, target.clamp(min=0).unsqueeze(-1)).squeeze(-1)
```

【8.5】QKV、Self-Attention、MHA

```python

import torch
import torch.nn as nn
import math
def scaled_dot_attention(Q, K, V, attn_mask=None):
    d_k = Q.size(-1)
    scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(d_k)
    if attn_mask is not None:
        scores = scores.masked_fill(attn_mask, float('-inf'))
    attn = torch.softmax(scores, dim=-1)
    return torch.matmul(attn, V), attn
class MultiHeadAttention(nn.Module):
    def __init__(self, d_model, num_heads):
        super().__init__()
        assert d_model % num_heads == 0
        self.d_head = d_model // num_heads
        self.num_heads = num_heads
        self.Wq = nn.Linear(d_model, d_model)
        self.Wk = nn.Linear(d_model, d_model)
        self.Wv = nn.Linear(d_model, d_model)
        self.Wo = nn.Linear(d_model, d_model)
    def forward(self, x, attn_mask=None):
        B, T, _ = x.shape
        Q = self.Wq(x).view(B, T, self.num_heads, self.d_head).transpose(1, 2)
        K = self.Wk(x).view(B, T, self.num_heads, self.d_head).transpose(1, 2)
        V = self.Wv(x).view(B, T, self.num_heads, self.d_head).transpose(1, 2)
        out, _ = scaled_dot_attention(Q, K, V, attn_mask)
        out = out.transpose(1, 2).contiguous().view(B, T, -1)
        return self.Wo(out)

```

【8.6】RoPE（旋转位置编码，核心思想）

对 Q、K 每对维度施 2D 旋转，角度随位置 m 增大；相对位置 k 体现为角度差 θ_k。 简写：x 分偶奇两两成对，乘以 cos/sin（位置 m 对应角度）。

```python

import torch
def precompute_freqs(dim, seq_len, theta=10000.0, device=None):
    inv_freq = 1.0 / (theta ** (torch.arange(0, dim, 2, device=device).float() / dim))
    t = torch.arange(seq_len, device=device)
    freqs = torch.einsum('i,j->ij', t, inv_freq)
    return torch.cat((freqs, freqs), dim=-1)
def apply_rope(x, freqs):
    # x: [..., dim], freqs: [seq, dim]
    x1 = x[..., ::2]
    x2 = x[..., 1::2]
    freqs_cos = freqs[..., ::2].cos()
    freqs_sin = freqs[..., ::2].sin()
    y1 = x1 * freqs_cos - x2 * freqs_sin
    y2 = x1 * freqs_sin + x2 * freqs_cos
    return torch.stack((y1, y2), dim=-1).flatten(-2)

```

【8.7】PPO / DPO / GRPO — Loss 轮廓

**DPO**（σ 为 sigmoid，β 为温度）：

  L = - E[ log σ( β ( log π_θ(y_w|x) - log π_θ(y_l|x)
                     - log π_ref(y_w|x) + log π_ref(y_l|x) ) ) ]

**PPO**：

  L_clip = E[ min( r A, clip(r,1-ε,1+ε) A ) ]，常再减 β KL

**GRPO**（示意，组内相对奖励 R_i）： 对同一 prompt 一组输出 {y_i}，用 R_i - mean(R) 或标准化作优势，再策略梯度 / PPO 式裁剪；具体以论文实现为准。

二、Traditional ML — 原理、优缺点、Loss（公式与直觉/推导要点）

【2.1 线性回归 Linear Regression】
模型: y = w^T x + b
Loss (MSE): L = (1/N) Σ (y_i - ŷ_i)^2
解析解（正规方程）: w* = (X^T X)^{-1} X^T y （需 X^T X 可逆；大数据用梯度下降）
梯度: ∂L/∂w = (2/N) X^T (Xw - y)
Pros: 可解释、训练快、baseline 强。
Cons: 线性可分假设；异常值敏感（MSE 对大误差惩罚平方级）。
【2.2 Logistic Regression（二分类）】
模型: P(y=1|x) = σ(w^T x + b), σ(z)=1/(1+e^{-z})
Loss（对数损失 / 负对数似然，单样本）:
ℓ = -[ y log p + (1-y) log(1-p) ]
梯度（对 logit z = w^T x + b）:
∂ℓ/∂z = p - y → ∂ℓ/∂w = (p-y) x （链式法则）
Pros: 概率输出、校准好、线性可解释。
Cons: 线性决策边界（需显式特征交叉或核；工业上常用特征工程补救）。
【2.3 Softmax + 交叉熵（多分类）】
logit: z ∈ R^K, p_k = exp(z_k) / Σ_j exp(z_j)
CE: ℓ = -Σ_k y_k log p_k （one-hot y）
性质: 与最大似然等价；梯度对 z 有简洁形式（与 softmax 组合时稳定用 log-sum-exp）。
【2.4 支持向量机 SVM】

思想: 最大间隔；可核方法处理非线性。

Hinge loss（软间隔）直觉: max(0, 1 - y f(x))；关注“难例”。
Pros: 小数据/高维稀疏有时很强；理论成熟。
Cons: 大数据训练贵；概率需 Platt scaling 等后处理。
【2.5 决策树 Decision Tree】

分裂准则: Gini / Entropy（信息增益）；回归树用 MSE 等。

Pros: 非线性、可解释、缺省值易处理。
Cons: 高方差、易过拟合 → 集成（RF / GBDT）是主战场。
【2.6 随机森林 Random Forest】

Bagging + 随机子特征；降低方差。

【2.7 KNN】
Pros: 无训练、非参数。
Cons: 推理慢、维度灾难；需标准化。
【2.8 朴素贝叶斯】

条件独立假设；文本分类经典 baseline。

Loss 小结（面试常问“为什么用这个 loss”）
  • MSE: 高斯噪声假设下的 MLE；对大误差惩罚重。
  • MAE: 对异常值更鲁棒；零点处不可微（次梯度）。
  • Huber: MSE 与 MAE 折中。
  • Logloss / CE: 分类概率建模；与 MLE 一致。
  • Focal Loss（分类，γ≥0）: FL = -(1-p_t)^γ log(p_t)，p_t 为正确类概率；

γ=0 退化为 CE；γ 越大越强调难分类样本（易例 down-weight）。

【2.9 正则与 Bias–Variance】

L2: λ||w||_2^2 → 权重衰减、平滑。 L1: λ||w||_1 → 稀疏（特征选择）。 Elastic Net（常见写法之一）: minimize ||y - Xw||_2^2 + λ [ α||w||_1 + (1-α)||w||_2^2 ] α∈[0,1]：α=1 退化为 Lasso；α=0 为 Ridge。相关特征成组时常比纯 L1 稳。

【2.10 Huber Loss（回归鲁棒）】
对残差 a = y - ŷ:
L_δ(a) = a^2/2 若 |a| ≤ δ
L_δ(a) = δ(|a| - δ/2) 若 |a| > δ

大残差上从平方惩罚退化为线性，减轻离群主导；δ 控制过渡点。

【2.11 数值小例：Logistic 单步梯度下降】

设二分类 CE，单样本 x=[1,2]^T, y=1, 当前 w=[0,0]^T, b=0。

z = w^T x + b = 0, p=σ(0)=0.5。
∇_w ℓ = (p - y) x = (0.5-1)·[1,2]^T = [-0.5, -1]^T。
若学习率 η=0.1，更新 w ← w - η∇_w ℓ = [0,0] - 0.1·[-0.5,-1] = [0.05, 0.1]。

（口述：错得自信时梯度大；对 w 的梯度来自链式法则 (p-y)x。）

【2.12 Multinomial Logistic（与 softmax 一致）】
P(y=k|x) ∝ exp(w_k^T x + b_k)；CE 同 §2.3；“one-vs-rest”不是此形式，勿与 softmax 多分类混淆。

三、Deep Learning 基础

【3.1 前向网络 MLP】
层: h^{(l)} = φ( W^{(l)} h^{(l-1)} + b^{(l)} )

通用近似定理: 宽度足够大可逼近连续函数（条件与限制了解即可）。

【3.2 反向传播】
链式法则: ∂L/∂W = (∂L/∂h)(∂h/∂W)；计算图上从输出向输入传播。
【3.3 梯度消失 / 爆炸】

原因: 深链式连乘许多 <1 或 >1 的项；sigmoid/tanh 饱和区导数很小。 缓解: 更好的初始化（Xavier/He）、残差、BN/LN、ReLU 系、梯度裁剪、更短路径（ResNet）。

【3.4 批量归一化 BN（批维）】

对 batch 通道归一化后仿射变换；训练用 batch 统计，推理用滑动平均。 作用: 稳定训练、允许更大 LR；但对小 batch / RNN 不友好。

【3.5 Dropout】

训练随机置零；推理按比例缩放或 weight scaling。 解读: 类似 bagging / 正则。

【3.6 激活函数】
Sigmoid: σ(x)=1/(1+e^{-x}), σ' = σ(1-σ)。输出(0,1)，饱和→消失；非零中心。

Tanh: 输出(-1,1)，仍可能饱和。 ReLU: max(0,x)，导数 1_{x>0}。稀疏激活动、算快；死神经元（永不更新）。 Leaky ReLU / PReLU / ELU: 缓解死 ReLU。 GELU: 平滑，Transformer 常用。

【3.7 Optimizers】
SGD + Momentum: v_t = μ v_{t-1} + g_t; θ -= lr * v。

RMSprop / Adagrad: 自适应逐参数步长。 Adam: 一阶矩 + 二阶矩估计 + bias correction。

  m_t = β1 m_{t-1} + (1-β1) g_t
  v_t = β2 v_{t-1} + (1-β2) g_t^2
m̂_t = m_t/(1-β1^t), v̂_t = v_t/(1-β2^t)
θ -= lr * m̂_t / (sqrt(v̂_t)+ε)

AdamW: 解耦权重衰减（不把 L2 混进自适应梯度里），Transformer 常用。

面试要点: Adam 对学习率不那么敏感；大数据 long-run 有时 SGD+Momentum 泛化更好（讨论较多，回答时提“任务依赖、调参、wd 策略”）。

【3.8 学习率调度】

Step/cosine/warmup；Transformer 常 warmup + cosine decay。

【3.9 参数初始化（Xavier / He，方差口径背一句）】

目标: 让前向每层激活方差、反向梯度方差不随深度指数爆/缩（简化分析下）。

Xavier/Glorot（tanh/sigmoid，常见一种）:

  fan_in = 输入维度, fan_out = 输出维度

W ~ Uniform[-a, a] 或 Normal, 其中 a ∝ sqrt(6 / (fan_in + fan_out)) （或有的实现用方差 2/(fan_in+fan_out) 的 Gaussian）

He（ReLU）: 若用正态，常取 Var(W)=2/fan_in（ReLU 约一半激活为 0，补偿方差减半）。

【3.10 Dropout（inverted dropout 训练式，与推理一致）】
训练: 掩码 m~Bernoulli(p) 逐元素；y = (m⊙a) / p。

推理: 不 dropout；于是 E[y_train] 与推理前向一致（期望匹配）。 若不除以 p（非 inverted），推理需乘 p 对齐期望。

【3.11 梯度裁剪（RNN/大模型常用）】

global norm: g ← g · min(1, clip / ||g||_2)；防一步炸飞；不改正负号方向。

五、Transformer 系列（核心组件）

【5.1 Attention 直觉】

Query “查什么”，Key “索引标签”，Value “内容”。相似度高的 key 对应的 value 权重大。

【5.2 Scaled Dot-Product Attention】
Attention(Q,K,V) = softmax( (Q K^T) / sqrt(d_k) ) V
为什么 scale: 防止 d_k 大时点积方差大，softmax 极端化（梯度不稳）。
【5.3 Self-Attention】

Q,K,V 均来自同一序列（不同线性投影）；全局依赖一步建模；并行好。

【5.4 Multi-Head Attention】

多组 (W^Q_i, W^K_i, W^V_i) 学不同子空间关系；拼接再投影。

【5.5 Masked Attention】

因果语言模型: 掩掉未来位置，避免信息泄露（上三角 -inf）。 Padding mask: 忽略 pad token。

【5.6 Positional Encoding】

因自注意力置换等变性，需注入位置。 学习式或固定正弦:

PE(pos,2i) = sin(pos / 10000^{2i/d})
PE(pos,2i+1) = cos(pos / 10000^{2i/d})

RoPE / ALiBi 等为后续改进（可按岗位深度准备）。

【5.7 Residual Connection】

x + SubLayer(x)；缓解退化、梯度路径更短。

【5.8 Layer Normalization】

对特征维归一化（与 BN 维度不同）；Transformer 默认 Pre-LN 或 Post-LN（实现各异）。

【5.9 代码级：因果 mask（numpy 示意）】

scores 形状 (L, L)，第 i 行对所有 key j 的 logits（实现里常广播生成）:

import numpy as np
from scipy.special import softmax
L = scores.shape[0]
mask = np.triu(np.ones((L, L)), k=1) * (-1e9)   # 上三角：未来位置
scores = scores + mask
attn = softmax(scores, axis=-1)                 # 每行对列 softmax
解释: softmax(-∞)→0，位置 i 对 j>i 的注意力权重为 0。
【5.10 Padding mask】
batch 内不等长序列 pad 到同一长度；对 pad 位置在 softmax 前把 score 置 -∞，
或在注意力权重上乘 valid_mask∈{0,1}（实现须与 softmax 数值稳定写法一致）。

六、XGBoost 原理（面试高频）

本质: 梯度提升树（GBDT）的高效实现 + 正则 + 近似算法。

【6.1 加法模型】
ŷ^{(t)} = ŷ^{(t-1)} + f_t(x)，f_t 为一颗 CART。
【6.2 目标（二阶 Taylor）】
在第 t 步近似 loss 为关于 ŷ^{(t-1)} 的一二阶导数（g_i, h_i），

【6.4 Logistic loss 下的 g_i, h_i（口述）】

七、模型量化（简要）

动机: 推理更快、更小、更低功耗。 类别: PTQ（训练后校准）vs QAT（训练中量化）。 对称整型量化（直观公式）:

八、CNN / ResNet / ViT / BERT（训练方式要点）

【9.0 第一步：能否记住训练集？】

Checklist（按顺序快速扫）:

  • 数据: label 对错、shuffle、归一化、train/eval 分布、batch 是否全一类。
  • Loss: 分类是否用 CE、是否 softmax+log 数值稳定；回归尺度是否过大。
  • 模型容量: 太弱拟合不动 vs 太强难优化。
  • LR: 过大震荡/NaN，过小几乎不动；试 LR range test / 减半或 warmup。
  • 初始化与死亡 ReLU: 换激活、换初始化。
  • BatchNorm: batch 太小统计不稳；推理模式是否弄错。
  • 梯度: clip；检查是否梯度全 0（dead graph、detach 误用）。
  • 正则过强: dropout/L2 过大。
  • 代码 bug: softmax 二次、维度 transpose、loss reduction（mean vs sum）。
  • 优化器状态: Adam 加载错 checkpoint、β 累积错误；混合精度 scaler 未 scale/unscale。
验证 loss vs train loss:
  • 二者都高: underfitting / 数据难 / 模型太小。
  • train 低 val 高: overfitting。
  • 都不收敛: 优先查 LR、bug、标签。

十四、CHEAT SHEET（面试前 10 分钟速扫）

[A] 任务 ↔ 指标（默记）
二分类排序能力 → ROC-AUC（概率排序，与阈值无关；类分布变化时仍常稳定）
不平衡 + 关心正类 → PR-AUC / F1（看 precision-recall trade-off）

需要可解释代价 → confusion matrix + cost matrix → 调阈值 多分类平衡 → macro-F1；长尾/不平衡 → weighted-F1 回归默认 → RMSE / MAE；异常值敏感 → Huber / MAE

排序学习 → NDCG / pairwise loss（了解名词）
[B] 激活函数（形状 / 导数 / 何时用）

Sigmoid : (0,1), σ'=σ(1-σ), 饱和/非零均值 → 多用输出层概率或二分类 legacy Tanh : (-1,1), 仍饱和 → RNN 时代常见；深层不如 ReLU 系 ReLU : max(0,x), 导数 1_{x>0} → 默认首选；小心 dead ReLU LeakyReLU: x<0 小斜率 α → 减轻死亡 GELU/SiLU: 平滑 → Transformer / 现代 CNN

Softmax : 向量→单纯形 → 多分类输出分布（数值用 log-softmax）
[C] 初始化（口径）

Xavier/Glorot: 适合 tanh/sigmoid（保持方差层级不过度放大/缩小） He init : 适合 ReLU（考虑 ReLU 干掉一半激活） 大意: 坏的初始化 → 前几层输出方差漂移 → 梯度难传播

[D] Normalization（BN vs LN）

BN : 沿 batch 维 Norm（CNN 常用）；batch 小不稳；train/infer 行为不同 LN : 沿特征维 Norm（Transformer/RNN）；batch 大小不敏感；train/infer 一致性强 备注: Transformer 常见 Pre-LN（更稳定）vs Post-LN（原版）

[E] Optimizer 一句话

SGD+Momentum: 动量平滑噪声，利于泛化讨论 Adam : 自适应 LR；默认 β1=0.9, β2=0.999 AdamW : 权重衰减解耦；Transformer 标配之一 误区 : Adam 不是“不用调 LR”，大模型仍需 warmup / wd

[F] Tree/Boosting 三角

RF : Bagging → 降方差；训练并行度好；精度常不如 boosting GBDT : Boosting → 逐步修正残差；表格数据极强 XGB/LGB/Cat: 工程实现差异（分裂策略、类别特征、直方图、GPU）；面试能说 3 条差异即可

[G] Attention 复杂度
序列长度 n，隐层 d：self-attention 主体 O(n^2 · d)（两两 token）
推理加速关键词: KV cache（自回归）、FlashAttention（IO-aware）
[H] 泄露（leakage）红线

· 用全量数据 fit StandardScaler 再切分 → 泄露（必须在 train 内 fit） · 时间序列随机 KFold → 泄露（应用 TimeSeriesSplit / purged CV） · 目标编码用未来标签 → 泄露

十五、Deep Dive — 推导与“能白板写”的细节

【D1】二分类 Logistic：MLE → log loss
设 y∈{0,1}，p=σ(z)，似然 L_i = p^y (1-p)^{1-y}
log L_i = y log p + (1-y) log(1-p)
负对数似然 NLL = -log L_i，对全体样本求和即交叉熵形式。
对 z 的导数: ∂NLL/∂z = p - y（面试高频）。
【D2】Softmax + CE 对 logits 的梯度（向量形式）
设 logits z∈R^K，p=softmax(z)，y 为 one-hot。
CE = -Σ_k y_k log p_k = -log p_c（c 为真类）。
结论: ∂CE/∂z = p - y（与二分类同形推广）。

直觉: 真类 logit 被往上拉，其它类被压低，直到 p≈y。

数值稳定实现（工程必会）:

· 用 log_softmax(z) 再与 y 做 dot（或 nll_loss）
· 手写 softmax 先减 max(z)，防 exp 溢出
【D3】线性回归 MSE 的闭式解何时用 GD
闭式解需求逆，复杂度约 O(D^3)；特征维高或样本极大时用迭代法。

岭回归 (X^T X + λI)^{-1} X^T y 改善病态。

【D4】Bias–Variance 分解（回归 MSE，口头版）
E[(y-ŷ)^2] = bias^2 + variance + noise

· 高 bias：欠拟合（模型太简单） · 高 variance：过拟合（对训练集敏感） 对应策略: 增模型复杂度 vs 正则/数据/集成。

【D5】SVM 间隔与 hinge（直觉）

硬间隔: 约束 y_i (w^T x_i + b) ≥ 1 软间隔引入 slack ξ_i，最小化 ||w||^2 + C Σ ξ_i；hinge 是其对偶形式的损失视角。 大 C：更惩罚误分 → 间隔可能更窄 → 更像硬间隔。

【D6】Entropy / Gini（树分裂）
Entropy = -Σ p_k log p_k；Gini = 1 - Σ p_k^2
信息增益 = 父节点熵 - 加权子节点熵。

分裂目标: 让子节点更“纯”（同类占比更高）。

【D7】Bagging vs Boosting 方差偏差视角（口语）

Bagging（RF）: 平均多棵高方差树 → 降方差为主。 Boosting : 序列修正偏差（逐步减小训练误差）；过强可过拟合；需 shrinkage、子采样。

【D8】Batch Norm 的训练与推理（为什么 moving average）
训练: μ_B, σ_B 来自当前 mini-batch；y = γ (x-μ)/σ + β

推理: μ, σ 用训练期滑动平均（running_mean/var），避免依赖 batch。 原因: 推理单样本或小 batch 统计噪声大。

【D9】Layer Norm 公式（Transformer cell）

对单个样本在 hidden 维度上:

LN(x) = γ ⊙ (x - μ)/(σ+ε) + β

μ,σ 在同一层、同一样本跨特征计算（与 BN 轴向相反）。

【D10】Dot-product attention 的 scale 推导（口头）

若 q,k 分量独立零均值方差 1，则 q·k = Σ_{i=1}^{d_k} q_i k_i 方差约为 d_k。
除以 sqrt(d_k) 使点积方差回到 O(1)，softmax 不过度尖锐。
【D11】Masked self-attention（因果 LM）
scores_{ij} = (q_i · k_j)/sqrt(d)；若 j>i（看未来），设 score → -∞。
softmax 后未来位置权重为 0。保证位置 i 预测只依赖 ≤i 的信息。
【D12】Generic GBDT（梯度提升）
前向加法模型 ŷ^{(t)} = ŷ^{(t-1)} + η f_t
f_t 用弱学习器拟合“伪残差”（对 loss 的一阶负梯度）；

XGBoost 在此基础上用二阶导数近似 + 正则化项做分裂评分（更精确更快）。

【D13】KL 与交叉熵（校准/知识蒸馏铺垫）
CE(p,q) = -Σ p log q；KL(p||q) = Σ p log(p/q) = CE(p,q) - H(p)

最小化 KL(p||q)（p 固定）等价于最小化 CE（因 H(p) 常数）。

【D14】Temperature scaling（校准）
推理用 z/T 再 softmax，T>1 分布更平；常用于改善概率校准。

训练模型不变，用小验证集拟合 T。

【D15】量化 PTQ vs QAT（追问时用）

PTQ: 训练后离线量化权重/激活；快但精度掉的可能大。 QAT: 训练中 fake quant；精度更好；成本高。 感知量化（per-channel）、对称 vs 非对称：权衡精度与硬件。

十六、Detailed Q&A Bank（深挖问答）

[Q1] 为什么分类常用 CE 而不是 MSE？
[A] CE 对应 Bernoulli/categorical 的负对数似然，梯度在错误置信度高时更大；
对 sigmoid/softmax 饱和区配合更好。MSE 对概率建模不是典型 MLE；梯度形式也不如 CE “干净”。
[Q2] ROC-AUC 与阈值无关是什么意思？
[A] AUC 衡量排序能力：随机抽一对正负样本，正样本分数高于负样本的概率。
它不依赖单个阈值；改变阈值只改混淆矩阵，不改 AUC。
[Q3] 不平衡数据为什么 ROC 可能“好看”但业务不行？
[A] 负样本极大时，假阳性绝对数量仍可很高；ROC 对负类基数不敏感。

PR 曲线更反映正类查全/查准折中；先看 business 指标与代价。

[Q4] L1 为什么稀疏？（几何/优化双解释口）
[A] 几何：L1 球的角落在坐标轴上，解常落角点→权重为 0。

优化：近端梯度/次梯度促使小系数收缩到 0（与 L2 圆滑球不同）。

[Q5] Ridge vs Lasso 何时用？
[A] 多共线性、全特征保留→Ridge；需要稀疏可解释/特征选择→Lasso；折中 ElasticNet。

表格高维稀疏：树模型也常强。

[Q6] 随机森林 OOB 是什么？能干什么？
[A] Bootstrap 采样下每棵树约见 63% 数据；剩余 OOB 样本可当验证估计泛化；

还能给特征重要度（permutation importance 常与 OOB/val 搭配说）。

[Q7] 为什么说 XGBoost 分裂用二阶信息？
[A] 对损失在 ŷ^{(t-1)} 处二阶 Taylor，得到每个样本的 g_i、h_i；

分裂增益可写成与 g、h 相关的闭式评分，比只用一阶残差更准更稳。

[Q8] LightGBM vs XGBoost（答 4 点够）
[A] Leaf-wise（更可能过拟合需调参）vs level-wise；

直方图算法/互斥特征捆绑（EFB）、GOSS/EFB（LGB 常用）； 类别特征处理差异；速度与内存取舍；GPU 路线差异（版本迭代快，回答以原则为主）。

[Q9] CatBoost 为啥强调类别特征？
[A] Ordered target statistics 降低目标泄露的有序编码；对称树等减少过拟合；

类别强的表格数据常用 CatBoost。

[Q10] 梯度消失 vs 爆炸分别常见于什么？
[A] 消失：sigmoid/tanh 深层链式乘积；爆炸：权重>1 反复连乘 & 长序列。

实务：换激活、残差、Norm、裁剪、初始化。

[Q11] BN 为何能加速训练？（不只“归一化”）
[A] 允许更大学习率；减轻内部协变量偏移（每层输入分布漂移变小）；

有轻微正则效应（batch 噪声）。

[Q12] Dropout 训练和推理差别？为什么？
[A] 训练随机丢弃；推理用全部权重但输出期望一致：要么 inference 乘保留概率，

要么训练时用 inverted dropout（除以 p）。避免推理随机性。

[Q13] Transformer 为何需要位置编码？
[A] Self-attention 本身是置换不变（打乱 token 顺序输出会变但结构不编码顺序），

必须显式注入位置信息。

[Q14] Multi-head 到底学到什么？
[A] 理论：多头在不同子空间并行关注不同依赖（语法/远距离/局部）。

实践： heads 可能有冗余；可视化 attention 只做直觉，不靠单头下结论。

[Q15] BERT MLM 的 [MASK] 有什么训练-推理 mismatch？
[A] fine-tune 下游常无 [MASK]；BERT 用 80% mask、10% random、10% original 缓解。
[Q16] GPT 类自回归为何能生成？
[A] 每一步 predict next token distribution，采样（greedy/beam/top-p）
自回归展开；KV cache 加速逐步生成。
[Q17] ViT 为啥要大数据？
[A] CNN 有归纳偏置（局部性、平移不变）；ViT 更数据饥渴；

小数据可用 hybrid、强增强、预训练微调。

[Q18] ResNet 解决了什么退化问题？
[A] 更深网络训练误差反而变差（优化难）；残差让网络至少可学 identity，

优化路径更平滑。

[Q19] “训练 loss 不降”第一步做什么？
[A] overfit 单个 batch（应能逼近 0）。若做不到：标签/维度/代码 bug；

若能：再看 LR、正则、容量、数据难度。

[Q20] val loss 上升但 train 下降？
[A] 典型过拟合：增数据、正则、早停、降复杂度、标签噪声治理。
[Q21] 如何用 learning rate finder（概念）？
[A] 从小到大扫 LR，记录 loss；选 loss 最快下降未到爆炸区间；

Transformer 还要 warmup 防一开始二阶矩估计不稳。

[Q22] AdamW 的 weight decay 和 L2 正则有什么本质区别？
[A] L2+Adam 传统写法会把衰减项混进自适应分母路径；AdamW 对权重做 decoupled decay，

实践更稳定，尤其 Transformer。

[Q23] calibration 不好会有什么问题？
[A] 阈值/期望代价决策错；排序指标好但概率不可信；可用 reliability diagram 看。
[Q24] A/B 测试与 offline 指标冲突怎么办？

二十一、Deep Dive — Softmax / Log-Softmax / Cross-Entropy 反向（白板逐步）

【21.1 向量 softmax】

输入 logits: z ∈ R^K

u_k = z_k - max_j z_j（数值稳定；softmax 不变）
p_k = exp(u_k) / Σ_j exp(u_j)
性质: p_k > 0, Σ p_k = 1。
【21.2 交叉熵（one-hot 目标）】

真类索引 c；y 为 one-hot。

CE(z) = -log p_c = -u_c + log(Σ_j exp(u_j))

常称 "log-sum-exp" 与 true class 项。

【21.3 雅可比 ∂p/∂z（全导数矩阵）】
记 s = softmax(z)，有:
∂s_i / ∂z_j = s_i (δ_{ij} - s_j)

其中 δ_ij 为 Kronecker delta。

推导要点: s_i = exp(z_i)/Z, Z=Σ exp；∂s_i/∂z_j 对 Z 与 exp 链式拆开即得。
【21.4 CE 对 logits 的梯度（可直接背）】
CE = -Σ_k y_k log s_k（y 为任意分布时亦成立）。
∂CE/∂z = s - y

当 y one-hot，第 j 分量: s_j - y_j。

【21.5 单类导数按元素核对（口述用）】
对真类 c: CE = -log s_c
∂CE/∂z_c = -(1/s_c)(∂s_c/∂z_c) = ... → 叠加所有 k 对 z_c 的通路 → 仍为 s_c - 1。

对非真类 j≠c 同理叠加 → s_j - 0。 合并即 s - y。

【21.6 log-softmax + NLL（工程等价形式）】
PyTorch: nll_loss(log_softmax(z), target)
log_softmax(z)_k = z_k - logsumexp(z)
对 z 的梯度仍是 softmax(z) - one_hot(target)。
【21.7 Softmax 温度 T】

p ∝ exp(z/T)；T>1 分布更平，梯度更温和；T<1 更尖。

二十二、Deep Dive — XGBoost / GBDT 白板公式（分裂增益与结构分数）

【22.1 加法模型】
第 t 棵树加入后: ŷ_i^{(t)} = ŷ_i^{(t-1)} + f_t(x_i)

f_t 为 CART，叶子权重向量 w ∈ R^T，T 为叶子数。

【22.2 正则（树复杂度）常见写法】
Ω(f_t) = γ·T + (λ/2)·Σ_{j=1..T} w_j^2

γ: 叶子数惩罚；λ: 叶子权重 L2。

【22.3 损失二阶近似（第 t 步）】

对样本 i，在 ŷ^{(t-1)} 处 Taylor: L_i ≈ g_i·f_t(x_i) + (1/2)·h_i·f_t(x_i)^2 + const

其中（依具体目标函数定义）:

g_i = ∂L/∂ŷ |_{ŷ=ŷ^{(t-1)}}
h_i = ∂²L/∂ŷ² |_{ŷ=ŷ^{(t-1)}}
平方损失回归例子: L=(y-ŷ)^2 → g_i = 2(ŷ_i - y_i), h_i = 2（常数）

Logistic 按概率导出不背也可，会说“用一二阶导”即可。

【22.4 叶子权重闭式（给定结构 q(x)→叶子索引）】
叶子 j 上样本集合 I_j；定义 G_j = Σ_{i∈I_j} g_i, H_j = Σ_{i∈I_j} h_i

目标（叶子二次项）近似为 Σ_j [ G_j w_j + (1/2)(H_j+λ) w_j^2 ] + γ·T

对固定结构，最优叶子权重:

w_j* = - G_j / (H_j + λ)

最小目标值（结构分数，常用形式）: Obj ≈ - (1/2)·Σ_j [ G_j^2 / (H_j + λ) ] + γ·T

【22.5 分裂增益 Gain（左右子 L/R）】

把叶子 I 分裂为 I_L 与 I_R。

定义 GL, HL, GR, HR 为左右和的 g、h。

Gain = (1/2)[ GL^2/(HL+λ) + GR^2/(HR+λ) - G^2/(H+λ) ] - γ

口述: “左叶分数 + 右叶分数 − 不分裂分数 − 复杂度 γ”。 γ 阻止无意义分裂。

【22.6 缺失值默认方向】

训练学一个默认左或右；推理缺失走默认 → 速度快。

【22.7 近似算法关键词】

Weighted quantile sketch：近似找候选分裂点（大数据）；稀疏感知。

【22.8 vs 经典 GBM（sklearn GradientBoosting）】

经典 GBM 多用一阶梯度（伪残差）；XGBoost 引入二阶 + 更强的树正则与工程优化。

二十四、指标与矩阵 — 公式 + 构造步骤（面试可推导）

【24.1 混淆矩阵（二分类，正类=1）】
              Predicted 0   Predicted 1

Actual 0 TN FP Actual 1 FN TP

Precision = TP / (TP + FP)     （预测为正的里头多少真）
Recall    = TP / (TP + FN)     （所有正例里找出多少）
Specificity = TN / (TN + FP)
F1 = 2PR / (P+R)
【24.2 MCC（不平衡时常报告）】
MCC = (TP·TN - FP·FN) / sqrt((TP+FP)(TP+FN)(TN+FP)(TN+FN))

范围 [-1,1]；比 accuracy 难刷。

【24.3 ROC-AUC 构造（口头算法）】

所有样本按分数降序排列；从高阈值扫到低阈值，记录 (FPR, TPR) 折线；

AUC = 正负可排序对的胜率 = Wilcoxon-Mann-Whitney 等价形式。
【24.4 PR-AUC（Average Precision 近似思想）】

PR 曲线下面面积；正类极少时比 ROC 更有信息量。

【24.5 LogLoss（二分类）】
logloss = -(1/N) Σ [ y log p + (1-y) log(1-p) ]

注意 clip p 防止 log(0)。

【24.6 RMSE / MAE / R²】
RMSE = sqrt(mean((y-ŷ)^2))  （同量纲原目标）
MAE  = mean(|y-ŷ|)          （对离群更稳）
R² = 1 - SSE/SST           （解释方差比例；可为负）
【24.7 NDCG@k（排序，直觉）】

Discounted CG: Σ rel / log2(rank+1)

NDCG = DCG / IDCG（理想排序归一化）；相关性分级时常用。

二十五、Transformer / BERT / GPT — 对照与块级前向（形状口径）

【25.1 单头 Attention 形状】

batch B, seq n, dim d_model X: (B, n, d_model)

Wq,Wk,Wv: (d_model, d_k), 多头时常 d_model = h·d_k
Q = X Wq → (B, n, d_k)
K = X Wk → (B, n, d_k)
V = X Wv → (B, n, d_v)
Scores = Q K^T / sqrt(d_k) → (B, n, n)
Attn = softmax(Scores, dim=-1)  （causal 前对 j>i mask）
Out = Attn V → (B, n, d_v)

多头: 头 i 输出拼回 d_model，再线性投影。

【25.2 Transformer Encoder 块（BERT 类）】

x ← x + MHA(LN(x)) x ← x + FFN(LN(x)) FFN: 两层 MLP，常 4d 隐藏，GELU。

【25.3 Decoder-only（GPT）】

因果 mask 的 MHA；无 encoder；自回归 LM head: 线性到词表 logits。

【25.4 BERT vs GPT（表）】

| | BERT (encoder) | GPT (decoder-only) | |--------|----------------|----------------------| | 训练 | MLM (+原NSP) | next token CE | | 方向性 | 双向上下文 | 单向（左到右） | | 下游 | 分类/序列标注 | 生成 / 指令微调 |

【25.5 CLS / pooling】

BERT 句级: 取 CLS 向量 + dropout + 线性；或 mean max over tokens。

二十六、概率校准（Platt / Isotonic）— 步骤级

【26.1 为何需要】

模型输出的 p 不一定等于真实频率；阈值与期望损失决策会偏。

【26.2 Platt scaling（sigmoid 校准）】

在验证集上，对 logit s 拟合:

P(y=1|s) = σ(a·s + b)

a,b 通过 NLL 优化得到；相当于温度+平移的一维情况。

【26.3 Isotonic regression】

单调非参数映射 raw_score → calibrated_score；小样本易过拟合验证集。

【26.4 评估】

Reliability diagram: 按预测概率分桶，对比桶内真实正例率； ECE（期望校准误差）常用。

二十七、经典算法补充 — Naive Bayes / PCA / k-means（简述）

【27.1 Gaussian Naive Bayes（连续特征）】

假设给定类别下特征条件独立: P(y|x) ∝ P(y) Π_k P(x_k|y) 每类每维常假设高斯分布，参数用 MLE 由训练集估计。

Pros: 极快、小数据可用；文本 MN 多项式 NB 常用。
Cons: 条件独立通常不成立。
【27.2 PCA（主成分分析）】

数据中心化后，求协方差矩阵 top 特征向量；投影方差最大方向。 实现常用 SVD(X) 避免显式构造大协方差。

用途: 降维、去噪、可视化；线性。

【27.3 k-means】

交替: (1) 分配点到最近中心 (2) 更新中心为簇均值。 目标: 最小化簇内平方和；敏感初始化与 k；非凸。

【31.1 数值例子：3 类 softmax + CE 的梯度】
设 z = [2, 1, 0]^T，真类为第 2 类（索引 1，one-hot y=[0,1,0]^T）。
稳定 softmax: u=z-max(z)=[1,0,-1]^T，exp(u)≈[e^1,e^0,e^-1]，归一化得 p≈[0.576,0.212,0.078]（示意）。
∂CE/∂z = p - y ≈ [0.576, -0.788, 0.078]^T。

直观: 真类 logit 梯度为负 → 梯度下降会增大该类 logit（因更新方向 −grad）。

【31.2 BN：moving average 推理统计（常用更新式）】

【31.3 ECE（Expected Calibration Error）概念公式】

把 [0,1] 切成 M 个 bin，B_m 为落入第 m 桶的样本集合。

acc(B_m) = 桶内正例比例，conf(B_m) = 桶内预测概率均值。
ECE = Σ_m (|B_m|/N) |acc(B_m) - conf(B_m)|

越小越好；可与 reliability diagram 一起答。

三十三、XGBoost：从一轮 boosting 到 Gain（口头一条链）

(1) 当前预测 ŷ^{(t-1)}；要在其上加一棵小树 f_t。 (2) 原损失 L 对 ŷ 在样本 i 处展开到二阶:

    L_i ≈ const + g_i f_t(x_i) + (h_i/2) f_t(x_i)^2

(3) 叶子 j 上样本集合 I_j，叶权重 w_j；树结构给定后目标近似为

    Σ_j [ G_j w_j + (H_j+λ)/2 · w_j^2 ] + γ·(#leaves)
    G_j=Σ_{i∈I_j} g_i, H_j=Σ_{i∈I_j} h_i。
(4) 对 w_j 求导得闭式 w_j* = -G_j/(H_j+λ)。

(5) 代回得“结构分数”；分裂把父叶 I 分成 L/R，Gain 用 §二十二 公式。 面试: 强调“二阶使目标对叶权重二次可解 + 正则 γ、λ”。

三十四、学习率调度（公式背常用的）

【34.1 Linear warmup（步 t，共 T_w）】
lr(t) = lr_max · (t / T_w)，t < T_w；之后接 cosine/step。
【34.2 Cosine decay（常见一种）】
lr(t) = lr_min + 0.5 (lr_max - lr_min)(1 + cos(π·(t - T_w)/(T - T_w)))，t ≥ T_w

T 为总训练步（或 epoch 映射到步）。

【34.3 指数衰减】
lr_t = lr_0 · γ^{t/k}（每 k 步乘 γ）。
【34.4 One-cycle（概念）】

lr 先升后降一度；Super-convergence 讨论用；能快速试大 LR 上界。

G1. Ray Core
【Knowledge】

Distributed Python runtime that schedules parallel work across a cluster: unify training, batch ETL, and serving control plane under one abstraction (tasks/actors/object store).

【Basics】

· Driver submits work to a Ray cluster (local or multi-machine). · Task: stateless remote function (can return futures/object refs). · Actor: stateful worker with mutable internal state (handles/concurrency control). · Object store (Plasma): shared immutable objects across workers; reduces redundant copies when used well. · Placement groups: reserve bundles of resources (e.g. 4 GPUs on one node) for gang scheduling. · Head vs worker nodes; GCS holds cluster metadata — head failure is operational concern.

【Detail — workflow, logic & mechanics】
  • Connect: ray.init(address="auto") joins an existing cluster (KubeRay sets RAY_ADDRESS); local dev

uses ray.init() implicit single-node.

  • Remote task: @ray.remote def f(x): return x+1 → ref = f.remote(1) schedules on a worker;

ray.get(ref) blocks and returns value. Async pattern: gather many refs then ray.get(list) for parallelism — logic: scheduler picks worker with free CPU/GPU per resource request in @ray.remote.

  • Object refs: large numpy array passed via object store — worker A ray.put(x), worker B receives ref

without pickling giant blobs over IPC repeatedly. Pitfall: holding references prevents GC → memory pressure on plasma.

  • Actor: @ray.remote class Store: ... → a = Store.remote() ; a.method.remote() serializes calls to

same process — use when you need mutable state (cache, shard counter, model handle).

  • Placement group: ray.util.placement_group([{"GPU": 1, "CPU": 8}] * 4, strategy="PACK") reserves 4

GPUs; start actors/tasks only after pg.ready() — logic: gang scheduling so distributed train workers launch together rather than fragmenting GPUs across idle nodes.

【Cheatsheet】
“Ray = distributed task/actor scheduler + shared object store + placement groups for gang GPU jobs.”
G2. Ray Train
【Knowledge】

Library on Ray for scaling ML training loops (PyTorch/TF/XGBoost-style integrations) without hand- rolling process groups for every cluster topology.

【Basics】

· DistributedDataParallel-style patterns; backends talk to accelerated frameworks. · Checkpointing to durable storage for fault tolerance and hyperparameter trials. · Works with placement groups so workers launch together on right GPU sets. · Often paired with Ray Tune for parallel experiments (separate library).

【Detail — how distributed training actually runs on Ray Train (PyTorch mental model)】

Goal: you write ONE train_loop_per_worker(config) that looks like single-process PyTorch, but Ray launches N copies (one per GPU/worker) and wires NCCL process groups for you.

Typical flow (API names follow Ray Train v2 style; exact imports evolve — interview is about logic):

  • Define train_loop_per_worker(config): inside, read train.get_context().get_world_size() and

get_local_rank() (or equivalent) to know global rank / local rank — logic: each process must know its rank to partition data and set device cuda:local_rank.

  • Build model, wrap with DistributedDataParallel(device_ids=[local_rank]) — same as bare multi-GPU

except Ray started the processes and set env vars (MASTER_ADDR/PORT, WORLD_SIZE, RANK).

  • Build dataloader — often shard dataset by rank so each worker sees disjoint batches (no duplicate

work); or use Ray Train’s dataset integration to distribute shards.

  • Loop: forward → loss.backward() → optimizer.step(). Gradients: DDP hooks all-reduce grads across

ranks each step — mathematically same as one big batch if you scale learning rate / batch size rules.

  • ScalingConfig(num_workers=4, use_gpu=True, resources_per_worker={"GPU": 1}) tells Ray how many

workers and what resources each needs — scheduler uses placement groups so four GPUs land together.

  • TorchTrainer(train_loop_per_worker, scaling_config=..., datasets=...) .fit() — Ray handles process

spawn, failure callbacks, and RunConfig(checkpoint_config=CheckpointConfig(storage_path="s3://..."))

   for periodic checkpoints — logic: on pod kill, restart from last artifact instead of epoch 0.
  • Why Ray vs raw torchrun? Same math; Ray adds unified cluster scheduling (same cluster as Serve/Data),

elastic patterns, and integration with Tune for HPO without new infra.

G3. Ray Data

Distributed dataset and transform layer: read parquet/csv/partitions, apply UDF/map/batch transforms, feed Ray Train or batch jobs — “Spark-like” ergonomics but Python-first and Ray-scheduled.

【Basics】

· Lazy execution with operators; parallelism from partitions. · Shuffle/repartition trades cost vs skew handling. · Spilling to disk if object store/backpressure exceeds memory — tune batch size.

【Detail — pipeline logic from files to Train】
  • Read: ds = ray.data.read_parquet("s3://bucket/prefix"). Repartition or read with parallelism tied to

object store listing — logic: more partitions → more parallel tasks, but overhead if files tiny.

  • Transform: ds = ds.map_batches(fn, batch_size=1024, num_cpus=1) — fn receives pandas/pyarrow batches;

transfer with compute via prefetch if framework allows.

  • End-to-end argument: Ray Data keeps preprocessing on Ray without copying everything to a separate

【Knowledge】

【Detail — serving process & knobs】

G11. PyTorch Distributed (DDP / NCCL)
【Knowledge】

Standard way to train PyTorch models across GPUs/nodes; Ray Train often wraps similar patterns.

【Basics】

· DDP: gradient all-reduce per step; requires processes per GPU aligned. · NCCL: GPU collective communication library (rings, bandwidth-bound). · Fault tolerance usually via checkpoint restart — not magically transparent.

【Detail — one step of DDP on paper】

Suppose 4 GPUs. Each rank holds identical model params. Forward on local batch (different data shard).

Loss backward computes local gradients. AllReduce(mean) gradients across ranks — after step every rank

has identical averaged grad → identical weight update → mathematically equivalent to one large batch (with caveats on batch norm stats).

NCCL chooses ring/tree algorithm based on topology — NVLink vs PCIe vs cross-node Ethernet changes all-reduce time; straggler GPU slows everyone because collective waits.

Gradient accumulation: K micro-batches before one optimizer step simulates larger batch without more memory — logic: divide learning rate rule accordingly.

Why mention with Ray: Ray Train sets env vars so torch.distributed.init_process_group finds peers — without that, manual rendezvous on master IP.

【Cheatsheet】

“DDP + NCCL for multi-GPU train; bandwidth + stragglers dominate — profile before blaming ‘Ray’.”

I1. RRF (Reciprocal Rank Fusion) — explicit form
For document d, with rank r_i(d) in ranker i (1 = best), common score:
RRF(d) = Σ_i 1 / (k + r_i(d))

k is a constant (often 60) so that high ranks don’t dominate only the first list.

Properties: no need to calibrate score scales between ANN distance and BM25; bad if a list is junk entirely — still need per-list quality controls or gating.

Tuning: lower k → more weight to top positions; higher k → flatter — A/B on NDCG@10 with your traffic mix.
I2. BM25 intuition (lexical ranker you’re pairing with dense)

Score(q,d) ∝ Σ idf(t) · (f(t,d) · (k1+1)) / (f(t,d) + k1(1 - b + b·|d|/avgdl))

f = term frequency in doc, |d| = doc length, k1/b tune term sat + length norm.

Why it still matters: exact product codes, brand names, file names — dense models can miss OOV rare tokens.

Interview: “We keep lexical for head + exact; dense for tail semantic.”

I3. Cosine vs inner product in ANN

J1. InfoNCE (contrastive) — loss shape

For a batch with one positive pair (query, doc+) and N-1 in-batch negatives:

  L = -log( exp(sim(q,d+)/τ) / ( exp(sim(q,d+)/τ) + Σ_j exp(sim(q,d_j)/τ) ) )
τ = temperature — lower → sharper distribution, harder training; higher → softer.

Why in-batch negatives: free negatives at scale; need large batch for hard coverage.

Domain fine-tune: move τ and batch size together in ablations.

J2. NDCG@K (Normalized Discounted Cumulative Gain)
DCG@K = Σ_i=1..K (2^{rel_i} - 1) / log2(i+1) (commonly used gain for graded rel; binary variant exists)
NDCG@K = DCG@K / IDCG@K (IDCG = perfect ordering)
Interpreting: accounts for position (log discount) so errors at top hurt more — matches “user looks at

top 5” product reality.

Offline: need labeled relevance grades; multi-level (0-4) better than binary if you have editor labels.

J3. MRR (Mean Reciprocal Rank) & MAP (one line each)

MRR: average 1/rank of first relevant document — good for “one right answer” tasks.

MAP: mean of average precisions per query — classic when multiple rel docs; heavier to explain in 30s;

use NDCG if grades exist, MRR for single correct item.
J4. Softmax temperature (inference, not only training)
On logits z: p = softmax(z/τ). τ>1 flattens distribution (less overconfident) — used in **knowledge
distillation** student matches teacher soft targets; in **LLM sampling** higher temperature = more random

creativity.

Link to calibration: not same as Platt scaling on single logit but same “sharpen/soften” intuition.

PART K — HARDWARE/NET · CUDA/DATALOADER · EXPERIMENT DESIGN · RELIABILITY PATTERNS

K1. PCIe vs NVLink vs Network — bandwidth order-of-magnitude literacy

PCIe Gen4 x16 ~32 GB/s theoretical — often bottleneck moving tensors host↔device if pipeline not overlapped.

NVLink GPU-GPU within node ~hundreds GB/s aggregate (SKU dependent) — NCCL prefers NVLink rings for allreduce.

Cross-node: 100 Gbps Ethernet ≈ 12.5 GB/s — dwarfed by local NVLink — large tensor parallel across nodes stresses NIC + RDMA.

Say: “We profiled before blaming the model — network was the straggler.”

K2. InfiniBand vs RoCE vs TCP for NCCL

InfiniBand: traditional HPC fabric + verbs — low latency.

RoCE: RDMA over Converged Ethernet — common in GPU clouds.

TCP fallback: higher latency — misconfigured RoCE shows up as slow allreduce — check NCCL_IB_DISABLE / interface selection.

K3. CUDA streams & overlap

Independent streams allow H2D transfer overlapping with kernels — hides PCIe when pipelined.

Sync everywhere → idle GPU gaps visible in nsys timeline.

K4. pinned_memory=True & DataLoader workers

Pinned host memory speeds async DMA to GPU.

num_workers tuned so CPU decode keeps GPU fed — image/JPEG decode often CPU-heavy — too few workers → GPU starvation; too many → contention.

K5. Gradient accumulation & LR scaling rule
Effective batch = micro_batch × accum_steps × world_size.

LR often scaled ~linearly with effective batch when doubling batch — not always safe at extreme batch — warmup + optimizers like LAMB sometimes used.

Interview: cite that you validated loss curves after scaling LR, didn’t blindly apply linear rule.
K6. Offline eval leakage — near-duplicate queries/assets

Duplicates between train/eval inflate metrics — dedupe queries (hash text, perceptual hash images) before split.

Same asset shown in multiple editorial labels — cluster-level split sometimes safer.

K7. Stratified metrics by locale / surface
Headline NDCG hides regressions in non-English or Stock-only surfaces — slice dashboards — leadership

questions follow tails.

K8. A/B statistics — honesty

Peeking at results daily inflates false positives — use fixed horizons or proper sequential tests when possible.

Say: “We predetermined minimum sample size for key metrics.”

K9. Circuit breaker on dependency calls

Closed → open after failure threshold → half-open trial — prevents slow OpenSearch/Triton from blocking all threads — triggers degradation matrix (J19).

K10. Bulkhead isolation

Separate pools per downstream — slow ranker can’t exhaust pool for retrieval — analogous to separate Kube deployments with caps.

K11. Hedged reads (dangerous but real pattern)

Repeat read to second replica after Δ latency — take fastest — reduces tail **only if idempotent** — adds load — use sparingly.

K12. Tail-at-scale + fan-out

many shards ⇒ probability of one slow shard high — mitigation: hedge sparingly, deadlines, fewer shards with better routing , bulkhead timeouts per hop.

K13. Object storage consistency & publish

Know whether GET-after-PUT is immediately consistent in your region — index pointer write only after validated segment reads succeed — avoids partial mixed old/new.

K14. JVM GC pauses (OpenSearch) — symptom

Huge heap → long GC STW pauses → unexplained latency spikes — tune heap vs off-heap / G1GC logs — qualifies you as infra-aware.

K15. TLS handshake storms & connection reuse

Churning short connections under load wastes CPU on handshakes — HTTP keep-alive + pooled gRPC channels.

K16. Feature flags for model tiers
Evaluate flag once per request → route cheap vs premium path → log variant_id on search_event — rollback is

flip not redeploy.

K17. Shadow / mirror traffic to new ranker

Duplicate scoring path without user impact — compares offline — costs 2× compute — sample fraction only.

K18. Kafka DLQ for embedding failures

Retry exhausted → DLQ topic — replay after fix — prevents silent holes in corpus coverage.

K19. Clock skew & distributed tracing

Bad NTP → spans misordered — troubleshoot tail incorrectly — ops hygiene.

K20. Audit / compliance language (honest)

Reference access controls + change logs + search_event replay — **do not claim SOC2 certified unless you are** — say “aligned with practices.”

K21. “Anything else?” breadth closer

“I care about the full stack from data contracts through training, retrieval architecture, GPU serving, observability, and graceful degradation when things break — that’s where search products succeed or fail.”

L-A — EXTRA MATH & METRICS (beyond J1–J3)

L-A1. Precision@K & Recall@K (set-based, binary relevance)

Precision@K = (# relevant in top K) / K.
Recall@K = (# relevant in top K) / (total relevant in corpus for this query).
Tradeoff speech: high recall@100 might still have bad UX if precision@5 low — product cares about top of
page — pair headline NDCG@10 with abuse metrics on position 1–3.

L-A2. Average Precision (AP) — intuition

AP = average of precision values at each rank where a relevant doc appears.
Example: rel at ranks 2 and 5 → AP = (1/2 + 2/5)/2 — rewards bringing relevant items earlier.
MAP = mean AP across queries — classic when multiple relevant docs exist.

L-A3. Wilson score interval (rating / CTR confidence)

Better than raw p̂ = clicks/impressions for small samples — bounds conservative for low counts — used in

quality dashboards to avoid ranking by noisy ratios.

Say: “We didn’t rank surfaces by raw CTR when traffic thin — used smoothed or Wilson-style confidence.”

L-A4. Jensen–Shannon divergence (distribution drift — literacy)

Symmetric measure between two probability distributions — common in drift detection between training query distribution vs production — name-drop if interviewer asks “how detect drift besides PSI.”

L-A5. CKA (Centered Kernel Alignment) — representation similarity

Compares activation matrices across layers/models — higher CKA ⇒ more similar representations — cite if asked how you verified two embedding models behave similarly during migration shadow period.

L-A6. Expected Calibration Error (ECE) — repeat for interviews

Bucket predictions by confidence; within each bucket compare avg predicted vs empirical accuracy — weighted

average gap = ECE — ties calibration discussion for rankers outputting probabilities.
L-A7. Pairwise ranking loss (RankNet sketch)

For pair (i,j) where i should rank above j: optimize σ(s_i - s_j) with cross-entropy toward 1 — bridges to

LambdaMART in GBDT land — enough to say “pairwise surrogates NDCG indirectly.”

PART M — AI INFERENCE SCHEDULING (vs OS scheduling) + 详细读懂（原文思想整合进本项目）

来源思想: 传统 OS 调度以线程/CPU 时间为中心；LLM serving 调度以 request/token/batch/session 与显存现实 为中心，是多目标联合优化。下文含【读懂了吗】白话解释，便于你对照原文段落。

M0. 结论先行（英文面试可用）

AI scheduling is **not** “who runs first” (classic OS). It is: **who advances, in what state, under what memory/GPU reality, at what granularity (token/step), with what SLA/QoS.**

The scheduler is simultaneously: **resource allocator**, **experience shaper** (first-token, jitter, long- job stability), and **service-class enforcer** (priority/budget actually honored).

**Prefill** and **decode** are two **different workload regimes** — batching policies good for one can destroy latency for the other if you treat “inference” as one lump.

M1. Why classic Linux-style scheduling succeeded (and what changed)
【原文要点】CPU 通用计算年代：竞争焦点主要是 **CPU 时间片**；可运行实体清晰（进程/线程）；阻塞/唤醒是节奏。
【What changed in AI serving】
· Dominant scarce resources are often **GPU compute + HBM bandwidth + KV cache residency**, not just CPU time.

· A “single request” is not one short task — it is a **pipeline**: prefill burst → many decode steps. · Hundreds of requests may multiplex on **few host threads** feeding the GPU — watching only pthreads misses the real contention.

【读懂了吗 · ELI5】

把餐厅比作 GPU：传统调度像“哪个服务员先接单”。LLM 服务里真正堵的是**厨房产能**和**每道菜要连续上很多小盘（token）**， 盯“服务员线程”看不到厨房里的排队形态。

M2. “可运行实体”迁移：thread → request / token / batch / session
【原文要点】宿主线程/协程仍在，但决定用户体感节拍的是 **request、token 流、batch、prefill job、decode step**。
【Detail】

· **Request**: carries SLA, tenant id, priority, timeout budget. · **Token (decode)**: each step is a measurable unit of GPU work + user-visible output cadence. · **Batch**: not only an optimization — it is **what the scheduler assembled this iteration**.

· **Session**: multi-turn context sharing affects KV footprint and fairness across turns.
【读懂了吗】

线程是“工人”，request/token 是“工单上的工序”。工人数量少，但工单上千 —— 调度逻辑必须在**工单层**做，否则只看到闲忙的线程， 看不到 GPU 里到底在跑谁的第几个 token。

M3. Prefill vs decode — 为什么必须当成两类负载（本项目已多次强调，此处系统化）
【Prefill 特征（输入阶段）】

· Often **long prompt** → large matmul in parallel across sequence positions (depends on implementation).

· More **compute-heavy per forward** to fill KV cache for all prompt tokens (conceptually “one big swallow”).

· Throughput-oriented: **batching prefill** can be efficient.

【Decode 特征（生成阶段）】
· **Autoregressive**: one (or few) new tokens per step; **must read huge KV history** each step.
· Often **memory-bandwidth / KV-access bound**, not just FLOPs.

· Latency-sensitive: user sees **time-per-token** and **stalls** between tokens.

【调度上若混为一谈会怎样】

· “大 batch 优先”利于吞吐，可能让 decode 排队 → **首 token / 中间卡顿**变差。 · 一味切小 batch 追延迟 → GPU 空转、吞吐崩溃。

【读懂了吗 · 类比】
Prefill 像**一次性把原料倒进反应釜搅匀**；decode 像**一滴一滴往外接产物**，每一滴都要看**整个反应历史（KV）**。

两种工序抢同一套釜（GPU），调度策略不可能完全相同。

M4. 为什么 token 会成为调度单位（原文“时代感”）
【机制链】decode 每步：读历史 KV → 前向得到下一 token logits → 写回 KV → 决定是否结束。
【因此 naturally】

· **Fairness** 不仅是“请求之间排队”，还是“下一步 token 给谁算”。 · **插队/优先级**会改变用户看到的输出流是否连贯。 · **长请求** vs **短请求**：是否允许打断/轮转（类似 OS 时间片，但单位是 token step）。

【读懂了吗】

用户肉眼看到的是 token 流 —— 所以调度节奏直接塑造“卡不卡、顺不顺”。这和只看平均 QPS 完全不同。

M5. Batch：不只是技巧，更是调度结果（原文核心）
【区别传统离线 batch】离线：凑一批 → 跑完 → 再凑一批。在线 decode：**请求随时进出**，长度不一。
【因此】某一时刻的 batch 组成 = 调度器在 **显存约束 + 优先级 + 等待时间 + shape 兼容性** 下的决策结果。
Continuous / dynamic batching = **联合调度**：每一轮都在决定“这一拍 GPU 上拼哪些请求一起跑”。
【读懂了吗】

不是“先调度队列再执行”两步；在很多系统里 **调度决策就是在决定执行形态（batch 形状）**。

M6. Continuous batching 的本质（对照 vLLM / 业界实现）
【概念】decode 步进过程中，新请求可以加入、完成的请求可以退出 —— 系统在**流动状态**里做实时折中。
【Why hard】不是静态优化一批任务；是**在线控制**：到达过程随机、优先级变化、显存块刚释放、某些请求已等太久。
【读懂了吗】

把它想成机场登机口不断有人插队/改签/航班取消 —— 不是排一次队就结束。

M7. 调度为什么必须和内存/KV 绑定（原文：不是多维队列那么简单）
【Decision depends on state】
· 能否把请求加入本轮 decode batch？→ 取决于 **KV 是否已 resident**、加入后 **显存是否爆**、是否需要 **驱逐/换页**

（PagedAttention 一类）。

· **抢占/取消**请求：废弃的 KV 块能否回收？会不会浪费已算前缀？
【读懂了吗】
经典 CPU 调度大多假设“任务无重状态”或状态很小；LLM 请求携带**巨大有状态 KV** —— 调度 = **算力 + 状态放置**联合问题。
M8. 多目标冲突：吞吐 vs 尾延迟 vs 公平 vs 高优保障（原文）
【典型张力】
· 大 batch ↑吞吐，可能 ↑排队 → **p99/p999** 变差。

· 保障尾延迟 → 可能牺牲 GPU 利用率。 · 高优插队 → 损害公平性与整体吞吐。

【面试一句】

“There is no single static policy — we tier policies by workload (prefill vs decode), tenant SLA, and load regime; feature flags switch strategies.”

M9. 多租户 QoS：什么时候 AI 调度像 OS 资源治理
【机制词汇】priority queues, token buckets / budgets, minimum guarantees, congestion degradation,

(preemption-like) cancellation when user disconnects.

【读懂了吗】

租户买的是“延迟 SLA”或“吞吐美元价” —— 调度器执行的是**合同**，不是单纯公平队列。

M10. 调度必须理解 compiler/kernel 现实（原文跨层）
【Why】kernel 对 batch size / sequence length 的收益非线性；某些 shape 触发慢 kernel；graph capture 需要稳定形状；

长上下文 attention **显存非线性**。

【面试】

“If the scheduler ignores executable reality, you get ‘smart’ queueing with an underfed GPU or mysterious tail spikes.”

M11. 为什么说调度正在变成一个独立子系统（原文）
因为它同时覆盖：request 生命周期、token 节律、batch 形成、prefill/decode 差异、显存与 KV、QoS、分布式并行

（TP/PP）。复杂度逼出独立组件，而不是框架里几行 if-else。

M12. 抢占、取消、准实时（原文 OS 熟悉问题在 AI 重现）
【场景】用户关页面应停止生成；高优 agent 插入；租户预算耗尽降级。
【难点】打断是否安全？KV partial 怎么处理？抢占会不会让整体吞吐崩盘？
【读懂了吗】
这和杀线程不一样：杀线程回收小；杀 LLM 请求可能浪费大量已生成 KV —— **取消策略本身要计费/优化**。
M13. 预算与成本：调度进入“经济学”（原文）
GPU 小时、显存占用、长上下文 KV、网络多卡通信 —— 都可计费。调度 = **在技术资源里落实商业条款**（admission control:

预算耗尽拒绝/降级）。

【读懂了吗】

调度器有时是 **budget enforcer**：不是想跑就跑，而是“你还剩多少 quota / 这条请求值不值得抢 GPU”。

M14. “智能体气质 / 体验”（原文产品观察 —— 与本项目检索平台相通）

用户感知：**首 token**、中间卡顿、长答是否掉速、多轮是否稳定 —— 很大比例由**调度节拍**决定，不只是模型分数。

排查路径原文建议：**体验差一口气 → 先看调度/配额/队列，而非只看 benchmark。**

【读懂了吗】

模型很强但“用起来别扭”，可能是**节拍（scheduler）**问题，不是智商（weights）问题。

M15. Glossary — 读完原文若仍模糊的术语（逐项解释）

· **CFS (Completely Fair Scheduler)**：Linux 默认公平 CPU 调度器思想 —— 长期平分 CPU 时间给线程。迁移对比：AI 里 “公平”要在 **token/租户 SLA** 上重新定义。

· **Time slice / quantum**：OS 分配给线程连续运行的 CPU 时间 —— 类比 decode 里“每个请求轮流得到一个 token step” 的思想实验（真实系统更复杂）。

· **Prefill**：提示词阶段，把 KV **装满**，为生成做准备 —— 通常并行度高、单次 forward “块”较大。
· **Decode**：自回归生成 —— 每步读全历史 KV，输出下一 token。
· **KV cache**：每层、每 token 存 K/V —— **显存大头**；调度与内存分配（PagedAttention 等）强耦合。

· **Continuous batching**：decode 过程中动态增减 batch 成员 —— 连续联合调度，而非静态批。

· **Admission control**：请求入口是否允许进入系统（配额、限流、优先级） —— 防止过载拖死所有人。

· **QoS (Quality of Service)**：服务等级 —— 延迟/吞吐保障，常通过队列层级、预算、抢占策略实现。

· **Tail latency (p99/p999)**：慢请求尾部 —— 交互产品体感往往由尾部决定。

· **Goodput**：有效吞吐 —— 对用户有用的完成速率，而非 raw FLOPs。

· **Head-of-line blocking**：队头一个大请求堵住后面 —— 需要多级队列/分级调度缓解。

· **Token bucket / Leaky bucket**：经典限流算法 —— 平滑突发，映射到“租户每秒可消耗 token 预算”。

· **Graph capture / CUDA graphs**：把一串 kernel 录成静态图减少 launch 开销 —— 要求形状相对稳定，调度侧避免 剧烈抖动 shape。

· **Tensor parallel / Pipeline parallel**：模型切多 GPU —— 调度还需叠 **分布式步** 与单机调度交互。

M16. 如何把这段讲进你的项目故事（Ray Serve / Triton / vLLM）

· **Ray Serve**：宿主层并发、路由、租户隔离、超时 —— **request/session 级 admission + 编排**。 · **Triton**：GPU 上 **batch 形成**（dynamic batching）直接改变吞吐/延迟权衡。

· **vLLM**：**PagedAttention + continuous batching** —— decode 调度与 KV 内存强绑定。

一句话：**OS 线程调度解决“谁占 CPU”；AI 调度解决“谁的 token 下一步上 GPU、拼成什么 batch、剩多少显存给高优”。**
