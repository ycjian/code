End-to-End Architecture: Context-Aware Hybrid Multimodal Engine

hybrid_multimodal_engine_architecture.md ? 8 top sections ? Hybrid engine

This document outlines the end-to-end system design, logical architecture, and core codebase for the Context-Aware Hybrid Multimodal Engine. This system intelligently paralyzes and delegates multimodal workflows (indexing, retrieving, processing, rendering) across Edge devices (user hardware) and Cloud clusters (Amsterdam CPU/GPU grids) based on real-time environmental context.

1. High-Level Architecture Concept

The core premise of this project is that pushing everything to the cloud is inefficient (high bandwidth/latency for raw video/3D), and processing everything on the edge is impossible (limited VRAM, battery, compute).

The solution is a dynamic, Context-Aware Dispatcher that utilizes a Directed Acyclic Graph (DAG) for task execution.

The 4 Primary Tiers:

  1. Edge Profiling Service: Runs on the client SDK. Monitors network bandwidth, device GPU/CPU load, thermal throttling, and user presets (e.g., "power saving vs. high performance").
  2. Hybrid Dispatcher (The Brain): Evaluates the task payload (e.g., "Index this 4K video") against the Edge Profile. Decides what executes locally and what offloads.
  3. Edge Execution Engine: Local WebAssembly/ONNX runtime or native C++ rendering engine capable of executing partial blocks (e.g., extracting keyframes locally to save upload bandwidth).
  4. Cloud Heterogeneous Clusters: Amsterdam’s elastic compute. CPU nodes for basic transformations; GPU nodes (A100s/H100s) for heavy embedding extraction, neural reranking, or heavy 3D rendering.

3. Workflow: End-to-End Execution Path

Let's trace a highly complex modern workflow through these components: Creating a 3D multimodal search index from high-res video.

  1. Ingress & Profiling (The Edge): The Client App issues a request to index a 4K, 10GB video. The Telemetry Profiler evaluates the local machine. It detects an Apple M3 Max (high compute) but an unstable cellular hotspot (terrible network).
  2. Context-Aware Dispaching (API Gateway): The Edge Context is sent via gRPC to the API Gateway. The Hybrid Dispatcher determines that uploading 10GB will fail or take hours. It commands the Edge to perform "Keyframe Extraction & Local Quantized Embedding".
  3. Edge Execution: The Local Edge Runtime (via ONNX/WebNN) processes the raw 10GB video down into an array of 256-dimension float values (just 10MB of text/vector data), persisting temp files to the Local SQLite index.
  4. Cloud Offload & Queueing: The 10MB vector payload is successfully transmitted to the API Gateway. The Dispatcher accepts the payload, caching the session state in Redis, and drops an indexing event into Kafka.
  5. Compute & Storage (The Cloud): The Cloud Execution tier consumes the Kafka message. The Envoy Service Mesh routes the payload to the GPU T4 Cluster. The GPU cluster uses a specialized LoRA (pulled dynamically from the LoRA store) to re-align the quantized edge vectors into the native massive cloud representation, committing the final vectors into the Vector Database (Qdrant/Milvus) and updating the central Doc DB with the metadata.

Let's look at the foundational logical code that powers this engine. This is essentially the code that your backend API and edge client use to communicate state.

A. The Environment Profile

The edge client must serialize its state in every major request.

B. The Decision Matrix (The Dispatcher)

When a request hits the Dispatcher, it calculates a cost function. Should it process on Edge, Cloud, or Hybrid?

class CostOptimizer:
    def __init__(self, cloud_gpu_cost_per_ms, edge_battery_penalty):
        self.cloud_cost = cloud_gpu_cost_per_ms
        self.battery_penalty = edge_battery_penalty

    def evaluate_execution_plan(self, task: MultimodalTask, profile: EdgeProfile) -> str:
        # 1. Hard Constraints (Cannot execute on edge)
        if task.modality == "3D_substance" and profile.available_vram_mb < 4000:
            return "cloud_gpu_heavy"
        
        # 2. Network Constraints
        # If payload is huge (e.g., 500MB video) and uplink is terrible, but the edge
        # device is powerful, do local processing (e.g., frame extraction).
        estimated_upload_time = task.payload_size_mb / profile.network_uplink_mbps
        
        if task.modality == "video" and task.operation == "index":
            if estimated_upload_time > 5.0 and profile.device_type == "desktop_high_end":
                return "hybrid_split" # Extract frames locally, send frames to cloud
        
        # 3. Energy / User Preset constraints
        if profile.user_preset == "eco" or (not profile.is_charging and profile.battery_level < 0.2):
            return "cloud_cpu" if task.modality == "text" else "cloud_gpu_light"

        # Default to highest performance path
        return "cloud_gpu_heavy"

C. The DAG Execution Logic (Hybrid Split Example)

This is where the magic happens. A "video indexing" task gets split. Instead of uploading a raw 4K video, the engine instructs the client to run a local WebAssembly process, extract embedding frames, and only upload the frames for vector indexing.

class HybridTaskOrchestrator:
    
    def process_task(self, task: MultimodalTask, profile: EdgeProfile):
        optimizer = CostOptimizer(cloud_gpu_cost_per_ms=0.01, edge_battery_penalty=0.5)
        execution_plan = optimizer.evaluate_execution_plan(task, profile)
        
        if execution_plan == "hybrid_split":
            return self._execute_hybrid_video_index(task, profile)
        elif execution_plan.startswith("cloud"):
            return self._upload_and_execute_cloud(task, execution_plan)
        else:
            return self._execute_edge_only(task)

    def _execute_hybrid_video_index(self, task: MultimodalTask, profile: EdgeProfile):
        print(f"Executing Hybrid Split for Task {task.task_id} via {profile.device_type}")
        
        # Step 1: Assign Local Task
        local_task = {
            "action": "extract_keyframes",
            "fps": 1,
            "max_resolution": "720p"
        }
        # Edge executes via ONNX/WebGPU, returns byte array of compressed frames
        extracted_frames = edge_client.execute(local_task, task.raw_data_uri)
        
        # Step 2: Cloud takes over for heavy embedding extraction
        cloud_payload_size = len(extracted_frames) / (1024 * 1024)
        print(f"Reduced upload payload from {task.payload_size_mb}MB to {cloud_payload_size}MB")
        
        cloud_task = {
            "action": "extract_clip_embeddings_and_index",
            "data": extracted_frames
        }
        
        # Dispatch to specific Amsterdam Cloud cluster based on latency SLO

4. Deep Dive into Specific Workflows

Scenario 1: High-Fidelity 3D Rendering & Retrieval (Commerce)

  • The User: A shopper on a mobile device (low battery, poor network) wants to search for and view a 3D substance/texture for a piece of furniture.
  • The Logic:
  1. Profiler detects mobile, VRAM < 2GB, and network=slow.
  2. Dispatcher recognizes that the device cannot render the raw 3D mesh + high-res raytraced textures locally.
  3. Action: The Cloud GPU cluster renders a 360-degree high-res video proxy of the 3D model, streaming it to the user. The edge device only handles lightweight video playback UI.
  • Business Impact: Zero app crashes. High conversion rate because the visual fidelity remains perfect, bypassing hardware limitations.

Scenario 2: Massive Image Ingestion (Creative Cloud SDK)

  • The User: A photographer using Photoshop on a MacBook Pro M2 (high VRAM, connected to gigabit Wi-Fi, user_preset=max_performance). They want to "Semantic Search" across 10,000 local RAW photos.
  • The Logic:
  1. Uploading 10,000 RAW files (300GB+) to the cloud to extract embeddings would take hours and cost massive cloud ingress bandwidth.
  2. Profiler detects Apple Neural Engine / M2 Mac.
  3. Action: The Dispatcher sends the CLIP (or Amsterdam Firefly) embedding model weights down to the client. The edge device iterates through the 10,000 images, extracts the 768-dimensional float vectors locally, and uploads only the arrays of vectors to the Cloud Unified Vector Index.
  • Business Impact: Privacy is preserved (raw assets never leave device), cloud costs drop by 99%, and indexing happens instantly utilizing the edge's idle Apple Silicon.

5. Modern Complexities & Engineering Considerations

When discussing this project in interviews, highlight these modern systems architectural challenges:

  1. State Synchronization & Eventual Consistency via Kafka: If a user jumps from Wi-Fi to Cellular mid-process, the Dispatcher must dynamically failover. By using Kafka, if the client drops connection, the cloud consumer doesn't crash; it finishes the indexing, writes to the Vector DB, and updates a "Notification State" in Redis, allowing the client to resync state gracefully upon reconnection.
  2. Envoy & KEDA Auto-Scaling: We don't just route randomly. Envoy acts as a service mesh measuring exact saturation on the GPU nodes. We pair this with KEDA (Kubernetes Event-driven Autoscaling) to spin up expensive H100 nodes only when the Kafka backlog of GenAI/3D rendering tasks exceeds our p95 SLA targets, minimizing idle GPU burn rate.
  3. Model Drift / Parity: You must ensure the ONNX embedding model running on the Mac/Windows edge yields the exact same tensor outputs as the massive PyTorch model running on the cloud GPUs, otherwise the vector similarity search breaks.
  4. Presigned URLs & Blob Storage: To avoid choking the API Gateway with massive binary ingress, if a task does require heavy cloud rendering (e.g. generating a 3D mesh), the API Gateway issues a temporary S3 Pre-Signed Upload URL to the client. The client blasts the file directly to S3. S3 triggers an event to Kafka, which unblocks the GPU tier.
