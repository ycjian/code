Core AI Infrastructure Framing

这一节帮助你把 Project 1 讲成 core AI infrastructure，而不是泛泛的 DevOps 或普通平台工具。面试时重点不是“我会用 Kubernetes”，而是“我设计了产品 AI 功能背后的共享执行层”。

Q: What is core AI infrastructure?

什么是 core AI infrastructure？

A:

  • Core AI infrastructure is the runtime substrate that makes product-facing AI capabilities reliable, scalable, observable, and cost-aware.

Core AI infrastructure 是让面向产品的 AI 能力可靠、可扩展、可观测、成本可控的运行时底座。

  • It includes serving contracts, intent/admission control, model routing, workflow orchestration, LLM-as-a-service, GPU/CPU resource control, rollout safety, and observability.

它包括 serving contract、intent/admission control、model routing、workflow orchestration、LLM-as-a-service、GPU/CPU resource control、rollout safety 和 observability。

  • Interview line: "I worked on the shared execution layer that turns product-level multimodal and LLM intents into reliable execution plans across edge, CPU, GPU, and model-serving backends."

面试表达：“我做的是共享执行层，把产品层的 multimodal 和 LLM intent 转换成跨端侧、CPU、GPU 与 model-serving backend 的可靠 execution plan。”

Q: Core AI infrastructure vs AI platform vs DevOps?

Core AI infrastructure、AI platform、DevOps 有什么区别？

A:

  • DevOps often sounds like operating clusters, CI/CD, deployments, and keeping services alive.

DevOps 听起来更像维护集群、CI/CD、部署和保障服务运行。

  • AI platform can mean a broad internal toolkit: notebooks, data pipelines, model registry, feature store, experiment tracking, evaluation tools, and serving tools.

AI platform 可以是很宽的内部工具集合：notebook、数据流水线、模型注册、特征库、实验跟踪、评估工具和 serving 工具。

  • Core AI infrastructure is closer to the critical runtime path: how product AI requests are admitted, routed, served, retried, evaluated, scaled, and rolled back.

Core AI infrastructure 更接近关键运行时路径：产品 AI 请求如何准入、路由、服务、重试、评估、扩缩容和回滚。

  • Strong interview phrasing: "It was product-facing infrastructure, not generic infrastructure. Product teams owned feature semantics; the infra layer owned execution reliability, serving policy, and resource control."

强面试表达：“这是面向产品的基础设施，不是泛泛的基础设施。产品团队拥有功能语义，infra layer 拥有执行可靠性、serving policy 和资源控制。”

Q: Why does this not mean your team served every Amsterdam AI request?

为什么这不意味着你的团队服务了所有 Amsterdam AI 请求？

A:

  • The credible claim is that the team provided shared primitives and selected execution paths for workflows that needed common routing, serving, GPU scheduling, evaluation, and observability.

可信的说法是，团队提供共享原语和选定执行路径，用于需要通用 routing、serving、GPU scheduling、evaluation 和 observability 的 workflow。

  • Product teams still own UX, API semantics, prompt intent, product-specific business logic, and final user experience.

产品团队仍然拥有 UX、API 语义、prompt intent、产品特定业务逻辑和最终用户体验。

  • The infrastructure layer owns reusable runtime concerns: admission, planning, model serving integration, failure recovery, quota, cost, telemetry, and rollout control.

基础设施层拥有可复用的运行时问题：admission、planning、model serving integration、failure recovery、quota、cost、telemetry 和 rollout control。

Q: Control plane vs data plane in AI infra?

AI infra 中的 control plane 和 data plane 有什么区别？

A:

  • The control plane decides what should happen: validate intent, enforce policy, select runtime, choose model version, set token budget, attach tracing, and create an execution plan.

Control plane 决定应该发生什么：校验 intent、执行策略、选择 runtime、选择 model version、设置 token budget、附加 tracing，并创建 execution plan。

  • The data plane does the expensive work: upload artifacts, run preprocessing, call embedding/ranking/generation models, query retrieval systems, stream tokens, and persist results.

Data plane 执行昂贵工作：上传 artifact、运行 preprocessing、调用 embedding/ranking/generation model、查询 retrieval system、stream token，并持久化结果。

  • Interview angle: "Most reliability and cost mistakes happen when control-plane decisions are scattered across product services."

面试角度：“很多可靠性和成本问题来自 control-plane decision 分散在各个产品服务里。”

Q: What is LLM-as-a-service in this context?

在这个上下文中，LLM-as-a-service 是什么？

A:

  • LLM-as-a-service is an internal generation service boundary, not just a model endpoint.

LLM-as-a-service 是内部 generation service boundary，不只是一个 model endpoint。

  • It gives product teams a stable way to request generation, summarization, query rewrite, explanation, evaluation, or tool planning without owning runtime placement and GPU scheduling.

它让产品团队可以稳定请求 generation、summarization、query rewrite、explanation、evaluation 或 tool planning，而不需要自己拥有 runtime placement 和 GPU scheduling。

  • It usually owns model-tier routing, prompt/template versioning, context assembly, token budget, streaming contract, TTFT/TPOT telemetry, safety gates, fallback behavior, and rollout control.

它通常拥有 model-tier routing、prompt/template versioning、context assembly、token budget、streaming contract、TTFT/TPOT telemetry、safety gate、fallback behavior 和 rollout control。

Q: Why is LLM-as-a-service harder than wrapping vLLM with HTTP?

为什么 LLM-as-a-service 比给 vLLM 包一层 HTTP 更难？

A:

  • LLM requests have variable cost: a short rewrite and a long-context assistant turn can differ massively in prefill cost, decode time, KV-cache pressure, and output length.

LLM request 的成本不固定：短 rewrite 和长上下文 assistant turn 在 prefill cost、decode time、KV-cache pressure 和 output length 上可能差异巨大。

  • Prompt, retrieval context, model version, tokenizer, decoding config, and safety policy interact; rolling out one without versioning the others can create silent quality regressions.

Prompt、retrieval context、model version、tokenizer、decoding config 和 safety policy 相互影响；如果只发布其中一个而不对其他部分版本化，可能产生静默质量回归。

  • Multi-tenant serving needs quotas, priority classes, fallback models, admission control, and cost attribution so one product surface does not starve others.

多租户 serving 需要 quota、priority class、fallback model、admission control 和 cost attribution，避免一个产品入口挤占其他入口资源。

Core AI infrastructure 的好指标有哪些？

产品侧指标：按 intent 统计 p95/p99 latency、success rate、degraded-result rate、rollout rollback time 和 user-visible error rate。

  • LLM metrics: TTFT, TPOT, generated tokens/sec, waiting tokens, active sequences, KV-cache pressure, token admission rejects, and fallback-model usage.

LLM 指标：TTFT、TPOT、generated tokens/sec、waiting tokens、active sequences、KV-cache pressure、token admission reject 和 fallback-model usage。

  • Resource metrics: GPU utilization, GPU memory, MIG fragmentation, queue depth, queue age, warm-capacity coverage, CPU orchestration saturation, and cost per successful workflow.

资源指标：GPU utilization、GPU memory、MIG fragmentation、queue depth、queue age、warm-capacity coverage、CPU orchestration saturation 和 cost per successful workflow。

  • Platform metrics: onboarding time for a new intent, percentage of workflows using shared guardrails, number of product teams using the shared path, and incident reduction from standardized fallbacks.

平台指标：新 intent 接入时间、使用共享 guardrail 的 workflow 占比、使用共享路径的产品团队数量，以及标准化 fallback 带来的 incident reduction。

Project Control Plane Concepts (Intent / Planner / Dispatcher)

这一节解释 Project 1 的核心平台抽象：intent registry、dispatcher、execution planner。重点是把产品需求和底层执行解耦。

Q: What is an intent registry, and why is it not just a config table?

什么是意图注册表，为什么它不仅仅是一个配置表？

A:

  • An intent registry is the contract between product features and the core AI infrastructure execution layer.

意图注册表是产品功能与 core AI infrastructure 执行层之间的契约。

  • It stores what the product wants, not exactly which server to call: input/output schema, owner, modality, privacy class, latency class, allowed execution targets, model/index/prompt dependencies, quota, fallback policy, and evaluation gates.

它存储产品想要的内容，而不是确切地调用哪个服务器：输入/输出 schema、所有者、模态、隐私类、延迟类、允许的执行目标、模型/索引/prompt依赖项、配额、fallback策略和评估门。

  • It is more than config because it creates a versioned control-plane abstraction: rollout, rollback, policy enforcement, and observability can all attach to intent versions.

它不仅仅是配置，因为它创建了版本化的控制平面抽象：推出、回滚、策略执行和可观测性都可以附加到意图版本。

Q: What should be versioned in an intent?

意图中应该对什么进行版本控制？

A:

  • Intent schema version.

意图架构版本。

  • Model version and tokenizer/version-specific runtime config.

模型版本和标记器/版本特定的运行时配置。

  • Prompt template and decoding config.

prompt 模板和解码配置。

  • Retrieval index and embedding model version.

检索索引和嵌入模型版本。

  • Tool schema version for agent/RAG workflows.

适用于agent/RAG 工作流程的工具架构版本。

  • Fallback policy, quota policy, privacy policy, and evaluation gate version.

fallback策略、配额策略、隐私策略和评估门版本。

  • Interview line: "If it can change behavior or affect rollback, it should be versioned."

访谈台词：“如果它可以改变行为或影响回滚，那么就应该对其进行版本控制。”

Q: What does the dispatcher do?

调度员做什么的？

A:

  • The dispatcher turns an intent request into an execution plan under current context.

调度程序将意图请求转换为当前上下文下的执行计划。

  • It reads static policy from the intent registry and dynamic signals from device context, payload size, network quality, queue pressure, model profile, GPU capacity, tenant quota, and privacy policy.

它从意图注册表中读取静态策略，并从设备上下文、有效负载大小、网络质量、队列压力、模型配置文件、GPU 容量、tenant配额和隐私策略中读取动态信号。

  • Its output is not just "call model X"; it is a plan with steps, placements, retries, fallbacks, checkpoint requirements, and observability metadata.

它的输出不仅仅是“调用模型X”；它是一个包含步骤、放置、重试、fallback、检查点要求和可观测性元数据的计划。

Q: Dispatcher vs execution planner: how would you explain the difference?

调度程序与执行计划程序：您如何解释其中的差异？

A:

  • Dispatcher: decides the high-level route and target execution mode for an incoming request.

调度程序：决定传入请求的高级路由和目标执行模式。

  • Execution planner: expands that decision into concrete steps: preprocess, upload, embed, retrieve, rerank, generate, evaluate, persist, return.

执行规划器：将该决策扩展为具体步骤：预处理、上传、嵌入、检索、重新排序、生成、评估、持久、返回。

  • In a simple system they may live together; in a mature platform they are often separated so routing policy and workflow expansion can evolve independently.

在一个简单的系统中，他们可以生活在一起；在成熟的平台中，它们通常是分开的，因此路由策略和工作流扩展可以独立发展。

Q: What is a model profile?

什么是模型配置文件？

A:

  • A model profile summarizes runtime requirements and constraints: model family, runtime engine, memory footprint, max input/output tokens, batchability, latency sensitivity, GPU type, MIG compatibility, precision, warmup time, and version compatibility.

模型配置文件总结了运行时要求和约束：模型系列、运行时引擎、内存占用、最大输入/输出token、批处理能力、延迟敏感性、GPU 类型、MIG 兼容性、精度、预热时间和版本兼容性。

  • The planner uses it to choose Ray/Triton/vLLM, CPU/GPU/MIG/full GPU, interactive/batch pool, and fallback behavior.

规划器使用它来选择 Ray/Triton/vLLM、CPU/GPU/MIG/全 GPU、交互式/批处理池和fallback行为。

Q: What does a good execution plan contain?

一个好的执行计划包含哪些内容？

A:

  • Steps with dependencies and timeouts.

具有依赖性和超时的步骤。

  • Placement hints for CPU, GPU, MIG slice, vLLM pool, Triton pool, or batch queue.

CPU、GPU、MIG 切片、vLLM 池、Triton 池或批处理队列的放置prompt。

  • Artifact references rather than raw media payloads.

工件引用而不是原始媒体负载。

  • Checkpoint requirements and idempotency keys.

检查点要求和幂等性密钥。

  • Retry/fallback/degradation policy.

重试/fallback/降级策略。

  • Model/index/prompt/tool versions.

模型/索引/prompt/工具版本。

  • Trace IDs and metrics labels for debugging.

用于调试的跟踪 ID 和指标标签。

Q: What is a workflow DAG, and why use it for multimodal jobs?

什么是工作流 DAG，为什么将其用于多模态作业？

A:

  • A workflow DAG is a directed acyclic graph of steps where each step depends on completed upstream artifacts.

工作流 DAG 是步骤的有向非循环图，其中每个步骤都取决于已完成的上游工件。

  • It fits multimodal jobs because preprocessing, upload, embedding, retrieval, generation, moderation, and persistence can have different runtimes and failure modes.

它适合多模态作业，因为预处理、上传、嵌入、检索、生成、审核和持久性可以具有不同的运行时和故障模式。

  • Checkpoint each DAG step so a network drop, GPU timeout, or client disconnect does not force the workflow to restart from raw media.

检查每个 DAG 步骤，以便网络丢失、GPU 超时或客户端断开连接不会强制工作流程从原始媒体重新启动。

Q: Why is a control plane important in AI infrastructure?

为什么控制平面在人工智能基础设施中很重要？

A:

  • AI systems change faster than traditional services: model versions, prompts, indexes, routing policies, safety gates, and cost controls can all change behavior.

人工智能系统比传统服务变化得更快：模型版本、prompt、索引、路由策略、安全门和成本控制都可以改变行为。

  • A control plane centralizes those decisions so product teams do not copy routing and rollout logic.

控制平面集中这些决策，因此产品团队不会复制路由和推出逻辑。

  • Interview angle: "The control plane decides what should happen; the data plane does the expensive work."

面试角度：“控制平面决定应该发生什么；数据平面完成昂贵的工作。”

Q: How do tenant quota and policy fit into the control plane?

tenant配额和策略如何融入控制平面？

A:

  • Quota prevents one product/team/tenant from consuming all expensive GPU or token capacity.

配额可防止一个产品/团队/tenant消耗所有昂贵的 GPU 或token容量。

  • Policy enforces privacy, region, retention, safety, eligibility, and allowed-runtime constraints before expensive execution starts.

在昂贵的执行开始之前，策略强制执行隐私、区域、保留、安全、资格和允许的运行时约束。

  • This should happen at admission/planning time, not after the system has already uploaded media or queued GPU work.

这应该在准入/计划时发生，而不是在系统已经上传媒体或排队 GPU 工作之后发生。

Q: What metrics prove this control plane is working?

哪些指标可以证明该控制平面正在工作？

A:

  • Intent onboarding time.

意向入职时间。

  • Percentage of workflows using shared routing/eval/fallback instead of custom code.

使用共享路由/评估/fallback而不是自定义代码的工作流程的百分比。

  • Policy violation rate and blocked requests.

策略违反率和被阻止的请求。

  • Rollback time by intent version.

按意图版本回滚时间。

  • Per-intent p95 latency and error rate.

每个意图的 p95 延迟和错误率。

  • GPU/token cost per successful workflow.

每个成功工作流程的 GPU/token成本。

Kubernetes / K8s Core (Production AI Infrastructure)

这一节是 Kubernetes 基础和生产实践。面试时要强调 K8s 是部署、调度、服务发现、健康检查和资源隔离的底座，但 AI workload 还需要更上层的策略。

Q: Why is Kubernetes central in an AI platform?

为什么 Kubernetes 是 AI 平台的核心？

A:

  • Kubernetes is the common substrate for running CPU services, GPU model servers, stream processors, indexing jobs, batch workers, monitoring agents, and control-plane components.

Kubernetes 是运行 CPU 服务、GPU 模型服务器、流处理器、索引作业、批处理工作线程、监控agent和控制平面组件的通用基础。

  • It provides scheduling, deployment, service discovery, health checks, resource isolation, config/secrets delivery, rollout mechanics, and cluster operations.

它提供调度、部署、服务发现、运行状况检查、资源隔离、配置/机密交付、部署机制和集群操作。

  • In interviews, emphasize that Kubernetes does not solve AI serving by itself; it gives the operating system for the platform, while workload-aware controllers and serving engines handle AI-specific bottlenecks.

在面试中强调 Kubernetes 本身并不能解决 AI 服务问题；它为平台提供操作系统，而工作负载感知控制器和服务引擎则处理人工智能特定的瓶颈。

Q: What are the core Kubernetes control-plane components?

Kubernetes 控制平面的核心组件是什么？

A:

  • API server: the front door for cluster state changes.

API服务器：集群状态变更的前门。

  • etcd: the durable key-value store for cluster state.

etcd：集群状态的持久键值存储。

  • Scheduler: assigns Pods to Nodes based on constraints and scoring.

调度程序：根据约束和评分将 Pod 分配给节点。

  • Controller manager: runs reconciliation loops for Deployments, ReplicaSets, Jobs, Nodes, and more.

控制器管理器：运行部署、副本集、作业、节点等的协调循环。

  • Kubelet: node agent that starts containers and reports Pod/Node status.

Kubelet：启动容器并报告 Pod/Node 状态的节点agent。

  • CNI / kube-proxy or equivalent: networking and service routing primitives.

CNI / kube-proxy 或同等版本：网络和服务路由原语。

Q: Pod vs Deployment vs StatefulSet vs DaemonSet vs Job?

Pod、Deployment、StatefulSet、DaemonSet、Job？

A:

  • Pod: smallest schedulable unit, one or more containers sharing network/storage namespace.

Pod：最小的可调度单元，一个或多个共享网络/存储命名空间的容器。

  • Deployment: manages stateless replicated Pods through ReplicaSets and rolling updates.

部署：通过 ReplicaSet 和滚动更新来管理无状态复制 Pod。

  • StatefulSet: stable identity and storage for stateful systems.

StatefulSet：有状态系统的稳定身份和存储。

  • DaemonSet: one Pod per selected Node, common for agents such as log collectors, CNI, or GPU telemetry.

DaemonSet：每个选定节点一个 Pod，常见于日志收集器、CNI 或 GPU 遥测等agent。

  • Job/CronJob: finite batch work such as backfills, embedding jobs, evaluations, and index maintenance.

Job/CronJob：有限批处理工作，例如回填、嵌入作业、评估和索引维护。

Q: Why are Deployments important for model-serving systems?

为什么部署对于模型服务系统很重要？

A:

  • They define desired replica count, rollout strategy, Pod template, and update behavior.

它们定义所需的副本数量、部署策略、Pod 模板和更新行为。

  • For CPU services and many stateless model-serving frontends, Deployments are the normal abstraction.

对于 CPU 服务和许多无状态模型服务前端，部署是正常的抽象。

  • For GPU services, Deployments still work, but you must carefully set resource requests, node selection, tolerations, readiness probes, and rollout limits to avoid draining too much warm capacity at once.

Time-slicing 按时间共享 GPU，可以提升利用率，但隔离性和性能可预测性弱于 MIG。

Q: What is proactive bin-packing for GPU workloads?

GPU workload 的 proactive bin-packing 是什么？

  • Bin-packing means placing workloads to reduce wasted GPU/MIG capacity while respecting latency, memory, tenant isolation, and rollback constraints.

Bin-packing 指在满足 latency、memory、tenant isolation 和 rollback constraint 的前提下放置 workload，以减少 GPU/MIG 容量浪费。

目标不是 100% 利用率，而是在 p95 latency、queue age 和 rollback safety 健康的前提下提高有用利用率。

Q: Bin-packing vs spreading: how do you choose?

Bin-packing 和 spreading 如何选择？

  • Bin-packing improves utilization and leaves larger empty blocks for future workloads, but it can increase blast radius if one node fails.

Bin-packing 提高利用率，并为未来 workload 留出更大的空闲块，但如果某个节点故障，blast radius 可能更大。

Spreading 提高可用性并降低相关故障风险，但可能加重 fragmentation，降低装箱效率。

好的策略会结合两者：容量紧张时在 pool 内 pack，高价值交互服务跨 zone/node spread，batch job 保持可抢占。

如何在不伤害延迟的情况下把 GPU 利用率从 45% 提升到 80%？

为高价值交互路径保留 warm capacity，让低优先级 batch/evaluation 工作填充空闲容量，并允许抢占或限流。

扩缩容要看 queue age、p95 latency、token backlog 和 GPU memory pressure，而不是只看 utilization。

Kafka (Event Streaming)

这一节讲 Kafka 的日志、分区、consumer group、offset、exactly/effectively-once 和常见故障。Project 2 的 ingestion/indexing 很依赖这些概念。

Q: What is Kafka “really”, and why do teams choose it?

Kafka“真正”是什么？为什么团队选择它？

A:

  • Kafka is a distributed append-only log: producers append records; consumers pull records by offset.

Kafka是一个分布式仅追加日志：生产者追加记录；消费者通过偏移量拉取记录。

  • It scales throughput via partitioning, keeps ordering within a partition, and supports durability through replication.

它通过分区扩展吞吐量，保持分区内的排序，并通过复制支持持久性。

  • Teams pick Kafka when they need high-throughput fan-out, replayability, and decoupling between producers and multiple consumer pipelines.

当团队需要高吞吐量扇出、可重玩性以及生产者和多个消费者管道之间的解耦时，他们会选择 Kafka。

Q: What guarantees does Kafka give about ordering?

Kafka 对排序有什么保证？

A:

  • Ordering is per partition, not global across the whole topic.

排序是按分区进行的，而不是整个主题的全局排序。

  • If you key messages consistently, all messages for a key go to the same partition and preserve per-key ordering.

如果您一致地键入消息，则某个键的所有消息都会进入同一分区并保留每个键的顺序。

  • If you increase partitions later, be ready to reason about how keys map to partitions and what ordering you actually require.

如果稍后增加分区，请准备好推理键如何映射到分区以及您实际需要的顺序。

Q: What’s a consumer group, and why does it matter for scaling?

什么是消费者群体？为什么它对扩展很重要？

A:

  • A consumer group is how Kafka achieves parallel consumption: partitions are assigned among group members.

消费者组是Kafka实现并行消费的方式：分区在组成员之间分配。

  • Max parallelism per topic is bounded by partition count (one active consumer per partition per group).

每个主题的最大并行度受分区计数限制（每组每个分区一个活动使用者）。

  • Group membership implies rebalancing; rebalances can impact tail latency and error rate if your app isn’t designed for them.

团体成员身份意味着重新平衡；如果您的应用程序不是专为重新平衡而设计的，则重新平衡可能会影响尾部延迟和错误率。

Q: What is an offset, and what does “committing offsets” actually mean?

什么是偏移量，“提交偏移量”实际上意味着什么？

A:

  • Offset is the position in a partition log.

偏移量是分区日志中的位置。

  • Committing offsets is persisting “I have processed up to offset N” for a group/partition.

提交偏移量是对组/分区持久化“我已处理至偏移量 N”。

  • Commit timing defines your processing semantics:

提交时间定义了您的处理语义： - Commit before processing: at-most-once risk (loss on failure). 处理前提交：至多一次风险（失败损失）。 - Commit after processing: at-least-once risk (duplicates on retry). 处理后提交：至少一次风险（重试时重复）。

Q: “Exactly once” in Kafka: real or marketing?

卡夫卡中的“恰好一次”：真实还是营销？

A:

  • “Exactly once” is achievable for specific pipelines, but it’s not magic.

对于特定的管道来说，“恰好一次”是可以实现的，但这并不神奇。

  • You still need to define what “once” means end-to-end: reading, processing, and writing outputs.

您仍然需要定义“一次”意味着端到端：读取、处理和写入输出。

  • A common pattern is: write outputs and commit consumed offsets atomically (transactions), plus idempotent processing downstream.

常见模式是：以原子方式写入输出并提交消耗的偏移量（事务），以及下游的幂等处理。

  • In many real systems, the practical goal is “effectively once”: at-least-once delivery with idempotent writes.

在许多实际系统中，实际目标是“有效一次”：通过幂等写入进行至少一次交付。

Q: When should I use log compaction?

我什么时候应该使用日志压缩？

A:

  • Use compaction when the topic represents the latest value for a key (like a changelog table).

当主题表示键的最新值（如更改日志表）时，请使用压缩。

  • Tombstones (key with null value) represent deletes; compaction eventually removes older versions.

墓碑（具有空值的键）代表删除；压缩最终会删除旧版本。

  • Tradeoff: compaction is asynchronous and depends on segment/cleanup settings; deletes are not instantaneous.

权衡：压缩是异步的，并且取决于段/清理设置；删除不是即时的。

Q: What’s a good interview answer for “Kafka vs RabbitMQ”?

“Kafka vs RabbitMQ”的面试最佳答案是什么？

A:

  • Kafka: log + replay + high throughput + consumer pull model; good for event streams and multiple independent consumers.

Kafka：日志+重播+高吞吐量+消费者拉动模型；适合事件流和多个独立消费者。

  • RabbitMQ/queues: queue semantics + push delivery; good for work queues and per-message ack workflows.

RabbitMQ/queues：队列语义+推送传递；适合工作队列和每消息确认工作流程。

  • Ask what the system needs: replay, multiple consumers, long retention, and stream processing point toward Kafka.

询问系统需要什么：重放、多个消费者、长期保留以及面向 Kafka 的流处理点。

Q: Common failure modes and what to say out loud

常见的故障模式以及要大声说出来的内容

A:

  • Consumer lag grows: inspect partitions, max.poll.interval, processing time, downstream dependency latency, and rebalance churn.

消费者滞后增长：检查分区、max.poll.interval、处理时间、下游依赖延迟和重新平衡流失。

  • Hot partition: fix keying strategy, increase partitions carefully, or introduce a second-stage sharding.

热分区：修复keying策略，谨慎增加分区，或者引入第二阶段分片。

  • “Poison pill” messages: use DLQ, per-record try/catch, and idempotent retries; never stall an entire partition forever.

“毒丸”消息：使用 DLQ、每条记录的 try/catch 和幂等重试；永远不要永远停止整个分区。

  • Exactly-once confusion: clarify semantics and introduce idempotency keys or transactional boundaries.

恰好一次混淆：澄清语义并引入幂等键或事务边界。

Redis (Cache + Coordination)

这一节讲 Redis 在缓存、短期状态、幂等键、限流和协调中的作用，以及 eviction、持久化和 cluster 的取舍。

Q: When do you use Redis in production systems?

什么时候在生产系统中使用Redis？

A:

  • As a cache for hot metadata/features, to reduce p95 latency and database load.

作为热门元数据/功能的缓存，以减少 p95 延迟和数据库负载。

  • As a coordination store: distributed locks/leasing, rate limiting, idempotency keys, short-lived workflow state.

作为协调存储：分布式锁/租赁、速率限制、幂等性密钥、短期工作流状态。

  • Sometimes as a lightweight pub/sub or queue, but for durable pipelines Kafka is usually preferred.

有时作为轻量级的发布/订阅或队列，但对于持久管道，Kafka 通常是首选。

Q: What are Redis eviction policies and why do they matter?

什么是 Redis 驱逐政策以及它们为何重要？

A:

  • Eviction is what Redis does when memory is full (maxmemory reached).

驱逐是 Redis 在内存已满（达到“maxmemory”）时执行的操作。

  • Policies include:

政策包括： - noeviction: writes fail (good when you want strong safety for non-cache keys). noeviction：写入失败（当您希望非缓存键具有强大的安全性时，这很好）。 - allkeys-lru / allkeys-lfu: evict across all keys (classic cache behavior). allkeys-lru / allkeys-lfu：逐出所有键（经典缓存行为）。 - volatile-*: only evict keys with TTL (useful when mixing cache and “persistent” keys). 易失性-*：仅逐出具有 TTL 的键（在混合缓存和“持久”键时有用）。

  • Interview tradeoff: eviction policy changes correctness. If eviction can drop required keys, your app must tolerate it.

面试权衡：驱逐政策改变了正确性。如果驱逐可能会删除所需的密钥，那么您的应用程序必须容忍它。

Q: How does Redis LRU/LFU behave in practice?

Redis LRU/LFU 在实践中表现如何？

A:

  • Redis eviction is approximate: it samples keys rather than tracking perfect LRU across everything.

Redis 驱逐是近似的：它对键进行采样，而不是在所有内容上跟踪完美的 LRU。

  • Increasing sample size improves eviction quality but may increase CPU overhead.

增加样本大小可以提高逐出质量，但可能会增加 CPU 开销。

Q: Redis persistence: RDB vs AOF?

Redis 持久化：RDB 与 AOF？

A:

  • RDB snapshots: periodic point-in-time dumps (fast recovery, potential data loss since last snapshot).

RDB 快照：定期时间点转储（快速恢复，自上次快照以来潜在的数据丢失）。

  • AOF: append-only log of writes (better durability, can be heavier I/O).

AOF：仅附加写入日志（更好的耐用性，可以是更重的 I/O）。

  • For pure cache, persistence is often unnecessary; for coordination/state, pick durability based on business tolerance.

对于纯粹的缓存来说，持久化往往是不必要的；对于协调/状态，根据业务容忍度选择持久性。

Q: Redis Cluster: what do I need to know for interviews?

Redis Cluster：面试时我需要了解什么？

A:

  • Redis Cluster shards by hash slots (16384 slots), not ring-based consistent hashing.

Redis Cluster 按哈希槽（16384 个槽）进行分片，而不是基于环的一致性哈希。

  • Clients are cluster-aware and follow redirections (MOVED / ASK) during resharding.

客户端具有集群感知能力，并在重新分片期间遵循重定向（“MOVED”/“ASK”）。

  • Replication is async; failover is possible but implies eventual consistency tradeoffs.

复制是异步的；故障转移是可能的，但意味着最终的一致性权衡。

  • Multi-key operations work only when keys map to the same slot (use hash tags when you must).

仅当键映射到同一槽时，多键操作才有效（必要时使用哈希标签）。

Q: Common Redis pitfalls

常见的 Redis 陷阱

A:

  • Hot keys: one key becomes a bottleneck; shard/partition keys and avoid large fan-in.

热键：一键成为瓶颈；分片/分区键并避免大量扇入。

  • Unbounded keys/TTL: missing TTL on cache keys causes memory blowups.

无界键/TTL：缓存键上缺少 TTL 会导致内存爆炸。

  • High cardinality metrics in Redis: don’t turn Redis into a metrics DB.

Redis 中的高基数指标：不要将 Redis 变成指标数据库。

  • Using Redis as “source of truth” without durability/replication plan.

使用 Redis 作为“事实来源”，但没有持久性/复制计划。

这一节讲为什么大媒体文件要走对象存储和 presigned URL，而不是穿过编排服务。重点是 payload、恢复、成本和生命周期管理。

如何围绕对象存储设计长期运行的多模态作业？

Distributed Systems Patterns for AI Platforms

这一节总结 AI 平台中的幂等、重试、backpressure、熔断、bulkhead、load shedding 和 tail latency 分析。

Q: What does idempotency mean in an AI workflow?

幂等性在人工智能工作流程中意味着什么？

A:

  • Repeating the same step with the same idempotency key should not create duplicate side effects.

使用相同的幂等键重复相同的步骤不应产生重复的副作用。

  • Examples: do not insert duplicate vectors, do not charge twice, do not write two result objects for the same completed step.

示例：不要插入重复的向量、不要充电两次、不要为同一已完成的步骤写入两个结果对象。

  • Use stable request IDs, step IDs, content hashes, and conditional writes.

使用稳定的请求 ID、步骤 ID、内容哈希和条件写入。

Q: How do you design retries safely?

如何安全地设计重试？

A:

  • Retry only when the error is likely transient.

仅当错误可能是暂时性的时才重试。

  • Use exponential backoff and jitter.

使用指数退避和抖动。

  • Put a cap on attempts and total time.

限制尝试次数和总时间。

  • Avoid retrying non-idempotent side effects without a dedupe key.

避免在没有重复数据删除密钥的情况下重试非幂等副作用。

  • Use DLQs or manual repair paths for poison messages.

对有害消息使用 DLQ 或手动修复路径。

Q: What is backpressure?

什么是背压？

A:

  • Backpressure is how a system tells upstream producers to slow down when downstream capacity is saturated.

背压是当下游产能饱和时系统告诉上游生产商放慢速度的方式。

  • In AI systems, downstream saturation may be GPU memory, token backlog, vector DB latency, object storage bandwidth, or external API quotas.

在人工智能系统中，下游饱和可能是 GPU 内存、代币积压、向量数据库延迟、对象存储带宽或外部 API 配额。

  • Mechanisms include bounded queues, admission control, rate limits, load shedding, and degraded execution modes.

机制包括有界队列、准入控制、速率限制、load shedding和降级执行模式。

Q: Circuit breaker vs timeout vs bulkhead?

熔断器、超时、隔板？

A:

  • Timeout: stop waiting after a budget.

超时：预算结束后停止等待。

  • Circuit breaker: stop sending traffic to a failing dependency for a period.

熔断器：在一段时间内停止向失败的依赖项发送流量。

  • Bulkhead: isolate capacity so one dependency or tenant cannot consume everything.

Bulkhead：隔离容量，以便一个依赖项或tenant无法消耗所有内容。

  • In AI platforms, these protect scarce GPU and retrieval resources from cascading failures.

在人工智能平台中，这些可以保护稀缺的 GPU 和检索资源免受级联故障的影响。

Q: What is load shedding in inference?

推理中的减载是什么？

A:

  • Refuse or degrade lower-priority work before the whole system collapses.

在整个系统崩溃之前拒绝或降低优先级较低的工作。

  • Examples: reduce top-k, skip reranking, shorten max generation tokens, route to smaller model, defer batch jobs, or return cached results.

示例：减少 top-k、跳过重新排名、缩短最大生成token、路由到更小的模型、推迟批处理作业或返回缓存结果。

  • The key is to define degradation policy before an incident.

关键是在事件发生之前定义降级策略。

Q: How do you reason about p95/p99 latency?

您如何推断 p95/p99 延迟？

A:

  • Tail latency is usually queueing plus stragglers, not just average service time.

尾部延迟通常是队列加上落后者，而不仅仅是平均服务时间。

  • Break latency into client upload, admission, queue wait, prefill, decode, retrieval, rerank, postprocess, and download.

将延迟分解为客户端上传、准入、队列等待、预填充、解码、检索、重新排序、后处理和下载。

  • Optimize the dominant term; do not blindly scale GPUs if queueing comes from object storage or retrieval.

优化主导项；如果队列来自对象存储或检索，请勿盲目扩展 GPU。

Layered Inference Stack (Ray Serve + Triton + vLLM)

这一节讲为什么 Ray Serve、Triton、vLLM 要分层使用。重点是不同 workload 的瓶颈不同：CPU 编排、tensor inference、LLM token/KV cache。

Q: Why build a layered inference stack instead of “one serving system for everything”?

为什么要构建分层推理堆栈而不是“适用于一切的一个服务系统”？

A:

  • Because bottlenecks differ by workload:

因为瓶颈因工作负载而异： - Orchestration/fan-out: CPU + async I/O + retries + business logic. 编排/扇出：CPU + 异步 I/O + 重试 + 业务逻辑。 - Embeddings/rankers: dense tensor math, throughput-driven batching. 嵌入/排名器：密集张量数学、吞吐量驱动的批处理。 - LLM generation: KV-cache memory, token scheduling, TTFT/TPOT tradeoffs. LLM 生成：KV 缓存、token调度、TTFT/TPOT 权衡。

  • Layering lets each tier optimize for its dominant constraints without forcing a least-common-denominator architecture.

分层使每一层能够针对其主要约束进行优化，而无需强制采用最小公分母架构。

Q: What’s the clean boundary between Ray Serve and Triton/vLLM?

Ray Serve 和 Triton/vLLM 之间的明确界限是什么？

A:

  • Ray Serve: request shaping, routing, DAG composition, fan-out/fan-in, per-step timeouts, and backpressure.

Ray Serve：请求整形、路由、DAG 组合、扇出/扇入、每步超时和背压。

  • Triton: high-throughput stateless tensor inference (embeddings, rankers, CV), using dynamic batching + concurrent model execution.

Triton：高吞吐量无状态张量推理（嵌入、排序、CV），使用动态批处理+并发模型执行。

  • vLLM: autoregressive generation with KV-cache aware scheduling (continuous batching), streaming tokens, and memory-first optimization.

vLLM：自回归生成，具有 KV 缓存感知调度（连续批处理）、流token和内存优先优化。

Q: Where does a layered stack usually fail first?

分层堆栈通常首先在哪里失败？

A:

  • Ray tier: too much synchronous work per request, event-loop blocking, unbounded fan-out, or “retry storms” when downstream is slow.

Ray 层：每个请求的同步工作过多、事件循环阻塞、无限制的扇出或下游缓慢时的“重试风暴”。

  • Triton tier: poor batching config, too many model instances fighting for VRAM, or too much host<->device transfer.

Triton 层：批处理配置不佳、争夺 VRAM 的模型实例过多或主机 <-> 设备传输过多。

  • vLLM tier: KV-cache exhaustion (OOM), long-context requests starving short ones (fairness), or token backlog mis-signaled by QPS-only autoscaling.

vLLM 层：KV 缓存耗尽 (OOM)、长上下文请求饥饿短请求（公平性）或仅 QPS 自动缩放错误发出token积压信号。

Q: What are the “right” scaling signals for LLM-heavy workloads?

对于 LLM 密集型工作负载来说，“正确”的扩展信号是什么？

A:

  • Prefer token-based signals (waiting tokens, decode tokens/s, TTFT/TPOT) over QPS, because work per request varies wildly.

优先选择基于token的信号（等待token、解码token、TTFT/TPOT）而不是 QPS，因为每个请求的工作量差异很大。

  • Separate pools by latency class (interactive vs long-context vs batch) to avoid “one giant queue” pathologies.

按延迟类别（交互式、长上下文、批处理）分隔池，以避免“一个巨型队列”病态。

A:

  • Client batching is fragile (clients must coordinate batch sizes and timing).

客户端批处理很脆弱（客户端必须协调批处理大小和时间）。

  • Triton dynamic batching forms batches inside the server to improve throughput while trading some micro-latency for better GPU efficiency.

Triton 动态批处理在服务器内部形成批处理，以提高吞吐量，同时牺牲一些微延迟以获得更好的 GPU 效率。

您可以根据 SLO 进行调整：过于激进的批处理可能会损害 p95 延迟。

Q: When do you reach for Triton ensembles vs Ray Serve composition?

你什么时候会选择 Triton 合奏与 Ray Serve 组合？

A:

Ray Serve 组合更适合包括 I/O、检索、分支逻辑和异构计算（CPU + GPU + 外部服务）的多步骤工作流程。

  • Host<->device transfer dominates because preprocessing isn’t co-located or inputs are oversized.

vLLM + LLM Serving (GenAI Inference)

这一节讲 vLLM、KV cache、PagedAttention、TTFT、TPOT、continuous batching 和 token-aware scaling。

Q: Prefill vs decode: what’s the difference and why do we care?

预填充与解码：有什么区别以及我们为什么关心？

A:

  • Prefill: process the prompt/context to build KV cache (high parallelism, compute-heavy).

预填充：处理prompt/上下文以构建 KV 缓存（高并行性、计算量大）。

  • Decode: generate tokens one-by-one (less parallel per request, often memory/latency sensitive).

解码：一对一生成token（每个请求的并行性较低，通常对内存/延迟敏感）。

  • Serving systems optimize by batching across requests (continuous batching) to keep GPU busy.

服务系统通过跨请求批处理（连续批处理）来优化，以保持 GPU 繁忙。

Q: What is the KV cache and why is it the bottleneck?

什么是 KV 缓存？为什么它是瓶颈？

A:

  • KV cache stores attention keys/values for past tokens so you don’t recompute them every step.

KV 缓存存储过去标记的注意键/值，因此您不必每一步都重新计算它们。

  • Memory grows with: batch size * sequence length * layers * heads * head_dim * dtype.

内存随着以下因素增长：批量大小 * 序列长度 * 层数 * 头数 * head_dim * dtype。

  • The bottleneck is often GPU memory capacity and bandwidth, not FLOPS.

瓶颈通常是 GPU 内存容量和带宽，而不是 FLOPS。

Q: TTFT vs TPOT: how do you explain them?

TTFT 与 TPOT：您如何解释它们？

A:

  • TTFT (time to first token): user-perceived responsiveness.

TTFT（第一个token的时间）：用户感知的响应能力。

  • TPOT (time per output token): throughput during generation; impacts total completion time.

TPOT（每个输出token的时间）：生成期间的吞吐量；影响总完成时间。

  • Good interview habit: discuss both; optimizing one can harm the other.

良好的面试习惯：两者都讨论；优化其中一个可能会损害另一个。

Q: What is PagedAttention (vLLM) in one minute?

一分钟了解什么是 PagedAttention (vLLM)？

A:

  • PagedAttention manages KV cache memory in fixed-size blocks (pages) rather than requiring contiguous allocations.

PagedAttention 以固定大小的块（页）管理 KV 缓存内存，而不需要连续分配。

  • This reduces fragmentation and KV “waste”, so the system can pack more concurrent sequences.

这减少了碎片和 KV“浪费”，因此系统可以打包更多并发序列。

  • It borrows concepts from OS virtual memory: an indirection table maps logical tokens to physical blocks.

它借用了操作系统虚拟内存的概念：间接表将逻辑标记映射到物理块。

Q: What are the tradeoffs of PagedAttention?

PagedAttention 的权衡是什么？

A:

  • Requires attention kernel support for non-contiguous KV blocks (more software complexity).

需要注意内核对非连续 KV 块的支持（软件复杂性更高）。

  • Can introduce overhead/portability constraints compared to contiguous KV approaches.

与连续 KV 方法相比，可能会引入开销/可移植性限制。

  • There are alternative approaches (e.g., designs that keep contiguous virtual memory and use demand paging) with different tradeoffs.

还有一些具有不同权衡的替代方法（例如，保留连续虚拟内存并使用需求分页的设计）。

Q: What’s a credible alternative to PagedAttention, and why might someone choose it?

PagedAttention 的可靠替代方案是什么？为什么有人会选择它？

A:

  • One alternative line of work is to keep the KV cache contiguous in *virtual* memory and rely on demand paging / runtime memory management to reduce fragmentation, instead of modifying attention kernels to operate on non-contiguous blocks.

一种替代工作是保持 KV 缓存在“虚拟”内存中连续，并依靠需求分页/运行时内存管理来减少碎片，而不是修改注意力内核以在非连续块上操作。

  • Interview framing: the trade is typically “kernel complexity and portability” vs “memory manager complexity and driver/runtime dependence”; the right answer depends on your deployment constraints and supported kernels.

面试框架：交易通常是“内核复杂性和可移植性”与“内存管理器复杂性和驱动程序/运行时依赖性”；正确的答案取决于您的部署限制和支持的内核。

Q: How does vLLM schedule requests, and why are token-based signals important?

vLLM 如何安排请求，以及为什么基于token的信号很重要？

A:

  • In LLM serving, “one request” is not a stable unit of work: token lengths vary wildly.

在 LLM 服务中，“一个请求”并不是一个稳定的工作单元：token长度变化很大。

  • Better scaling/routing signals include waiting tokens, TTFT, TPOT, and KV-cache usage, not just QPS.

更好的扩展/路由信号包括等待token、TTFT、TPOT 和 KV 缓存使用，而不仅仅是 QPS。

  • Pools are often separated by workload: interactive short, long-context, batch eval, etc.

池通常按工作负载分开：交互式短上下文、长上下文、批量评估等。

Q: Chunked prefill: what problem does it solve?

分块预填充：它解决了什么问题？

A:

  • Very long prompts can monopolize prefill capacity and worsen tail latency.

很长的prompt会独占预填充容量并恶化尾部延迟。

  • Chunked prefill limits how much prompt work happens per scheduling step so shorter prompts can “jump the queue”.

分块预填充限制了每个调度步骤发生的prompt工作量，因此较短的prompt可以“插队”。

  • This is a fairness/latency optimization; it can trade throughput for better p95 responsiveness.

这是公平/延迟优化；它可以用吞吐量换取更好的 p95 响应能力。

Q: What metrics do you watch for vLLM health?

您关注 vLLM 健康状况的哪些指标？

A:

  • TTFT histogram (p50/p95/p99).

TTFT 直方图 (p50/p95/p99)。

  • TPOT or inter-token latency.

TPOT 或token间延迟。

  • Waiting tokens/backlog.

等待代币/积压。

  • KV cache usage/residency.

KV 缓存使用/驻留。

  • GPU memory pressure and OOM events.

GPU 内存压力和 OOM 事件。

  • Error rate/timeouts.

错误率/超时。

Q: When do you use vLLM vs Triton?

什么时候使用 vLLM 和 Triton？

A:

  • vLLM: autoregressive generation (KV cache + token scheduling dominate).

vLLM：自回归生成（KV缓存+token调度主导）。

  • Triton: stateless tensor inference (embeddings, rankers, CV/classifiers) where batching and per-model scheduling help.

Triton：无状态张量推理（嵌入、排名器、CV/分类器），批处理和按模型调度有帮助。

  • In a layered inference stack, Ray Serve composes the graph, while vLLM/Triton own GPU-heavy inference.

在分层推理堆栈中，Ray Serve 组成图形，而 vLLM/Triton 拥有 GPU 密集型推理。

Q: What’s the most common “gotcha” when debugging vLLM latency?

调试 vLLM 延迟时最常见的“陷阱”是什么？

A:

  • People over-focus on average QPS and ignore token variability.

人们过度关注平均 QPS 而忽略了 token 的可变性。

  • A better approach is to split the problem into:

更好的方法是将问题分解为： - Prefill saturation (prompt-heavy): TTFT rises, GPU compute may be the limiter. 预填充饱和（prompt重）：TTFT 上升，GPU 计算可能是限制因素。 - Decode saturation (generation-heavy): TPOT rises, KV-cache/memory bandwidth is often the limiter. 解码饱和（生成重）：TPOT 上升，KV 缓存/内存带宽通常是限制因素。 - Queueing: num_requests_waiting / waiting tokens rise; autoscaling or admission control is needed. 排队：num_requests_waiting/等待token上升；需要自动缩放或准入控制。

Q: What knobs usually matter most for stabilizing LLM serving under load?

对于稳定负载下的 LLM 服务，哪些旋钮通常最重要？

A:

  • Admission control by tokens/context length (protect TTFT for interactive workloads).

通过token/上下文长度进行准入控制（保护交互式工作负载的 TTFT）。

  • Pool separation (interactive vs batch).

池分离（交互式与批量）。

  • Prompt caching / prefix caching when traffic has shared prefixes.

当流量具有共享前缀时prompt缓存/前缀缓存。

  • Chunked prefill / fairness scheduling to prevent long prompts from monopolizing prefill.

分块预填充/公平调度，以防止长prompt垄断预填充。

GPU Hardware: MIG + Multi-Tenancy

这一节讲 MIG、time-slicing、多tenant隔离和 GPU 资源切分。重点是利用率、隔离、灵活性之间的权衡。

Q: What is MIG and when is it useful?

什么是 MIG？什么时候有用？

A:

  • MIG (Multi-Instance GPU) partitions one physical GPU into multiple hardware-isolated GPU instances with dedicated resources.

MIG（多实例 GPU）将一个物理 GPU 划分为多个具有专用资源的硬件隔离的 GPU 实例。

  • It is useful for multi-tenant inference where models don’t need a full GPU and you want better packing/isolation.

它对于多tenant推理非常有用，其中模型不需要完整的 GPU 并且您需要更好的打包/隔离。

Q: MIG vs time-slicing: how do you explain the tradeoff?

MIG 与时间切片：您如何解释其中的权衡？

A:

  • MIG: spatial partitioning (hard isolation of memory/SM slices), predictable performance, less interference.

MIG：空间分区（内存/SM片的硬隔离），性能可预测，干扰更少。

  • Time-slicing: temporal sharing, easier to allocate full memory to one job at a time, but more contention and less predictability.

时间切片：时间共享，更容易一次将完整内存分配给一项作业，但竞争更多，可预测性更低。

  • Interview framing: choose based on workload shapes, latency SLOs, and isolation requirements.

面试框架：根据工作负载形状、延迟 SLO 和隔离要求进行选择。

Q: MIG gotchas to mention

MIG 值得一提的问题

A:

  • Not all GPUs support MIG; it’s generally datacenter GPUs.

并非所有 GPU 都支持 MIG；通常是数据中心 GPU。

  • Partitioning can reduce flexibility for large models and long-context requests.

分区会降低大型模型和长上下文请求的灵活性。

  • Operational complexity: provisioning profiles, scheduling, and ensuring the right workloads go to the right slices.

操作复杂性：配置配置文件、调度并确保正确的工作负载分配到正确的切片。

Q: Kubernetes + MIG: what are the “strategies” people talk about (none/single/mixed)?

Kubernetes + MIG：人们谈论的“策略”是什么（无/单一/混合）？

A:

  • none: no MIG devices exposed; pods request full GPUs.

none：没有暴露 MIG 设备； Pod 请求完整的 GPU。

  • single: a node exposes a single MIG geometry (all GPUs on the node partitioned the same way).

single：节点公开单个 MIG 几何体（节点上的所有 GPU 以相同方式分区）。

  • mixed: a node can expose a mixture of MIG profiles (and potentially non-MIG GPUs depending on configuration), giving more packing flexibility but increasing operational complexity.

“混合”：节点可以公开 MIG 配置文件的混合（以及可能的非 MIG GPU，具体取决于配置），提供更多的打包灵活性，但增加了操作复杂性。

Q: Time-slicing vs MIG: what extra operational footguns show up?

时间切片 vs 米格：会出现哪些额外的作战枪械？

A:

  • Time-slicing improves utilization for small models but does not provide memory isolation; noisy neighbors can hurt tail latency.

时间切片提高了小模型的利用率，但不提供内存隔离；嘈杂的邻居会损害尾部延迟。

  • Telemetry attribution is harder in shared modes; you may not be able to reliably attribute per-pod utilization depending on your monitoring stack.

在共享模式下，遥测归因更加困难；根据您的监控堆栈，您可能无法可靠地确定每个 Pod 的利用率。

Kubernetes Autoscaling (KEDA/HPA/KubeRay) + Rollout Patterns

这一节讲 HPA、KEDA、KubeRay 和 Ray Serve autoscaling 如何协同，以及 GPU/LLM workload 的冷启动和扩缩容风险。

Q: What’s the interview-friendly difference between HPA and KEDA?

HPA 和 KEDA 之间的面试友好性有何区别？

A:

  • HPA is Kubernetes-native autoscaling driven by resource metrics (and optionally external metrics with adapters).

HPA 是由资源指标（以及带有适配器的可选外部指标）驱动的 Kubernetes 原生自动缩放。

  • KEDA specializes in event-driven autoscaling (Kafka lag, queue depth, etc.) and supports scale-to-zero patterns more naturally.

KEDA 专注于事件驱动的自动缩放（Kafka 滞后、队列深度等），并更自然地支持缩放到零模式。

Q: What’s a real “scale-to-zero” gotcha with Kafka lag scaling?

Kafka 滞后缩放真正的“缩放到零”问题是什么？

A:

  • If a consumer group has no committed offsets (new group or deleted offsets), autoscaling logic can misinterpret lag (or treat it as invalid) and fail to scale up.

如果消费者组没有提交的偏移量（新组或已删除的偏移量），则自动缩放逻辑可能会误解滞后（或将其视为无效）并无法扩展。

  • In practice you must define how “no offset” should behave (earliest vs latest) and configure your scaler accordingly.

在实践中，您必须定义“无偏移”的行为方式（最早与最新）并相应地配置缩放器。

Q: How do you talk about Ray upgrades on Kubernetes without sounding hand-wavy?

您如何谈论 Kubernetes 上的 Ray 升级而不听起来很夸张？

A:

  • Mention that RayService supports zero-downtime upgrade strategies, but pure blue/green can be too expensive for GPU-heavy deployments.

值得一提的是，RayService 支持零停机升级策略，但纯蓝/绿对于 GPU 密集型部署来说可能过于昂贵。

  • Incremental upgrade strategies exist to avoid duplicating the full cluster capacity while still maintaining availability.

增量升级策略的存在是为了避免复制整个集群容量，同时仍然保持可用性。

Kubernetes + GPU Scheduling Deep Dive

这一节深入 GPU 在 Kubernetes 中的调度、node pool、warm pool、fragmentation、多个 autoscaler 的反馈回路。

Q: How does Kubernetes schedule GPUs?

Kubernetes 如何调度 GPU？

A:

  • Kubernetes does not understand "GPU memory" or "model size" by default. It schedules based on resources exposed by device plugins, such as nvidia.com/gpu or MIG-specific resources.

默认情况下，Kubernetes 不理解“GPU 内存”或“模型大小”。它根据设备插件公开的资源进行调度，例如“nvidia.com/gpu”或特定于 MIG 的资源。

  • GPU requests usually behave like integer resources; you request one or more devices, and the device plugin allocates them to the pod.

GPU 请求通常表现为整数资源；您请求一台或多台设备，设备插件会将它们分配给 pod。

  • Workload-aware placement still needs policy: node labels, taints/tolerations, affinity, topology spread, PriorityClass, custom schedulers, or controller logic.

工作负载感知放置仍然需要策略：节点标签、污点/容忍、亲和性、拓扑扩展、PriorityClass、自定义调度程序或控制器逻辑。

Q: What does a good GPU placement decision consider?

一个好的 GPU 放置决策会考虑什么？

A:

  • Model memory footprint, context length, latency class, batchability, expected token rate, tenant priority, and isolation requirements.

对内存占用、上下文长度、延迟类别、可批处理性、预期token率、tenant优先级和隔离要求进行建模。

  • Hardware capability: GPU generation, VRAM, MIG profile, NVLink topology, local NVMe cache, and network proximity to storage/retrieval systems.

硬件功能：GPU 生成、VRAM、MIG 配置文件、NVLink 拓扑、本地 NVMe 缓存以及与存储/检索系统的网络邻近度。

  • Operational constraints: warm capacity, upgrade domain, current queue pressure, and whether the workload can tolerate preemption or cold start.

操作约束：热容量、升级域、当前队列压力以及工作负载是否可以容忍抢占或冷启动。

Q: Why is GPU fragmentation a platform problem?

为什么 GPU 碎片是一个平台问题？

A:

  • If many small workloads occupy whole GPUs, large LLM or video workloads cannot find contiguous full-GPU capacity.

如果许多小型工作负载占用整个 GPU，则大型 LLM 或视频工作负载将无法找到连续的完整 GPU 容量。

  • MIG can improve packing for small/medium inference models, but it can make full-GPU workloads harder to place later.

MIG 可以改进中小型推理模型的打包，但它可能会使以后更难放置全 GPU 工作负载。

  • A good resource controller reserves certain pools for full-GPU jobs and uses MIG/time-slicing only where latency and memory isolation allow it.

一个好的资源控制器会为全 GPU 作业保留某些池，并仅在延迟和内存隔离允许的情况下使用 MIG/时间切片。

Q: How do warm pools help AI serving?

温池如何助力AI服务？

A:

  • AI model cold start is expensive: pull image, download weights, initialize runtime, load model to GPU, compile kernels/engines, warm caches.

AI 模型冷启动成本高昂：拉取镜像、下载权重、初始化运行时、将模型加载到 GPU、编译内核/引擎、热缓存。

  • Warm pools keep a minimum number of ready replicas or partially initialized nodes so interactive requests avoid cold-start latency.

温池保留最少数量的就绪副本或部分初始化的节点，以便交互式请求避免冷启动延迟。

  • Tradeoff: warm capacity costs money even when idle, so you reserve it for high-value or high-variance workloads.

权衡：即使闲置时，热容量也会产生费用，因此您可以将其保留用于高价值或高变化的工作负载。

Q: What are the autoscaling layers in a real AI platform?

真正的人工智能平台中的自动缩放层是什么？

A:

  • Request-level: admission control and queueing.

请求级：准入控制和排队。

  • Serving-level: Ray Serve replicas, Triton model instances, vLLM engines.

服务级别：Ray Serve 副本、Triton 模型实例、vLLM 引擎。

  • Pod-level: HPA/KEDA based on CPU, queue depth, request concurrency, token backlog, or Prometheus metrics.

Pod 级别：基于 CPU、队列深度、请求并发度、token积压或 Prometheus 指标的 HPA/KEDA。

  • Cluster-level: Ray worker nodes and Kubernetes node pools.

集群级别：Ray工作节点和Kubernetes节点池。

  • Capacity-level: cloud GPU provisioning, reserved instances, spot/preemptible pools, and warm nodes.

容量级别：云GPU配置、预留实例、现货/抢占池和热节点。

Q: What can go wrong when several autoscalers act at once?

当多个自动缩放器同时运行时会出现什么问题？

A:

  • Feedback loops: one layer scales up while another scales down.

反馈循环：一层放大，而另一层缩小。

  • Cold-start blindness: the scaler sees queue growth but new GPUs are not ready for minutes.

冷启动盲目性：缩放器看到队列增长，但新的 GPU 几分钟内还没有准备好。

  • Oscillation: aggressive scale-down removes capacity just before the next traffic burst.

振荡：在下一次流量爆发之前，激进的缩减会消除容量。

  • Fixes: dampening windows, cooldowns, min capacity, per-class queues, and clear ownership of each scaling decision.

修复：抑制窗口、冷却时间、最小容量、每类队列以及每个扩展决策的明确所有权。

Q: How do you explain KEDA vs KubeRay vs Ray Serve autoscaling?

您如何解释 KEDA、KubeRay 和 Ray Serve 自动缩放？

A:

  • KEDA scales Kubernetes workloads from external/event metrics such as Kafka lag or Prometheus metrics.

KEDA 根据外部/事件指标（例如 Kafka 滞后或 Prometheus 指标）扩展 Kubernetes 工作负载。

  • KubeRay manages Ray clusters and can grow/shrink Ray worker groups.

KubeRay 管理 Ray 集群并可以增长/收缩 Ray 工作组。

  • Ray Serve scales application deployments based on serving-level metrics such as ongoing requests per replica.

Ray Serve 根据服务级别指标（例如每个副本的持续请求）来扩展应用程序部署。

  • In a layered platform, KEDA may scale worker pods, KubeRay may add cluster capacity, and Ray Serve may adjust replicas inside the Ray application.

在分层平台中，KEDA 可以扩展工作节点，KubeRay 可以增加集群容量，Ray Serve 可以调整 Ray 应用程序内的副本。

Project-Critical Resource Control Q&A

这一节是 Project 1 简历 bullet 的核心问答：MIG-aware placement、bin-packing、warm capacity、token telemetry、latency class、model profile 和 cost budget。

Q: What does “MIG-aware placement” mean in this project?

“MIG 感知布局”在此项目中意味着什么？

A:

  • It means the platform does not treat every GPU as a generic nvidia.com/gpu unit.

这意味着该平台不会将每个 GPU 视为通用的“nvidia.com/gpu”单元。

  • It understands whether a workload can run on a MIG slice, needs a full GPU, or should avoid MIG because it needs full memory bandwidth, large KV cache, or flexible batching.

它了解工作负载是否可以在 MIG 切片上运行、需要完整的 GPU，还是应该避免 MIG，因为它需要完整的内存带宽、大型 KV 缓存或灵活的批处理。

  • Example: small embedding/ranking models may fit on 1g/2g/3g MIG profiles, while long-context LLM generation or heavy video models may need full A100/H100 capacity.

例如：小型嵌入/排名模型可能适合“1g/2g/3g”MIG 配置文件，而长上下文 LLM 生成或重型视频模型可能需要完整的 A100/H100 容量。

  • The placement layer maps model profile + latency class + memory footprint + tenant priority to an appropriate GPU pool or MIG resource type.

放置层将模型配置文件+延迟类别+内存占用+tenant优先级映射到适当的GPU池或MIG资源类型。

Q: How would you implement MIG-aware placement on Kubernetes?

您将如何在 Kubernetes 上实现 MIG 感知放置？

A:

  • Configure NVIDIA GPU Operator/device plugin with a MIG strategy such as single or mixed.

使用 MIG 策略（例如“单一”或“混合”）配置 NVIDIA GPU Operator/设备插件。

  • Label/taint GPU node pools by GPU type, MIG profile, workload class, and tenant isolation policy.

按 GPU 类型、MIG 配置文件、工作负载类和tenant隔离策略标记/污染 GPU 节点池。

  • Expose MIG resources such as nvidia.com/mig-1g.10gb or similar device-plugin resource names.

公开 MIG 资源，例如“nvidia.com/mig-1g.10gb”或类似的设备插件资源名称。

  • Have the platform planner choose the target resource class, then emit Kubernetes/Ray/KubeRay placement hints through node selectors, affinity, tolerations, priority class, and resource requests.

让平台规划者选择目标资源类，然后通过节点选择器、关联性、容忍度、优先级和资源请求发出 Kubernetes/Ray/KubeRay 放置prompt。

  • Keep a placement inventory so the planner knows current free slices, full GPUs, warm replicas, and fragmentation risk.

保留放置库存，以便规划人员了解当前的空闲切片、完整的 GPU、热副本和碎片风险。

Q: What is the difference between Kubernetes scheduling and your resource controller placement?

Kubernetes 调度和资源控制器放置有什么区别？

A:

  • Kubernetes scheduling places Pods on Nodes based on declared resources and constraints.

Kubernetes 调度根据声明的资源和约束将 Pod 放置在节点上。

  • The resource controller decides what the Pod should ask for in the first place: full GPU vs MIG slice, interactive pool vs batch pool, warm replica vs cold start, vLLM vs Triton pool, and fallback path.

资源控制器决定 Pod 首先应该要求什么：完整 GPU 与 MIG 切片、交互池与批处理池、热副本与冷启动、vLLM 与 Triton 池以及fallback路径。

  • In interview language: Kubernetes is the low-level scheduler; the AI platform resource controller is the workload-aware policy layer above it.

用面试语言来说：Kubernetes 是底层调度器； AI 平台资源控制器是其上方的工作负载感知策略层。

Q: What does “proactive workload bin-packing” mean?

“主动工作负载装箱”是什么意思？

A:

  • Bin-packing means consolidating compatible workloads onto fewer GPUs/nodes or smaller slices to reduce waste and free scarce full-GPU capacity.

装箱意味着将兼容的工作负载整合到更少的 GPU/节点或更小的切片上，以减少浪费并释放稀缺的完整 GPU 容量。

  • “Proactive” means doing it before capacity is exhausted: analyzing placement efficiency, queue pressure, model profiles, and warm-pool needs, then shifting batch/low-priority workloads or changing scale targets.

“主动”意味着在容量耗尽之前采取行动：分析放置效率、队列压力、模型配置文件和热池需求，然后转移批量/低优先级工作负载或更改规模目标。

  • The goal is not simply “pack everything tightly”; it is to improve utilization while preserving p95 latency, isolation, and rollback safety.

目标不仅仅是“把所有东西都包紧”；它是为了提高利用率，同时保留 p95 延迟、隔离和回滚安全性。

Q: How does Kubernetes support bin-packing?

Kubernetes 如何支持装箱？

A:

  • The kube-scheduler has NodeResourcesFit scoring strategies such as MostAllocated and RequestedToCapacityRatio that can favor nodes with higher existing allocation.

kube-scheduler 具有“NodeResourcesFit”评分策略，例如“MostAlowned”和“RequestedToCapacityRatio”，可以有利于具有更高现有分配的节点。

  • This helps consolidate general CPU/memory or extended-resource workloads.

这有助于整合一般 CPU/内存或扩展资源工作负载。

  • For AI workloads, native bin-packing is not enough because the scheduler does not understand model shape, KV cache, batching behavior, tenant priority, or whether a model should reserve full GPU memory.

对于 AI 工作负载，原生装箱是不够的，因为调度程序不了解模型形状、KV 缓存、批处理行为、tenant优先级或模型是否应保留完整的 GPU 内存。

Q: What are the constraints in GPU bin-packing?

GPU装箱有哪些限制？

A:

  • Hard constraints: GPU/MIG profile, VRAM, CUDA capability, model compatibility, tenant isolation, privacy policy, and availability zone/region.

硬约束：GPU/MIG 配置文件、VRAM、CUDA 功能、模型兼容性、tenant隔离、隐私策略和可用区域/区域。

  • Soft constraints: latency class, expected queue pressure, batchability, warm-cache benefit, cost budget, and co-location with retrieval/storage.

软约束：延迟等级、预期队列压力、批处理能力、热缓存优势、成本预算以及检索/存储的共置。

  • Anti-goals: do not pack an interactive vLLM service next to a noisy batch job if it hurts TTFT/TPOT or p95 latency.

反目标：如果会损害 TTFT/TPOT 或 p95 延迟，请勿将交互式 vLLM 服务打包在嘈杂的批处理作业旁边。

Q: When can bin-packing make things worse?

什么时候垃圾箱会让事情变得更糟？

A:

  • It can create noisy-neighbor interference under time-slicing or shared CPU/network paths.

它可能会在时间切片或共享 CPU/网络路径下产生噪声邻居干扰。

  • It can fragment GPUs so large LLM/video jobs cannot find full devices.

它会使 GPU 碎片化，导致大型 LLM/视频作业无法找到完整的设备。

  • It can reduce fault isolation by putting too many important replicas on one node or zone.

它可以通过在一个节点或区域上放置过多的重要副本来减少故障隔离。

  • It can make rollouts risky if many models share the same physical failure domain.

如果许多模型共享相同的物理故障域，那么部署可能会存在风险。

Q: How do you explain “model profile” in the resource controller?

您如何解释资源控制器中的“模型配置文件”？

A:

  • A model profile is the platform’s compact description of runtime needs.

模型配置文件是平台对运行时需求的简洁描述。

  • It can include model type, memory footprint, preferred runtime, batchability, precision, max input/output tokens, typical sequence length, GPU type, MIG compatibility, cold-start time, and SLO class.

它可以包括模型类型、内存占用、首选运行时间、批处理能力、精度、最大输入/输出标记、典型序列长度、GPU 类型、MIG 兼容性、冷启动时间和 SLO 类别。

  • The planner uses this profile to decide routing, admission, autoscaling, and placement.

规划者使用此配置文件来决定路由、准入、自动缩放和放置。

Q: How do “latency class” and “queue pressure” affect routing?

“延迟等级”和“队列压力”如何影响路由？

A:

  • Latency class says how urgent the request is: interactive, async, batch, backfill, evaluation, or low-priority maintenance.

延迟类别表示请求的紧急程度：交互式、异步、批处理、回填、评估或低优先级维护。

  • Queue pressure says whether the target pool is already overloaded.

队列压力表示目标池是否已经过载。

  • If an interactive pool has high queue pressure, the platform may route to a smaller/faster model, a warmer pool, a regional fallback, or reject/defer low-priority work before user-facing p95 degrades.

如果交互池具有较高的队列压力，平台可能会在面向用户的 p95 降级之前路由到更小/更快的模型、更温暖的池、区域fallback或拒绝/推迟低优先级工作。

Q: What is “cost budget” in placement?

什么是安置中的“成本预算”？

A:

  • Cost budget is a policy constraint that prevents every request from taking the most expensive GPU path.

成本预算是一种策略约束，可防止每个请求都采用最昂贵的 GPU 路径。

  • Example: a low-priority batch embedding job may wait for cheap capacity, while an interactive creative-editing workflow may use warm H100/A100 capacity.

示例：低优先级批量嵌入作业可能会等待廉价容量，而交互式创意编辑工作流程可能会使用热 H100/A100 容量。

  • It can be enforced by tenant quota, feature tier, latency class, max tokens, max batch size, or allowed runtime targets.

它可以通过tenant配额、特征层、延迟类别、最大token、最大批量大小或允许的运行时目标来强制执行。

Q: Why use live token telemetry for autoscaling?

为什么使用实时token遥测进行自动缩放？

A:

  • LLM request cost is token-shaped, not request-shaped.

LLM 请求成本是代币形状的，而不是请求形状的。

  • Two requests can have the same QPS but wildly different prompt length, output length, KV-cache pressure, and decode time.

两个请求可以具有相同的 QPS，但prompt长度、输出长度、KV 缓存压力和解码时间截然不同。

  • Live token telemetry gives better scaling signals: waiting tokens, active sequences, TTFT, TPOT, decode tokens/sec, prefill backlog, and KV-cache utilization.

实时token遥测提供更好的扩展信号：等待token、活动序列、TTFT、TPOT、解码token/秒、预填充积压和 KV 缓存利用率。

Q: How would KEDA/KubeRay use token telemetry?

KEDA/KubeRay 如何使用token遥测？

A:

  • KEDA can scale Kubernetes workloads from external metrics such as Prometheus queries over token backlog or queue depth.

KEDA 可以根据外部指标（例如 Prometheus 对token积压或队列深度的查询）来扩展 Kubernetes 工作负载。

  • KubeRay can add Ray worker capacity when Ray Serve deployments need more replicas or resources.

当 Ray Serve 部署需要更多副本或资源时，KubeRay 可以添加 Ray Worker 容量。

  • Ray Serve can scale per deployment based on ongoing requests, while the platform controller adjusts pool-level min/max replicas and warm capacity using token-specific signals.

Ray Serve 可以根据持续请求对每个部署进行扩展，而平台控制器则使用特定于token的信号调整池级最小/最大副本和热容量。

Q: What is warm capacity, and how do you decide how much to keep?

什么是热容量？您如何决定保留多少？

A:

  • Warm capacity means model servers are already running, weights are loaded, kernels are initialized, and health checks are passing.

热容量意味着模型服务器已经在运行，权重已加载，内核已初始化，并且运行状况检查正在通过。

  • Decide warm capacity from traffic baseline, burstiness, model cold-start time, business priority, and cost budget.

根据流量基线、突发性、模型冷启动时间、业务优先级和成本预算来确定热容量。

  • Keep more warm capacity for interactive, high-value, or slow-to-load models; use colder capacity for batch/backfill/evaluation.

为交互式、高价值或加载缓慢的模型保留更多热容量；使用较冷的容量进行批量/回填/评估。

Q: How do you improve GPU utilization without hurting p95 latency?

如何在不影响 p95 延迟的情况下提高 GPU 利用率？

A:

  • Separate interactive and batch pools.

独立的交互池和批处理池。

  • Use dynamic batching where latency budget allows.

在延迟预算允许的情况下使用动态批处理。

  • Use MIG slices for small stable models.

对小型稳定模型使用 MIG 切片。

  • Keep full-GPU pools for large or variable-memory workloads.

为大型或可变内存工作负载保留完整的 GPU 池。

  • Use admission control before queues explode.

在队列爆炸之前使用准入控制。

  • Track p95/p99, not only average utilization.

跟踪 p95/p99，而不仅仅是平均利用率。

  • Canary placement changes because utilization improvements can hide tail-latency regressions.

金丝雀布局发生变化，因为利用率的提高可以隐藏尾部延迟回归。

Q: If an interviewer asks “how did you increase GPU utilization from 45% to 80%?”, what is a credible answer?

如果面试官问“你们是如何将 GPU 利用率从 45% 提高到 80% 的？”，什么是可信的答案？

A:

  • “We attacked idle and fragmented capacity rather than just increasing load. We separated CPU orchestration from GPU inference, moved small models onto MIG-compatible pools, kept full GPUs for large models, tuned batching by latency class, scaled from queue/token metrics, and introduced warm capacity only where cold starts hurt user-facing latency. We validated the change through p95 latency, error rate, queue depth, and rollout metrics, not GPU utilization alone.”

“我们攻击空闲和碎片化的容量，而不仅仅是增加负载。我们将 CPU 编排与 GPU 推理分开，将小型模型移至 MIG 兼容池，为大型模型保留完整的 GPU，按延迟类别调整批处理，从队列/token指标进行扩展，并且仅在冷启动损害面向用户的延迟时才引入热容量。我们通过 p95 延迟、错误率、队列深度和推出指标验证了这一变化，而不仅仅是 GPU 利用率。”

Q: What are admission controls for vLLM/LLM serving?

vLLM/LLM 服务的准入控制是什么？

A:

  • Max input tokens and max output tokens.

最大输入token和最大输出token。

  • Max active sequences.

最大活动序列。

  • Per-tenant token budget.

每个tenant的代币预算。

  • Per-latency-class queue limits.

每个延迟类别的队列限制。

  • Context-length based routing.

基于上下文长度的路由。

  • Reject/defer low-priority requests when KV cache or token backlog is unsafe.

当 KV 缓存或token积压不安全时，拒绝/推迟低优先级请求。

Q: Why not just autoscale on GPU utilization?

为什么不直接根据 GPU 利用率自动缩放？

A:

  • GPU utilization can be high while users still see poor latency because the queue is too long.

GPU 利用率可能很高，但用户仍然会看到很差的延迟，因为队列太长。

  • GPU utilization can be low during decode if the bottleneck is KV-cache memory bandwidth or small batch occupancy.

如果瓶颈是 KV 缓存带宽或小批量占用，则解码期间 GPU 利用率可能较低。

  • For LLMs, better signals include TTFT, TPOT, waiting tokens, active sequences, KV-cache utilization, and preemption count.

对于 LLM，更好的信号包括 TTFT、TPOT、等待token、活动序列、KV 缓存利用率和抢占计数。

Q: What does “backpressure before overload” mean?

“过载前的背压”是什么意思？

A:

  • The system should slow, reject, degrade, or reroute work before GPU pools collapse.

在 GPU 池崩溃之前，系统应该减慢、拒绝、降级或重新路由工作。

  • Examples: reduce top-k, skip rerank, route to smaller model, defer batch jobs, cap tokens, or return cached/partial results.

示例：减少 top-k、跳过重新排名、路由到较小的模型、推迟批处理作业、限制token或返回缓存/部分结果。

  • Backpressure is a controlled degradation strategy; overload is an uncontrolled failure mode.

背压是一种受控的降级策略；过载是一种不受控制的故障模式。

Q: How do tenant quota and priority class work together?

tenant配额和优先级如何协同工作？

A:

  • Tenant quota limits how much capacity a product/team/customer can consume.

tenant配额限制产品/团队/客户可以消耗的容量。

  • Priority class decides which work survives when capacity is scarce.

优先级决定在容量稀缺时哪些工作能够生存。

  • A high-priority interactive request may preempt or bypass low-priority batch work, but quota prevents one tenant from permanently starving others.

高优先级的交互请求可能会抢占或绕过低优先级的批处理工作，但配额可以防止一个tenant永远让其他tenant陷入饥饿。

Q: What metrics should a resource controller consume?

资源控制器应该使用哪些指标？

A:

  • Queue depth, waiting time, p95/p99 latency, TTFT, TPOT, tokens/sec, active sequences, KV-cache utilization.

队列深度、等待时间、p95/p99 延迟、TTFT、TPOT、token/秒、活动序列、KV 缓存利用率。

  • GPU utilization, GPU memory, MIG occupancy, preemption count, cold-start duration, model load failures.

GPU利用率、GPU内存、MIG占用、抢占计数、冷启动持续时间、模型加载失败。

  • Kubernetes signals: Pending Pods, scheduling failures, node pressure, restarts, readiness failures.

Kubernetes 信号：待处理的 Pod、调度失败、节点压力、重新启动、就绪失败。

  • Cost signals: node type, spot/on-demand mix, warm capacity cost, tenant budget burn.

成本信号：节点类型、现货/按需混合、热容量成本、tenant预算消耗。

Q: What are the resource controller loops mentioned in the code doc?

代码文档中提到的资源控制器循环是什么？

A:

  • Fast loop: reacts to live queue, latency, token, and GPU metrics to adjust scale, admission, or routing.

快速循环：对实时队列、延迟、token和 GPU 指标做出反应，以调整规模、准入或路由。

  • Slow loop: analyzes placement efficiency, fragmentation, warm-pool strategy, and bin-packing opportunities over longer windows.

慢循环：分析较长窗口内的放置效率、碎片、热池策略和装箱机会。

  • This split avoids overreacting to short spikes while still improving long-term utilization.

这种划分避免了对短期峰值反应过度，同时仍提高了长期利用率。

Q: How do you canary a new placement or bin-packing policy?

如何制定新的安置或装箱政策？

A:

  • Start with shadow simulation using historical traffic and current cluster inventory.

从使用历史流量和当前集群库存的影子模拟开始。

  • Enable policy for a small tenant, model, region, or latency class.

为小型tenant、模型、区域或延迟类别启用策略。

  • Watch p95/p99 latency, queue depth, preemptions, OOMs, GPU memory, failed scheduling, and rollback rate.

观察 p95/p99 延迟、队列深度、抢占、OOM、GPU 内存、调度失败和回滚率。

  • Keep a rollback path to the previous placement policy.

保留先前放置策略的回滚路径。

Q: Why can model/index versioning affect resource control?

为什么模型/索引版本控制会影响资源控制？

A:

  • A new model version may need more memory, different precision, different max context, or different serving engine.

新的模型版本可能需要更多的内存、不同的精度、不同的最大上下文或不同的服务引擎。

  • A new embedding model requires a compatible vector index; serving mixed embedding/index versions can hurt retrieval quality.

新的嵌入模型需要兼容的向量索引；提供混合嵌入/索引版本可能会损害检索质量。

  • The resource controller must place and scale by version, not just by generic service name.

资源控制器必须按版本放置和缩放，而不仅仅是按通用服务名称。

Q: How does edge-first preprocessing reduce GPU and network pressure?

边缘优先预处理如何减轻GPU和网络压力？

A:

  • The edge/client extracts compact artifacts such as frames, thumbnails, embeddings, OCR text, audio segments, or metadata before cloud upload.

边缘/客户端在云上传之前提取紧凑的工件，例如帧、缩略图、嵌入、OCR 文本、音频片段或元数据。

  • Cloud services receive smaller payloads and can skip raw-media preprocessing when artifacts are valid.

云服务接收较小的有效负载，并且在工件有效时可以跳过原始媒体预处理。

  • This reduces object-storage bandwidth, network transfer, CPU preprocessing, and sometimes GPU inference cost.

这减少了对象存储带宽、网络传输、CPU 预处理，有时还减少了 GPU 推理成本。

Q: Why do checkpointed DAGs matter for resource management?

为什么检查点 DAG 对于资源管理很重要？

A:

  • Long workflows may fail due to network drop, client disconnect, GPU preemption, timeout, or model server restart.

长时间的工作流程可能会因网络中断、客户端断开连接、GPU 抢占、超时或模型服务器重新启动而失败。

  • Checkpoints let the platform resume from the last completed step instead of restarting from raw media.

检查点让平台从最后完成的步骤恢复，而不是从原始媒体重新启动。

  • Idempotent step keys prevent duplicate writes to object storage, vector indexes, result tables, or evaluation logs.

幂等步骤键可防止重复写入对象存储、向量索引、结果表或评估日志。

Q: What is the difference between fallback, retry, and degradation?

fallback、重试和降级之间有什么区别？

A:

  • Retry: same operation again, usually for transient failures.

重试：再次进行相同的操作，通常用于暂时性故障。

  • Fallback: switch to an alternate path, model, region, or runtime.

fallback：切换到备用路径、模型、区域或运行时。

  • Degradation: intentionally reduce quality/cost/latency target, such as smaller model, lower top-k, no rerank, shorter generation, or edge-only mode.

降级：故意降低质量/成本/延迟目标，例如更小的模型、更低的 top-k、不重新排序、更短的生成或仅边缘模式。

  • Strong platforms define all three in policy rather than improvising during incidents.

强大的平台在政策中定义了这三者，而不是在事件发生时即兴发挥。

Q: How would you draw the resource-control architecture in an interview?

在面试中你会如何绘制资源控制架构？

A:

  • Intent Gateway validates tenant, quota, latency class, and policy.

意图网关验证tenant、配额、延迟类别和策略。

  • Planner reads model profile, payload shape, privacy, cost budget, and current capacity.

Planner 读取模型配置文件、有效负载形状、隐私、成本预算和当前容量。

  • Resource Controller reads queue/token/GPU/Kubernetes metrics.

资源控制器读取队列/token/GPU/Kubernetes 指标。

可观测性和推出控制器将结果反馈到未来的布局决策中。

LLM / Transformer Internals

这一节讲 transformer/LLM serving 必备基础：self-attention、prefill/decode、KV cache、GQA/MQA、FlashAttention、speculative decoding。

Q: What is self-attention in practical serving terms?

在实际服务中什么是自我关注？

A:

  • Each token builds a query vector and attends to key/value vectors from other tokens.

每个token都会构建一个查询向量并关注来自其他token的键/值向量。

  • In prefill, the model processes the prompt and builds KV state for all prompt tokens.

在预填充中，模型处理prompt并为所有prompt标记构建 KV 状态。

  • In decode, each new token attends over previous KV state and appends one more KV entry.

在解码时，每个新token都会覆盖先前的 KV 状态并附加一个以上的 KV 条目。

Q: Why does attention become expensive for long context?

为什么对于长上下文来说注意力会变得昂贵？

A:

  • Prefill attention has quadratic behavior in sequence length for standard full attention.

对于标准的完全注意力，预填充注意力在序列长度上具有二次行为。

  • Decode with KV cache avoids recomputing old keys/values, but each new token still reads a growing KV cache.

使用 KV 缓存进行解码可以避免重新计算旧的键/值，但每个新token仍然会读取不断增长的 KV 缓存。

  • Long context pressures GPU memory capacity, memory bandwidth, and scheduling fairness.

长上下文对 GPU 内存容量、内存带宽和调度公平性造成压力。

Q: How do you estimate KV-cache memory?

你如何估计 KV 缓存内存？

A:

  • A useful approximation is: layers * 2 * sequence_length * kv_heads * head_dim * bytes_per_element * batch_size.

一个有用的近似值是：“layers * 2 *equence_length * kv_heads * head_dim * bytes_per_element * batch_size”。

  • The 2 is for keys and values.

“2”代表键和值。

  • GQA/MQA reduce KV heads, which lowers KV-cache memory compared with full multi-head attention.

GQA/MQA 减少了 KV 头，与完整的多头注意力相比，这降低了 KV 缓存内存。

Q: What are MHA, MQA, and GQA?

什么是 MHA、MQA 和 GQA？

A:

  • MHA: each attention head has its own query/key/value projection.

MHA：每个注意力头都有自己的查询/键/值投影。

  • MQA: multiple query heads share one key/value head, reducing KV-cache memory and bandwidth.

MQA：多个查询头共享一个键/值头，减少 KV 缓存内存和带宽。

  • GQA: query heads are grouped so each group shares key/value heads; it is a middle ground between quality and memory efficiency.

GQA：查询头被分组，因此每个组共享键/值头；它是质量和内存效率之间的中间立场。

Q: What is FlashAttention, and how is it different from PagedAttention?

什么是 FlashAttention，它与 PagedAttention 有何不同？

A:

  • FlashAttention optimizes attention computation by reducing reads/writes between GPU HBM and faster on-chip memory.

FlashAttention 通过减少 GPU HBM 和更快的片上内存之间的读/写来优化注意力计算。

  • PagedAttention optimizes KV-cache memory management for serving by storing KV blocks in page-like units.

PagedAttention 通过将 KV 块存储在类似页面的单元中来优化 KV 缓存管理。

  • Interview framing: FlashAttention is about attention kernel I/O efficiency; PagedAttention is about serving-time KV allocation and batching efficiency.

面试框架：FlashAttention是关于注意力内核I/O效率的； PagedAttention 是关于服务时间 KV 分配和批处理效率的。

Q: What is continuous batching?

什么是连续配料？

A:

  • Traditional batching waits for a batch, runs it together, and returns when every request finishes.

传统的批处理等待一个批次，一起运行它，并在每个请求完成时返回。

  • Continuous batching lets new requests enter and completed requests leave at token boundaries.

连续批处理允许新请求进入，并让已完成的请求在token边界离开。

  • This matters for LLM serving because output lengths vary; otherwise short requests get stuck behind long generations.

这对于 LLM 服务很重要，因为输出长度各不相同；否则，短请求会陷入长代的困境。

Q: What is speculative decoding?

什么是推测解码？

A:

  • A small draft model proposes several tokens, and the larger target model verifies them in parallel.

一个小的草稿模型提出了几个token，更大的目标模型并行地验证它们。

  • If the draft tokens are accepted, the system emits multiple tokens per expensive target-model call.

如果草稿token被接受，系统将为每个昂贵的目标模型调用发出多个token。

  • Tradeoff: it helps when the draft model is fast and accurate enough; bad drafts waste compute and add complexity.

权衡：当草稿模型足够快和准确时会有帮助；糟糕的草稿会浪费计算并增加复杂性。

Q: What are temperature, top-k, top-p, and beam search?

什么是温度、top-k、top-p 和波束搜索？

A:

  • Temperature rescales logits; higher temperature increases randomness.

温度重新调整对数；较高的温度会增加随机性。

  • Top-k samples from the k most likely tokens.

来自 k 个最有可能的标记的前 k 个样本。

  • Top-p samples from the smallest token set whose cumulative probability crosses p.

Top-p 从累积概率与 p 相交的最小 token 集中进行采样。

  • Beam search explores multiple high-probability continuations and is common in older seq2seq tasks, but for open-ended generation it can sound bland or over-optimized.

集束搜索探索多个高概率延续，在较旧的 seq2seq 任务中很常见，但对于开放式生成，它可能听起来平淡或过度优化。

Q: Why is LLM serving often memory-bandwidth bound rather than FLOPS-bound?

为什么 LLM 服务经常受到内存带宽限制而不是 FLOPS 限制？

A:

  • During decode, each new token may require reading large weights and a growing KV cache.

在解码期间，每个新token可能需要读取较大的权重和不断增长的 KV 缓存。

  • Batch size may be limited by KV-cache memory, not arithmetic throughput.

批量大小可能受 KV 缓存内存限制，而不是算术吞吐量。

  • This is why quantization, KV-cache management, prefix caching, and batching policy often matter more than raw peak TFLOPS.

这就是为什么量化、KV 缓存管理、前缀缓存和批处理策略通常比原始峰值 TFLOPS 更重要。

Quantization + Inference Optimization

这一节讲 FP16/BF16/FP8/INT8/INT4、PTQ/QAT、weight-only quantization、LoRA 和 runtime 选择。

Q: FP32 vs FP16 vs BF16 vs FP8 vs INT8/INT4: how do you explain the tradeoff?

FP32 vs FP16 vs BF16 vs FP8 vs INT8/INT4：您如何解释这种权衡？

A:

  • FP32: high precision, high memory and bandwidth cost.

FP32：精度高，内存和带宽成本高。

  • FP16: lower memory and faster tensor cores, but smaller exponent range can cause overflow/underflow.

FP16：较低的内存和更快的张量核心，但较小的指数范围可能会导致上溢/下溢。

  • BF16: similar memory to FP16 but wider exponent range, often more stable for training.

BF16：与 FP16 类似的内存，但指数范围更宽，通常训练更稳定。

  • FP8/INT8/INT4: reduce memory and bandwidth further, but require calibration, quantization-aware training, or careful model-specific validation.

FP8/INT8/INT4：进一步减少内存和带宽，但需要校准、量化感知训练或仔细的特定于模型的验证。

Q: What is PTQ vs QAT?

什么是 PTQ 与 QAT？

A:

  • PTQ (post-training quantization): quantize an already trained model using calibration or static statistics; cheaper, but may lose accuracy.

PTQ（训练后量化）：使用校准或静态统计对已经训练好的模型进行量化；更便宜，但可能会失去准确性。

  • QAT (quantization-aware training): simulate quantization during training/fine-tuning; more expensive, but usually more robust for sensitive models.

QAT（量化感知训练）：在训练/微调期间模拟量化；更昂贵，但通常对于敏感模型来说更稳健。

Q: What is weight-only quantization?

什么是仅权重量化？

A:

  • Only weights are quantized; activations may remain FP16/BF16.

仅量化权重；激活可能仍为 FP16/BF16。

  • It reduces model memory footprint and bandwidth for loading weights.

它减少了模型内存占用和加载权重的带宽。

  • It may be easier to deploy than full activation+weight quantization, but speedups depend on kernel support and hardware.

它可能比完全激活+权重量化更容易部署，但加速取决于内核支持和硬件。

Q: When would you choose TensorRT, Triton, vLLM, or PyTorch eager/compile?

您什么时候会选择 TensorRT、Triton、vLLM 或 PyTorch eager/compile？

A:

  • PyTorch eager: fastest iteration and debugging.

PyTorch eager：最快的迭代和调试。

  • torch.compile: useful when Python overhead/kernel fusion matters and graph breaks are manageable.

torch.compile：当 Python 开销/内核融合很重要并且图形中断可以管理时很有用。

  • TensorRT: optimized NVIDIA inference engine when model shapes/ops are compatible and deployment can tolerate engine build complexity.

TensorRT：当模型形状/操作兼容并且部署可以容忍引擎构建复杂性时，优化的 NVIDIA 推理引擎。

  • Triton: production serving wrapper for many model types and frameworks, with batching, model repository, and metrics.

Triton：许多模型类型和框架的生产服务包装器，具有批处理、模型存储库和指标。

  • vLLM: LLM generation where KV-cache management and token scheduling dominate.

vLLM：LLM 生成，其中 KV 缓存管理和token调度占主导地位。

Q: What is LoRA, and why does it matter for platforms?

LoRA 是什么？为什么它对平台很重要？

A:

  • LoRA freezes the base model and trains small low-rank adapter matrices.

LoRA 冻结基本模型并训练小型低秩适配器矩阵。

  • Platform value: many teams or tenants can customize behavior without serving a full copy of every model.

平台价值：许多团队或tenant可以自定义行为，而无需提供每个模型的完整副本。

  • Serving challenge: dynamic adapter loading, adapter caching, memory accounting, and quality/version control.

服务挑战：动态适配器加载、适配器缓存、内存核算和质量/版本控制。

Q: How do you safely serve many adapters?

如何安全地为许多适配器提供服务？

A:

  • Keep base models warm and load adapters on demand.

保持基础模型温暖并按需加载适配器。

  • Cache popular adapters and evict cold ones by memory pressure.

缓存流行的适配器并通过内存压力驱逐冷的适配器。

  • Track adapter version, tenant, evaluation status, and compatibility with the base model.

跟踪适配器版本、tenant、评估状态以及与基本模型的兼容性。

  • Protect latency by separating interactive adapter workloads from batch/eval workloads.

通过将交互式适配器工作负载与批处理/评估工作负载分开来保护延迟。

Q: PyTorch eager vs TorchScript/torch.compile vs ONNX vs TensorRT?

PyTorch eager vs TorchScript/torch.compile vs ONNX vs TensorRT？

A:

  • PyTorch eager: easiest to debug and iterate, but may leave performance on the table.

PyTorch eager：最容易调试和迭代，但可能会影响性能。

  • torch.compile: accelerates PyTorch by graph capture/fusion, but graph breaks and dynamic shapes can reduce benefit.

torch.compile：通过图形捕获/融合加速 PyTorch，但图形中断和动态形状会降低收益。

  • ONNX: interchange format for moving models across runtimes and hardware providers.

ONNX：用于跨运行时和硬件提供商移动模型的交换格式。

  • TensorRT: NVIDIA-optimized inference runtime; strong performance when ops, precision, and shapes are supported.

TensorRT：NVIDIA 优化的推理运行时；当支持运算、精度和形状时，性能强劲。

Q: What is ONNX Runtime’s execution-provider idea?

ONNX Runtime 的执行提供者理念是什么？

A:

  • ONNX Runtime can partition a model graph across execution providers such as CPU, CUDA, TensorRT, or other accelerators.

ONNX 运行时可以跨执行提供程序（例如 CPU、CUDA、TensorRT 或其他加速器）对模型图进行分区。

  • This helps with heterogeneous deployment, but unsupported ops can fall back to another provider and silently hurt latency.

这有助于异构部署，但不受支持的操作可能会fallback到另一个提供商并默默地损害延迟。

  • Interview answer: always inspect provider placement, graph optimization level, and actual runtime profile.

面试答案：始终检查提供程序布局、图形优化级别和实际运行时配置文件。

Q: What are common ONNX export pitfalls?

ONNX 导出的常见陷阱有哪些？

A:

  • Unsupported operators or custom ops.

不支持的运算符或自定义操作。

  • Dynamic axes not exported correctly.

动态轴未正确导出。

  • Shape inference mismatch between training and serving.

训练和服务之间的形状推理不匹配。

  • Numerics differ from PyTorch due to fused ops or precision changes.

由于融合操作或精度变化，数值与 PyTorch 不同。

  • Pre/postprocessing left outside the model graph and implemented inconsistently across clients.

预处理/后处理留在模型图之外，并且在客户端之间实施不一致。

Q: What are TensorRT optimization profiles?

什么是 TensorRT 优化配置文件？

A:

  • For dynamic shapes, TensorRT builds engines with profiles specifying min/optimal/max input shapes.

对于动态形状，TensorRT 使用指定最小/最佳/最大输入形状的配置文件构建引擎。

  • Performance is best near the optimized shape and may degrade near profile edges.

性能在优化形状附近最佳，但在轮廓边缘附近可能会降低。

  • For image/video/LLM-adjacent workloads, choose profiles based on real production shape distributions, not only benchmark samples.

对于图像/视频/LLM 相关工作负载，请根据实际生产形状分布（而不仅仅是基准样本）选择配置文件。

Q: Why is model artifact versioning hard in multimodal systems?

为什么模型工件版本控制在多模态系统中很难？

A:

  • Retrieval quality depends on the embedding model and the index built from it.

检索质量取决于嵌入模型和从中构建的索引。

  • Ranking quality depends on features, feature normalization, and ranker version.

排名质量取决于特征、特征标准化和排名器版本。

  • LLM behavior depends on model weights, tokenizer, prompt, tools, safety policy, and decoding config.

LLM 行为取决于模型权重、分词器、prompt、工具、安全策略和解码配置。

  • Rollback must version all coupled artifacts, not just the neural network file.

回滚必须对所有耦合工件进行版本控制，而不仅仅是神经网络文件。

Q: How do you validate runtime conversion?

如何验证运行时转换？

A:

  • Compare outputs against a PyTorch/eager baseline on golden inputs.

将输出与黄金输入的 PyTorch/eager 基线进行比较。

  • Measure latency, throughput, memory, and p95/p99 behavior under realistic batch/shape distributions.

测量实际批次/形状分布下的延迟、吞吐量、内存和 p95/p99 行为。

  • Test edge cases: empty inputs, long inputs, unusual image sizes, Unicode text, large batches, and precision-sensitive examples.

测试边缘情况：空输入、长输入、异常图像大小、Unicode 文本、大批量和精度敏感示例。

  • Run canary traffic before full rollout because offline equivalence does not guarantee production stability.

在全面推出之前运行金丝雀流量，因为离线等效并不能保证生产稳定性。

Vector Search + Hybrid Retrieval (Milvus / Elasticsearch)

Q: What’s the production rationale for hybrid retrieval (BM25 + vectors)?

混合检索（BM25 + 向量）的生产原理是什么？

A:

  • Dense vectors improve semantic recall; lexical retrieval (BM25) is strong for exact entities, numbers, and rare terms.

密集向量提高语义召回率；词汇检索 (BM25) 对于精确实体、数字和罕见术语非常有用。

  • Hybrid fusion (e.g., RRF) is a pragmatic way to blend two signals without requiring perfectly calibrated scores.

混合融合（例如 RRF）是一种混合两个信号的实用方法，无需完美校准 score。

Q: How do you explain RRF (reciprocal rank fusion) simply?

如何简单解释一下RRF（倒数秩融合）？

A:

  • RRF combines ranked lists by giving higher weight to items that rank near the top in either list.

RRF 通过对任一列表中排名接近顶部的项目给予更高的权重来组合排名列表。

  • It’s robust when score scales aren’t directly comparable across retrievers.

当检索器之间的评分量表不能直接比较时，它是稳健的。

Q: Milvus “segments” and why you should care

Milvus“细分”以及您为何应该关心

A:

  • Milvus ingests data into segments; indexing and compaction happen at the segment level.

Milvus 将数据摄入分段；索引和压缩发生在段级别。

  • Segment lifecycle and compaction policies influence latency, freshness, and storage costs (especially at large scale).

分段生命周期和压缩策略会影响延迟、freshness和存储成本（尤其是大规模时）。

Q: How does “index building” work at scale in a vector DB like Milvus?

在像 Milvus 这样的向量数据库中，“索引构建”是如何大规模工作的？

A:

  • You should expect storage/compute separation patterns: index build reads data/log snapshots for a segment, builds an index, and writes index artifacts back to object storage.

您应该期望存储/计算分离模式：索引构建读取段的数据/日志快照，构建索引，并将索引工件写回对象存储。

  • Interview framing: index build is a background pipeline competing for CPU/RAM/I/O; you plan capacity and throttling so ingestion freshness doesn’t collapse query latency.

面试框架：索引构建是一个竞争CPU/RAM/I/O的后台管道；您可以规划容量和限制，以便ingestion freshness 不会降低查询延迟。

Q: What is query understanding in a multimodal search platform?

什么是多模态搜索平台中的查询理解？

A:

  • Query understanding converts raw user input into a retrieval plan.

查询理解将原始用户输入转换为检索计划。

  • It identifies intent (exact search, visual similarity, semantic discovery, template search, document search), modality, filters, entities, language, and whether lexical/vector/hybrid retrieval should run.

它识别意图（精确搜索、视觉相似性、语义发现、模板搜索、文档搜索）、模态、过滤器、实体、语言以及是否应该运行词汇/向量/混合检索。

  • It also protects precision by treating structured constraints such as license, file type, product source, orientation, duration, and tenant visibility differently from free-text semantic terms.

它还通过与自由文本语义术语不同的方式处理结构化约束（例如许可证、文件类型、产品来源、方向、持续时间和tenant可见性）来保护准确性。

Q: What is a shared embedding space?

什么是共享嵌入空间？

A:

  • A shared embedding space maps different modalities into comparable vector representations.

共享嵌入空间将不同的模态映射为可比较的向量表示。

  • Example: text query, image thumbnail, video keyframe, audio segment, 3D rendered view, or document chunk can be compared by similarity.

例如：文本查询、图像缩略图、视频关键帧、音频片段、3D 渲染视图或文档块可以通过相似性进行比较。

  • The hard part is alignment: embeddings from different encoders must preserve semantic relationships well enough for retrieval, and version changes require index compatibility planning.

困难的部分是对齐：来自不同编码器的嵌入必须很好地保留语义关系以供检索，并且版本更改需要索引兼容性规划。

Q: What is cross-modal retrieval?

什么是跨模态检索？

A:

  • Cross-modal retrieval means querying in one modality and retrieving another, such as text-to-image, image-to-template, text-to-video-shot, or image-to-similar-stock-asset.

跨模态检索意味着以一种模态查询并检索另一种模态，例如文本到图像、图像到模板、文本到视频镜头或图像到相似库存资产。

  • It depends on aligned embeddings, modality-specific preprocessing, and metadata/ranking layers that understand the returned asset type.

它取决于对齐的嵌入、特定于模态的预处理以及了解返回的资产类型的元数据/排名层。

Q: What is a feature store / feature service in ranking?

什么是排名中的特征存储/特征服务？

A:

  • A feature service joins candidate assets with ranking features at request time or nearline.

特征服务在请求时或nearline将候选资产与排名要素结合起来。

  • Features can include semantic score, lexical score, quality score, freshness, popularity, license eligibility, creator/source metadata, personalization signals, and policy flags.

特征可以包括语义score、词汇score、质量score、freshness、流行度、许可证资格、创建者/源元数据、个性化信号和策略标志。

  • In interviews, mention that the feature layer can become a p95 bottleneck and should have caching/defaults/fallback behavior.

在面试中，提到特征层可能成为 p95 瓶颈，并且应该具有缓存/默认/fallback行为。

Q: Candidate overlap: why is it useful for debugging retrieval?

候选重叠：为什么它对于调试检索有用？

A:

  • Candidate overlap compares the candidates returned by old vs new retrieval pipelines.

候选重叠比较旧检索管道与新检索管道返回的候选。

  • If overlap drops sharply after an embedding/index/query-understanding change, quality may shift even before CTR data is available.

如果嵌入/索引/查询理解更改后重叠急剧下降，即使在点击率数据可用之前，质量也可能会发生变化。

  • It helps localize regressions: vector recall changed, lexical filters changed, fusion changed, or ranker changed.

它有助于本地化回归：向量召回更改、词法过滤器更改、融合更改或排序器更改。

Q: What is filter drop rate?

什么是过滤掉率？

A:

  • Filter drop rate measures how many retrieved candidates are removed by filters such as license, tenant visibility, region, asset type, orientation, safety, or product source.

过滤器丢弃率衡量有多少检索到的候选者被许可证、tenant可见性、区域、资产类型、方向、安全性或产品来源等过滤器删除。

  • A sudden increase may mean query understanding is too broad, index metadata is stale, or policy fields changed.

突然增加可能意味着查询理解太广泛、索引元数据过时或策略字段发生更改。

  • For high-risk filters such as license/permissions, fail closed and alert on anomalies.

对于许可证/权限等高风险过滤器，关闭故障并对异常发出警报。

Q: How do you debug a search freshness issue?

如何调试搜索freshness问题？

A:

  • Trace one asset version through ingestion event, Kafka topic, Flink job, embedding queue, vector index build, lexical index refresh, feature store update, cache invalidation, and serving read path.

通过ingestion事件、Kafka 主题、Flink 作业、嵌入队列、向量索引构建、词法索引刷新、特征存储更新、缓存失效和服务读取路径来跟踪一项资产版本。

  • Useful metrics: Kafka lag, Flink checkpoint duration, embedding queue delay, index publish delay, cache age, and time-to-searchable.

有用的指标：Kafka 延迟、Flink 检查点持续时间、嵌入队列延迟、索引发布延迟、缓存年龄和可搜索时间。

Q: How do you handle near-duplicate results?

您如何处理几乎重复的结果？

A:

  • Detect near-duplicates using embedding similarity, perceptual hashes, asset lineage, creator/source metadata, or cluster IDs.

使用嵌入相似性、感知哈希、资产沿袭、创建者/源元数据或集群 ID 来检测近似重复项。

  • Use diversity-aware reranking or grouping so the first page does not collapse into copies of the same asset.

使用多样性感知的重新排名或分组，以便第一页不会折叠成同一资产的副本。

  • Tune diversity by query intent: exact search can show similar items; discovery needs broader exploration.

通过查询意图调整多样性：精确搜索可以显示相似的项目；发现需要更广泛的探索。

Q: Diversity vs relevance: how do you explain the tradeoff?

多样性与相关性：您如何解释这种权衡？

A:

  • Pure relevance can over-rank near-duplicates and reduce exploration.

纯粹的相关性可能会超过近似重复的排名并减少探索。

  • Too much diversity can hide the best matching result.

太多的多样性会隐藏最佳匹配结果。

  • A practical system blends relevance, diversity, freshness, business policy, and user context; the right balance is measured with CTR, saves/downloads, reformulation rate, and human relevance judgments.

实用的系统融合了相关性、多样性、freshness、业务政策和用户环境；正确的平衡是通过点击率、保存/下载、重新制定率和人类相关性判断来衡量的。

RAG + Retrieval Quality

这一节讲 RAG 流程、chunking、embedding/reranker、groundedness、citation、PDF/document RAG 和 RAG failure debugging。

Q: What is the end-to-end RAG pipeline?

什么是端到端 RAG 管道？

A:

  • Ingest documents or assets.

ingestion文档或资产。

  • Parse and normalize content.

解析和规范化内容。

  • Chunk content and attach metadata.

分块内容并附加元数据。

  • Embed chunks and build lexical/vector indexes.

嵌入块并构建词汇/向量索引。

  • Retrieve candidates for a query.

检索查询的候选者。

  • Rerank candidates.

重新排列候选人。

  • Build grounded context.

建立扎根的背景。

  • Generate an answer with citations or evidence.

生成带有引文或证据的答案。

  • Evaluate answer quality and retrieval quality separately.

分别评估答案质量和检索质量。

Q: What are the biggest RAG failure modes?

最大的 RAG 故障模式是什么？

A:

  • Bad parsing: tables, PDFs, OCR, video transcripts, or image captions are malformed.

解析错误：表格、PDF、OCR、视频转录或图像标题格式错误。

  • Bad chunking: chunks are too small to contain answer context or too large to retrieve precisely.

错误的分块：块太小而无法包含答案上下文，或者太大而无法精确检索。

  • Bad retrieval: relevant evidence is not in top-k.

检索不良：相关证据不在 top-k 中。

  • Bad reranking: good candidates are present but pushed down.

糟糕的重新排名：存在好的候选人，但被推低了。

  • Bad generation: the LLM ignores evidence, overgeneralizes, or fabricates.

生成失败：LLM 忽略证据、过度概括或编造。

Q: How do you choose chunk size?

如何选择块大小？

A:

  • Small chunks improve precision but may lose context.

小块可以提高精度，但可能会丢失上下文。

  • Large chunks preserve context but reduce retrieval specificity and increase prompt cost.

大块可以保留上下文，但会降低检索特异性并增加prompt成本。

  • Practical answer: tune chunking with retrieval eval, use overlap carefully, and attach hierarchy metadata so the system can expand from chunk to section/document when needed.

实际答案：通过检索评估调整分块，小心使用重叠，并附加层次结构元数据，以便系统可以在需要时从块扩展到部分/文档。

Q: Embedding model vs reranker: what is the difference?

嵌入模型与重新排序器：有什么区别？

A:

  • Embedding retrieval maps query and documents into vectors and searches quickly at large scale.

嵌入检索将查询和文档映射到向量中并进行大规模快速搜索。

  • A reranker, often a cross-encoder or LLM scorer, looks at query-document pairs more deeply but is slower.

重新排序器（通常是交叉编码器或 LLM 评分器）会更深入地查看查询文档对，但速度较慢。

  • Pattern: high-recall retrieval first, high-precision reranking second.

schema：高召回率检索第一，高精度重排序第二。

Q: Cosine similarity vs dot product?

余弦相似度与点积？

A:

  • Cosine compares direction and is common when vector magnitude should not dominate.

余弦比较方向，当向量大小不应占主导地位时很常见。

  • Dot product includes magnitude and can be efficient when embeddings are trained for it.

点积包含量值，并且在训练嵌入时可以非常有效。

  • If vectors are normalized, cosine similarity and dot product ranking are equivalent up to scale.

如果向量被归一化，则余弦相似度和点积排序在尺度上是等效的。

Q: HNSW vs IVF vs PQ: how do you explain vector index tradeoffs?

HNSW vs IVF vs PQ：您如何解释向量索引权衡？

A:

  • HNSW: graph-based, strong recall/latency, memory-heavy, tuning via graph degree/search depth.

HNSW：基于图、强召回/延迟、占用大量内存、通过图度/搜索深度进行调整。

  • IVF: clusters vectors into coarse lists; query searches selected lists, trading recall for speed.

IVF：将向量聚类成粗列表；查询搜索选定的列表，以召回换取速度。

  • PQ: compresses vectors to reduce memory/storage; faster/cheaper but can lose precision.

PQ：压缩向量以减少内存/存储；更快/更便宜，但可能会失去精度。

  • Real systems often combine approaches and tune for recall@k, latency, memory, and index-build cost.

真实的系统通常会结合多种方法并针对recall@k、延迟、内存和索引构建成本进行调整。

Q: Why use hybrid retrieval instead of vector search only?

为什么使用混合检索而不是仅使用向量搜索？

A:

  • Vector search is good for semantic meaning.

向量搜索有利于语义意义。

  • BM25/sparse search is strong for exact names, IDs, rare terms, error codes, and product-specific vocabulary.

BM25/稀疏搜索对于精确名称、ID、罕见术语、错误代码和产品特定词汇具有强大的功能。

  • Hybrid retrieval gives better robustness when query intent varies.

当查询意图变化时，混合检索可以提供更好的鲁棒性。

Q: What metrics do you use for retrieval?

您使用什么指标进行检索？

A:

  • Recall@k: did the right answer appear in top-k?

Recall@k：正确答案是否出现在 top-k 中？

  • MRR: how early was the first relevant result?

MRR：第一个相关结果多久出现？

  • nDCG: how good is the ranked order with graded relevance?

nDCG：相关性分级的排名顺序有多好？

  • Latency/cost: retrieval quality is not enough if p95 breaks the product experience.

延迟/成本：如果p95破坏了产品体验，检索质量就不够。

Q: What is groundedness in RAG?

RAG 中的接地性是什么？

A:

  • Groundedness means the generated answer is supported by retrieved evidence.

扎根性意味着生成的答案得到检索到的证据的支持。

  • A grounded answer should cite or point to the source passages, pages, timestamps, or asset segments that justify it.

接地气的答案应该引用或指出源段落、页面、时间戳或证明其合理性的资产片段。

  • In document AI, fluency is not enough; the answer must be faithful to the PDF/document/video transcript evidence.

在文档AI中，光流畅还不够；答案必须忠实于 PDF/文档/视频记录证据。

Q: Citation coverage vs citation correctness?

引文覆盖率与引文正确性？

A:

  • Citation coverage asks whether important claims have citations.

引文覆盖率询问重要的声明是否有引文。

  • Citation correctness asks whether the cited source actually supports the claim.

引文正确性询问所引用的来源是否确实支持该主张。

  • A system can have high coverage but poor correctness if citations are attached loosely or hallucinated.

如果引用松散或存在幻觉，系统可能具有较高的覆盖率，但正确性较差。

Q: Why is PDF/document RAG hard?

为什么 PDF/文档 RAG 很难？

A:

  • PDFs can contain multi-column layouts, tables, footnotes, scanned pages, forms, figures, captions, and OCR errors.

PDF 可以包含多列布局、表格、脚注、扫描页面、表格、图形、标题和 OCR 错误。

  • Bad parsing creates bad chunks; bad chunks create bad retrieval; bad retrieval creates hallucinated answers.

错误的解析会产生错误的块；坏块会导致坏检索；糟糕的检索会产生幻觉答案。

  • Strong systems track page/section/table metadata and preserve enough structure for citation and debugging.

强大的系统跟踪页面/节/表元数据并保留足够的结构以供引用和调试。

Q: What is context precision vs context recall?

什么是上下文精确度与上下文召回率？

A:

  • Context recall: did retrieval include all evidence needed to answer?

上下文回忆：检索是否包括回答所需的所有证据？

  • Context precision: how much retrieved context was actually relevant?

上下文精度：检索到的上下文有多少实际上是相关的？

  • Low recall causes missing answers; low precision wastes context budget and distracts the model.

召回率低导致答案缺失；低精度会浪费上下文预算并分散模型的注意力。

Q: How do you debug a RAG answer failure?

如何调试 RAG 应答失败？

A:

  • Ask where the failure happened: parsing, chunking, embedding, retrieval, reranking, context assembly, prompt, generation, or citation rendering.

询问失败发生在哪里：解析、分块、嵌入、检索、重新排名、上下文组装、prompt、生成或引文渲染。

  • Replay the query with trace IDs and inspect retrieved chunks, scores, ranker output, final context, and model answer.

使用跟踪 ID 重放查询并检查检索到的块、score、排名器输出、最终上下文和模型答案。

  • Add the failure to the golden dataset so it becomes a regression test.

将失败添加到黄金数据集，使其成为回归测试。

Content Authenticity / Provenance / Trust Metadata

这一节讲 AI 内容的 provenance、content authenticity、trust metadata 和可审计性。适合连接 Amsterdam 内容安全与商业使用场景。

Q: What is content authenticity in an AI platform?

人工智能平台中的内容真实性是什么？

A:

  • Content authenticity is the ability to understand and communicate how an asset was created, edited, generated, or transformed.

内容真实性是理解和传达资产如何创建、编辑、生成或转换的能力。

  • Platform metadata can include source asset reference, model version, prompt/edit recipe, timestamp, tenant, policy decision, safety checks, and rollout version.

平台元数据可以包括源资产引用、模型版本、prompt/编辑配方、时间戳、tenant、策略决策、安全检查和推出版本。

  • Interview angle: provenance should be part of the artifact data model, not UI text added later.

面试角度：出处应该是工件数据模型的一部分，而不是后来添加的 UI 文本。

Q: What is provenance metadata?

什么是来源元数据？

A:

  • Provenance metadata records the lineage of an output: where it came from, which transformations happened, and which versions/policies were involved.

来源元数据记录输出的沿袭：它来自哪里、发生了哪些转换以及涉及哪些版本/策略。

  • For AI-generated or AI-edited content, provenance helps audit, rollback, reproduce, evaluate, and enforce downstream policy.

对于人工智能生成或人工智能编辑的内容，出处有助于审核、回滚、复制、评估和执行下游策略。

Q: Why is trust metadata hard to maintain?

为什么信任元数据难以维护？

A:

  • It must survive retries, exports, derived edits, batch processing, and cross-service handoff.

它必须能够承受重试、导出、派生编辑、批处理和跨服务切换。

  • It must balance auditability with privacy and retention policy.

它必须平衡可审计性与隐私和保留政策。

  • It must be versioned because model, prompt, policy, and safety logic can all change.

它必须进行版本控制，因为模型、prompt、策略和安全逻辑都可能发生变化。

Agents, Prompting, Evaluation, And Rollout Safety

本文件覆盖 AI agent、tool use、prompt engineering、in-context learning、harness、evaluation、LLM-as-judge 和 rollout safety。

AI Agents + Tool Use

这一节讲 agent 的生产定义、workflow vs agent、ReAct、tool calling、MCP、memory、durable execution 和指标。

Q: What is an AI agent in production engineering terms?

从生产工程的角度来说，什么是AI agent？

A:

  • An agent is not just an LLM prompt. It is a control loop that can observe context, decide next actions, call tools, update state, and stop based on success/failure criteria.

agent不仅仅是LLMprompt。它是一个控制循环，可以观察上下文、决定下一步操作、调用工具、更新状态以及根据成功/失败标准停止。

  • Production agent systems need tool schemas, permissions, state, memory, tracing, timeouts, retries, guardrails, and evaluation.

生产级 agent 系统需要工具 schema、权限、状态、内存、跟踪、超时、重试、安全护栏和评估。

Q: Agent vs workflow: what is the clean distinction?

agent与工作流程：明显的区别是什么？

A:

  • Workflow: predefined steps and transitions; more reliable and easier to test.

工作流程：预定义的步骤和转换；更可靠且更容易测试。

  • Agent: the model chooses some steps/tools dynamically; more flexible but harder to bound.

Agent：模型动态选择一些步骤/工具；更灵活但更难约束。

  • Interview answer: use deterministic workflows for known business processes and agentic decision-making only where flexibility is worth the risk.

面试答案：只有在灵活性值得冒险的情况下，才对已知的业务流程和agent决策使用确定性工作流程。

Q: What is ReAct, and why is it important?

什么是 ReAct，为什么它很重要？

A:

  • ReAct interleaves reasoning and acting: the model reasons about a plan, calls a tool/action, observes the result, and continues.

ReAct 交错推理和行动：模型对计划进行推理，调用工具/操作，观察结果，然后继续。

  • The platform lesson is that tool traces matter. You evaluate not only final text, but also action choice, tool arguments, intermediate observations, and stopping behavior.

平台的教训是工具跟踪很重要。您不仅评估最终文本，还评估操作选择、工具参数、中间观察和停止行为。

Q: What are tool-calling failure modes?

什么是工具调用故障模式？

A:

  • Wrong tool selection.

工具选择错误。

  • Invalid or hallucinated arguments.

无效或幻觉的参数。

  • Retrying non-idempotent actions.

重试非幂等操作。

  • Calling tools with stale context.

使用陈旧的上下文调用工具。

  • Infinite loops or unbounded token/tool cost.

无限循环或无限的token/工具成本。

  • Prompt injection from retrieved documents or tool outputs.

从检索到的文档或工具输出中prompt injection。

Q: How do you make tool calls safer?

如何使工具调用更安全？

A:

  • Use strict schemas and local validation.

使用严格的 schema 和本地验证。

  • Make tools idempotent where possible.

尽可能使工具具有幂等性。

  • Require approval for destructive actions.

破坏性行为需要获得批准。

  • Enforce per-tool permissions and tenant boundaries.

强制执行每个工具的权限和tenant边界。

  • Add timeout, retry, and circuit-breaker policy.

添加超时、重试和熔断策略。

  • Log traces for replay/evaluation.

记录重播/评估的跟踪。

Q: What is MCP, and what should you know for interviews?

什么是 MCP？面试时您应该了解哪些知识？

A:

  • MCP standardizes how AI applications connect to external tools, resources, and prompts.

MCP 标准化了 AI 应用程序连接到外部工具、资源和prompt的方式。

  • The interview point is not "MCP solves agents"; it is that tool/resource boundaries, user consent, permissioning, and prompt-injection handling become platform concerns.

面试要点不是“MCP解决agent商”；工具/资源边界、用户同意、许可和prompt injection处理成为平台关注的问题。

Q: What is agent memory?

什么是agent记忆？

A:

  • Short-term memory: current conversation/task state.

短期记忆：当前对话/任务状态。

  • Long-term memory: durable user/project facts, preferences, or learned artifacts.

长期记忆：持久的用户/项目事实、偏好或学到的工件。

  • Retrieval memory: searchable documents, prior traces, or embeddings.

检索记忆：可搜索的文档、先前的痕迹或嵌入。

  • Risk: stale or poisoned memory can make an agent confidently wrong, so memory needs provenance, TTL, write policy, and deletion controls.

风险：陈旧或中毒的内存可能会让agent确信错误，因此内存需要出处、TTL、写入策略和删除控制。

Q: Why is durable execution important for agents?

为什么持久执行力对于agent商来说很重要？

A:

  • Agent tasks may involve long-running tool calls, human approval, retries, and partial failures.

agent任务可能涉及长时间运行的工具调用、人工批准、重试和部分失败。

  • Durable execution checkpoints graph state so the system can resume after timeout, crash, or manual intervention.

持久执行检查点图形状态，以便系统可以在超时、崩溃或手动干预后恢复。

  • This is similar in spirit to workflow checkpointing in AI infra: save state at step boundaries, not only at final output.

这在本质上类似于人工智能基础设施中的工作流检查点：在步骤边界保存状态，而不仅仅是在最终输出时保存状态。

Q: What metrics do you track for agents?

您跟踪agent的哪些指标？

A:

  • Task success rate and user correction rate.

任务成功率和用户纠正率。

  • Tool-call error rate and invalid argument rate.

工具调用错误率和无效参数率。

  • Average tool calls per task.

每个任务的平均工具调用次数。

  • Loop/timeout rate.

循环/超时率。

  • Cost per successful task.

每成功任务的成本。

  • Groundedness, policy violations, and human escalation rate.

groundedness、策略违规和人工升级率。

Prompt / In-Context / Harness Engineering

这一节讲 prompt 模板、few-shot/in-context learning、context engineering、prompt harness、golden dataset、shadow/canary 和 prompt injection。

Q: What is prompt engineering in production terms?

生产环境中的 prompt engineering 是什么？

A:

  • Prompt engineering is designing the instruction, context, examples, output schema, and constraints that steer model behavior.

prompt engineering 是设计指导模型行为的指令、上下文、示例、输出 schema 和约束。

  • In production, a prompt is not a string hidden in code; it is a versioned artifact with owner, evaluation set, rollout plan, and rollback path.

在生产中，prompt 不是隐藏在代码中的字符串；它是一个版本化的工件，具有所有者、评估集、部署计划和回滚路径。

Q: What is in-context learning?

什么是情境学习？

A:

  • In-context learning is when the model adapts its behavior from instructions and examples inside the current context window without weight updates.

上下文学习是指模型根据当前上下文窗口内的指令和示例调整其行为，而无需更新权重。

  • Examples can teach format, style, edge cases, refusal behavior, or domain-specific reasoning patterns.

示例可以教授格式、风格、边缘情况、拒绝行为或特定领域的推理模式。

  • Tradeoff: examples consume context budget and may overfit behavior to the examples you selected.

权衡：示例会消耗上下文预算，并且可能会导致行为与您选择的示例过度匹配。

Q: Zero-shot vs few-shot vs many-shot prompting?

零样本prompt、少样本prompt、多样本prompt？

A:

  • Zero-shot: instruction only, no examples.

零样本：仅说明，无示例。

  • Few-shot: a small number of examples showing desired behavior.

Few-shot：显示所需行为的少量示例。

  • Many-shot: many examples when context length allows and the task benefits from pattern induction.

many-shot：上下文长度允许且任务受益于模式归纳的许多示例。

  • In interviews: start simple, then add examples only when evals show the prompt needs them.

在面试中：从简单开始，然后仅在评估显示prompt需要时添加示例。

Q: What should be in a good prompt template?

一个好的prompt 模板应该包含哪些内容？

A:

  • Role/task definition.

角色/任务定义。

  • Success criteria and constraints.

成功的标准和限制。

  • Input variables with clear boundaries.

边界清晰的输入变量。

  • Context/evidence section.

背景/证据部分。

  • Output format/schema.

输出格式/架构。

  • Examples for tricky cases.

棘手案例的示例。

  • Safety/refusal/fallback behavior.

安全/拒绝/后备行为。

  • Version and owner metadata outside the prompt.

prompt之外的版本和所有者元数据。

Q: What is context engineering?

什么是情境工程？

A:

  • Context engineering is deciding what information enters the model context, in what order, with what formatting, and under what token budget.

上下文工程正在决定哪些信息以什么顺序、采用什么格式以及在什么token预算下进入模型上下文。

  • It includes retrieval selection, deduplication, chunk ordering, summarization, citation metadata, tool outputs, conversation history, and policy instructions.

它包括检索选择、重复数据删除、块排序、摘要、引文元数据、工具输出、对话历史记录和策略指令。

  • In RAG systems, bad context engineering can make a good model look bad.

在 RAG 系统中，糟糕的上下文工程可能会让一个好的模型看起来很糟糕。

Q: What is a prompt harness?

什么是prompt线束？

A:

  • A prompt harness is the surrounding test and runtime system for prompts.

prompt工具是prompt的周围测试和运行时系统。

  • It runs fixed datasets through prompt versions, captures outputs, compares metrics, records traces, and blocks rollout when quality or safety regresses.

它通过prompt版本运行固定数据集，捕获输出，比较指标，记录跟踪，并在质量或安全性下降时阻止推出。

  • Think of it as unit/integration testing for probabilistic AI behavior.

将其视为概率人工智能行为的单元/集成测试。

Q: What should a prompt evaluation harness measure?

及时评估工具应该衡量什么？

A:

  • Task correctness or rubric score.

任务正确性或评分标准。

  • Format/schema validity.

格式/schema 有效性。

  • Groundedness and citation quality.

基础性和引用质量。

  • Safety/policy violations.

违反安全/政策。

  • Latency and cost.

延迟和成本。

  • Regression against previous prompt/model version.

针对先前prompt/模型版本的回归。

  • Slice metrics by language, tenant, product surface, modality, and query class.

按语言、tenant、产品入口、schema 和查询类别对指标进行切片。

Q: What is a golden dataset?

什么是黄金数据集？

A:

  • A curated set of representative inputs with expected behavior, labels, rubrics, or human-reviewed outputs.

一组精心策划的代表性输入，其中包含预期行为、标签、标题或人工审核的输出。

  • It should contain common cases, edge cases, adversarial cases, safety cases, and known historical failures.

它应该包含常见案例、边缘案例、对抗案例、安全案例和已知的历史故障。

  • Golden datasets should grow from production traces: every resolved incident should become a regression test.

黄金数据集应该从生产跟踪中增长：每个解决的事件都应该成为回归测试。

Q: Shadow traffic vs canary for prompts/models?

影子流量与金丝雀prompt/模型？

A:

  • Shadow traffic runs the new version in parallel without user-facing impact; useful for comparing outputs, latency, and cost.

影子流量并行运行新版本，不会对用户造成影响；对于比较输出、延迟和成本很有用。

  • Canary sends a small real traffic slice to the new version; useful for measuring real user behavior and production effects.

金丝雀向新版本发送一小部分真实流量切片；对于衡量真实的用户行为和生产效果很有用。

  • Use shadow before canary when output quality or safety is uncertain.

当输出质量或安全性不确定时，在金丝雀之前使用影子。

Q: Why is structured output important?

为什么结构化输出很重要？

A:

  • It makes downstream systems safer and easier to validate.

它使下游系统更安全、更容易验证。

  • Instead of parsing free text, require JSON/schema-constrained outputs or tool-call arguments.

不需要解析自由文本，而是需要 JSON/架构约束的输出或工具调用参数。

  • Still validate everything server-side; structured output reduces risk but does not replace policy checks.

仍然在服务器端验证所有内容；结构化产出可以降低风险，但不能取代政策检查。

Q: How do you handle prompt injection?

您如何处理即时注入？

A:

  • Treat retrieved documents, web pages, PDFs, and tool outputs as untrusted data.

将检索到的文档、网页、PDF 和工具输出视为不可信数据。

  • Separate instructions from data using clear boundaries and schemas.

使用清晰的边界和schema将指令与数据分开。

  • Limit tool permissions, validate tool arguments, and enforce policy in code.

限制工具权限、验证工具参数并在代码中强制执行策略。

  • Add eval cases where malicious retrieved content tries to override system instructions or exfiltrate data.

添加恶意检索内容尝试覆盖系统指令或窃取数据的评估案例。

Q: What is prompt drift?

什么是瞬时漂移？

A:

  • Prompt drift happens when prompt, context, model version, retrieval behavior, or tool schema changes cause output behavior to shift.

当prompt、上下文、模型版本、检索行为或工具 schema更改导致输出行为发生变化时，就会发生prompt漂移。

  • Track prompt/model/index/tool versions in traces so a bad output can be reproduced.

跟踪跟踪中的prompt/模型/索引/工具版本，以便可以重现错误的输出。

  • Rollout prompt changes the same way you roll out code: offline eval, shadow, canary, monitor, rollback.

转出prompt的更改方式与转出代码的方式相同：离线评估、影子、金丝雀、监视器、回滚。

Q: What is the difference between prompt engineering and fine-tuning?

prompt engineering 和微调有什么区别？

A:

  • Prompt engineering changes behavior through instructions/context/examples.

通过说明/上下文/示例prompt engineering改变行为。

  • Fine-tuning changes model weights for repeated patterns or domain behavior.

微调会更改重复模式或域行为的模型权重。

  • RAG changes behavior by retrieving relevant knowledge at runtime.

RAG 通过在运行时检索相关知识来改变行为。

  • Interview answer: use prompt/RAG first for controllability and speed; fine-tune when you need consistent learned behavior across many examples and can support evaluation/maintenance.

面试答案：先用prompt/RAG，可控性和速度；当您需要在许多示例中保持一致的学习行为时进行微调，并且可以支持评估/维护。

Q: What is harness engineering beyond prompts?

除了prompt之外，线束工程是什么？

A:

  • Harness engineering includes prompt versioning, model routing, tool mocks, trace replay, evaluation datasets, judge rubrics, policy checks, and CI/CD gates.

线束工程包括prompt版本控制、模型路由、工具模拟、跟踪重放、评估数据集、判断标准、策略检查和 CI/CD 门。

  • For agents, the harness also captures tool-call sequences, intermediate state, human approvals, and stopping conditions.

对于agent来说，该工具还捕获工具调用序列、中间状态、人工批准和停止条件。

  • This is what turns a clever demo into an operable AI system.

这就是将一个聪明的演示变成一个可操作的人工智能系统的原因。

AI Agent System Design / Harness Q&A

这一节讲 agent platform 的组件、tool registry、trace、guardrails、human-in-the-loop 和防止 agent loop。

Q: What are the core components of an agent platform?

agent平台的核心组件有哪些？

A:

  • Model runtime.

模型运行时间。

  • Tool registry and schema validation.

工具注册表和schema 验证。

  • State/memory store.

状态/内存存储。

  • Planner or workflow graph.

规划器或工作流程图。

  • Policy/guardrail layer.

策略/安全护栏层。

  • Trace and evaluation system.

跟踪和评估系统。

  • Human approval/escalation path.

人工批准/升级路径。

  • Cost, quota, and timeout controls.

成本、配额和超时控制。

Q: Agent loop vs graph workflow: when do you choose which?

agent循环与图形工作流程：什么时候选择哪个？

A:

  • Agent loop: flexible, model decides next step repeatedly; useful for open-ended tasks but harder to bound.

Agent循环：灵活，模型反复决定下一步；对于开放式任务很有用，但更难约束。

  • Graph workflow: explicit states/transitions; better for production workflows that need reliability, checkpoints, approval, and auditability.

图形工作流程：显式状态/转换；更适合需要可靠性、检查点、批准和可审核性的生产工作流程。

  • Interview line: "I prefer graph/workflow structure for production and use agentic decisions only at bounded decision points."

面试台词：“我更喜欢用于生产的图形/工作流结构，并且仅在有限的决策点使用agent决策。”

Q: How do you design a tool registry?

如何设计工具注册表？

A:

  • Each tool should have a name, description, input schema, output schema, owner, permissions, timeout, retry policy, idempotency behavior, rate limit, and audit requirements.

每个工具都应该有名称、描述、输入 schema、输出 schema、所有者、权限、超时、重试策略、幂等行为、速率限制和审核要求。

  • Tools should be narrow and typed, not arbitrary shell/API access.

工具应该是狭窄的和类型化的，而不是任意的 shell/API 访问。

  • Dangerous tools require approval or policy checks before execution.

危险工具在执行之前需要批准或策略检查。

Q: What is a tool-call trace?

什么是工具调用跟踪？

A:

  • A trace records model input, selected tool, tool arguments, tool result, intermediate reasoning/state summary, final output, latency, cost, and policy decisions.

跟踪记录模型输入、所选工具、工具参数、工具结果、中间推理/状态摘要、最终输出、延迟、成本和策略决策。

  • It is essential for debugging why an agent failed: wrong tool, bad arguments, stale memory, bad retrieval, or missing guardrail.

对于调试agent失败的原因至关重要：错误的工具、错误的参数、过时的内存、错误的检索或缺少安全护栏。

Q: How do you evaluate an agent?

您如何评价经纪人？

A:

  • Final answer quality is not enough.

最终的答案质量不够。

  • Evaluate task success, tool correctness, unnecessary tool calls, invalid arguments, loop count, timeout rate, cost, human escalation, and policy violations.

评估任务成功、工具正确性、不必要的工具调用、无效参数、循环计数、超时率、成本、人工升级和策略违规。

  • For RAG agents, evaluate retrieval quality separately from generation and tool behavior.

对于 RAG agent，将检索质量与生成和工具行为分开评估。

Q: What are guardrails in agent systems?

agent 系统中的安全护栏是什么？

A:

  • Input guardrails classify or block unsafe/unrelated requests before expensive execution.

输入安全护栏在昂贵的执行之前对不安全/不相关的请求进行分类或阻止。

  • Tool guardrails validate arguments and permissions before tool calls.

工具安全护栏在工具调用之前验证参数和权限。

  • Output guardrails check final response safety, format, privacy, and policy compliance.

输出安全护栏检查最终响应的安全性、格式、隐私和策略合规性。

  • Code/rules should enforce critical constraints; do not rely only on prompt instructions.

规范/规则应强制实施关键约束；不要仅依赖及时的指示。

Q: What does human-in-the-loop mean beyond an Approve button?

除了批准按钮之外，人机交互意味着什么？

A:

  • The human needs enough context to make a decision: proposed action, evidence, tool outputs, risk, alternatives, and rollback path.

人类需要足够的背景来做出决定：建议的行动、证据、工具输出、风险、替代方案和回滚路径。

  • The system should checkpoint state before approval, resume after approval, and record the decision for audit/evaluation.

系统应在批准前检查点状态，批准后恢复，并记录审核/评估的决定。

Q: How do you prevent agent loops?

如何防止agent循环？

A:

  • Set max steps, max tool calls, max tokens, and total wall-clock budget.

设置最大步数、最大工具调用数、最大token数和总挂钟预算。

  • Detect repeated tool calls or repeated state.

检测重复的工具调用或重复的状态。

  • Require explicit stopping conditions.

需要明确的停止条件。

  • Escalate to human or return partial/fallback answer when progress stalls.

当进度停滞时，升级为人工或返回部分/后备答案。

Q: What is the biggest production risk with agents?

agent商最大的生产风险是什么？

A:

  • Agents can take actions, not just produce text.

agent可以采取行动，而不仅仅是生成文本。

  • The high-risk failures are wrong side effects, permission leaks, prompt injection through tools/retrieved content, runaway cost, and non-reproducible behavior.

高风险的失败是错误的副作用、权限泄漏、通过工具/检索内容的prompt injection、失控的成本和不可重现的行为。

  • Mitigate with least privilege, schemas, approval, audit logs, evals, and deterministic workflow boundaries.

使用最低权限、schema、批准、审核日志、评估和确定性工作流程边界进行缓解。

Model Evaluation, LLM-as-Judge, and Rollout Safety

这一节讲评估体系：LLM-as-judge、rubric、offline/online eval、shadow traffic、canary、versioning 和 trace-to-dataset。

Q: When is LLM-as-judge useful?

LLM-as-judge 什么时候有用？

A:

  • It is useful for scalable qualitative evaluation: answer helpfulness, groundedness, instruction following, tone, safety, or side-by-side preference.

它对于可扩展的定性评估非常有用：回答有用性、基础性、遵循指令、语气、安全性或并排偏好。

  • It is especially useful as a rollout gate when human evaluation is too slow for every model/prompt/index change.

当人工评估对于每个模型/prompt/索引更改都太慢时，它作为推出门特别有用。

Q: What are LLM-as-judge risks?

LLM-as-judge的风险是什么？

A:

  • Judge bias toward verbose answers.

法官偏向于冗长的答案。

  • Sensitivity to prompt wording and rubric.

对prompt措辞和标题的敏感性。

  • Position bias in pairwise comparisons.

成对比较中的位置偏差。

  • Poor calibration on domain-specific correctness.

对特定领域正确性的校准较差。

  • Leakage if the judge sees labels or unsafe context.

如果法官看到标签或不安全的环境，就会发生泄漏。

Q: How do you make LLM-as-judge credible?

如何让LLM-as-judge变得可信？

A:

  • Use a fixed rubric with examples.

使用带有示例的固定标题。

  • Compare against human-labeled gold sets.

与人类标记的黄金套装进行比较。

  • Track judge agreement and drift.

跟踪判断一致性和漂移。

  • Blind pairwise evaluations where possible.

尽可能进行盲配对评估。

  • Keep hard metrics such as retrieval recall, latency, error rate, and policy violations alongside judge scores.

将检索召回、延迟、错误率和策略违规等硬指标与评判score一起保留。

Q: What is a good AI rollout strategy?

什么是好的人工智能推广策略？

A:

  • Offline eval on golden sets.

对黄金套装进行离线评估。

  • Shadow traffic to check latency, error, and tool behavior.

影子流量以检查延迟、错误和工具行为。

  • Canary rollout by tenant, region, or feature flag.

按tenant、区域或功能标志进行金丝雀部署。

  • Online A/B test for product metrics.

产品指标的在线 A/B 测试。

  • Automated rollback based on SLOs and quality guardrails.

基于 SLO 和质量安全护栏的自动回滚。

Q: What should be versioned in an AI platform?

人工智能平台中应该对哪些内容进行版本控制？

A:

  • Model weights, tokenizer, prompt template, tool schema, retrieval index, embedding model, reranker, policy/routing config, evaluation dataset, and safety thresholds.

模型权重、分词器、prompt 模板、工具 schema、检索索引、嵌入模型、重新排序器、策略/路由配置、评估数据集和安全阈值。

  • Interview point: without versioning, you cannot reproduce a bad answer or safely rollback.

面试要点：如果没有版本控制，你就无法重现错误的答案或安全地回滚。

Q: What is an evaluation harness?

什么是评估工具？

A:

  • An evaluation harness runs a fixed set of inputs through a specific model/prompt/index/tool version and computes comparable metrics.

评估工具通过特定的模型/prompt/索引/工具版本运行一组固定的输入，并计算可比较的指标。

  • It should store input, output, retrieved context, tool trace, model version, prompt version, score, latency, cost, and reviewer notes.

它应该存储输入、输出、检索到的上下文、工具跟踪、模型版本、prompt版本、score、延迟、成本和审阅者注释。

  • It lets teams compare versions before rollout instead of relying on manual spot checks.

它允许团队在推出之前比较版本，而不是依赖手动抽查。

Q: Offline eval vs online eval?

离线评估 vs 在线评估？

A:

  • Offline eval uses curated datasets before deployment; it is fast, repeatable, and good for regression testing.

离线评估在部署前使用精选数据集；它速度快、可重复，并且适合回归测试。

  • Online eval monitors real production interactions; it catches distribution shift and user-behavior effects.

在线评估监控真实的生产交互；它捕捉到分布变化和用户行为影响。

  • Use both: offline eval protects release quality, online eval detects reality drifting away from your test set.

使用两者：离线评估保护发布质量，在线评估检测偏离测试集的现实。

Q: Deterministic evaluator vs LLM-as-judge vs human review?

确定性评估器 vs LLM-as-judge vs 人工审核？

A:

  • Deterministic evaluator: exact checks, schema validation, latency, policy, known-answer tests.

确定性评估器：精确检查、schema 验证、延迟、策略、已知答案测试。

  • LLM-as-judge: useful for fuzzy quality such as helpfulness, groundedness, coherence, and side-by-side preference.

LLM-as-judge：对于模糊质量很有用，例如helpfulness、groundedness、coherence和并排偏好。

  • Human review: expensive but necessary for calibration, policy-sensitive cases, and judge validation.

人工审查：成本昂贵，但对于校准、政策敏感案例和判断验证来说是必要的。

Q: What makes an LLM-as-judge rubric good?

是什么让LLM-as-judge的标题很好？

A:

  • Clear criteria and score scale.

明确的标准和评分标准。

  • Examples for each score band.

每个score段的示例。

  • Separate dimensions for correctness, groundedness, safety, concision, and style.

正确性、接地性、安全性、简洁性和风格的独立维度。

  • Blind pairwise comparison when possible.

尽可能进行盲配对比较。

  • Calibration against human labels and periodic drift checks.

根据人类标签进行校准并定期进行漂移检查。

Q: What is trace-to-dataset?

什么是追踪到数据集？

A:

  • When a production trace exposes a failure, add that trace to a regression dataset after labeling or defining expected behavior.

当生产跟踪暴露故障时，请在标记或定义预期行为后将该跟踪添加到回归数据集。

  • This turns incidents into durable tests.

这将事件转变为持久的考验。

  • It is especially useful for prompts, agents, RAG, and rollout gates because failures are often behavior-specific rather than crash-specific.

它对于prompt、agent、RAG 和转出门特别有用，因为故障通常是特定于行为的，而不是特定于崩溃的。

PyTorch + Training/Optimization Basics

这一节讲 SGD/Momentum/Adam/AdamW、weight decay、gradient clipping、DDP/FSDP 和 CUDA graphs。

Q: SGD vs Momentum vs Adam: what’s the real difference?

SGD、Momentum、Adam：真正的区别是什么？

A:

  • SGD: constant step size per parameter (with optional weight decay/L2), simple and often strong in generalization.

SGD：每个参数的恒定步长（具有可选的权重衰减/L2），简单且泛化能力强。

  • Momentum: adds a velocity term to smooth updates and accelerate along consistent gradients; helps traverse ravines.

Momentum：添加速度项以平滑更新并沿着一致的梯度加速；帮助穿越峡谷。

  • Adam: adaptive per-parameter learning rates using first/second moment estimates; converges fast but can be sensitive to hyperparameters and generalization behavior.

Adam：使用一阶/二阶矩估计的自适应每参数学习率；收敛速度快，但对超参数和泛化行为很敏感。

Q: Adam vs AdamW: why do people prefer AdamW for transformers?

Adam vs AdamW：为什么人们更喜欢 AdamW 作为变形金刚？

A:

  • AdamW decouples weight decay from Adam’s adaptive gradient scaling (weight decay becomes a direct parameter shrink).

AdamW 将权重衰减与 Adam 的自适应梯度缩放解耦（权重衰减变成直接参数收缩）。

  • In practice, AdamW is a common default for transformer training and fine-tuning.

在实践中，AdamW 是 Transformer 训练和微调的常见默认值。

  • Interview note: distinguish “weight decay” from “L2 regularization”; they are not always equivalent in adaptive optimizers.

面试笔记：区分“权重衰减”和“L2正则化”；它们在自适应优化器中并不总是等效的。

Q: What does “weight decay” do, and why can it hurt if misused?

“重量衰减”有什么作用？如果使用不当，为什么会造成伤害？

A:

  • Weight decay penalizes large weights; it can improve generalization and stability.

权重衰减会惩罚大权重；它可以提高泛化性和稳定性。

  • Too much weight decay can underfit or destabilize layer norms/embeddings if applied indiscriminately.

如果不加区别地应用，过多的权重衰减可能会导致层规范/嵌入不适合或不稳定。

  • Common practice: exclude bias and layer norm parameters from weight decay in transformer training.

常见做法：从变压器训练中的权重衰减中排除偏差和层规范参数。

Q: What is gradient clipping and why is it used?

什么是梯度裁剪以及为什么使用它？

A:

  • Clipping limits gradient norm or value to prevent exploding gradients.

裁剪限制梯度范数或值以防止梯度爆炸。

  • It’s especially useful in large models or long sequences where rare spikes can destabilize training.

它在大型模型或长序列中特别有用，因为罕见的峰值可能会破坏训练的稳定性。

Q: DDP vs FSDP: when do you use which?

DDP 与 FSDP：什么时候使用哪个？

A:

  • DDP: replicate model on each GPU, all-reduce gradients; simpler and often fastest at smaller model sizes.

DDP：在每个GPU上复制模型，all-reduce梯度；在较小的模型尺寸下更简单且通常最快。

  • FSDP: shard parameters/gradients/optimizer state; enables larger models per GPU at the cost of more communication and complexity.

FSDP：分片参数/梯度/优化器状态；每个 GPU 可以支持更大的模型，但代价是更多的通信和复杂性。

  • Interview: explain memory vs bandwidth tradeoff and why sharding changes the comm/compute profile.

面试：解释内存与带宽的权衡以及为什么分片会改变通信/计算配置文件。

Q: CUDA graphs: why do they matter for inference?

CUDA 图：为什么它们对推理很重要？

A:

  • CUDA graphs record and replay GPU work to reduce CPU launch overhead.

CUDA 图表记录和重放 GPU 工作以减少 CPU 启动开销。

  • They can improve latency/throughput for small batch or short sequence workloads, but require more static shapes and careful capture behavior.

它们可以改善小批量或短序列工作负载的延迟/吞吐量，但需要更多静态形状和仔细的捕获行为。

ML / DL / AI Basics For Platform Interviews

这一节讲 AI/ML/DL/GenAI、监督/自监督、embedding、contrastive learning、loss、backprop、CNN/Transformer、fine-tuning/RAG/prompting。

Q: AI vs ML vs deep learning vs GenAI?

AI、ML、深度学习、GenAI？

A:

  • AI: broad field of systems that perform tasks associated with intelligence.

人工智能：执行与智能相关的任务的广泛系统。

  • ML: systems learn patterns from data instead of only hand-coded rules.

ML：系统从数据中学习模式，而不仅仅是手工编码的规则。

  • Deep learning: ML using multi-layer neural networks.

深度学习：使用多层神经网络的机器学习。

  • GenAI: models that generate text, images, audio, video, code, or structured outputs.

GenAI：生成文本、图像、音频、视频、代码或结构化输出的模型。

  • Platform angle: GenAI adds serving, safety, evaluation, cost, latency, and rollout complexity beyond traditional prediction.

平台角度：GenAI 在传统预测之外增加了服务、安全、评估、成本、延迟和部署的复杂性。

Q: Supervised vs unsupervised vs self-supervised learning?

监督学习、无监督学习、自监督学习？

A:

  • Supervised: train from labeled inputs/outputs.

监督：根据标记的输入/输出进行训练。

  • Unsupervised: discover structure without explicit labels.

无监督：在没有明确标签的情况下发现结构。

  • Self-supervised: create training signals from the data itself, common in language and representation learning.

自监督：从数据本身创建训练信号，这在语言和表示学习中很常见。

  • Multimodal embeddings often use self-supervised or weakly supervised signals plus contrastive objectives.

多模态嵌入通常使用自监督或弱监督信号加上对比目标。

Q: What is an embedding?

什么是嵌入？

A:

  • An embedding is a dense vector representation of an object such as text, image, audio, video segment, user, or asset.

嵌入是对象（例如文本、图像、音频、视频片段、用户或资产）的密集向量表示。

  • Good embeddings put semantically similar objects close together.

良好的嵌入将语义相似的对象放在一起。

  • Search platforms use embeddings for retrieval, clustering, deduplication, recommendation, semantic tagging, and similarity search.

搜索平台使用嵌入进行检索、聚类、重复数据删除、推荐、语义标记和相似性搜索。

Q: What is contrastive learning?

什么是对比学习？

A:

  • Contrastive learning trains representations by pulling positive pairs together and pushing negative pairs apart.

对比学习通过将正对拉在一起并将负对分开来训练表征。

  • In multimodal systems, positives may be image-caption pairs, video-transcript pairs, or asset-query interaction pairs.

在多模态系统中，正例可能是图像-标题对、视频-脚本对或资产-查询交互对。

  • It is a natural fit for shared embedding spaces and cross-modal retrieval.

它非常适合共享嵌入空间和跨模态检索。

Q: What is a loss function?

什么是损失函数？

A:

  • A loss function measures how wrong the model is for a training example or batch.

损失函数衡量模型对于训练示例或批次的错误程度。

  • Cross-entropy is common for classification/language modeling; contrastive losses are common for embeddings; MSE/MAE are common for regression.

交叉熵在分类/语言建模中很常见；对比损失对于嵌入来说很常见； MSE/MAE 对于回归很常见。

  • Platform angle: offline loss does not always equal product quality; retrieval/CTR/latency/eval metrics still matter.

平台角度：线下损失并不一定等于产品质量；检索/点击率/延迟/评估指标仍然很重要。

Q: What is backpropagation?

什么是反向传播？

A:

  • Backpropagation computes gradients of the loss with respect to model parameters using the chain rule.

反向传播使用链式法则计算损失相对于模型参数的梯度。

  • Optimizers such as SGD or Adam use those gradients to update weights.

SGD 或 Adam 等优化器使用这些梯度来更新权重。

  • You rarely need to derive it in platform interviews, but you should know it explains training cost, gradient memory, and distributed training communication.

你很少需要在平台面试中推导出它，但你应该知道它解释了训练成本、梯度记忆和分布式训练通信。

Q: What is overfitting in production terms?

从生产角度来说，什么是过度拟合？

A:

  • A model performs well on training/eval data but poorly on real-world traffic.

模型在训练/评估数据上表现良好，但在实际流量上表现不佳。

  • In AI platforms, "overfitting" can also happen to prompts/evals: a prompt passes a small golden set but fails production distribution.

在人工智能平台中，“过度拟合”也可能发生在prompt/评估中：prompt通过了一个小的黄金集，但未能进行生产分发。

  • Mitigation: better data splits, harder eval slices, regularization, monitoring, and adding production failures back into regression datasets.

缓解措施：更好的数据分割、更难的评估切片、正则化、监控以及将生产故障添加回回归数据集中。

Q: CNNs vs transformers: what should you know?

CNN 与 Transformer：你应该知道什么？

A:

  • CNNs use convolutional inductive bias and are efficient for local visual patterns.

CNN 使用卷积归纳偏差，对于局部视觉模式非常有效。

  • Transformers use attention and scale well across text, vision, audio, and multimodal sequence problems.

Transformer 在文本、视觉、音频和多模态序列问题上很好地利用了注意力和规模。

  • Modern creative AI systems often combine transformer-based encoders/generators with specialized vision/audio preprocessing.

现代创意人工智能系统通常将基于变压器的编码器/生成器与专门的视觉/音频预处理相结合。

Q: Encoder vs decoder vs encoder-decoder?

编码器 vs 解码器 vs 编码器-解码器？

A:

  • Encoder: maps input to representation; common for embeddings, classification, retrieval.

编码器：将输入映射到表示；常见于嵌入、分类、检索。

  • Decoder: generates output autoregressively; common for LLM generation.

解码器：自回归生成输出； LLM一代很常见。

  • Encoder-decoder: maps one sequence/modality to another; common in translation/summarization-style tasks.

编码器-解码器：将一个序列/模态映射到另一个序列/模态；常见于翻译/摘要式任务。

  • Serving implication: encoders batch well; decoders require KV cache and token scheduling.

服务含义：编码器批处理良好；解码器需要KV缓存和token调度。

Q: Fine-tuning vs RAG vs prompting?

微调、RAG、prompt？

A:

  • Prompting: fastest way to steer behavior, but limited by context and model behavior.

prompt：引导行为的最快方法，但受到上下文和模型行为的限制。

  • RAG: injects external knowledge at runtime; good for private/current/domain data and citations.

RAG：运行时注入外部知识；适用于私人/当前/域数据和引文。

  • Fine-tuning: changes weights; good for style, structure, repeated task behavior, or domain adaptation, but needs data/eval/maintenance.

微调：改变权重；有利于风格、结构、重复任务行为或领域适应，但需要数据/评估/维护。

  • Production systems often combine all three.

生产系统通常将这三者结合起来。

Q: What is RLHF / preference tuning / DPO at a high level?

什么是高级 RLHF/偏好调整/DPO？

A:

  • RLHF aligns model behavior using human preference feedback and reinforcement-style optimization.

RLHF 使用人类偏好反馈和强化式优化来调整模型行为。

  • DPO-style methods optimize directly from preference pairs without the same RL loop complexity.

DPO 风格的方法直接从偏好对进行优化，而无需相同的 RL 循环复杂性。

  • Platform relevance: preference data, safety policy, and evaluation rubrics become important assets.

平台相关性：偏好数据、安全政策和评估标准成为重要资产。

Q: What is hallucination?

什么是幻觉？

A:

  • Hallucination is when a model produces plausible but unsupported or false output.

幻觉是指模型产生看似合理但不受支持或错误的输出。

  • In RAG, distinguish answer hallucination from retrieval failure: the model may hallucinate because evidence was missing, wrong, or ignored.

在 RAG 中，区分答案幻觉和检索失败：模型可能因为证据缺失、错误或被忽视而产生幻觉。

  • Mitigation includes better retrieval, grounded prompts, citations, refusal behavior, and eval harnesses.

缓解措施包括更好的检索、接地prompt、引用、拒绝行为和评估工具。

Common ML / AI Fundamentals

这一节讲 bias/variance、precision/recall、ROC-AUC/PR-AUC、calibration、data leakage、BatchNorm/LayerNorm、dropout、mixed precision 等常见基础题。

Q: Bias vs variance?

偏差与方差？

A:

  • High bias: model is too simple or underfit; training and validation errors are both high.

高偏差：模型过于简单或欠拟合；训练和验证错误都很高。

  • High variance: model overfits; training error is low but validation error is high.

高方差：模型过拟合；训练误差较低，但验证误差较高。

  • Remedies depend on which side dominates: better features/model capacity for bias, regularization/data/augmentation for variance.

补救措施取决于哪一方占主导地位：更好的特征/模型能力来应对偏差，正则化/数据/方差增强。

Q: Precision vs recall vs F1?

准确率、召回率、F1？

A:

  • Precision: of predicted positives, how many are correct?

精度：预测的阳性结果中，有多少是正确的？

  • Recall: of actual positives, how many did we find?

回想一下：在实际的阳性结果中，我们发现了多少？

  • F1: harmonic mean of precision and recall.

F1：精确率和召回率的调和平均值。

  • In retrieval/search, recall at candidate generation is often more important before reranking; precision matters more in final ranked results.

在检索/搜索中，候选生成时的召回率通常比重新排名更重要；在最终排名结果中，精度更为重要。

Q: ROC-AUC vs PR-AUC?

ROC-AUC 与 PR-AUC？

A:

  • ROC-AUC can look good on imbalanced datasets because true negatives dominate.

ROC-AUC 在不平衡数据集上看起来不错，因为真正的负数占主导地位。

  • PR-AUC focuses on positive-class precision/recall and is often more informative for rare-event detection.

PR-AUC 专注于正类精度/召回率，并且对于罕见事件检测通常提供更多信息。

Q: Calibration vs accuracy?

校准与精度？

A:

  • Accuracy measures whether predictions are correct.

准确性衡量预测是否正确。

  • Calibration measures whether predicted probabilities match actual frequencies.

校准测量预测概率是否与实际频率匹配。

  • A model can rank well but be poorly calibrated, which matters for thresholds, risk scoring, and business decisions.

模型可能排名良好，但校准不佳，这对于阈值、风险评分和业务决策很重要。

Q: Data leakage: what does it look like?

数据泄露：它是什么样的？

A:

  • Training features include information unavailable at prediction time.

训练特征包括在预测时不可用的信息。

  • Train/test split leaks user, document, session, or time-period overlap.

训练/测试拆分会泄漏用户、文档、会话或时间段重叠。

  • Labels are indirectly encoded in metadata.

标签间接编码在元数据中。

  • In search/recommendation, leakage often comes from future engagement signals.

在搜索/推荐中，泄漏通常来自未来的参与信号。

Q: BatchNorm vs LayerNorm?

BatchNorm 与 LayerNorm？

A:

  • BatchNorm normalizes using batch statistics and is common in CNN-style training.

BatchNorm 使用批量统计数据进行归一化，在 CNN 式训练中很常见。

  • LayerNorm normalizes within each sample across features and is standard in transformers.

LayerNorm 在每个样本中跨特征进行标准化，并且是 Transformer 中的标准。

  • For variable-length sequence models and small/distributed batches, LayerNorm is usually easier to use.

对于可变长度序列模型和小/分布式批次，LayerNorm 通常更容易使用。

Q: Dropout vs weight decay?

Dropout vs 体重衰减？

A:

  • Dropout randomly masks activations during training to reduce co-adaptation.

Dropout 在训练期间随机屏蔽激活以减少共同适应。

  • Weight decay penalizes large weights.

权重衰减会惩罚大权重。

  • Both regularize, but they act differently and are not interchangeable.

两者都是正则化的，但它们的作用不同并且不可互换。

Q: Cross-entropy vs MSE?

交叉熵 vs MSE？

A:

  • Cross-entropy is natural for classification because it compares predicted probability distributions with labels.

交叉熵对于分类来说是很自然的，因为它将预测的概率分布与标签进行比较。

  • MSE is natural for regression but can be poorly matched to classification probability learning.

MSE 对于回归来说是很自然的，但与分类概率学习的匹配度很差。

Q: What is gradient accumulation?

什么是梯度累积？

A:

  • Accumulate gradients across several micro-batches before optimizer step.

在优化器步骤之前累积多个微批次的梯度。

  • It simulates a larger effective batch size when GPU memory is limited.

当 GPU 内存有限时，它会模拟更大的有效批量大小。

  • Watch out for learning-rate scaling, loss normalization, and optimizer state memory.

注意学习率缩放、损失归一化和优化器状态记忆。

Q: What is mixed precision training?

什么是混合精准训练？

A:

  • Use lower precision for many tensor operations while keeping enough precision for stability.

对许多张量运算使用较低的精度，同时保持足够的精度以保证稳定性。

  • AMP/autocast chooses lower precision for safe ops, while gradient scaling helps avoid FP16 underflow.

AMP/autocast 选择较低的精度来实现安全操作，而梯度缩放有助于避免 FP16 下溢。

  • BF16 often reduces the need for explicit gradient scaling because it has wider exponent range.

BF16 通常会减少显式梯度缩放的需要，因为它具有更宽的指数范围。

Q: DDP vs FSDP vs tensor parallelism?

DDP vs FSDP vs 张量并行？

A:

  • DDP replicates the model on each GPU and all-reduces gradients.

DDP 在每个 GPU 上复制模型并全部减少梯度。

  • FSDP shards parameters, gradients, and optimizer state to reduce memory per GPU.

FSDP 对参数、梯度和优化器状态进行分片，以减少每个 GPU 的内存。

  • Tensor parallelism shards individual model layers across GPUs, often needed for very large inference/training.

张量并行性在 GPU 上分割各个模型层，这通常是非常大的推理/训练所需要的。

  • The tradeoff is always memory saved vs communication overhead and operational complexity.

权衡始终是节省内存与通信开销和操作复杂性。

Q: What is contrastive learning in multimodal systems?

什么是多模态系统中的对比学习？

A:

  • Contrastive learning pulls matched pairs together and pushes mismatched pairs apart in embedding space.

对比学习将匹配对拉到一起，并将不匹配对在嵌入空间中分开。

  • CLIP-style training aligns image/text representations so cross-modal retrieval becomes possible.

CLIP 式训练可以对齐图像/文本表示，从而使跨模态检索成为可能。

  • Platform relevance: model drift or modality mismatch shows up as retrieval-quality regressions, not only model-loss changes.

平台相关性：模型漂移或模态不匹配表现为检索质量回归，而不仅仅是模型损失变化。

Example: bounded fan-out with timeout

真实例子：带超时的受控 fan-out。

import asyncio
from dataclasses import dataclass

@dataclass(frozen=True)
class ModelCall:
    name: str
    payload: dict

async def call_model(call: ModelCall) -> dict:
    # Replace with aiohttp/httpx/gRPC client.
    await asyncio.sleep(0.05)
    return {"model": call.name, "ok": True}

async def run_fanout(calls: list[ModelCall], concurrency: int = 8) -> list[dict]:
    sem = asyncio.Semaphore(concurrency)

    async def guarded(call: ModelCall) -> dict:
        async with sem:
            async with asyncio.timeout(2.0):
                return await call_model(call)

    return await asyncio.gather(*(guarded(c) for c in calls))

Interview line: "For orchestration services, I avoid unbounded fan-out. I set concurrency limits, per-step timeouts, and cancellation behavior because downstream model servers and retrieval systems can saturate." 面试表达：“在 orchestration service 里，我会避免无限 fan-out。我会设置并发限制、每个 step 的 timeout 和 cancellation 行为，因为下游 model server 与 retrieval system 都可能饱和。”

Gotchas Interviewers Like

面试官喜欢追问的坑：

  • asyncio.gather without bounded concurrency can overload downstream services.

没有并发上限的 asyncio.gather 可能打爆下游服务。

  • Timeouts should use monotonic time or framework timeout primitives, not wall-clock assumptions.

timeout 应该用 monotonic time 或框架自带 timeout primitive，而不是依赖 wall-clock 假设。

  • CPU-bound work does not become faster just because it is in asyncio; use process pools, native extensions, vectorized libraries, or move it to the right runtime.

CPU-bound 工作不会因为放进 asyncio 就变快；应使用 process pool、native extension、向量化库，或把它放到正确 runtime。

Example: safe inference path

真实例子：安全的推理路径。

import torch

def predict(model: torch.nn.Module, batch: dict[str, torch.Tensor]) -> torch.Tensor:
    device = "cuda" if torch.cuda.is_available() else "cpu"
    model = model.to(device).eval()
    inputs = batch["input"].to(device, non_blocking=True)

    with torch.inference_mode(), torch.autocast(device_type=device, enabled=(device == "cuda")):
        logits = model(inputs)
        probs = torch.softmax(logits, dim=-1)

import torch

def log_cuda_memory(tag: str) -> None:
    if not torch.cuda.is_available():
        return
    torch.cuda.synchronize()
    allocated = torch.cuda.memory_allocated() / 1024**2
    reserved = torch.cuda.memory_reserved() / 1024**2
    peak = torch.cuda.max_memory_allocated() / 1024**2
    print(f"{tag}: allocated={allocated:.1f}MB reserved={reserved:.1f}MB peak={peak:.1f}MB")

def cache_key(intent: str, model_version: str, payload: dict) -> str:
